# SS-Helper 插件系统开发规范

本规范旨在指导开发者在 `SS-Helper` 体系内创建统一、稳定、美观的 SillyTavern 并联插件。

## 1. 目录结构规范
每个被列装在根目录下的插件被称之为一个“独立工作区”，建议从 `_TemplatePlugin` 直接复制一份作为基础盘起步。其标准结构如下：
```text
插件名/
├── src/                # TS 源码核心目录
│   ├── ui/             # 预置的标准 Settings UI 界面代码
│   └── index.ts        # 挂接在 SillyTavern 的前端入口
├── manifest.json       # 供 SillyTavern 平台读取的元数据与环境清单
├── changelog.json      # 构建期间自动注入到「关于插件」的更新日志库
└── tsconfig.json       # 指明打包与关联 SDK 路径的编译配置文件
```

## 2. 内容数据与 UI 解耦机制 (重要)
为了防止界面文件在维护中被反复污染，必须遵循以下的依赖注入设计：

- **标题与版本控制**：
  必须在插件目录下的 `manifest.json` 中配置 `display_name`、`version` 和 `author`。UI 代码会在构建期自动同步它并在设置抽屉的标题栏渲染，切勿硬编码。
  
- **联系与外链**：
  GitHub（或主页）请填入 `manifest.json` -> `homePage` (如 `"https://github.com/..."`), 邮箱填入 `email`。

- **更新日志分离**：
  编写和宣发更新通告请**唯一指定操作**插件目录下的 `changelog.json` 文件。
  只需在其中新增节点并填写 `Array`：
  ```json
  {
    "version": "v1.0.1",
    "changes": [
      "⚡ 修复了某个特定的缺陷",
      "🔗 新增了某个强大的能力"
    ]
  }
  ```
  打包系统会自动编译生成带高亮和滚动条限制的 UI 组件挂到《关于》面板中。

## 3. 面板 UI 隔离约束
由于所有插件在 ST 底层共用同一套 DOM 环境，编写 UI 的 ID 与 Class 需要高度当心：
- 各插件内部务必使用唯一的常量标识 `const NAMESPACE = 'stx-[你的标识词]';`。 
- 添加进页面的新元素均需声明以避免冲突的类名前缀（如预设模板中的 `stx-ui-`）。
- CSS 文件中全部被 `#${cardId}` 选择器包裹以隔离样式作用域，不破环他人的原生框架。

## 4. 插件编译投产路径
当前大项目由 `pnpm workspaces` 全局统筹。
当新插件写好或者更新后，您只需在根目录 (即含有 `build.js` 的 `SS-Helper` 根目录) 操作即可：

1. 打开 `g:\vue\SS-Helper\package.json`。
2. 在 `scripts` 字段内加入您自己的快速命令如 `"build:myplugin": "node build.js MyPluginName"`。
3. 执行打包，编译系统会自动提取 `manifest.json` 一同抛入您目录里的 `dist`，您只需将 `dist` 推送进入 SillyTavern 即可完成部署测试。

## 5. 输出与日志规范
为了让浏览器端控制面板具有统一且大厂水准的追踪溯源能力，严禁在业务逻辑中直接乱打 `console.log()`。

### 调用规范
请在文件头部统一引入 `SDK` 提供的标准 `Logger` 打印器：
```typescript
import { Logger } from '../../SDK/logger';

// 按照您的插件名为其分配具名空间
const logger = new Logger('MyPluginName'); 

// 以下为五种标准的输出层级，自带终端着色效果
logger.debug('琐碎的循环状态或中间量'); // 灰色
logger.info('业务节点启动或状态变更'); // 蓝色 
logger.success('关键操作写入或校验完成'); // 绿色
logger.warn('配置缺失使用了 fallback，或边缘拦截'); // 橙黄
logger.error('异常追踪栈与严重失败', err); // 红色高亮，走底层的 console.error
```
使用上述的打印工具，控制台呈现出来的效果将统一遵循如下样式，方便使用过滤条件检索排错：
`[SS-Helper]-[插件名字]-[报错类型] 具体打印内容`。

## 6. 跨插件微服务通信层（Broadcast 与 RPC）指南
从 V2 架构起，`SS-Helper` 原生的 `STX.bus` 已被升级为企业级的微服务总线底座。它由 `protocol`、`broadcast` 和 `rpc` 三大核心模块支撑。
**任何插件之间的状态联动、网络调用、甚至是 UI 红绿灯，均强制要求接入此总线标准，彻底告别以前直接读取 `window` 下全局变数的脏乱差操作。**

### 6.1 统一信封格式 (Envelope & 规约)
所有由于 `emit` 或 `request` 脱出的跨进程消息体，均必须为符合以下契约结构的 `JSON` 数据。
> **[WARNING]** 切忌在 `data` 载荷里放任何 DOM 节点或原生 Function 形参，否则可能引发克隆异常。

**单向状态广播 (Broadcast Envelope) 范例：**
```typescript
{ 
  v: 1, type: 'broadcast', 
  topic: 'plugin:broadcast:state_changed', 
  from: 'stx_myplugin', 
  ts: Date.now(), 
  data: { isEnabled: true } 
}
```

**双向响应式包裹 (RPC Envelope)：**
内部核心为您封装了附带全套 `reqId` 的 `request` -> `response` 防丢包和去重回流链路，同时自动过滤抛错：如果 `respond` 句柄体内代码抛出异常，外层会直接转为 `ok: false / error: {...}` 标准态，而绝不会在底座内默默报错消失！

### 6.2 怎么发？怎么收？ (最佳实践示例)
无论您的插件叫做什么，**引入 SDK 层级的总干线基础设施 (`../../SDK/bus/` 下的 `rpc.ts` 和 `broadcast.ts`)** 进行安全包装才是最高效的方法。（参考内置的 `MemoryOS` 的网络代码调用）

#### 场景 A：发送不含回执的通知 (例如: 报告我自己开启或关闭了功能)
```typescript
// 投递方向整网广播
broadcast('plugin:broadcast:state_changed', { isEnabled: true }, '我自己的namespace标识');

// 接收方 (订阅端)
const unsubscribe = subscribe('plugin:broadcast:state_changed', (data) => {
    console.log('别人的状态改成了:', data.isEnabled);
}, { from: '我想监听的特定目标标识' }); // 支持直接用 from 参数做数据源筛选防抖
```

#### 场景 B：向其他插件索要数据或指令它执行某动作 (异步 RPC)
**被叫方 (提供能力的 Server 端):**
```typescript
// 在你自己的逻辑代码中注册接口并等待响应别人
const disposeProvider = respond('plugin:request:hello', '我的namespace', async (payload, rawEnvelope) => {
    // rawEnvelope 里存有请求人的身份 from，你可以拒绝响应！
    return {
        replyMsg: `您好 ${rawEnvelope.from}，已经收到你发来的附言: ${payload.testPayload}`
    };
});

// 在卸载插件或关停时记得清理
// disposeProvider();
```

## 7. LLMHub 接入快速指引

如果你的插件需要使用 AI 能力（文本生成、向量化、重排序），请通过 LLMHub 统一接入，而非自行调用 LLM API。

### 7.1 三步接入

```typescript
const llm = (window as any).STX?.llm;

// 1. 注册消费方
llm.registerConsumer({
    pluginId: '你的_NAMESPACE',
    displayName: '你的插件名',
    registrationVersion: 1,
    tasks: [{
        taskId: 'my_task',
        taskKind: 'generation',          // 'generation' | 'embedding' | 'rerank'
        requiredCapabilities: ['chat'],   // 该任务需要的 Provider 能力
        backgroundEligible: false,        // 是否允许静默执行
    }],
});

// 2. 调用 AI
const result = await llm.runTask({ consumer: '你的_NAMESPACE', taskId: 'my_task', taskKind: 'generation', input: { messages } });

// 3. 卸载时注销
llm.unregisterConsumer('你的_NAMESPACE');
```

### 7.2 关键点

- **taskKind** 必填：决定路由走 generation / embedding / rerank 哪条链路。
- **requiredCapabilities** 诚实声明：设置页会据此过滤候选 Provider，乱填会导致运行时 `capability_mismatch` 错误。
- **backgroundEligible** 慎用：只有真正不需要用户确认的任务才设为 `true`，否则覆层会自动降级。
- **budget.maxLatencyMs** 优于外部 `Promise.race`：编排器内部超时会正确清理覆层和队列。
- **双 Promise**：`runTask()` 在 AI 结果到达时就 resolve；如需等覆层关闭, 使用 `await llm.waitForOverlayClose(result.meta.requestId)`。

> 完整 API 文档请参阅 **LLMHub/docs/integration-manual.md**。
```

**主叫方 (请求使用的 Client 端):**
```typescript
try {
    // 提供完善的 Promise 流、内置超时、及前置挂载防丢失回传
    // request(请求频道, 携带的数据负载, '我是谁', { 去找谁, 等多久 })
    const res = await request(
        'plugin:request:hello', 
        { testPayload: 'From Inspector' }, 
        'stx_memory_os', 
        { to: 'stx_llmhub', timeoutMs: 3000 }
    );
    
    console.log("顺利连通！收到了对方的回传数据", res.replyMsg);
} catch (e) {
    // 在这里优雅阻击诸如：“对方挂了 (Timeout)”，“对方没开没接到 (No Handler)”或者“对方里面报错了 (Handler Error)”
    console.error('RPC 请求异常拦截:', e.message); 
}
```

> **[IMPORTANT] 生命周期必须守备好** 
> *千万记得妥善保存诸如 `unsubscribe` 这类返回的回收句柄！并在插件重置或UI下线时触发一次释放防备内存溢出越积越多。*
