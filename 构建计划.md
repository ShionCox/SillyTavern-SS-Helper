# 以用户体验优先的统一存储重构方案

## 摘要
- 采用**接口统一、底层分层**的方案，不把所有数据都塞进 IndexedDB。
- **本地存储**只保留小而高频、需要同步即时生效的数据：
  - `pluginSettings`
  - `pluginUiState`
  - 主题、面板布局、轻量偏好
- **`ss-helper-db`** 统一保存所有聊天级数据和大体量数据：
  - Roll 的轮次、结果、状态、摘要、聊天级技能预设
  - MemoryOS 的事件、事实、世界状态、模板、向量
  - LLMHub 的聊天级上下文、会话快照
- 所有插件**只能通过 SDK 操作**，不再直接碰各自的存储实现。
- 不保留兼容层，不读取旧逻辑；旧存储代码在新链路稳定接入后直接删除。

## 核心设计
### 1. 存储分层
- 本地存储继续作为**同步设置层**：
  - 目标是首屏无闪动、设置页无延迟、主题切换即时生效
  - SDK 对外继续提供同步设置接口
- `ss-helper-db` 作为**统一聊天数据层**：
  - 一个数据库，所有插件共用
  - SDK 统一负责建库、升级、事务、索引、缓存、写入合并
- 固定表结构：
  - `chat_documents`
    - 每个聊天一条轻量主文档
    - 保存 `chatKey`、`entityKey`、`ref`、`meta`、`shared`、`updatedAt`
  - `chat_plugin_state`
    - 每个插件每个聊天一条当前快照
    - 保存 `pluginId`、`chatKey`、`schemaVersion`、`state`、`summary`、`updatedAt`
  - `chat_plugin_records`
    - 每个插件每个聊天的历史明细、结果流、时间线
    - 保存 `pluginId`、`chatKey`、`collection`、`recordId`、`payload`、`ts`、`updatedAt`
- 只有 MemoryOS 这类高索引模块保留专属表，但仍在同一个 `ss-helper-db` 中。

### 2. 公共聊天模板
- 主文档 `chat_documents` 保持轻量，公共区固定为小型共享字段：
  - `shared.labels`
  - `shared.flags`
  - `shared.notes`
  - `shared.signals`
- `shared.signals[pluginId]` 只放可共享摘要，不放完整业务数据。
- 插件完整数据一律放自己的 `chat_plugin_state.state` 或 `chat_plugin_records`。

### 3. SDK 接口
- 保留同步本地设置接口：
  - `createSdkPluginSettingsStore`
  - `read/writeSdkPluginSettings`
  - `read/writeSdkPluginUiState`
- 新增异步聊天数据接口：
  - `getSdkChatDocument`
  - `ensureSdkChatDocument`
  - `patchSdkChatShared`
  - `readSdkPluginChatState`
  - `writeSdkPluginChatState`
  - `listSdkPluginChatStateSummaries`
  - `appendSdkPluginChatRecord`
  - `querySdkPluginChatRecords`
  - `deleteSdkPluginChatState`
- SDK 内部固定实现：
  - 当前聊天内存缓存
  - 写入去抖合并
  - 单事务更新主文档与插件状态
  - 变更通知给 UI 层
- 目标是让 UI 感知仍然接近同步，不因 IDB 异步导致卡顿或闪动。

### 4. 跨插件访问规则
- 所有插件都可以通过 SDK 读取：
  - `chat_documents.shared`
  - 其他插件的 `summary`
  - 被允许公开的聊天摘要
- 插件只能直接写：
  - 自己的 `chat_plugin_state`
  - 自己的 `chat_plugin_records`
  - 公共区通过 `patchSdkChatShared` 写
- 插件若要修改其他插件的数据，必须走 SDK 能力调用：
  - 目标插件注册聊天能力接口
  - 调用方通过 SDK 发起请求
  - 由目标插件自己决定如何修改自己的状态
- 不允许插件直接改别人的 `state`。

## 各插件落地
### RollHelper
- `chat_plugin_state` 保存：
  - `activeStatuses`
  - `lastBaseRoll`
  - `pendingRound`
  - 当前聊天的 `skillPresetStoreText`
- `chat_plugin_records` 保存：
  - `roll_results`
  - `round_summaries`
  - 后续需要时再加 `event_history`
- `shared.signals.stx_rollhelper` 只写轻量摘要：
  - 最近一次掷骰概要
  - 待处理轮次标记
  - 当前状态数量
- 删除旧的：
  - `chatScopedStoreEvent.ts`
  - `chatScopedStoreV2Event.ts`
  - 旧迁移逻辑
  - 旧帮助文案里关于 `chatMetadata.lastRoll` 的描述

### MemoryOS
- 物理数据库并入 `ss-helper-db`
- 保留其专属高索引表，但数据库实例改由 SDK 创建和管理
- `shared.signals.stx_memory_os` 输出聊天摘要：
  - 模板状态
  - 最近摘要时间
  - 记忆条数概览

### LLMHub
- 设置继续本地存储
- 聊天级上下文、路由快照、预算快照接入 `chat_plugin_state`
- `shared.signals.stx_llmhub` 输出当前聊天路由/模型摘要

## 实施顺序
- 先完成 SDK 聊天数据层和 `ss-helper-db` 架构。
- 再先切 RollHelper 到新接口。
  - 原因：它最直接依赖聊天级持久化，最容易验证页面体验
- 然后切 MemoryOS 和 LLMHub 到统一数据库。
- 每个插件切完后立即删除该插件旧存储逻辑，不保留双写或兼容读取。

## 测试方案
- 设置体验
  - 设置页首次打开无闪动
  - 主题切换即时生效
  - 轻量设置读写无明显延迟
- RollHelper
  - 掷骰、生成事件卡、刷新页面、切聊天后能恢复当前轮次、结果、状态、摘要
  - 状态编辑器聊天列表和清理功能正常
- MemoryOS
  - 事件、事实、模板、向量仍可按 `chatKey` 正常读写
- LLMHub
  - 聊天级上下文和会话状态能正常落盘与读取
- 跨插件
  - 读取公共摘要正常
  - 直接写别人的状态被禁止
  - 通过 SDK 能力调用修改对方数据正常
- 构建
  - `pnpm.cmd build:roll`
  - `pnpm.cmd build:memory`
  - `pnpm.cmd build:llm`

## 默认决策
- 采用“本地设置 + 统一聊天库”的分层存储，不做全量 IDB。
- `ss-helper-db` 是唯一聊天级数据数据库。
- 所有插件聊天数据都通过 SDK 访问。
- 不保留旧逻辑兼容，不做旧链路读取。
