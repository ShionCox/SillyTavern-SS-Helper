

## 2) 你要的核心功能：LLMHub 增加“插件 → AI 选择”Tab（可落地方案）

你现在的路由配置是“手动填 consumer/task”。你想要的 Tab，本质是：

> 把 `consumer` 这件事“自动发现 + 列表化 + 一键配置默认 provider/profile”，让人不用懂内部路由结构也能配好。

### 2.1 先把路由能力补齐：支持“插件级默认”

现在 `TaskRouter.resolve()` 的优先级是：

1. consumer + task 精确匹配
2. consumer='*' + task
3. 全局 default ([GitHub][3])

要做“插件级默认”，你需要新增一个规则层：

* **consumer + `*`**：表示“该插件所有 task 的默认 provider/profile/fallback”
* 优先级建议变成：

1. consumer + task（最精确）
2. consumer + `*`（插件默认）
3. `*` + task（任务默认）
4. 全局 default

这样你新 Tab 配置的“插件默认 AI”，不会覆盖掉更精确的 per-task 规则（而你现有 Router tab 仍可用来做高级覆盖）。

> 这一步改动集中在：`LLMHub/src/router/router.ts`（resolve 匹配顺序 + 允许 task='*'） ([GitHub][3])

---

### 2.2 “检测到已安装插件”怎么做：给你一个可控且渐进增强的检测策略

你想要的是“当前检测到已安装的插件”。在 SillyTavern 生态里，这件事没有绝对标准 API，所以我建议做 **三层来源合并**（从“最可靠/可控”到“最大覆盖”）：

#### 来源 A：STX Registry（最推荐）

MemoryOS 已经创建了 `window.STX = { bus, registry, memory, llm }` 的全局底座。 ([GitHub][9])
但 registry 目前只是 stub（register 只 log）。你可以把它升级为“真实注册中心”：

* `STX.registry.register(manifest)` 存入 Map（key=pluginId）
* 增加 `list()` / `get()` / `onChange(cb)`（或者直接通过 STX.bus 广播 registry changed）
* 每个 SS-Helper 生态插件启动时调用 register（LLMHub、MemoryOS、RollHelper…）

这样 LLMHub 的新 Tab 能拿到 **“生态内插件”** 的完整列表（包括能力/任务声明）。

#### 来源 B：Ping 扫描（最贴近“已安装且正在运行”）

你已经实现了统一的 `plugin:request:ping` 端点：MemoryOS 和 LLMHub 都支持。 ([GitHub][9])
做法是：

* 从 `SillyTavern.getContext().extensionSettings` 里枚举 key（很多扩展都会在这里有设置项）
* 对每个 key（以及 registry 里的 pluginId）发起 `request('plugin:request:ping', ...)`
* 有响应的，判定为“运行中/可配置”
* 结合返回的 capabilities / version / isEnabled 给 UI 展示状态

这能覆盖“不是你生态，但也实现 ping 的插件”。

#### 来源 C：手动添加 consumer（兜底）

UI 提供一个输入框 “手动添加 pluginId/consumer”，让用户可以对未知插件也设默认 AI。

### 2.3 新 Tab 的 UI 形态（建议）

在 `LLMHub` 现有 4 tab 基础上加一个：

* Tab 名称建议：**“插件映射”** 或 **“插件默认路由”**
* 展示一个表格/列表，每行是一个插件 consumer：

每行字段建议：

* 插件名（manifest displayName 或 pluginId）
* pluginId（consumer）
* 状态：Alive / Not running（来自 ping）
* Enabled（来自 ping）
* 默认 Provider（select：openai/claude/gemini/groq + 你已注册 provider 列表）
* 默认 Profile（select：balanced/precise/creative/economy） ([GitHub][15])
* 默认 Fallback Provider（可选）
* （可选）预算 quick-set：maxRPM/maxTokens

按钮：

* “应用/保存”
* “清除该插件覆盖”
* “跳转到路由规则（高级）”——跳到 Router tab 并预填 consumer

实现落点文件（最少改动路径）：

* `LLMHub/src/ui/settingsCardTemplateTypes.ts`：新增 tab/panel 的 id 字段 ([GitHub][16])
* `LLMHub/src/ui/settingsCardHtmlTemplate.ts`：新增 tab button + panel 容器 ([GitHub][6])
* `LLMHub/src/ui/index.ts`：

  * `IDS` 增加新 id
  * `tabs` 数组加入新 tab
  * 新增 `renderConsumersPanel()`：做发现、渲染、绑定事件 ([GitHub][17])

### 2.4 配置怎么存：不破坏你现有结构的“最省事方案”

你当前 LLMHub settings 里已经用 `routePolicies?: RoutePolicy[]` 存规则。 ([GitHub][7])

最省事方案：**插件级默认直接写成 routePolicies 的一条规则**：

```ts
{ consumer: pluginId, task: "*", providerId, profileId?, fallbackProviderId? }
```

优点：

* 不用新建 settings 字段（少迁移）
* Router tab 也能看到这些规则（用户可高级编辑）
* 只要 TaskRouter.resolve 支持 consumer+`*` 即可

## 3) “记忆性”角度：你这套 MemoryOS 目前哪里更强、哪里该优先补

你问“记忆性上来说哪个更好”，如果只看架构上限：**MemoryOS 的上限更高**，因为你有：

* 事件流（可压缩/可回放） ([GitHub][18])
* 结构化 facts/state/summaries（可检索、可编辑、可写回世界书） ([GitHub][19])
* AI 提议制 + 审计（可控、不容易被模型胡写污染） ([GitHub][20])
* 世界模板（让抽取和注入“有 schema 可依”） ([GitHub][11])

但“实际体验的记忆效果”通常由 **注入与检索** 决定，你当前的注入策略（InjectionManager）还是偏“把最近的都塞进去”，并且 token 限制是粗略按字符截断。 ([GitHub][21])
这会导致两类问题：

1. 注入内容不够相关（噪声大）
2. 长期记忆无法被“命中”——尤其当对话很长时，仅靠最近 12 events/30 facts 远远不够

所以 MemoryOS 优化优先级建议是：

### 3.1 注入策略升级（最影响“记忆性”的第一优先级）

把注入从“按数量”升级为“按相关度 + token 预算”：

* 从当前 prompt 或最新用户消息提取 query
* 走 `memory.vector.search(query)` 做混合检索（你 SDK 已经预留 vector/hybrid 入口） ([GitHub][19])
* 注入时按 section 预算分配：

  * 最近事件（短）
  * 场景摘要（中）
  * 与 query 最相关的 facts/state（长记忆命中关键）
* 老事件更多依赖 Compaction summaries，而不是直接塞 events

### 3.2 AI 抽取触发策略优化（控制成本 + 减少重复污染）

ExtractManager 现在是“窗口>=5 就跑 summarize+extract”，窗口取最近 20。 ([GitHub][10])
建议加两层“防抖/去重”：

* 在 meta 里记录 `lastExtractedEventTs` 或 `lastExtractedEventId`
* 同一窗口不重复提取
* 触发条件改成：新增 user 消息达到 N 条/或 token 增量超过阈值/或特定事件类型出现

并且结合 LLMHub 的预算（consumer=stx_memory_os）做限流兜底。 ([GitHub][4])

### 3.3 提议闸门增强（长期稳定性）

ProposalManager 已经把“拒绝也要审计”做了，这是非常好的。 ([GitHub][20])
下一步把闸门做“更像数据库写入层”：

* 去重 gate：同 factKey/value 不重复写
* 冲突 gate：同一字段出现互斥值时要有策略（按置信度/按新鲜度/按人工锁定）
* 模板 gate：强制 fact.type/path 必须符合当前 world template 的允许范围

## 4) 给你一份“从现在代码出发”的完善计划（按里程碑拆解）

我不给你虚的“建议”，这里直接按“你现有代码落点”拆成可执行的 milestone（顺序就是建议的开发顺序）。

---

### P0：分发/安装一致性（不然用户装了也跑不起来）

**目标**：用户安装时永远能加载到正确 JS

* 明确产物目录：`LLMHub/dist`、`MemoryOS/dist` 是最终插件根目录（因为 manifest 会被复制进去） ([GitHub][2])
* 二选一：

  1. 在仓库内把 `manifest.json` 改成 `js: "dist/index.js"`（并让 SillyTavern 以包根目录为插件目录）
  2. 或者发布时（release/脚本）只打包 dist 目录给用户安装（推荐）

**验收**：安装后 ST 能从 manifest 指向路径加载成功（不再出现 “index.js 不存在” 类问题）。 ([GitHub][22])

---

### P1：LLMHub 新增“插件映射 Tab”（你点名要的）

**目标**：用户能在一个 tab 里，对“检测到的插件”设置默认 AI（provider/profile/fallback）

1. **TaskRouter 支持 consumer 默认**

* `resolve()` 增加 consumer + `*` 匹配层（优先级见上文 2.1） ([GitHub][3])

2. **插件发现模块**

* 实现 `discoverConsumers()`：

  * 从 `STX.registry.list()`（需要 P1.3）取
  * 从 `extensionSettings` key 枚举取
  * ping 探测 `plugin:request:ping` 合并结果 ([GitHub][9])

3. **LLMHub UI 扩展**

* 改 `settingsCardTemplateTypes.ts` 增 id
* 改 `settingsCardHtmlTemplate.ts` 增 tab/panel
* 改 `ui/index.ts` 增 panel 渲染与保存逻辑 ([GitHub][16])

4. **存储映射**

* 写入 `routePolicies` 形式：`{consumer: pluginId, task:'*', ...}`

**验收**：

* tab 列出 consumer（至少能看到 stx_memory_os / stx_llmhub）
* 每个 consumer 能设置 provider/profile，保存后立刻生效（runTask 路由命中） ([GitHub][4])

---

### P1.3：把 STX Registry 做成真“注册中心”（支撑生态扩展）

**目标**：SS-Helper 生态插件能被可靠发现，而不是靠猜 key

* 在 MemoryOS 初始化 STX 时，把 registry 从 stub 变成：

  * `register(manifest)`：写 Map
  * `list()`：返回 manifest 数组
  * `get(pluginId)`
  * `emit changed`（可用 STX.bus 广播） ([GitHub][9])

* 让 LLMHub、MemoryOS、RollHelper 等启动时 register 自己的 manifest（可从各自 manifest.json 读取） ([GitHub][22])

**验收**：LLMHub 插件映射 tab 不依赖“猜 extensionSettings key”，也能发现生态插件。

---

### P2：MemoryOS 注入策略升级（直接提升“记忆性”）

**目标**：注入内容更相关、更稳定、更省 token

* 把 InjectionManager 从“固定数量”改成“预算分配”：

  * section 预算：WORLD_STATE/FACTS/EVENTS/SUMMARY 各占多少
  * facts 优先注入与 query/当前场景相关的（而不是最近更新） ([GitHub][21])
* 引入“摘要优先”：

  * 旧内容来自 summaries（scene/arc）
  * 最新几轮才用 events
* token 估算从“字符截断”升级为更靠谱的近似（哪怕先用 word/中文字符比率，也比现在更稳） ([GitHub][21])

**验收**：长聊天时，模型仍能稳定命中关键设定，而不是被“最近事件 JSON”淹没。

---

### P3：AI 抽取编排与成本控制（让 AI 模式长期可用）

**目标**：ExtractManager 不重复跑、不会爆成本、失败可控

* meta 记录 lastExtract，避免重复对同一窗口抽取 ([GitHub][23])
* 触发规则改为：

  * “新增用户消息 N 条” 或 “事件数增量超过阈值”
  * 或者特定事件类型（例如世界模板变化/战斗结束）
* runTask 时显式传 budget（maxTokens/maxLatencyMs），并依赖 LLMHub BudgetManager 的 consumer 限流兜底 ([GitHub][4])

---

### P4：数据可移植性与回滚闭环（用户信任关键）

你 UI 已经有导出/导入/清空/审计快照入口了。 ([GitHub][14])
下一步是把这些能力做成“真可用”：

* Export：按 chatKey 打包（events/facts/state/summaries/templates/meta/binding）
* Import：支持 merge/replace 两种模式
* Snapshot 回滚：把 audit snapshot 做成可一键恢复（你已经有 auditManager 接口入口） ([GitHub][19])

## 5) 你问“我的有什么可以优化”：一句话总结优先级

如果你把“记忆性/长期体验”当第一目标，优先级我建议是：

1. **LLMHub 插件映射 Tab + Router wildcard**（先把生态配置体验打通）
2. **MemoryOS 注入策略（相关度 + token 预算）**（立刻提升“记忆命中率”）
3. **ExtractManager 触发去重 + 成本控制**（让 AI 记忆不会越用越贵越乱）
4. **GateValidator/冲突策略/人工审核开关**（保证长期稳定不污染）
5. **分发结构与 release 自动化**（让用户安装不踩坑）
