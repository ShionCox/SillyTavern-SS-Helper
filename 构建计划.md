# Memory OS（记忆插件）+ LLM Hub（通用 LLM 插件）详细实施计划（升级版）

> 目标：做成 **可扩展插件生态底座**  
> - **Memory OS**：统一记忆/状态/检索/注入/互通（**RULE 模式可独立运行**）  对应代码放到MemoryOS文件夹中
> - **LLM Hub**：统一管理模型与 AI 能力（按插件/任务路由，Memory OS 可选组合启用 AI 模式） ，对应代码放到LLMHub文件夹中
> - **SDK**：给骰子/地图/任务/关系/背包等后续插件提供标准接口，实现互联互通  
>
> 关键升级点（按你的要求改）  
> 1) **主存储从 JSON + localStorage 改为 IndexedDB（浏览器数据库）**  
> 2) **按 chatId/groupId 分区存储**，每个聊天可初始化一套独立“表/对象仓库”  
> 3) AI 模式下：**读取世界书(World Info)**，自动生成“世界模板（schema）”，用于不同世界观（魔幻/都市）不同表结构  
> 4) SDK 更细、更通用，负责插件联动与跨插件事件通信

## 0. 背景：从现有酒馆记忆插件吸取的优点

### 0.1 ST-Memory-Context（你研究的库）可复用的工程经验
- **会话隔离**：用 chatId/groupId/characterId 生成稳定分区键（避免串档）  
- **请求前注入管道**：发送前统一注入 summary/table/prompt（支持锚点变量替换）  
- **失败降级**：向量检索等耗时操作支持超时静默降级，不影响主对话  
- **多后端同步思路**：本地存 + chatMetadata 同步 + worldinfo 同步（但你们要改存储主策略）

### 0.2 其它插件值得吸收
- **MessageSummarize**：短期/长期分层，长期尽量“可确认/可编辑”，降低 LLM 误写风险  
- **MemoryBooks / ReMemory**：世界书作为兼容层，结构化产物（JSON 摘要）更稳定

## 2. Memory OS 设计（重点：IndexedDB + 按聊天建“表” + 世界模板）

### 2.1 存储策略：IndexedDB 为主、JSON 为辅
**为什么改：**
- localStorage 同步写 + 全量 JSON stringify 很容易卡 UI  
- IndexedDB 支持异步、容量更大、可以分 store、可索引、适合事件流/大数据量

**策略：**
- 主数据全部进入 IndexedDB（**事实/事件/状态/摘要/索引**）  
- localStorage 仅保存少量：UI 设置、当前启用的模板 ID、最后一次迁移版本等  
- chatMetadata 仅保存 **指针级信息**（例如：db版本/最近摘要 hash/同步状态），避免塞大块数据

### 2.2 分区键：按聊天隔离（chat-scoped namespace）
定义一个统一的 `ChatNamespace`：

- `chatId`: 当前聊天文件/会话 id  
- `groupId`（可选）  
- `characterId`（可选）  
- `scopeKey`: `Group_${groupId}_${chatId}` 或 `${characterId}_${chatId}` 或 `${chatId}`

**要求：**
- 同一个聊天永远映射到同一个 namespace  
- 不同聊天绝不共享同一份事实库（除非用户手动启用“跨聊天共享模板”）

### 2.3 IndexedDB Schema（建议：单 DB + 多 store，而不是每聊天一个 DB）
> IndexedDB 不建议每聊天建一个新数据库实例（会碎片化、难管理）。  
> 更稳的是：**一个 DB，多 object store，用 namespace 作为主键前缀/索引**。

#### DB：`st_memory_os`
Object Stores（最小可用版）：

1) `events`（事件流，append-only）
- key：`eventId`（uuid）  
- index：`byChatTs`（`chatKey + ts`），`byType`（`chatKey + type`），`bySourcePlugin`

2) `facts`（语义事实，结构化、可覆盖）
- key：`factKey`（`chatKey + path` 或 `chatKey + (type+id)`）  
- index：`byType`, `byUpdatedAt`, `byEntity`

3) `world_state`（世界状态 materialized view，可从事件回放重建）
- key：`stateKey`（`chatKey + path`）  
- index：`byPathPrefix`（用于快速取某实体树）

4) `summaries`（摘要层：message/scene/arc）
- key：`summaryId`  
- index：`byChatLevel`（chatKey + level）, `byMessageId`, `byTs`

5) `world_templates`（世界模板：从世界书/AI 生成的 schema）
- key：`templateId`（uuid）  
- index：`byChatKey`, `byWorldBookName`, `byType`（fantasy/urban/custom）

6) `vector_store`（可选：向量索引/embedding）
- key：`vecId`  
- index：`byChatKey`, `byBookId`, `byChunkId`

7) `meta`（迁移、版本、统计、同步指针）
- key：`chatKey`  
- value：`{schemaVersion, lastCompactionTs, activeTemplateId, ...}`

> 注意：`vector_store` 的 embedding/rerank 由 LLM Hub 提供能力；Memory OS 只负责存储与检索策略。

## 3. 世界模板（World Template）系统：AI 模式读取世界书生成“数据库表”

### 3.1 目标
让不同世界观在同一个 Memory OS 下拥有不同“数据结构与默认抽取规则”：

- 魔幻：阵营、魔法、神祇、遗物、怪物、城邦、任务线  
- 都市：公司、职位、人际关系、地点、案件、经济、日程

### 3.2 数据来源（AI 模式）
AI 模式启动时（或切换世界书时）：

1) Memory OS 读取当前聊天启用的 World Info（世界书）
2) 将世界书内容整理为 `WorldContextBundle`
3) 调用 LLM Hub 的 `world.template.build` 任务生成模板（结构化 schema）
4) 将模板写入 `world_templates` store，并设置该 chat 的 `activeTemplateId`
5) 后续抽取/写入/注入都以 `activeTemplateId` 的 schema 为准

### 3.3 模板结构（建议）
```json
{
  "templateId": "uuid",
  "chatKey": "Group_xxx_yyy",
  "name": "魔幻世界-阿格尼亚",
  "worldType": "fantasy",
  "entities": {
    "Character": {
      "primaryKey": "charId",
      "fields": ["name", "faction", "traits", "relations", "status", "location"]
    },
    "Faction": {
      "primaryKey": "factionId",
      "fields": ["name", "alignment", "territory", "leaders"]
    },
    "Quest": {
      "primaryKey": "questId",
      "fields": ["title", "stage", "giver", "targets", "status"]
    }
  },
  "factTypes": [
    {
      "type": "relationship",
      "pathPattern": "rel.{a}.{b}",
      "slots": ["a","b","value","reason"],
      "defaultInjection": "RELATIONSHIP"
    }
  ],
  "extractPolicies": {
    "memory.extract": { "minConfidence": 0.75, "allowWriteFacts": true },
    "world.update": { "allowPatches": true, "requireDiff": true }
  },
  "injectionLayout": {
    "WORLD_STATE": { "maxTokens": 600 },
    "FACTS": { "maxTokens": 600 },
    "EVENTS": { "maxItems": 12 },
    "SUMMARY": { "maxTokens": 400 }
  }
}
```

### 3.4 世界模板与“表”的关系（你提到的“每个聊天初始化不同表存储”）

IndexedDB 层不需要真的为每个世界建立不同 object store（会让迁移复杂）。
正确做法是：

* **同一套 store（events/facts/state/summaries）**
* 通过 `activeTemplateId` 决定：

  * 写入哪些 FactType
  * 允许哪些 pathPattern
  * UI 展示哪些“表视图”
  * 注入时按哪个 layout 拼接上下文

也就是说：
**“表”是逻辑表（模板驱动的视图），不是物理表。**

## 4. 双模式：RULE（无 AI）与 AI（组合 LLM Hub）

### 4.1 RULE 模式（默认必须能跑）

**特点：确定性 + 可回放 + 可审计**

* 所有插件事件都写 `events`（最重要）
* world_state 由规则更新（或由事件回放重建）
* facts 可由：

  * 用户手动标记/编辑（UI）
  * 插件规则推导（比如骰子暴击 → 记录“关键成功”事件）
* summaries 可选：只做简单规则摘要（非 LLM）

适合：掷骰/战斗/数值/地图等强规则场景

### 4.2 AI 模式（Memory OS + LLM Hub 组合启用）

**原则：AI 只能“提议”，不能直接写核心状态**
链路（示例：用户对话/骰子事件）：

1. 事件写入 `events`
2. Memory OS 构造 `ExtractionInput`（结合 activeTemplate + 最近消息 + worldinfo bundle）
3. 调用 LLM Hub：

   * `memory.extract`：提取结构化事实提议
   * `world.update`：生成 patch 提议（JSON Patch 风格）
   * `memory.summarize`：生成摘要（message/scene/arc）
4. Memory OS 做 **四道闸门**：

   * schema 校验（必须）
   * diff 校验（必须：无变化不落盘）
   * 权限校验（插件是否有权写该 scope）
   * 审计记录（写入 audit log）
5. 通过后写入 `facts/world_state/summaries`

失败/超时：静默降级回 RULE 模式（仍可继续聊天）

## 5. LLM Hub（独立插件）详细设计：按插件/任务指定 AI

### 5.1 为什么必须拆成独立插件

* 密钥与连接管理不要散落在每个业务插件里（更安全、更可控）
* 同一个用户可能希望：

  * 记忆抽取用一个更稳定模型
  * 骰子叙事用便宜快速模型
  * embedding 用本地模型
* 新插件接入时只要声明 task，就能复用 LLM Hub

### 5.2 Task-first：用“任务”路由到具体 provider

任务示例：

* `world.template.build`
* `memory.extract`
* `memory.summarize`
* `rag.embed`
* `rag.rerank`
* `dice.narrate`

路由优先级：

1. `consumer(pluginId)+task`
2. `task` 默认
3. 全局 default

### 5.3 Provider 抽象（支持云/本地/自建中转）

```ts
interface LLMProvider {
  id: string;
  kind: "openai" | "claude" | "gemini" | "local" | "custom";
  capabilities: { chat: boolean; json: boolean; tools: boolean; embeddings: boolean; rerank?: boolean };
  request(req: LLMRequest): Promise<LLMResponse>;
  embed?(req: EmbedRequest): Promise<EmbedResponse>;
  rerank?(req: RerankRequest): Promise<RerankResponse>;
}
```

### 5.4 结构化输出强约束（必须）

每个 task 都绑定 schema（Zod/Ajv）：

* `memory.extract` → `{facts:[...], confidence, notes?}`
* `world.update` → `{patches:[{op,path,value}], confidence}`
* `world.template.build` → `{template:{...}}`
* `dice.narrate` → `{title, summary, keywords[]}`

校验失败：不回写 Memory OS，只写审计/队列

### 5.5 预算/限流/熔断/回退

* 每 consumer 可配置：RPM、maxTokens、maxLatency、maxCost
* 熔断：连续失败触发禁用一段时间
* 回退：同 task 配备用 provider（例如主模型失败自动换备用模型）

## 6. Plugin SDK（必须更详细）：让未来所有插件“接得上”

> 你的要求是：SDK 要足够细，能兼容后续新插件，并承担联动与沟通。
> 所以 SDK 需要分三层：**Event、State、AI**，再加权限与生命周期。

### 6.1 SDK 形态（建议：全局注入 + 版本协商）

`window.STX`（SillyTavern eXtension Bus）：

* `STX.memory`：MemorySDK
* `STX.llm`：LLMSDK（如果 LLM Hub 未安装则为 stub）
* `STX.bus`：事件总线
* `STX.registry`：插件注册与能力声明
* `STX.version`：SDK 版本

插件启动时：

* `STX.registry.register(manifest)`
* Memory OS / LLM Hub 根据 manifest 发放能力

### 6.2 Plugin Manifest（生态插件必须声明）

```json
{
  "pluginId": "dice",
  "name": "Dice Plugin",
  "version": "0.1.0",
  "capabilities": {
    "events": ["emit:dice.roll", "emit:dice.check"],
    "memory": ["append:events", "read:world_state"],
    "llm": ["task:dice.narrate"]
  },
  "scopes": ["chat"],
  "requiresSDK": "^1.0.0"
}
```

### 6.3 Event Bus（跨插件互通的核心）

```ts
type EventEnvelope<T> = {
  id: string;
  ts: number;
  chatKey: string;
  source: { pluginId: string; version: string };
  type: string;
  payload: T;
};

interface STXBus {
  emit<T>(type: string, payload: T, opts?: { chatKey?: string }): void;
  on<T>(type: string, handler: (evt: EventEnvelope<T>) => void): () => void;
  once<T>(type: string, handler: (evt: EventEnvelope<T>) => void): () => void;
  off(type: string, handler: Function): void;
}
```

推荐事件命名规范：

* `dice.roll`, `dice.check`
* `combat.attack`, `combat.damage`
* `quest.update`
* `world.move`
* `memory.fact.upserted`
* `memory.template.changed`

### 6.4 MemorySDK（写事件、写事实、读状态、查询、注入控制）

```ts
interface MemorySDK {
  // 会话
  getChatKey(): string;
  getActiveTemplateId(): Promise<string | null>;
  setActiveTemplateId(templateId: string): Promise<void>;

  // 事件流
  events: {
    append<T>(type: string, payload: T, meta?: { sourceMessageId?: string; sourcePlugin?: string }): Promise<string>; // returns eventId
    query(opts: { type?: string; sinceTs?: number; limit?: number }): Promise<Array<EventEnvelope<any>>>;
  };

  // 事实（结构化）
  facts: {
    upsert(fact: {
      factKey?: string;
      type: string;
      entity?: { kind: string; id: string };
      path?: string;              // rel.a.b / char.x.trait...
      value: any;
      confidence?: number;
      provenance?: any;
    }): Promise<string>; // returns factKey
    get(factKey: string): Promise<any | null>;
    query(opts: { type?: string; entity?: { kind: string; id: string }; pathPrefix?: string; limit?: number }): Promise<any[]>;
    remove(factKey: string): Promise<void>;
  };

  // 世界状态（materialized view）
  state: {
    get(path: string): Promise<any | null>;
    set(path: string, value: any, meta?: { sourceEventId?: string }): Promise<void>;
    patch(patches: Array<{ op: "add"|"replace"|"remove"; path: string; value?: any }>, meta?: any): Promise<void>;
    query(prefix: string): Promise<Record<string, any>>;
  };

  // 摘要
  summaries: {
    upsert(summary: { level: "message"|"scene"|"arc"; messageId?: string; title?: string; content: string; keywords?: string[] }): Promise<string>;
    query(opts: { level?: string; sinceTs?: number; limit?: number }): Promise<any[]>;
  };

  // 注入控制（给 Prompt Builder）
  injection: {
    buildContext(opts?: { maxTokens?: number; sections?: Array<"WORLD_STATE"|"FACTS"|"EVENTS"|"SUMMARY"> }): Promise<string>;
    setAnchorPolicy(opts: { allowSystem?: boolean; allowUser?: boolean; defaultInsert?: "top"|"beforeStart"|"customAnchor" }): Promise<void>;
  };

  // 审计/回滚
  audit: {
    list(opts?: { sinceTs?: number; limit?: number }): Promise<any[]>;
    rollbackToSnapshot(snapshotId: string): Promise<void>;
    createSnapshot(note?: string): Promise<string>;
  };
}
```

### 6.5 LLMSDK（给所有插件用：任务调用 + 路由提示 + schema）

```ts
interface LLMSDK {
  runTask<T>(args: {
    consumer: string;               // pluginId
    task: string;
    input: any;
    schema?: object;                // JSON schema 或 zod 转 schema
    routeHint?: { provider?: string; profile?: string };
    budget?: { maxTokens?: number; maxLatencyMs?: number; maxCost?: number };
  }): Promise<
    | { ok: true; data: T; meta: { provider: string; latencyMs: number; cost?: number } }
    | { ok: false; error: string; retryable?: boolean; fallbackUsed?: boolean }
  >;

  // embedding / rerank 可选暴露
  embed?(args: { consumer: string; texts: string[]; routeHint?: any }): Promise<any>;
  rerank?(args: { consumer: string; query: string; docs: string[]; routeHint?: any }): Promise<any>;
}
```

### 6.6 权限模型（最小可用，但必须有）

* 默认：普通插件只能 `append events`，不能直接写 `world_state` / `facts`
* 写状态需要用户在 Memory OS UI 显式授权（或白名单）
* LLM 调用也要声明 capability，否则 LLM Hub 拒绝

## 7. 关键流程（你们要实现的“开机流程”和“写入流程”）

### 7.1 聊天启动 / 切换时（初始化）

1. 计算 `chatKey`
2. IndexedDB `init(chatKey)`：确保 meta 存在、store 可用
3. 读取 meta：activeTemplateId
4. 若 AI 模式开启：

   * 读取世界书内容 → `WorldContextBundle`
   * 若没有 activeTemplate 或世界书变更（hash 不一致）：

     * 调 LLM Hub `world.template.build`
     * 写入 `world_templates`
     * 更新 meta.activeTemplateId
5. 加载必要状态到内存缓存（只缓存热数据）

### 7.2 任意插件写事件（骰子示例）

Dice 插件：

* `STX.bus.emit("dice.roll", payload)`
* 或者直接 `MemorySDK.events.append("dice.roll", payload)`

Memory OS：

* 监听 `dice.roll`
* append 到 events
* RULE 模式：可能更新 state（例如记录 lastDice）
* AI 模式：调用 `dice.narrate` 或 `memory.extract` 生成摘要/事实提议 → 校验后落盘

## 8. 实施里程碑（可直接照着排期开发）

### Milestone 1：SDK + Bus + IndexedDB 主存储（最关键底座）

**交付**

* `STX.registry` 注册与版本协商
* `STX.bus`：emit/on/once/off（含 EventEnvelope）
* `IndexedDBProvider`：db 初始化 + events/facts/state/summaries/meta 基础 CRUD
* `MemorySDK` 最小闭环：append events / get state / set state

**验收**

* Dice 插件能发 `dice.roll`
* Memory OS 能在 IndexedDB 里看到事件记录（按 chatKey 隔离）
* 页面不卡顿（事件写入异步）

---

### Milestone 2：Prompt 注入器（Injection Planner）+ 纯 RULE 模式闭环

**交付**

* `MemorySDK.injection.buildContext()`：按 sections 拼装上下文
* 注入锚点策略：默认不强塞 system（可配置）
* 简单规则摘要（非AI）：最近 N 条事件 → 压缩成段落
* UI：查看/编辑 facts、查看 events、导入导出（JSON 作为“迁移/备份格式”，不是主存储）

**验收**

* 无 LLM Hub 也能稳定运行
* 不同聊天互不影响
* 记忆注入可控，不污染 prompt

---

### Milestone 3：LLM Hub MVP（任务路由 + schema 校验 + 熔断）

**交付**

* Provider 抽象 + 至少 1 个 provider
* `LLMSDK.runTask()`，支持按 consumer+task 路由
* schema 校验失败不回写
* 预算/限流/超时/熔断/回退
* 凭据管理：密钥集中放 LLM Hub（Memory OS 不保存明文 key）

**验收**

* Memory OS 能调用 `memory.summarize` 并落 summaries
* AI 失败不影响对话（自动降级）

---

### Milestone 4：世界模板系统（WorldInfo → Template → “逻辑表”）

**交付**

* `world.template.build` task + schema
* `world_templates` store + per-chat activeTemplateId
* 世界书 hash 检测（变更触发重建模板）
* UI：选择模板/锁定模板/查看模板结构
* 记忆抽取策略按 template 的 FactType 生效

**验收**

* 魔幻聊天与都市聊天能使用不同 template
* 同样的“关系/事件”在不同世界观有不同字段/展示/注入布局

---

### Milestone 5：向量层（可选）+ 混合检索

**交付**

* `rag.embed/rag.rerank` 通过 LLM Hub 提供能力

## 13. IndexedDB 实现方案

### 13.1 选型：原生 IndexedDB vs localforage vs Dexie

**推荐优先级**
1) **Dexie**（开发效率高、迁移友好、索引声明清晰）
2) **localforage**（ST 文档提到可用共享库，封装简单，但复杂索引能力弱）
3) 原生 IndexedDB（可控但代码冗长）

> 你要做“按 chatKey 多表/逻辑表 + 查询索引 + compaction”，Dexie 最舒服。

### 13.2 数据库版本与迁移策略（必须从 v1 就规划）

- DB 名：`stx_memory_os`
- 版本号：从 `1` 开始，每次 schema 增量 + 迁移函数
- 原则：**迁移只做结构变更与索引补齐，不做“全量重算”**（全量重算走后台任务/compaction）

#### v1（最小闭环）
Stores：
- `events`
- `facts`
- `world_state`
- `summaries`
- `templates`
- `audit`
- `meta`

#### v2（支持世界书解析缓存 + 模板 hash）
新增：
- `worldinfo_cache`（每 chatKey：世界书快照 hash + 解析产物）
- `template_bindings`（chatKey -> activeTemplateId + 绑定信息）

#### v3（向量层）
新增：
- `vector_chunks`
- `vector_embeddings`
- `vector_meta`

> 迁移要可回滚/可重建：如果升级失败，要保留旧 DB 或把失败状态写 meta。

### 13.3 Store 结构定义（推荐字段与索引）

#### 13.3.1 events（append-only，核心）
**用途**
- 任何插件的行为都先落到事件流
- RULE 模式世界状态可由事件回放得到一致性（强保证）

**记录结构**
```json
{
  "eventId": "uuid",
  "chatKey": "Group_xxx_yyy",
  "ts": 1710000000,
  "type": "dice.roll",
  "source": { "pluginId": "dice", "version": "0.1.0" },
  "payload": { "...": "..." },
  "refs": { "messageId": "m_123", "turn": 88 },
  "tags": ["combat","roll"],
  "hash": "sha256(payload+meta)"
}
```

**索引建议**

* `chatKey+ts`（用于按时间取最近 N 条）
* `chatKey+type+ts`（用于取某类事件）
* `chatKey+source.pluginId+ts`（用于某插件事件聚合）
* `chatKey+tags`（可选：tag 查询）

#### 13.3.2 facts（可覆盖的结构化事实）

**用途**

* 人物关系、任务进度、角色属性、地点别名等
* 允许“同 key 覆盖”，但要进 audit

**主键策略（强烈建议稳定可预测）**

* `factKey = ${chatKey}::${type}::${entityKind}:${entityId}::${path}`
* 或者 `factKey = ${chatKey}::${path}`（如果 path 足够唯一）

**记录结构**

```json
{
  "factKey": "Group...::relationship::Character:agni::rel.user",
  "chatKey": "Group_xxx_yyy",
  "type": "relationship",
  "entity": { "kind": "Character", "id": "agni" },
  "path": "rel.user",
  "value": { "level": 0.8, "reason": "..." },
  "confidence": 0.82,
  "provenance": { "eventId": "uuid", "messageId": "m_123", "extractor": "ai|rule|manual" },
  "updatedAt": 1710000001
}
```

**索引建议**

* `chatKey+type`
* `chatKey+entity.kind+entity.id`
* `chatKey+path`（或 `pathPrefix` 用于树状查询）
* `chatKey+updatedAt`

#### 13.3.3 world_state（materialized view）

**用途**

* 快速注入/快速 UI 展示
* 可从 facts/events 生成，但这里是“缓存视图”（可重建）

**记录结构**

```json
{
  "stateKey": "Group...::char.agni.hp",
  "chatKey": "Group_xxx_yyy",
  "path": "char.agni.hp",
  "value": 24,
  "sourceEventId": "uuid",
  "updatedAt": 1710000002
}
```

**索引建议**

* `chatKey+path`
* `chatKey+pathPrefix`（可用 path 前缀分段索引：`char.agni.`）

#### 13.3.4 summaries（短/中/长摘要）

**分层**

* `message`：绑定 messageId
* `scene`：绑定场景范围（startMsgId/endMsgId）
* `arc`：绑定章节/故事线

**记录结构**

```json
{
  "summaryId": "uuid",
  "chatKey": "Group_xxx_yyy",
  "level": "scene",
  "title": "酒馆冲突",
  "content": "....",
  "keywords": ["酒馆","争吵","雇佣"],
  "range": { "fromMessageId": "m_120", "toMessageId": "m_138" },
  "createdAt": 1710000003,
  "source": { "extractor": "ai|rule|manual", "provider": "openai_main" }
}
```

索引：

* `chatKey+level+createdAt`
* `chatKey+range.fromMessageId`（可选）

#### 13.3.5 templates（世界模板）

**记录结构**

```json
{
  "templateId": "uuid",
  "chatKey": "Group_xxx_yyy",
  "worldType": "fantasy|urban|custom",
  "name": "魔幻-阿格尼亚",
  "schema": { "...": "..." },
  "policies": { "...": "..." },
  "layout": { "...": "..." },
  "worldInfoRef": { "book": "xxx", "hash": "sha256" },
  "createdAt": 1710000004
}
```

索引：

* `chatKey+createdAt`
* `chatKey+worldType`
* `chatKey+worldInfoRef.hash`

#### 13.3.6 audit（不可或缺）

**原则**：任何跨 store 的写操作都生成 audit 记录（可回滚 / 可追责 / 可对比）

记录结构：

```json
{
  "auditId": "uuid",
  "chatKey": "Group_xxx_yyy",
  "ts": 1710000005,
  "action": "fact.upsert|state.patch|summary.upsert|template.change",
  "actor": { "pluginId": "memory-os", "mode": "ai|rule|manual" },
  "before": { "...": "..." },
  "after": { "...": "..." },
  "refs": { "eventId": "uuid" }
}
```

## 14. “按 chatKey 初始化不同表”的正确做法（逻辑表 + 模板驱动）

你说的“每个聊天初始化建立不同的表存储不同世界内容”，推荐实现为：

* **物理层固定（IndexedDB stores 不变）**
* **逻辑层按 template 动态生成 UI 表视图 + 写入规则**
* 每个 chatKey 绑定一个 `activeTemplateId`

### 14.1 逻辑表是什么？

比如魔幻世界模板会定义这些逻辑表：

* `Characters`（来自 facts：type=entity.character）
* `Factions`
* `Quests`
* `Locations`
* `Relationships`

而都市模板会是：

* `Persons`
* `Companies`
* `Cases`
* `Schedules`
* `Addresses`

这些逻辑表都映射到：

* facts（主体）
* world_state（快速展示）
* summaries（压缩）
* events（证据链）

### 15.1 输入：WorldContextBundle（Memory OS 负责拼）

```json
{
  "chatKey": "Group_xxx_yyy",
```

```json

      "type": "object",

        "type": "object",

          "primaryKey": { "type": "string" },

      "type": "array",
      "items": {
        "type": "object",

          "type": { "type": "string" },
          "pathPattern": { "type": "string" },
          "slots": { "type": "array", "items": { "type": "string" } },
          "defaultInjection": { "type": "string" }
        }
      }
    },
    "extractPolicies": {
      "type": "object",
      "properties": {
        "memory.extract": {
          "type": "object",
          "properties": {
            "minConfidence": { "type": "number" },
            "allowWriteFacts": { "type": "boolean" }

    "injectionLayout": {
      "type": "object",
      "properties": {
        "WORLD_STATE": { "type": "object", "properties": { "maxTokens": { "type": "number" } } },
```

### 15.3 两个世界类型的模板示例（用于验收）

#### A) 魔幻模板（精简示例）

* entities：Character/Faction/Quest/Location/Item
* factTypes：relationship、quest.progress、location.alias、magic.rule

#### B) 都市模板（精简示例）

* entities：Person/Company/Case/Place/Schedule
* factTypes：relationship、case.status、schedule.event、place.alias

> 验收标准：同样一段世界书内容，生成的模板 schema 明显不同；并且注入布局不同（都市更强调日程/地点，魔幻更强调阵营/任务线）。

## 16. AI 写入流程的“提议制”与四道闸门（必须）

> 这是为了让 AI 模式“可用但不毁档”。

### 16.1 提议结构（proposal）

所有 AI 任务返回统一 envelope：

```json
{
  "ok": true,
  "proposal": {
    "facts": [ ... ],
    "patches": [ ... ],
    "summaries": [ ... ],
    "notes": "..."
  },
  "confidence": 0.83
}
```

### 16.2 四道闸门

1. **Schema 校验**：JSON 格式、字段类型正确
2. **Diff 校验**：对比现存 facts/state，变化为空则丢弃
3. **权限校验**：consumer/plugin 是否有写权限（默认禁止）
4. **审计记录**：写 audit（before/after）

通过后才落盘到 IndexedDB。

## 17. Compaction / 快照 / 回放（解决“越用越大”和一致性）

### 17.1 事件流为什么必须 compaction？

长期使用后 events 会非常大（特别是骰子/战斗），要定期：

* 抽取已稳定事实 → facts
* 将旧事件压缩为 scene summaries
* 生成 world_state 快照
* 标记某个 ts 之前事件可“冷存储/归档”

### 17.2 Compaction 任务（RULE 模式也能跑）

触发条件：

* `events` 超过 N 条（如 5000）
* 距离上次 compaction 超过 T 时间（如 7 天）
* 用户手动触发

流程：

1. 取一段事件窗口（比如最近 1000 条）
2. 规则聚合：合并重复事件（同类型同 tag）
3. 生成 scene summary（RULE 简化版 or AI 模式走 LLM Hub）
4. 写入 summaries，更新 meta.lastCompactionTs
5. 可选：把已摘要覆盖范围的 events 标记 `archived=true`（不删除，先做软归档）

## 18. 世界书读写策略（AI 模式读取 + 可选写回）

### 18.1 读取（必须）

* chat 启动时读取“启用的世界书集合”
* 生成 `worldInfoHash`
* hash 未变：不重建模板

### 18.2 写回（可选，作为兼容层）

把“稳定事实/模板摘要”写回 worldinfo 的好处：

* 能被 ST 原生 worldinfo 引擎触发注入（兼容性强）
* 用户可在 ST UI 手改

写回策略：

* 只写 **summary 型条目**（信息密度高、避免指令性文本）
* 对每 chatKey 建一个 book（或条目前缀），例如：`[MEMORY_OS][chatKeyShort]`

## 19. SDK 再补全：面向“所有新插件”接入的通用规范

> 你说 SDK 要详细到可以兼容后续所有插件联动，这里把“规范/约束”补齐。

### 19.1 SDK 版本协商与降级

* `STX.version = "1.0.0"`
* 插件 manifest 指定 `requiresSDK: "^1.0.0"`
* 不满足：registry 拒绝注册（或降级只允许 emit events）

### 19.2 Scope 与命名空间（防串数据）

支持三种 scope：

* `chat`：默认（chatKey 隔离）
* `character`：角色卡专属（同 chat 可多角色）
* `global`：跨聊天共享（例如用户偏好）

API 所有读写都必须显式带 scope（默认 chat）。

### 19.3 Typed Events（建议提供类型包 stx.d.ts）

给生态插件开发者一个“强类型体验”，事件 payload 统一规范，减少乱写。

### 19.4 插件写入策略（建议默认策略）

* 插件只能写 `events.append`
* 只有 Memory OS（或用户授权插件）才能写 facts/state
* 插件可以“请求写入”：通过 `memory.requestWrite(proposal)` 让 Memory OS 决策

#### requestWrite（关键接口）

```ts
memory.requestWrite({
  source: { pluginId: "dice", version:"0.1.0" },
  chatKey,
  proposal: {
    facts?: [...],
    patches?: [...],
    summaries?: [...]
  },
  reason: "combat outcome"
})
```

Memory OS 返回：

* `accepted: true/false`
* `applied: [factKeys/statePaths/summaryIds]`
* `rejectedReasons: []`

这样骰子、任务、地图都能提交“结构化结果”，但最终落盘由 Memory OS 控制。
