
# 一、最该补强的方向：更贴近 SillyTavern 的“聊天行为模型”

## 1. 把“楼层”从单纯计数，升级成 ST 语义楼层

你现在已经有 `TurnTracker`，并且是按助手回复计楼，这非常对路 。
而且 `ExtractManager` 已经用 `assistantTurnCount + interval` 去触发总结，这比单纯按 event 数量靠谱很多 。

但对 ST 来说，还差一步：

### 现在的问题

当前“楼层”更像：

* 收到一条助手消息
* 且是 `normal`
* 且非系统
* 且没重复
* 就算 1 楼

这在一般场景成立，但对 ST 用户来说还不够，因为他们常做这些事：

* regenerate
* swipe
* edit 上一条消息
* 删除某一楼
* 插入系统楼
* 分支聊天
* 回滚到前几楼继续聊

这些行为不是“新增一楼”那么简单。

### 建议升级

把楼层定义成：

* `visible assistant turn`
* `committed assistant turn`
* `superseded turn`
* `branched turn`

也就是说，一条助手回复不只是“+1 楼”，而是要有状态：

```ts
type TurnLifecycle = 'active' | 'edited' | 'swiped_out' | 'deleted' | 'branch_root';
```

这样做的价值很大：

* 总结只基于 active 楼层
* 被 swipe 掉的回复不要持续污染长期摘要
* edited 楼层可以做差量重抽
* 分支后能知道哪部分记忆该继承，哪部分该冻结

---

## 2. 对齐 SillyTavern 的“消息可变性”

SillyTavern 跟普通 IM 最大区别之一，就是消息不是稳定不可改的。

你现在在 `index.ts` 里已经认真处理了：

* `CHAT_CHANGED`
* `CHAT_STARTED`
* `CHAT_NEW`
* 用户消息
* 助手消息
* `GENERATION_ENDED`
* `CHAT_COMPLETION_PROMPT_READY`

这些监听说明你已经在适配 ST 的消息生命周期 。

但还不够贴 ST 的地方在于：

### 缺少几类“会改变记忆解释”的动作

我目前没看到显式处理：

* 消息编辑
* swipe 切换
* 单条消息删除
* 分支/副本聊天
* 聊天重命名
* 角色卡替换后当前 chat 的 seed 失效

这些动作不一定都由 ST 提供稳定事件，但从产品上必须考虑。

### 建议

把消息接入从“只 append event”升级成“双层模型”：

#### A. 原始事件流

仍然保留你现在的 append 模式。

#### B. 逻辑消息视图

增加一个逻辑层，按 ST 用户眼里的当前聊天结构解释：

* 当前可见用户楼
* 当前可见助手楼
* 被 swipe 覆盖的旧候选
* 被 edit 替换的旧文本
* 已删除的楼层

然后：

* 摘要/抽取优先看逻辑消息视图
* 原始事件流只做审计和回放

这样 MemoryOS 会更像“贴着 ST 界面工作”，而不是只贴着事件总线工作。

---

## 3. 补全“删除聊天”的 ST 式体验

这个我前面提过，现在还是非常重要。

你当前代码里已经有保留策略和删除策略类型：

* `soft_delete`
* `immediate_purge` 

这说明概念已经准备好了。
但要更贴 ST，应该把它做成用户能感知的行为，而不是只停在类型定义。

### 最佳做法

当 ST 删除聊天时：

#### 默认模式：软删除

* 聊天从 ST 列表消失
* MemoryOS 标记 archived
* 不参与注入
* 不参与常规检索
* UI 可以“恢复记忆”

#### 高级模式：彻底清除

* 当前 chat 的 MemoryOS 数据 purge
* LLMHub chat snapshot 也清掉
* shared.signals 清掉
* 向量索引清掉
* audit 保留或可选保留

这会比现在更符合 ST 用户预期，因为 ST 用户经常会误删、试聊、复制聊天。

# 二、让它更贴 ST 的“卡 / 世界书 / 群聊”生态

## 4. 新聊天要真正理解“角色卡冷启动”

当前 `MemorySDKImpl.init()` 只做了：

* `meta.ensureInit()`
* `chatState.load()`

这说明新聊天初始化目前还是“数据库壳初始化”，不是“ST 角色语义初始化” 。

这对引擎来说没问题，但对 ST 来说还不够。

### 更贴 ST 的方式

新聊天初始化时，主动采集这些种子：

* 角色卡核心字段
* system prompt
* first message / opener
* 作者注释
* jailbreak / instruct
* 绑定 lorebook
* 群成员名单
* 当前 preset / 模型上下文风格

然后把这些分三类存：

#### A. 固定身份种子

角色名、别名、身份、阵营、口癖、关系锚点

#### B. 世界设定种子

地点、规则、世界观硬约束

#### C. 生成风格种子

这个角色更偏：

* 叙事
* RP
* 设定答疑
* 工具式响应

这样新聊天从第一轮起就更像“角色驱动对话”，而不是“空记忆库”。

---

## 5. 世界书不只是写回，还要参与“注入裁决”

你现在在模板、世界书读写、worldinfo writeback 上已经做了很多，甚至有 template 管理、worldinfo preview/writeback、逻辑表回写这些能力 。

但要更贴 ST，建议再补一层：

### 让 lorebook 参与注入选择，而不是只做数据源

ST 用户对世界书的直觉不是“数据库素材”，而是“当前这轮该不该出现”。

所以可以加一个 **Lorebook Relevance Gate**：

每轮注入前判断：

* 当前 query 是否是设定型问题
* 最近场景是否涉及 world entities
* 当前聊天是否处在剧情推进还是设定说明模式
* 当前 world state 是否与 lorebook 条目冲突

然后决定：

* lorebook 强注入
* lorebook 弱注入
* 只注摘要，不注原文
* 完全不注

这样可以减少很多“明明开着 lorebook，却总塞太多无关设定”的问题。

---

## 6. 群聊需要单独的记忆策略

你已经有：

* `chatType = group`
* `recentGroupSpeakerCount`
* speaker tracking / entity resolution level 这些结构  

这很好，但更贴 ST 的群聊体验，还应该再进一步：

### 群聊的 ST 特征

ST 群聊常见问题：

* 多角色轮流说话
* 用户会手动插楼
* 群成员卡会变
* 某个角色突然长期沉默
* swipe 可能只替换其中一名角色的回复逻辑

### 建议

群聊单独做：

#### A. speaker memory lanes

每个角色独立保留：

* 最近发言风格
* 最近情绪状态
* 最近目标
* 最近关系变化

#### B. scene-level shared memory

群共享：

* 当前场景
* 当前冲突
* 群体共识
* 未完成事件

#### C. actor salience

谁最近更重要，谁在当前 scene 应该优先进入注入

这样群聊就不会只是“把多人对话混成一锅事件”。

# 三、让它更贴 ST 的“生成流程”

## 7. Prompt 注入要更懂 ST 的系统提示结构

你现在注入是在 `CHAT_COMPLETION_PROMPT_READY` 阶段，把 Memory Context 包成：

`<MEMORY_OS_CONTEXT>...</MEMORY_OS_CONTEXT>`

然后按 anchor policy 插入 system message，这已经相当规范了 。

但更贴 ST 的地方还可以继续优化。

### 建议增强

#### A. 支持更多插入位

不只是：

* top
* beforeStart
* customAnchor

还建议支持：

* system 之后但在 persona 之前
* 作者注释之后
* lorebook 之后
* 最后一个系统消息之后
* 仅在 query 为设定类时插入

#### B. 支持不同注入风格

ST 用户其实很在意 prompt 污染感。

可以加：

* XML 风格
* Markdown 风格
* 注释风格
* 压缩 KV 风格
* 极简 bullet 风格

不同模型对这些结构容忍度不一样。

#### C. 支持角色扮演型软注入

对 RP 模型，很多时候不该把 MemoryOS 看起来像“系统硬命令”。

可以做成：

* Scene Note
* Continuity Note
* Character Anchor
* Hidden Context Summary

这会更自然。

---

## 8. 区分“生成前注入”和“生成后落记忆”

你现在在 `GENERATION_ENDED` 后触发 extract，这很合理 。
但更贴 ST，可以再细分：

### 生成前

关注：

* 这轮该注哪些 memory
* 预算怎么分配
* 是否要压缩当前上下文

### 生成后

关注：

* 这一楼值不值得进长期记忆
* 是剧情推进、设定确认、还是闲聊噪音
* 是否应该立刻重建 summary
* 是否要更新 world state

也就是说，生成前后应该有两套不同 gate，而不只是“一套 MemoryOS”。

# 四、让它更贴 ST 的“用户调参习惯”

## 9. 把设置从“技术参数”翻译成“聊天习惯参数”

你现在的聊天策略编辑器已经很强了，甚至有：

* chat type
* style preference
* memory strength
* extract strategy
* summary strategy
* deletion strategy
* vector thresholds
* retention settings
* quality diagnostics 

但对 ST 用户来说，还可以更顺手一点。

### 建议新增两层设置

#### A. 用户能看懂的场景预设

比如：

* 轻陪伴聊天
* 长剧情 RP
* 世界设定问答
* 跑团 / 多角色
* 工具 / 代码问答

选择后自动映射到底层参数。

#### B. 用户会常调的直白项

直接给这些选项：

* 多少楼总结一次
* 总结覆盖最近多少楼
* 多少楼刷新一次画像
* 多少楼重算一次质量分
* 多少条事实后启用向量
* 聊天删除时：归档 / 彻底删除
* 新聊天是否自动从角色卡初始化
* 群聊是否启用说话人分轨

这比要求用户直接理解 `activationFacts`、`vectorSearchStride` 更贴 ST。

---

## 10. 做“角色/聊天级 preset”

SillyTavern 用户很习惯“这张卡有这张卡的玩法”。

所以建议把覆盖策略支持三层：

* 全局默认
* 角色默认
* 聊天覆盖

例如：

* 某个角色永远高记忆、重关系追踪
* 某个工具角色永远低记忆、低摘要
* 某个具体聊天临时改成 timeline summary

这样用户会觉得 MemoryOS 真正在“贴着卡工作”。

---

# 五、让它更贴 ST 的“长期使用体验”

## 11. 需要“记忆维护感知”，而不只是后台机制

你已经有 quality score、maintenance advice，这非常好  。

下一步建议把它变得更像 ST 用户能直接理解的提示：

* 这个聊天记忆过载了，建议压缩
* 这个聊天太久没重建摘要
* 这个聊天向量命中很差，建议重建索引
* 这个聊天存在大量失效设定，建议整理
* 这个群聊角色状态分裂，建议做群聊维护

这类提示最好能出现在：

* 聊天策略面板
* 当前聊天顶部小诊断
* 执行某次摘要/压缩后 toast

---

## 12. 做“聊天生命周期管理”

这会非常贴 ST。

建议把聊天状态分成：

* new
* active
* stable
* long_running
* archived
* deleted

不同阶段默认策略不同：

### new

* 不急着重抽
* 不开向量
* 轻量 seed

### active

* 正常按楼层总结
* 观察画像

### stable

* 开启摘要/向量协同
* 进入质量监控

### long_running

* 定期压缩
* 清理低价值事实
* 优先 scene summary

### archived

* 不注入
* 可恢复

# 六、我最推荐你优先补的 8 项

按“最能让用户感觉像 ST 原生能力”的优先级排：

### 1. 聊天删除闭环

把 soft delete / immediate purge 真正接起来。

### 2. 消息编辑 / swipe / regenerate 生命周期

让 MemoryOS 不只是 append，而是真正理解“当前可见聊天”。

### 3. 新聊天 seed 冷启动

角色卡 / system / lorebook / 群成员自动挂种子。

### 4. 可调楼层策略

把下面这些直接做成可设置：

* 多少楼总结一次
* 总结窗口多大
* 多少楼刷新画像
* 多少楼重算质量

### 5. 群聊分轨

每个角色独立 lane + 共享 scene memory。

### 6. Prompt 注入位扩展

更贴 ST 的 prompt pipeline。

### 7. 角色级 preset

不是只有聊天级，还要有卡级默认策略。

### 8. 归档聊天与孤儿记忆巡检

长期使用体验会好很多。
