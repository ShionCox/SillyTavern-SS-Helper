# SillyTavern 结构化记忆插件开发文档

> 基于 `SillyTavern-SS-Helper` 的 `memoryos-settings-unified` 分支进行重构扩展。  
> 这份文档用于框架阶段定设计，核心目标是把现有 MemoryOS 往上整理成一套**可维护、可扩展、可兼容多世界观**的结构化记忆系统。  
> 文中示例统一使用 **JSONC** 风格，注释只用于解释字段含义，运行时仍需转回标准 JSON。

---

## 1. 文档定位

这套插件不是“给聊天加一层自动摘要”，而是把现有 MemoryOS 底盘提升成一套真正的**角色视角长期记忆系统**。它需要同时处理冷启动、世界观初始化、角色关系、事件演化、世界状态更新、总结前规划、单次 AI 变更提议、角色视角注入、自然语言渲染，以及多世界观角色卡兼容。

这次重构不推翻现有 MemoryOS。现有分支已经具备冷启动流程、聊天状态、自动总结运行态、角色画像、关系状态、召回缓存、遗忘阶段、维护与修复机制，这些仍然作为底层能力保留。新的插件更适合被理解为：**MemoryOS 上层的结构化记忆引擎 + 世界模板兼容层 + 角色视角注入系统**。这版文档是在你上传的 v6 基础上继续整理，重点补入：**更灵活的记忆召回、记忆度设定、embedding 开关式设计，以及关键词识别资源的独立维护目录**。

---

## 2. 核心设计思路

整个系统要先把职责分清楚，否则后面会很容易越做越乱。

第一层是**统一主数据层**。它负责存储角色、关系、事件、任务、地点、组织、物品，以及世界基础信息。这里的主数据不应该因为世界观不同就拆成多套数据库结构，因为现代都市、古风和魔幻世界，本质上都仍然有“人、地点、关系、事件、规则”这些核心对象。  
第二层是**世界模板层**。它不直接存记忆，而是描述“当前角色卡更像什么世界模板”“这种模板更偏向哪些世界面向”“总结时哪些类型应该更敏感”“哪些字段扩展更常用”“注入时该用什么语言风格”。这一层专门用于兼容现代都市、古风、魔幻以及混合世界观。  
第三层是**总结规划与变更执行层**。程序先分析总结窗口，识别候选类型，结合世界模板加载对应字段定义，再召回候选旧记录，最后才把上下文一次性交给 AI，让 AI 输出唯一一次 mutation document。  
第四层是**角色视角注入层**。它不直接拿主数据生拼硬凑，而是先构建当前角色此刻真正可见、可记得、可理解的内容，然后再渲染成 XML + Markdown 的自然语言上下文注入给模型。

这四层里面，最重要的不是“多加几个功能点”，而是要始终坚持两个边界：**程序先做确定性整理，AI 再做智能判断；结构化存储与自然语言注入必须分离。**

---

## 3. 记忆主数据模型

整套记忆建议仍然维持三类主数据：Core Memory、Episodic Memory、Interpretive Memory。

Core Memory 存放长期稳定、相对不频繁变化的内容，例如角色长期身份、世界核心设定、制度规则、固定关系、组织归属和长期目标。它更像整个世界与角色的骨架。  
Episodic Memory 存放“发生过什么”，也就是见面、争执、战斗、承诺、获得物品、旅行、任务推进、场景切换这类事件性信息。它会持续增长，也是最常被总结、合并和更新的一层。  
Interpretive Memory 存放角色的主观理解与情绪烙印，比如怀疑、误解、偏见、情绪性判断、自我叙事偏差等。它是“世界真实状态”和“角色所相信的状态”之间的缓冲层。

这里的关键是：**多世界观兼容不应该改变这三层的存在方式**。世界模板只决定“这轮更该关注什么、下发什么字段、用什么风格渲染”，而不应该让主数据层变成三四套互不兼容的结构。

## 4. 世界信息建模与基础注入策略

为了尽量减少用户后台操作，世界信息不应该通过 `alwaysOn` 这类开关交给用户自己配，而是由系统根据类型和分类自动决定是否属于基础注入层。

世界信息建议分成三类：`world_core_setting`、`world_hard_rule`、`world_global_state`。

`world_core_setting` 用来描述世界长期成立的基础事实，例如“魔法体系存在”“龙族可化形”“王权结构成立”“货币体系如何运作”。这类信息通常在冷启动时建立，之后变化不频繁，默认进入全局基础注入层。  
`world_hard_rule` 才是真正的硬规则，也就是世界运作法则、长期禁令或制度约束。例如“未登记施法者不得在主城区公开施法”“王都夜间普通平民不得擅自出城”。硬规则不属于具体角色，也不属于局部物件，默认进入全局基础注入层，并且只允许被重大世界事件修改。  
`world_global_state` 则表示某一时期的世界现实。例如“魔王统治下人心惶惶”。它不是法则，而是一个历史阶段的全局状态，所以它会变化。后续当魔王被击杀后，不应该把它理解成“硬规则被改写”，而应理解为旧的全局状态失效，再由新的全局状态取代。

```jsonc
{
  "schemaId": "world_global_state", // 类型：世界全局状态
  "title": "魔王统治下的社会恐慌", // 状态标题
  "summary": "在魔王长期统治下，多数地区民众对外界充满恐惧，夜间鲜少出行。", // 状态内容
  "scope": "global", // 作用域：全局
  "state": "active", // active | invalidated
  "supersededBy": null, // 若后续被替代，这里记录新条目
  "updatedAt": 1711111111111 // 最近更新时间
}
```

注入策略在系统内部规定即可，不应该暴露给用户手工管理。建议默认规则如下：`world_core_setting`、`world_hard_rule`、`world_global_state` 进入 `WORLD_BASE`；`scene_shared_state` 进入 `SCENE_SHARED`；`actor_profile`、`relationship`、`actor_visible_event`、`actor_private_interpretation` 进入 `ACTOR_VIEW`；其他类型由召回系统按需决定是否注入。这样做可以显著减少用户后台操作，同时保证注入行为稳定。

## 5. 多世界观兼容：世界模板层

每个角色卡的世界观可能完全不同。有些是现代都市，有些是古风，有些是魔幻，还有很多是混合型，例如现代都市 + 异能、古风 + 修仙、魔幻 + 政治。如果直接为不同世界观做不同 schema 或不同数据库表，架构会迅速碎裂。更稳的做法是：**主数据层统一，差异收束到世界模板层**。

世界模板层不直接存具体记忆，而是负责回答这些问题：这个角色卡当前最接近哪种世界模板；这种模板默认更关注哪些世界 facet；总结前候选类型识别应该给哪些类型加权；类型字段下发时，哪些扩展字段值得额外补充；注入渲染时应采用怎样的自然语言风格。

```jsonc
{
  "worldProfileId": "urban_modern", // 世界模板 ID
  "displayName": "现代都市", // 模板显示名
  "genre": "modern", // 主风格分类
  "subGenres": ["slice_of_life", "school", "workplace"], // 子类型
  "capabilities": {
    "hasMagic": false, // 是否存在魔法
    "hasCultivation": false, // 是否存在修炼体系
    "hasFantasyRace": false, // 是否存在幻想种族
    "hasModernTechnology": true, // 是否存在现代科技
    "hasFormalPoliticalOrder": true, // 是否存在正式公共秩序
    "hasSupernatural": false // 是否存在超自然
  },
  "preferredSchemas": [
    "character", // 常用结构化类型
    "relationship",
    "event",
    "location",
    "organization",
    "task",
    "world_global_state"
  ],
  "preferredFacets": [
    "social_order", // 常用世界面向
    "city_life",
    "public_rules",
    "career",
    "daily_relationship"
  ],
  "schemaFieldExtensions": {
    "location": ["district", "businessType", "publicAccess"], // 对 location 建议追加的扩展字段
    "organization": ["industry", "legalStatus"] // 对 organization 建议追加的扩展字段
  },
  "summaryBias": {
    "boostedTypes": ["relationship", "event", "task", "organization"], // 总结前提高权重的类型
    "suppressedTypes": ["artifact_rule", "race_rule"] // 降低权重的类型
  },
  "injectionStyle": "modern_realistic" // 注入时的自然语言风格
}
```

很多角色卡并不是纯类型，所以模板识别最好支持主模板 + 次模板，而不是只允许单模板。冷启动阶段可输出 `primaryProfile` 和 `secondaryProfiles`，后续在总结、字段下发和注入时共同产生偏置。

## 6. 结构化主数据示例

基础块 `MemoryBlockSchema` 负责定义某类记录允许有哪些字段、默认重要度是什么、默认遗忘模型是什么、注入时用什么渲染模板、总结时哪些字段允许 AI 修改。它是模板，不是实际记录。

```jsonc
{
  "schemaId": "event", // 基础块唯一标识
  "type": "event", // 所属大类
  "label": "事件", // 界面显示名称
  "description": "事件类记忆模板", // 类型用途描述
  "icon": "sparkles", // UI 图标
  "color": "#7E57C2", // UI 主色
  "defaultImportance": 0.75, // 默认重要度
  "defaultDecayModel": "medium", // 默认遗忘模型：slow | medium | fast
  "mergeStrategy": "field_merge", // 默认合并策略
  "renderTemplate": "{{title}}：{{summary}}", // 默认渲染模板
  "fields": [
    {
      "key": "title", // 字段键名
      "label": "标题", // 字段显示名称
      "kind": "text", // 字段类型
      "required": true // 是否必填
    },
    {
      "key": "participants",
      "label": "参与角色",
      "kind": "tags"
    },
    {
      "key": "location",
      "label": "地点",
      "kind": "text"
    },
    {
      "key": "outcome",
      "label": "结果",
      "kind": "textarea"
    }
  ]
}
```

记忆块 `MemoryRecord` 才是真正存储的数据。这里允许有结构化字段、数值、状态、索引、来源和 revision 信息。要特别注意的是，这一层是程序内部主数据，不应该原样进入提示词。

```jsonc
{
  "recordId": "mem_evt_001", // 记录主键，由程序生成
  "schemaId": "event", // 对应基础块类型
  "title": "桥上争执", // 记录标题
  "summary": "艾琳与男主因身份隐瞒问题发生了正面争执。", // 摘要
  "content": "更完整的事件描述正文。", // 扩展内容
  "ownerActorKey": "char_erin", // 主要归属角色
  "participants": ["char_erin", "char_mc"], // 参与角色
  "visibilityScope": "actor_private", // 可见范围：global | group | actor_private
  "memoryType": "event", // 记忆大类
  "memorySubtype": "major_plot_event", // 细分类
  "importance": 0.82, // 重要度
  "strength": 0.76, // 记忆强度
  "forgetBase": 0.22, // 基础遗忘概率
  "decayStage": "clear", // 遗忘阶段：clear | blur | distorted
  "truthStatus": "confirmed", // 真实性状态
  "emotionTag": "conflict", // 情绪标签
  "relationScope": "char_erin->char_mc", // 关系作用域
  "tags": ["争执", "身份隐瞒"], // 辅助标签
  "fields": {
    "location": "旧城区桥边", // 类型专属字段：地点
    "outcome": "关系紧张升级" // 类型专属字段：结果
  },
  "state": "active", // active | archived | invalidated
  "revision": 1, // 当前版本号
  "createdAt": 1711111111111, // 创建时间
  "updatedAt": 1711112222222, // 更新时间
  "lastRecalledAt": 1711113333333 // 最近一次召回时间
}
```

## 7. 冷启动设计：加入世界模板识别与统一提示词维护

冷启动的职责不再只是建立“世界骨架 + 角色骨架 + 初始关系 + 初始结构化记忆”，现在还要顺带建立**世界模板识别结果**。因为后续的总结偏置、字段扩展和注入风格，都要基于这一层来工作。

冷启动输入源包括角色卡、系统提示词、首条消息、作者注释、Jailbreak、Instruct，以及角色绑定的世界书和用户选中的世界书条目。冷启动阶段除了写入 `identity / worldBase / relationships / memoryRecords`，还应该输出 `worldProfileDetection`，并把结果写入 `WorldProfileBinding`。

更重要的是：**冷启动提示词和总结提示词应该放到同一个文件里集中维护**。不要把它们散在不同模块里到处拼字符串，否则后期修改规范、样例和字段说明会很痛苦。建议单独建一个 `memory-prompts/` 目录，并使用一个统一文件来维护整个 prompt pack。

建议结构如下：

```text
MemoryOS/src/
  memory-prompts/
    prompt-pack.md
    prompt-loader.ts
    prompt-renderer.ts
```

其中 `prompt-pack.md` 是唯一维护文件，里面按 section 划分：

- `COLD_START_SYSTEM`
- `COLD_START_SCHEMA`
- `COLD_START_OUTPUT_SAMPLE`
- `SUMMARY_SYSTEM`
- `SUMMARY_SCHEMA`
- `SUMMARY_OUTPUT_SAMPLE`

`prompt-loader.ts` 负责把这个文件按 section 读取出来，`prompt-renderer.ts` 负责在运行时把变量渲染进去。这样做的好处是：规范、正文、样例和字段说明全部集中在一个地方维护，不会出现冷启动改了一版字段，总结提示词却忘了同步的问题。

## 8. 冷启动提示词设计

冷启动提示词建议用 XML + Markdown 形式编写，目标不是让 AI 自由写设定，而是让它基于输入源构造一份标准化初始化 JSON。提示词正文和样例写法应当一起放在 `prompt-pack.md` 中维护。

### 8.1 冷启动提示词正文示意

```md
<!-- section: COLD_START_SYSTEM -->

你正在执行结构化记忆系统的冷启动初始化任务。

你的目标不是写小说，不是扩写内容，也不是自由补完设定。
你的任务是：根据角色卡、系统提示词、首条消息、作者注释、世界书内容，提取当前聊天开局阶段最稳定、最重要、最适合写入长期记忆系统的结构化信息。

请遵循以下原则：

1. 只提取高价值、稳定或高影响的信息。
2. 优先输出角色身份、世界基础设定、世界硬规则、世界全局状态、初始关系、初始事件。
3. 若能判断当前角色卡更接近哪种世界模板，请输出 worldProfileDetection。
4. 不要输出无依据的内容，不要强行补设定。
5. 只能返回 JSON，不要输出解释文本。
```

### 8.2 冷启动输出样例与字段说明

## 9. 总结前规划与总结提示词设计

AI 总结只有一次，所以在进入这一次总结之前，程序必须先做一次轻量但确定性的预分析。它要先读取本次要总结的窗口，分析窗口中有哪些变化信号，再结合世界模板偏置识别这轮涉及哪些候选类型，然后按候选类型加载对应字段定义，再按候选类型召回相关旧记录，最后才构造 `summary mutation context` 给 AI。

这里的关键不是“让 AI 猜库里应该改什么”，而是：**程序先把可能需要改动的内容缩小到一个候选集合，再让 AI 判断具体是 ADD、MERGE、UPDATE、INVALIDATE 还是 NOOP。**

### 9.1 总结前上下文包

```jsonc
{
  "task": "memory_summary_mutation", // 当前任务类型
  "schemaVersion": "1.0.0", // 协议版本
  "window": {
    "fromTurn": 20, // 本轮窗口起始楼层
    "toTurn": 28, // 本轮窗口结束楼层
    "summaryText": "艾琳发现男主隐瞒身份，并在桥上与其发生争执。" // 程序压缩后的窗口摘要
  },
  "detectedSignals": {
    "candidateTypes": ["relationship", "event"], // 本轮候选类型
    "actors": ["char_erin", "char_mc"], // 涉及角色
    "topics": ["identity_secret", "argument", "trust_shift"], // 涉及主题
    "relationshipKeys": ["relationship:char_erin:char_mc"] // 涉及关系键
  },
  "worldProfileBias": {
    "primaryProfile": "fantasy_magic", // 当前主模板
    "secondaryProfiles": ["medieval_political"] // 次模板
  },
  "typeSchemas": [
    {
      "schemaId": "relationship", // relationship 的字段白名单
      "editableFields": ["summary", "trust", "affection", "tension", "unresolvedConflict", "milestones"]
    },
    {
      "schemaId": "event", // event 的字段白名单
      "editableFields": ["title", "summary", "importance", "memorySubtype", "participants", "tags", "fields.location"]
    }
  ],
  "candidateRecords": [
    {
```

## 10. 记忆召回流程：第一版设计

这版要把召回逻辑讲清楚，而且不要做得太死。

记忆召回不是“固定拿几条事件 + 几条关系”，而是先构建当前角色能看到的记忆池，再从里面按优先级往提示词里放，直到达到当前回合的记忆上下文预算为止。也就是说，这里不建议做过于僵硬的“条数限制”，而是做**预算式召回**。  
程序每轮召回时，先确定当前视角角色和当前场景，再拿默认存在的世界基础层，然后从角色可见记忆里继续筛：当前角色自己的资料、当前角色和他人的关系、当前角色亲历或可知的事件、当前角色的主观理解。之后再根据当前输入主题、当前场景、当前 world profile 和相关性评分做排序。

为了让召回更符合“角色记性”的感觉，每条记忆都应带有一个**记忆度**或**记忆强度**概念。这个值不是单纯用来存着好看，而是直接参与召回。记忆度高的内容，在排序时更容易被提前拿出来；记忆度低的内容则更容易被放到后面，或者在渲染时变成模糊表达。也就是说，召回不仅要看“相关不相关”，还要看“当前角色还记得有多清楚”。

```jsonc
{
  "recordId": "mem_evt_001", // 记忆记录 ID
  "importance": 0.82, // 事件本身的重要度
  "strength": 0.76, // 当前整体记忆强度，可理解为记忆度
  "forgetBase": 0.22, // 基础遗忘概率
  "decayStage": "clear", // 当前记忆阶段
  "lastRecalledAt": 1711113333333 // 最近一次被召回时间
}
```

### 10.1 第一版召回评分思路

第一版不需要把召回做得太复杂，先做到可控即可。  
召回分数建议至少由四部分组成：文本相关度、角色相关度、时间连续性、记忆度。

- 文本相关度：当前输入和记忆内容是否相关  
- 角色相关度：是不是当前角色自己的、是不是当前角色直接参与的  
- 时间连续性：是不是最近发生、最近提到、最近刚强化过  
- 记忆度：这条记忆当前还记得有多清楚  

召回时先拿固定基础层，再从角色可见池里按得分往里放，直到达到预算即可。

### 10.2 第一版 embedding 开关设计

第一版建议把 embedding 做成**可开关项**，不是强依赖。

- 如果 `enableEmbedding = true`：先预留 embedding 检索接口，但第一阶段不强绑定具体模型服务  
- 如果 `enableEmbedding = false`：直接走本地轻量文本相似度方案

这样可以先把架构定好，而不用一开始就把本地模型、server plugin、模型生命周期全绑定死。

建议配置结构如下：

```jsonc
{
  "retrievalConfig": {
    "enableEmbedding": false, // 是否启用 embedding 检索
    "retrievalBudget": 1.0, // 当前回合召回预算占比，内部使用
    "fallbackMethod": "bm25", // 未启用 embedding 时采用的主相似度方法
    "allowNgramBoost": true, // 是否启用 n-gram 补充分
    "allowEditDistanceBoost": true // 是否启用编辑距离补充分
  }
}
```

### 10.3 第一版不启用 embedding 时的方案选择

你提到的这些方法里：

- BM25
- TF-IDF
- Jaccard
- 编辑距离
- n-gram 相似度

如果要选一个作为**第一版主方法**，我建议选 **BM25**。

原因很简单：BM25 在“短文本检索 + 关键词相关度”这个场景下更稳，通常比纯 TF-IDF 更适合做检索主分。  
Jaccard、编辑距离、n-gram 更适合作为补充信号，而不适合作为主检索方法。  
所以第一版建议：

- 主方法：**BM25**
- 补充增强：**n-gram 相似度 + 编辑距离**
- 暂时不把 Jaccard 作为主方法

这意味着第一版可以做成：

> 先用 BM25 排主序，再用 n-gram 和编辑距离给相近表达做小幅加权修正。

这样已经足够比纯关键词强，而且不需要 embedding。

### 10.4 embedding 接口预留思路

即便第一版不开启 embedding，也建议把接口先留出来，后续好升级。

```ts
interface RetrievalProvider {
  search(query: string, candidates: RetrievalCandidate[]): Promise<RetrievalResult[]>
}
```

然后实现两套：

- `LexicalRetrievalProvider`：第一版实际使用，内部走 BM25 + 辅助相似度
- `EmbeddingRetrievalProvider`：先预留，占位不强制接入

这样后期如果你想切换到本地 embedding，只需要替换 provider，不需要推翻整个召回层。

## 12. 提示词注入模型：XML + Markdown + 自然语言渲染

注入给模型的提示词不应使用原始 JSON，也不应直接拼接一大堆散乱字段。最适合的形式是：**XML 外壳 + Markdown 内容块**。

整个注入上下文建议固定成三层：`<world_base>` 负责世界基础层，`<scene_shared>` 负责当前场景共享层，`<actor_view>` 负责当前角色视角层。`<world_base>` 默认注入世界核心设定、世界硬规则和世界全局状态；`<scene_shared>` 注入当前公共场景状态；`<actor_view>` 则根据当前 `activeActorKey` 动态构建当前角色可见的信息。

多世界观兼容不仅体现在总结和字段下发上，也体现在注入时的语言风格上。同样一条世界状态，在现代都市模板、古风模板和魔幻模板下，系统应当使用不同的表达方式，但底层主数据不应变化。因此，注入渲染器应根据 `worldProfile.injectionStyle` 选择不同的 narrative renderer。

```xml
<memory_context version="1.0">
  <world_base>
## 世界基础设定
- 王都夜间禁止普通平民擅自出城。
- 未登记施法者不得在主城区公开施法。
- 魔王死后，社会秩序正在缓慢重建。
  </world_base>

  <scene_shared>
## 当前共享场景
- 地点：旧城区桥边
- 当前冲突：身份隐瞒导致争执升级
- 当前参与者：艾琳、男主
  </scene_shared>

  <actor_view actor="char_erin">
## 当前视角角色
- 艾琳，王都情报员，性格冷静、多疑、克制。
- 当前目标：确认男主真实身份；继续调查失踪案。

### 当前关系认知
- 她对男主有一定好感，但信任刚刚下降，内心戒备明显上升。

### 当前可见事件
- 她记得：自己与男主因身份隐瞒问题发生了正面争执。
- 她隐约记得：男主似乎曾在雨夜帮过自己一次。

### 当前主观理解
- 她怀疑男主仍在隐瞒更重要的身份信息。
- 这次争执让她短期内更难完全信任对方。
  </actor_view>
</memory_context>
```

## 13. 角色可见记忆与自然语言渲染

每轮注入时，系统不应问“数据库里现在总共有多少记忆”，而应问：**当前角色此刻能够得到哪些记忆信息**。

这意味着数据库中的主数据需要在注入前被转换成一个临时视图对象，可以称为 `ActorVisibleMemoryContext`。它不是主存储，而是每轮推理前根据 `activeActorKey`、当前场景、当前意图和世界模板动态生成的视图。它至少要包含四部分：当前角色自己的长期资料、当前角色对其他人的关系认知、当前角色可见的近期事件，以及当前角色的主观理解、怀疑、误解和失真记忆。

同时，角色属性、关系值、遗忘概率、失真倾向等数值应该保留在存储层和策略层，用于排序、裁剪、判断和渲染，但**不应直接进入提示词**。模型看到 `trust = 0.42`、`forgetProbability = 0.67` 这种字段，帮助并不大，反而会让提示词机械化。正确方式是先在程序内部算数值，再将其转成自然语言描述。

因此，渲染层应提供专门的自然语言转换函数，例如 `renderRelationshipNarrative()`、`renderActorMemoryNarrative()`、`renderEventMemoryNarrative()`、`renderWorldStateNarrative()`。而且这些 renderer 最好继续拆成“基础 renderer + 模板 renderer”，也就是先有通用叙述逻辑，再允许 `urban-modern.ts`、`fantasy-magic.ts` 这种模板实现覆写措辞。

## 14. 遗忘与失真机制

遗忘建议维持三阶段模型：`clear / blur / distorted`，并且遗忘状态应按角色独立计算，而不是全局统一。也就是说，同一条事件，对 A 来说可能依旧清晰，对 B 来说已经模糊，对 C 来说甚至已经失真。

```jsonc
{
  "perActorRetention": {
    "char_erin": {
      "forgetProbability": 0.12, // 艾琳视角下的遗忘概率
      "stage": "clear", // 当前阶段
      "rehearsalCount": 4 // 已被强化次数
    },
    "char_mc": {
      "forgetProbability": 0.54, // 男主视角下的遗忘概率
      "stage": "blur",
      "rehearsalCount": 1
    }
  }
}
```

遗忘计算可继续参考现有组合公式，但对模型输出时，一律走渲染层。也就是说，系统内部可以保留 `forgetProbability`、`distortionTendency` 这些数值，而模型最终只看到自然语言，例如“她隐约记得……”或“她模糊地认为……”。

## 16. 开发规范

整个重构阶段的开发规范需要继续收束。UI 不直接操作 DB；总结模块不直接写 Prompt；召回与注入构建只读主数据，不负责写入；planner 只做候选分析与上下文构造，不直接落库；schema 模块只提供类型定义和字段白名单；world-profile 模块只负责模板识别与偏置决策；renderer 模块只负责自然语言输出；prompt-loader 只负责读取和分发 prompt asset；retrieval provider 只负责检索分数；AI 输出永远不能直接落库，必须经过解析、校验、匹配、去重和 mutation apply。

所有 AI 返回对象必须带 `schemaVersion`；所有数值字段都要 clamp；所有数组必须去重；AI 不允许直接指定内部主键去覆盖现有记录；AI 只能写本轮实际下发的 `editableFields`；所有 JSONC 示例在运行前都必须转回标准 JSON。  
每个世界模板必须独立文件；模板文件不允许直接操作数据库；模板文件只导出模板定义、字段扩展和偏置策略；模板注册统一走 `registry.ts`；新增模板时必须同时补充对应注入风格 renderer。  
关键词识别资源必须独立目录维护，不允许在业务逻辑文件中散落大量硬编码关键词。  
第一版 retrieval 默认不依赖 embedding，embedding 只作为 provider 接口预留，不作为强制依赖。  
日志与追踪必须覆盖冷启动开始/成功/失败、世界模板识别结果、总结触发与结果、candidateTypes 检测结果、typeSchemas 下发结果、candidateRecords 命中结果、retrieval provider 选择结果、mutation 执行结果、retention 判定、注入构建结果和 maintenance 结果。  
测试层面，单元测试优先覆盖 schema 注册、world-profile detector、candidateTypes 检测、字段白名单校验、BM25 检索、关键词 matcher、mutation validator、合并规则、forgetting 公式和 narrative render；集成测试重点覆盖冷启动、world-profile 识别、planner、总结、召回、注入构建、repair 和角色切换；回归测试重点覆盖消息编辑、swipe、删除、分支聊天、角色切换、模板切换和冷启动重建。

## 17. 第一阶段验收标准

第一版不需要一步做到所有高级能力，但至少应形成完整闭环：冷启动可以建立角色、关系、世界基础信息和初始事件，并识别世界模板；冷启动提示词和总结提示词已经被统一放到同一个 prompt file 里维护，并带有输出 JSON 样例和字段说明；总结前可以先识别候选类型，再按候选类型与世界模板偏置下发字段定义，再召回候选旧记录；召回层默认使用 BM25 为主、n-gram 与编辑距离为辅，不强制依赖 embedding；系统可以根据 `activeActorKey` 和 `worldProfile` 构建角色可见记忆视图；召回不采用过于僵硬的条数限制，而采用预算式召回；每条记忆的记忆度会参与召回排序和渲染；注入层可以用 XML + Markdown 输出自然语言形式的世界基础层、共享场景层和角色视角层；遗忘至少支持 `clear / blur / distorted` 三种输出效果；关键词识别资源已独立目录维护，方便后续继续添加；模板层至少支持现代都市、古风、魔幻三类基础模板，并允许继续扩展。

## 18. 结论

这版继续往前收束之后，框架主线已经很清楚了。

第一，世界信息要分层，世界核心设定、硬规则和全局状态必须分开建模，并由系统自动决定基础注入策略，而不是让用户在后台手工设置开关。  
第二，多世界观兼容要通过世界模板层解决，而不是为现代都市、古风、魔幻分别做三套数据库系统。统一主 schema，模板决定偏置、扩展字段和渲染风格，这样架构才不会裂开。  
第三，AI 总结前必须先经过候选分析，先识别类型，再结合模板下发字段，再召回旧记录，最后才执行那一次唯一的 AI mutation 总结。  
第四，记忆召回第一版采用预算式而不是死板条数式，并且记忆度直接参与召回排序与最终渲染。  
第五，embedding 在第一版里不是强依赖，而是通过 provider 接口预留；默认关闭时，召回主方法采用 BM25，并用 n-gram 与编辑距离做补充增强。  
第六，关键词识别资源、世界模板、渲染器和提示词资产都必须拆文件夹、拆文件维护，不要把所有词典、偏置、风格逻辑和 prompt 规则塞进一个大文件。  
第七，冷启动提示词和总结提示词要作为一套完整协议包，放在同一个文件里维护；Prompt Pack 内统一维护 system prompt、JSON Schema、样例和字段说明，而真正的结构化请求统一通过 `window.STX.llm.runTask(..., schema)` 发起。  
第八，数据库中的结构化数值不能原样进入提示词，提示词必须使用 XML + Markdown，并且由渲染层把数值和状态转换成自然语言。  
第九，每轮注入给模型的不是全部记忆，而是基础世界层 + 当前角色可见层，并且这套视图还要受当前世界模板影响，从而保证不同世界观下的表达方式自然一致。

如果按这个方向推进，这套插件会更像一个真正的“角色视角长期记忆系统 + 多世界观兼容框架 + 提示词资产集中维护体系 + 可渐进升级的召回架构”，而不只是一个会自动总结的插件。



## 9.4 LLMHub 统一结构化调用入口

在真正实现时，记忆插件**不应该自己直接区分 OpenAI、Claude、Gemini、DeepSeek**，也不应该在插件内部手写多套 provider 调用逻辑。对外统一只使用一个入口：`window.STX.llm`。  
也就是说，冷启动和总结阶段虽然都由插件自己维护 Prompt Pack、JSON Schema、样例输出和字段说明，但真正发请求时，统一通过 LLMHub 暴露的 SDK 来完成。这样做有两个直接好处：一是业务层不会被 provider 差异污染；二是用户切换模型资源时，记忆插件本身不需要改代码。

这一层的职责边界要明确：

- **记忆插件负责**：Prompt Pack、冷启动/总结用到的 JSON Schema、字段说明、输出样例、validator/repair 规则
- **LLMHub 负责**：检测当前模型资源能力、把统一 schema 转成各 provider 所需格式、执行 `runTask`、统一返回结果

换句话说，插件维护的是“业务协议”，LLMHub 维护的是“模型适配协议”。

### 9.4.1 对外只用 `window.STX.llm`

LLMHub 对外最主要的统一入口就是：

```ts
const llm = (window as any).STX?.llm;
if (!llm) throw new Error('LLMHub 不可用');
```

插件真正需要用的，基本就三步：

第一步先注册消费方和任务；第二步调用 `runTask`；第三步读取统一返回结果。  
这套入口在 `llm-sdk.ts` 和 `schema/types.ts` 里已经定义好了，记忆插件不需要自己再包一层 provider-specific SDK。

### 9.4.2 注册任务

记忆插件启动时，应先向 LLMHub 注册自己的插件与任务，尤其是结构化输出任务要声明 `json` 能力。冷启动任务和总结任务都属于 `taskKind: "generation"`。

```ts
const llm = (window as any).STX?.llm;
if (!llm) throw new Error('LLMHub 不可用');

llm.registerConsumer({
  pluginId: 'stx_memory_plugin',
  displayName: 'Structured Memory Plugin',
  registrationVersion: 1,
  tasks: [
    {
      taskId: 'memory_cold_start',
      taskKind: 'generation',
      requiredCapabilities: ['chat', 'json'],
      description: '冷启动结构化记忆初始化',
      backgroundEligible: false,
    },
    {
      taskId: 'memory_summary_mutation',
      taskKind: 'generation',
      requiredCapabilities: ['chat', 'json'],
      description: '增量总结并输出 mutation document',
      backgroundEligible: false,
    },
  ],
});
```

这里最重要的是两点：  
一是 `pluginId` 要稳定；二是所有需要结构化 JSON 输出的任务，都要在 `requiredCapabilities` 里声明 `json`。这样 LLMHub 才能优先为这些任务路由到支持结构化输出的资源，或者在必要时执行降级。

### 9.4.3 统一通过 `runTask` 发起结构化任务

冷启动和总结这两类任务，最终都建议统一走：

```ts
await llm.runTask<MyType>({
  consumer: '你的插件ID',
  taskId: '你的任务ID',
  taskKind: 'generation',
  input: {
    messages: [
      { role: 'system', content: '你的系统提示词' },
      { role: 'user', content: '你的用户输入' },
    ],
  },
  schema: { ...JSON Schema... },
});
```

也就是说，记忆插件只需要关心：

- `consumer`
- `taskId`
- `taskKind`
- `input.messages`
- `schema`
- 可选的 `budget`
- 可选的 `enqueue`

结构化输出的关键就在于：**Prompt Pack 里维护的 JSON Schema，最终由插件作为 `schema` 直接传给 `runTask`**。  
这也是为什么前面文档强调要把冷启动提示词、总结提示词、输出样例、字段说明和 JSON Schema 全部集中维护在同一个 Prompt Pack 文件里。因为真正调用时，system prompt 和 schema 会一起成为结构化任务输入的一部分。

### 9.4.4 冷启动调用示例

冷启动任务最终建议这样调用：

```ts
const result = await llm.runTask<{
  schemaVersion: string;
  identity: {
    actorKey: string;
    displayName: string;
    aliases: string[];
    identityFacts: string[];
    originFacts: string[];
    traits: string[];
  };
  worldProfileDetection?: {
    primaryProfile: string;
    secondaryProfiles: string[];
    confidence: number;
    reasonCodes: string[];
  };
  worldBase: Array<{
    schemaId: string;
    title: string;
    summary: string;
    scope: string;
  }>;
  relationships: Array<{
    sourceActorKey: string;
    targetActorKey: string;
    summary: string;
    trust?: number;
    affection?: number;
    tension?: number;
  }>;
  memoryRecords: Array<{
    schemaId: string;
    title: string;
    summary: string;
    importance?: number;
  }>;
}>({
  consumer: 'stx_memory_plugin',
  taskId: 'memory_cold_start',
  taskKind: 'generation',
  input: {
    messages: [
      {
        role: 'system',
        content: coldStartSystemPrompt, // 来自 Prompt Pack 中的 COLD_START_SYSTEM
      },
      {
        role: 'user',
        content: coldStartUserPayload, // 角色卡、世界书、首条消息等拼出的用户侧上下文
      },
    ],
  },
  schema: coldStartJsonSchema, // 来自 Prompt Pack 中的 COLD_START_SCHEMA
  budget: {
    maxLatencyMs: 12000,
  },
  enqueue: {
    displayMode: 'compact',
  },
});
```

如果 `result.ok === true`，就直接使用 `result.data`。  
它的结构就是你在冷启动 schema 中定义的对象，不需要插件再从一大段文本里自己 parse 一次。

### 9.4.5 总结调用示例

总结任务同样统一走 `runTask`：

```ts
const result = await llm.runTask<{
  schemaVersion: string;
  window: {
    fromTurn: number;
    toTurn: number;
  };
  actions: Array<{
    action: 'ADD' | 'MERGE' | 'UPDATE' | 'INVALIDATE' | 'DELETE' | 'NOOP';
    targetKind: string;
    candidateId?: string;
    compareKey?: string;
    payload?: Record<string, unknown>;
    reasonCodes?: string[];
  }>;
}>({
  consumer: 'stx_memory_plugin',
  taskId: 'memory_summary_mutation',
  taskKind: 'generation',
  input: {
    messages: [
      {
        role: 'system',
        content: summarySystemPrompt, // 来自 Prompt Pack 中的 SUMMARY_SYSTEM
      },
      {
        role: 'user',
        content: summaryUserPayload, // summary mutation context
      },
    ],
  },
  schema: summaryJsonSchema, // 来自 Prompt Pack 中的 SUMMARY_SCHEMA
  budget: {
    maxLatencyMs: 10000,
  },
  enqueue: {
    displayMode: 'compact',
  },
});
```

总结任务的重点不是让模型自由发挥，而是：**Prompt Pack 提供 system 约束与字段说明，planner 提供窗口、候选类型、候选旧记录和字段白名单，LLMHub 负责把 schema 交给当前模型资源执行。**

### 9.4.6 返回结果如何处理

LLMHub 的统一返回结构是：

```ts
type LLMRunResult<T> =
  | { ok: true; data: T; meta: LLMRunMeta }
  | { ok: false; error: string; reasonCode?: string; ... }
```

所以处理逻辑非常简单：

- 成功时，直接使用 `result.data`
- 失败时，读取 `result.error` 和 `result.reasonCode`
- 不需要自己再按 provider 分支判断返回结构

```ts
if (!result.ok) {
  console.error(result.error, result.reasonCode);
  return;
}

console.log(result.data);
```

这也是使用 LLMHub 的主要价值之一：**插件层不需要处理各家接口不同的返回格式。**

### 9.4.7 JSON Schema 放在 system 提示词里，还是单独作为 schema 传入？

这部分要讲清楚，因为前面你特别提到了这一点。

通用原则是：

- **业务上，JSON Schema 属于 Prompt Pack 的一部分，和 prompt、样例、字段说明一起维护**
- **调用上，JSON Schema 应作为 `schema` 参数传给 `runTask`**
- system prompt 里仍然要保留“你必须按 schema 返回 JSON”的文字说明，但不需要把整份 schema 再手写拼进 system 文本里

也就是说，文档里可以继续保留“通用：把 JSON schema 放到 SYSTEM 提示词中，通过描述让 AI 根据要求回复”这个原则，但在真正实现时，应理解成：

> **Schema 与 system 提示词在同一个 Prompt Pack 中统一维护；真正发起结构化任务时，system prompt 负责语义约束，schema 字段负责正式结构约束。**

这样做比把整份 JSON Schema 直接塞进 system 文本更稳定，也更符合 LLMHub 当前的调用方式。

### 9.4.8 不同模型资源由 LLMHub 自动适配

记忆插件不需要自己区分 OpenAI、Claude、Gemini、DeepSeek。  
LLMHub 会根据当前资源的 `apiType` 自动处理结构化输出：

- `OpenAI`：走 `chat/completions + response_format json_schema/json_object`
- `Claude`：走原生 `/messages`
- `Gemini`：走原生 `generateContent + responseJsonSchema`
- `DeepSeek`：优先 `json_object`，必要时附加强 system 约束
- `Generic / Tavern`：自动降级到“system/prompt 强制 JSON”模式

因此，记忆插件只需要定义“我要什么 schema、我要什么 messages”，不需要自己维护一套 provider 兼容层。

### 9.4.9 这一层在文档中的最终落点

所以这里的架构结论应该明确写成：

- Prompt Pack 统一归记忆插件维护
- 冷启动与总结任务统一通过 `window.STX.llm` 发起
- 结构化输出统一使用 `runTask(..., schema)`
- provider 能力探测与结构化适配由 LLMHub 负责
- 插件层不自己分 OpenAI / Claude / Gemini / DeepSeek
- 成功时直接使用 `result.data`
- 失败时读取 `result.error / reasonCode` 并进入 repair / retry 流程
