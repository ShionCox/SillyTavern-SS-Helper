# MemoryOS 旧聊天接管与分批补建完整实施方案（MD）
> 说明：本方案按照你当前明确提出的要求设计：**不做 token 限制**，而是把控制权交给用户，由用户决定**处理哪些楼层**、以及**每批处理多少层**。

---

## 一、目标与原则

## 1.1 总目标

当用户安装插件后，打开一个已经存在大量历史消息的旧聊天时，系统不再把它当成普通新聊天，而是进入一套独立的“旧聊天接管初始化”流程。该流程需要支持：

1. 识别旧聊天并进入接管模式。
2. 允许用户选择要处理的楼层范围。
3. 允许用户设置每批处理多少层。
4. 按批次完成历史分析、摘要、提炼和整合。
5. 支持暂停、继续、失败重试、中断恢复。
6. 在处理过程中保持当前聊天可继续使用。
7. 最终生成可供 MemoryOS 持续使用的稳定记忆结构。

---

## 1.2 核心设计原则

### 原则 A：不限制 token，但限制处理边界

这里不再以 token 上限作为产品层限制条件，而是把边界控制交给：

- 用户选中的楼层范围
- 用户设置的每批楼层数

也就是说，系统不主动拦 token；系统只负责：

- 按你选中的范围读取历史
- 按你设置的批次大小切分
- 一批一批地执行

这样做的好处是：

- 产品逻辑更符合用户预期，尤其适合高配模型、高上下文模型使用者
- 不需要在 UI 中暴露复杂 token 参数
- 处理成本由用户对“范围”和“批次大小”自行权衡

---

### 原则 B：旧聊天接管与新聊天冷启动彻底分离

必须把这两类流程拆开：

#### 新聊天冷启动

处理的是：

- 角色卡
- 世界书
- 初始设定
- 基础身份
- 初始规则

本质是“从设定建盘”。

#### 旧聊天接管初始化

处理的是：

- 已经发生过大量剧情的聊天历史
- 已经变化过的关系、任务、状态
- 已经形成的阶段性世界状态

本质是“接管已有盘面”。

这两个流程不能共用同一套入口逻辑，也不能让旧聊天继续走普通冷启动。

---

### 原则 C：历史批处理的中间产物优先是“章节摘要”，不是大量细碎长期记忆

历史分批处理时，每一批的直接输出不应该是一大堆碎片 memory entry，而应该优先产出：

- `batchDigest`
- `chapterDigest`
- `factCandidates`
- `relationTransitions`
- `taskTransitions`
- `unresolvedThreads`

最后再由统一整合阶段决定：

- 哪些进入长期记忆
- 哪些进入当前活跃快照
- 哪些只保留为章节索引

这样可以显著降低：

- 重复记忆
- 冲突记忆
- 批次间互相覆盖
- 后续维护难度

---

### 原则 D：优先保证“可接着聊”，再追求“历史全补齐”

旧聊天接管不能让用户一上来就长时间等待全部历史跑完。

所以流程必须分两层：

#### 第一层：立即可用层

先建立：

- 静态设定基线
- 当前活跃状态
- 最近阶段摘要

使聊天可以尽快进入“可继续聊”的状态。

#### 第二层：深度补建层

再按批次处理更老的历史，并逐步完成全量历史补建。

---

## 二、完整功能范围

本次一次性实现，建议直接覆盖以下功能范围，不再拆成半成品：

### 2.1 入口与识别

- 识别当前会话是否为“旧聊天”
- 识别当前会话是否已有 MemoryOS 初始化记录
- 针对旧聊天弹出接管配置面板

### 2.2 用户配置

- 选择处理范围
- 选择是否处理全部楼层
- 选择自定义起止楼层
- 设置每批处理多少层
- 设置是否自动继续下一批
- 设置是否在失败时自动暂停
- 设置是否优先处理最近区间

### 2.3 处理执行

- 静态基线提取
- 最近活跃区间处理
- 历史批次处理
- 批次结果持久化
- 中途暂停 / 继续 / 重试
- 最终整合

### 2.4 数据与状态

- 接管任务状态管理
- 批次任务状态管理
- 检查点持久化
- 去重与冲突处理
- 最终记忆结构产物

### 2.5 可视化与可观测性

- 总进度
- 当前批次进度
- 已完成楼层范围
- 错误批次
- 重试入口
- 预计剩余批次数
- 关键摘要预览

### 2.6 后续复用

- 接管完成后继续纳入正常 MemoryOS 工作流
- 支持未来重新补建某一段历史
- 支持重新整合

## 三、用户侧交互方案

## 3.1 打开旧聊天时的触发逻辑

当满足以下条件时，触发“旧聊天接管”提示：

1. 当前聊天消息数超过阈值（例如 > 30 或 > 50）
2. 当前聊天未发现已完成的 MemoryOS 接管记录
3. 当前聊天不是刚创建的新会话

触发后不直接自动处理，而是打开“旧聊天接管配置面板”。

---

## 3.2 接管配置面板

建议配置项如下。

### 基础信息区

- 当前聊天总楼层数
- 当前是否已有历史接管记录
- 当前角色卡 / 世界书是否可用
- 上次中断位置（如果存在）

### 楼层范围设置

用户可选择：

1. **全部楼层**
2. **最近 N 层**
3. **自定义楼层区间**
   - 起始楼层
   - 结束楼层

### 批处理设置

- **每批处理多少层**（必填或有默认值）
- 是否自动继续下一批
- 是否处理完成后自动执行最终整合
- 是否优先先处理最近一段，再回补更早历史

### 运行策略

- 失败时自动暂停
- 允许手动重试失败批次
- 允许处理中继续聊天

---

## 3.3 推荐默认值

虽然不限制 token，但仍建议给出默认值：

- 默认处理范围：全部楼层
- 默认每批处理：`30` 层
- 默认自动继续：开启
- 默认优先处理最近区间：开启
- 默认失败自动暂停：开启
- 默认处理完自动整合：开启

说明：这里不是限制用户，而是提供一个不会太极端的默认起点。

---

## 3.4 用户体验原则

### 不要让用户配置 token

用户只需要理解“楼层”和“批次”，不需要理解 token。

### 不要把“分批”做成隐藏逻辑

要明确告诉用户：

- 你可以处理全部历史
- 也可以自定义范围
- 你可以设置每批处理多少层
- 批次越大，单批耗时和单次消耗通常越高

### 不要在 UI 中暴露过多底层参数

不建议暴露：

- 模型内部摘要层级数
- 压缩算法级别
- prompt token 估算参数

这些应该是系统内部实现细节。

---

## 四、总体处理流程

## 4.1 总体执行阶段

完整流程建议拆成 6 个阶段：

1. 会话识别
2. 配置确认
3. 静态基线构建
4. 当前活跃状态接管
5. 历史分批补建
6. 最终整合与落库

---

## 4.2 阶段 1：会话识别

输入：当前聊天元信息、历史消息数、接管状态记录。

输出：

- 是否进入旧聊天接管模式
- 是否存在中断任务
- 是否允许恢复

---

## 4.3 阶段 2：配置确认

根据用户设置生成：

- 目标楼层范围
- 分批切片计划
- 最近优先区间
- 自动执行策略

输出一个 `takeoverPlan`。

---

## 4.4 阶段 3：静态基线构建

这一步不依赖完整聊天历史，而是优先读取：

- 角色卡
- 世界书
- 硬规则
- 系统设定
- 持久设定类内容

输出：

- `staticBaseline`
- `personaBaseline`
- `worldBaseline`
- `ruleBaseline`

这部分是后续历史理解的参照系。

---

## 4.5 阶段 4：当前活跃状态接管

即使用户选择处理全部历史，也建议先处理最近一段，用于快速建立当前状态。

建议最近活跃区间为：

- 最近 `N` 层（例如默认 40 或 60）
- 如果总楼层本来就少，则直接全部

这一段优先提炼：

- 当前关系状态
- 当前任务状态
- 当前世界状态
- 当前场景位置
- 当前未决问题
- 最近阶段摘要

输出：

- `activeSnapshot`
- `recentDigest`
- `openThreads`

完成这一步后，即使后面的历史补建还没跑完，系统也已经具备“可继续聊”的能力。

---

## 4.6 阶段 5：历史分批补建

这是本方案的重点。

### 分批切分方式

按用户设置的“每批处理多少层”切片，例如：

- 总楼层：320
- 处理范围：1 ~ 320
- 每批：30

则得到：

- 批次 1：1 ~ 30
- 批次 2：31 ~ 60
- 批次 3：61 ~ 90
- ...
- 批次 11：301 ~ 320

### 推荐执行顺序

若开启“优先处理最近区间”：

1. 先处理最近活跃区间
2. 再从最早批次向后批量补建

这样更容易形成稳定的时间线。

### 每批处理内容

每一批不直接生成最终记忆，而是先产出：

- 本批摘要
- 稳定事实候选
- 关系变化候选
- 任务推进候选
- 未决线索候选
- 本批与上一阶段的衔接说明

输出结构见后文数据结构章节。

---

## 4.7 阶段 6：最终整合与落库

当所有目标批次处理完成后，执行一次 consolidation：

- 合并全部 `batchDigest`
- 去重 `factCandidates`
- 合并时间线上重复事件
- 决定哪些写入长期记忆
- 决定哪些作为章节索引保留
- 刷新最终 `activeSnapshot`

最终产物：

- `staticMemory`
- `chapterDigestIndex`
- `activeSnapshot`
- `longTermFacts`
- `relationState`
- `taskState`
- `worldState`

---

## 五、楼层分批策略设计

## 5.1 为什么改为“按楼层分批”而不是“按 token 分批”

你的要求非常明确：不限制 token，而是让用户控制楼层。

这个设计在产品层是合理的，因为用户最直观能理解的是：

- 我要处理哪些楼层
- 每批处理多少楼

而不是：

- 每批多少 token
- 每批上下文预算多少

因此本方案中：

- **产品配置层**：按楼层分批
- **内部执行层**：允许任意 token 使用，不做产品拦截

---

## 5.2 分批模式

### 模式 A：处理全部楼层

- 用户选择全部范围
- 系统按楼层数切片
- 批次自动执行完毕

### 模式 B：处理最近 N 层

- 用户只关心当前可继续聊
- 可先用较小范围建立状态

### 模式 C：自定义区间

适合：

- 想单独补建某段剧情
- 想修复某段历史记忆
- 想避免处理整段旧历史

---

## 5.3 批次大小建议

虽然不限制 token，但建议在 UI 中提示：

- 批次越大，单次处理更慢
- 批次越大，单次成本通常越高
- 批次越大，失败后重试粒度越粗
- 批次越小，整体更稳，但批次数更多

可提示推荐值：

- 轻量聊天：20 ~ 30 层 / 批
- 中等聊天：30 ~ 50 层 / 批
- 超长聊天：50 层以上由用户自行选择

---

## 5.4 是否允许“一批处理全部”

允许，但不推荐作为默认值。

实现上可以约定：

- 若用户将“每批处理楼层数”设置为大于等于所选范围总层数，则等价于单批全处理

UI 中应给出提示：

> 你可以一次性处理全部选中楼层，但单次耗时与消耗可能显著增加，且失败后需要重跑整批。

## 七、执行引擎设计

## 7.1 核心模块划分

建议拆成以下模块：

1. `takeoverDetector`：识别旧聊天、判断是否需要接管
2. `takeoverPlanner`：根据用户配置生成切片计划
3. `baselineBuilder`：提取静态基线
4. `recentBootstrapRunner`：处理最近活跃区间
5. `historyBatchRunner`：执行单批历史分析
6. `takeoverScheduler`：批次调度、暂停、恢复、重试
7. `takeoverStore`：持久化状态与中间产物
8. `takeoverConsolidator`：最终整合
9. `takeoverDiagnostics`：日志、错误、统计

---

## 7.2 调度流程

伪代码如下：

```ts
async function runTakeover(plan) {
  await buildStaticBaseline(plan)
  await bootstrapRecentActiveState(plan)

  const batches = buildBatches(plan)

  for (const batch of batches) {
    if (isPaused(plan)) break

    try {
      await runHistoryBatch(batch)
      await saveBatchResult(batch)
      await markBatchCompleted(batch)
    } catch (error) {
      await markBatchFailed(batch, error)
      if (plan.pauseOnError) {
        await pauseTakeover(plan)
        break
      }
    }
  }

  if (allBatchesCompleted(plan) && plan.autoConsolidate) {
    await runConsolidation(plan)
  }
}
```

---

## 7.3 恢复执行逻辑

系统重启、页面刷新或插件重新加载后，应能根据 `takeoverStore` 恢复：

- 当前任务状态
- 已完成批次
- 当前失败批次
- 最后检查点

恢复时可给用户三个按钮：

- 继续执行
- 从失败批次重试
- 放弃本次接管并重新配置

## 八、Prompt / 提取契约设计

## 8.1 静态基线提取契约

目标：从角色卡、世界书、硬设定中提取稳定不易变化的信息。

输出重点：

- 角色身份
- 世界规则
- 固定设定
- 长期约束
- 不应被剧情轻易覆盖的内容

不应输出：

- 当前场景
- 临时任务
- 阶段性情绪

---

## 8.2 最近活跃状态提取契约

目标：优先建立“现在是什么状态”。

输出重点：

- 当前位置 / 场景
- 当前关系状态
- 当前任务推进状态
- 最近未决问题
- 最近情绪基调

---

## 8.3 历史批次提取契约

目标：把一段楼层压缩成阶段性可整合信息，而不是直接入库。

每批重点提炼：

- 本批发生的关键事件
- 哪些事实是稳定的
- 哪些关系发生了变化
- 哪些任务推进了
- 哪些问题遗留到后面
- 这段内容属于哪种章节标签

禁止倾向：

- 不要把所有细节都写进 `stableFacts`
- 不要把单次台词当成长期设定
- 不要把临时情绪当作永久关系
- 不要把未证实推测当作事实写死

---

## 8.4 最终整合契约

目标：从多个批次结果中抽取真正长期有效的信息。

整合阶段重点做：

- 去重
- 冲突处理
- 时间线修正
- 章节压缩
- 生成最终活跃状态

---

## 九、去重与冲突处理

## 9.1 为什么必须做统一整合

因为不同批次可能会出现：

- 同一个事实重复出现
- 旧状态被新状态覆盖
- 关系先升温后降温
- 任务从未开始到进行中再到完成

如果不统一整合，记忆层会被大量重复与矛盾污染。

---

## 9.2 去重规则

### 事实去重

对于同主体、同谓词、同值的条目进行合并。

### 状态覆盖

对于具有明显时间推进属性的状态：

- 任务状态以最新为准
- 关系状态以最终收敛结果为准
- 世界状态以当前有效状态为准

### 候选保留

若冲突信息无法明确判定，则不直接入长期记忆，而保留在章节摘要中。

---

## 9.3 冲突决议优先级

建议优先级：

1. 明确的近期状态 > 早期状态
2. 多批次重复确认的稳定事实 > 单批次单句暗示
3. 静态设定基线 > 临时剧情误差
4. 最终整合判断 > 单批次原始候选

---

## 十、冷启动优化策略

虽然这里不做 token 限制，但冷启动优化仍然非常重要，核心不是“省 token”本身，而是：

- 更快进入可用状态
- 减少首次等待
- 避免长历史导致首次体验过慢

---

## 10.1 先建当前状态，再回补历史

这是最重要的优化。

旧聊天打开时，优先做：

1. 静态基线
2. 最近活跃区间

完成后，用户即可继续聊天。更老的历史批次在后续继续执行。

---

## 10.2 处理中允许继续聊天

建议接管任务与当前对话解耦：

- 接管任务在后台调度
- 当前聊天仍可发送新消息
- 新消息进入正常 MemoryOS 处理链

为了避免冲突，可在接管期间采用：

- 新消息只写入实时层
- 等整合完成后再统一校准

---

## 10.3 增量整合而不是每批全量重建

每批完成后只写入：

- 批次结果
- 简要可视预览
- 批次状态

不要每完成一批就全量重建全部记忆。  
真正的全量整合放到：

- 全批次完成后
- 或用户手动点击“立即整合”时执行

---

## 12.1 必须持久化的内容

1. 接管计划
2. 批次列表
3. 每批状态
4. 每批分析结果
5. 静态基线结果

7. 最终整合结果
8. 错误日志
9. 最后检查点

## 12.2 中断场景

必须支持以下场景恢复：

- 浏览器刷新
- 插件重载
- 页面切换
- 手动暂停后下次再继续
- 某批失败后下一次继续

---

## 12.3 重入安全

若用户重复打开同一旧聊天，应优先检查：

- 是否已有进行中的接管任务
- 是否已有已完成接管结果
- 是否已有失败且可恢复的任务

不要重复创建多个并行接管任务污染状态。

## 十三、与现有 MemoryOS 主流程的衔接

## 13.1 接管完成前

当前聊天仍可继续使用，但主注入建议采用：

- `staticBaseline`
- `activeSnapshot`
- 已完成的最近阶段摘要

不要把所有中间批次原文直接灌到主 prompt。

---

## 13.2 接管完成后

切换到正常工作流：

- 静态记忆层
- 当前活跃状态层
- 长期事实层
- 章节索引层

以后新消息继续走普通增量更新，不再重复历史补建。

---

## 13.3 重新补建某一段历史

建议保留二次操作入口：

- 选择某段楼层重新补建
- 仅替换该段 `chapterDigest`
- 再次执行整合

这对于修复旧数据、验证 prompt 升级效果很有价值。

## 十八、风险点与规避方案

## 18.1 单批过大导致耗时过长

虽然不限制 token，但这仍然会发生。

规避方式：

- UI 清晰提示“批次越大，单次越慢”
- 支持暂停 / 重试
- 支持把已失败批次单独重跑

---

## 18.2 历史细节过多导致摘要污染

规避方式：

- 每批只产出阶段摘要和候选，不直接大量入长期记忆
- 最终由 consolidation 再次净化

---

## 18.3 旧状态覆盖当前状态

规避方式：

- 关系、任务、世界状态都采用“时间推进优先”规则
- 当前活跃状态必须由最近区间优先建立

---

## 18.4 重复接管任务并发污染

规避方式：

- 同一聊天同一时间只允许一个 takeover task 处于 running 状态
- 新任务启动前先检查旧任务状态

---

## 十九、最终推荐的产品表现

从产品完成态来看，我建议最终呈现为：

### 进入旧聊天时

系统提示：检测到这是一个已有大量历史的聊天，是否开始接管历史？

### 用户配置时

用户只需要决定三件事：

1. 处理哪些楼层
2. 每批处理多少层
3. 是否自动连续处理

### 接管进行时

用户能看到：

- 已处理到哪一段
- 当前正在处理哪一批
- 哪一批失败了
- 当前摘要已经提炼出什么

### 接管完成后

MemoryOS 获得：

- 静态设定基线
- 当前活跃状态
- 历史章节索引
- 长期稳定事实
- 关系 / 任务 / 世界状态

这时整个旧聊天才算真正被系统“接住”。

## 二十、结论

如果按你的要求来定方向，那么这套能力的最佳落地方式不是“限制 token”，而是：

### 最终结论

1. **不限制 token**，不把 token 作为产品层配置项。
2. **把控制权交给用户的楼层范围与批次大小**。
3. **旧聊天必须单独走接管初始化，不再混入普通冷启动。**
4. **分批处理中间结果优先做章节摘要，不要直接把每批都写成大量长期碎片记忆。**
5. **先建立当前状态，再回补全历史。**
6. **必须一次性把暂停、恢复、重试、最终整合一起做完，否则这个功能很容易停留在 demo 状态。**

---

## 二十一、建议你接下来直接开工的顺序

如果你要按“完整功能一次性实现”的思路推进，我建议就按下面顺序动手：

1. 先把 `takeover` 数据结构、store、repository 定义完整。
2. 再做“旧聊天接管配置面板”，把楼层范围和每批楼层数配起来。
3. 然后落地“静态基线 + 最近活跃区间接管”。
4. 接着实现“按楼层切片的批次执行器”。
5. 再补齐“暂停 / 恢复 / 重试 / 中断恢复”。
6. 最后做“统一整合 + 接入主记忆流 + 可视化诊断”。

这样做，整套系统会一次成型，而且后续很好扩展。
