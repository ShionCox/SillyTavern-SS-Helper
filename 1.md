# MemoryOS 世界画像系统融入整体链路改造方案

## 1. 文档目标

本文档用于指导 **SillyTavern-SS-Helper / MemoryOS** 对当前“世界画像 / 世界模板（World Profile）”系统进行深化改造，使其不再只是：

- 识别一个题材标签
- 选择一套注入文案风格

而是真正融入到以下主链：

- Prompt 注入
- 检索召回
- 自动总结
- Takeover 旧聊天处理
- Cold Start 初始化
- 梦境系统
- 结构化抽取
- 诊断与工作台

本方案的核心目标是：

> **让世界画像从“风格标签”升级为“跨链路的策略配置中心”。**

---

## 2. 当前现状审查结论

## 2.1 当前世界画像系统还在起作用
根据当前代码，世界画像系统并不是摆设。

当前至少明确有三类作用：

### A. 参与世界类型识别
系统会通过 `detectWorldProfile()`，根据输入文本和内置关键词识别当前世界类型。  
例如 `urban_modern` 会用：

- 都市
- 公司
- 学校
- 地铁
- 互联网
- 办公
- 写字楼

等关键词进行匹配。fileciteturn129file0L1-L1 fileciteturn128file0L1-L1

### B. 参与 Prompt 注入风格选择
在 `PromptAssemblyService` 中，系统会先：

- `detectWorldProfile()`
- `resolveWorldProfile()`

然后把 `worldProfile.primary.injectionStyle` 传给 XML/Markdown 注入渲染器。fileciteturn118file0L1-L1

### C. 决定叙事风格映射
不同 `styleId` 会映射到不同 `NarrativeStyleProfile`，影响：
- 标题名
- 区块显示文案
- 注入风格选择

例如：

- `urban_modern`
- `ancient_traditional`
- `fantasy_magic`
- `supernatural_hidden`

都参与最终叙事风格映射。fileciteturn116file0L1-L1

---

## 2.2 当前世界画像系统还没有真正深入主链
虽然世界画像定义里已经包含很多高级字段，例如：

- `capabilities`
- `preferredSchemas`
- `preferredFacets`
- `schemaFieldExtensions`
- `summaryBias`
- `injectionStyle`
- `detectionKeywords` fileciteturn125file0L1-L1

而 `resolveWorldProfile()` 也已经会把这些能力合并出来：

- `mergedPreferredSchemas`
- `mergedPreferredFacets`
- `mergedFieldExtensions`
- `mergedSummaryBias` fileciteturn126file0L1-L1

但从当前主链来看，这些能力**大多没有被深度消费**。

目前真正被稳定用到的，主要还是：

- `detectionKeywords`
- `injectionStyle`

换句话说：

> **现在的世界画像系统更像“模板元数据 + 风格选择器”，还不是 MemoryOS 的策略中枢。**

---

## 3. 当前问题

## 3.1 问题一：世界画像只影响“看起来怎么说”，很少影响“实际怎么处理”
当前世界画像更偏：
- UI/Prompt 风格层
- 识别层

但还没有真正影响：

- 哪些条目应该更优先保留
- 哪些 schema 更应该被抽取
- 哪些事件在当前题材下更重要
- 梦境系统如何进行维护偏置
- Summary 应优先强化哪些信息

---

## 3.2 问题二：世界画像字段定义很多，但没有形成闭环
例如：

### `capabilities`
当前定义了：
- hasMagic
- hasCultivation
- hasFantasyRace
- hasModernTechnology
- hasFormalPoliticalOrder
- hasSupernatural fileciteturn125file0L1-L1

但目前没有看到这些能力直接控制：

- 允许哪些 schema 出现
- 优先哪些抽取字段
- 某些题材下的召回偏置
- 提示词风格差异化

---

### `preferredSchemas / preferredFacets`
现在只是合并出来，但没有真正进入：
- 检索排序
- 总结优先级
- Takeover 批处理偏置
- Dream 维护策略

---

### `summaryBias`
当前只是定义了 `boostedTypes / suppressedTypes`，但没有系统性作用到 Summary 和 Dream。

---

### `schemaFieldExtensions`
这个本来特别适合做：
- 不同题材抽取不同结构字段

但当前也没有真正驱动抽取链。

---

## 3.3 问题三：世界画像识别只在注入阶段用得最明显，缺少“聊天级绑定后的持续生效”
当前 Prompt 组装里会读取：

- `worldBinding?.primaryProfile`
- 若无则再 `detectWorldProfile(...)` fileciteturn118file0L1-L1

这说明它还没有完全成为：

> **一个聊天级长期稳定的策略基线**

而更像“每次组装时顺手拿一下”的运行时配置。

---

## 3.4 问题四：世界画像没有成为工作台中可见、可调、可解释的中心配置
现在虽然 diagnostics 会带：
- `world_profile:${worldProfile.primary.worldProfileId}` fileciteturn118file0L1-L1

但对用户和开发者来说，通常还看不到：

- 当前世界画像为什么被识别成这个类型
- 它对哪些链路产生了实际影响
- 如果切换画像，会导致哪些策略变化

这就导致世界画像的存在感和实际价值都不够强。

---

## 4. 改造目标

本次改造建议把世界画像系统升级为：

# **聊天级策略中枢（World Strategy Core）**

它要同时承担以下职责：

1. **题材识别与绑定**
2. **注入风格选择**
3. **召回排序偏置**
4. **总结抽取偏置**
5. **Takeover/Cold Start 抽取偏置**
6. **Dream 维护偏置**
7. **结构化字段扩展**
8. **诊断与工作台解释**

---

## 5. 总体改造思路

## 5.1 从“World Profile”升级为“World Strategy”
当前 `WorldProfileDefinition` 已经有不少策略字段，但使用方式还不够。

建议统一理解成：

- `worldProfile`：识别出来的世界类型
- `worldStrategy`：这个世界类型在系统里的完整行为策略

也就是说：

> `worldProfileId` 只是入口，真正应该驱动全链路的是解析后的 `ResolvedWorldProfile`。

---

## 5.2 建立“识别 -> 绑定 -> 应用 -> 诊断”的完整闭环

建议把世界画像主链固定成：

```text
文本线索
-> detectWorldProfile()
-> resolveWorldProfile()
-> 写入聊天级 world binding
-> 所有核心链路统一消费 resolved world strategy
-> diagnostics / workbench 展示当前策略影响
```

这样世界画像才不只是“注入时临时判断一下”。

---

## 6. 分链路改造建议

# 6.1 Prompt 注入链

## 当前状态
当前已经会根据 `worldProfile.primary.injectionStyle` 选择叙事风格。fileciteturn118file0L1-L1

## 建议增强
不仅影响标题风格，还应影响：

### A. 注入层级优先级
例如：

- `urban_modern`
  - 更强调 organization / task / city_life / public_rules
- `fantasy_magic`
  - 更强调 world_rule / power_rule / race / artifact / territory
- `ancient_traditional`
  - 更强调 clan / hierarchy / ritual / political_order

### B. 区块展示偏好
不同世界类型，注入块优先级应不同。

例如：

#### 现代都市
优先：
- 当前活跃场景
- 关系推进事件
- 组织 / 社会规则

#### 奇幻魔法
优先：
- 世界规则
- 阵营 / 种族 / 法则
- 魔法体系 / 危险区域

#### 古代传统
优先：
- 身份关系
- 礼法等级
- 势力 / 门第 / 朝堂秩序

---

# 6.2 检索召回链

## 当前状态
当前检索里没有明显看到世界画像对召回排序进行深度影响。

## 建议增强
### A. 用 `preferredSchemas` 调整召回得分
例如：

- `urban_modern` 提升：
  - event
  - location
  - organization
  - task
- `fantasy_magic` 提升：
  - world_rule
  - race_rule
  - territory
  - artifact_rule
- `ancient_traditional` 提升：
  - family
  - clan
  - political_order
  - etiquette

### B. 用 `preferredFacets` 调整检索路由
例如：
- 现代都市更偏 social_order / career / daily_relationship
- 古代传统更偏 hierarchy / status / clan_politics
- 奇幻更偏 power_system / race / territory / taboo

### C. 用 `capabilities` 做检索过滤补偿
例如：
- `hasMagic = false`
  - 遇到魔法类 schema 时可降低召回权重
- `hasModernTechnology = false`
  - 遇到现代工业/互联网类 schema 可降低权重

---

# 6.3 Summary 自动总结链

## 当前问题
Summary 现在没有明显根据世界画像来调整抽取重点和摘要偏向。

## 建议增强
### A. 把 `summaryBias` 真正接入 Summary
例如：

#### 现代都市
- `boostedTypes`: event / task / organization
- `suppressedTypes`: artifact_rule / race_rule

#### 奇幻魔法
- `boostedTypes`: world_rule / territory / race_rule / artifact_rule
- `suppressedTypes`: office_task / company_process

#### 古代传统
- `boostedTypes`: hierarchy / clan / etiquette / political_order
- `suppressedTypes`: internet_daily / modern_workflow

### B. 控制总结文案偏重点
例如：

- 现代都市：强调现实关系推进、公共秩序、组织角色
- 古代传统：强调身份等级、关系称谓、礼制与立场
- 奇幻魔法：强调规则、禁忌、力量体系、阵营边界

### C. 控制摘要粒度
例如：
- 现代都市：允许更生活流、更细碎的关系进展
- 奇幻魔法：更倾向于抽取规则变化与设定性事件
- 古代传统：更倾向于抽取身份关系与政治站位变化

---

# 6.4 Takeover / Cold Start 链

## 当前问题
旧聊天接管和冷启动里，世界画像还没有成为正式抽取策略。

## 建议增强
### A. Takeover 先做世界画像识别，再进入批处理
旧聊天处理开始前，应尽早识别 world profile，并写入 chat binding。

### B. Takeover 抽取时根据 `preferredSchemas` 和 `fieldExtensions` 调整抽取目标
例如：

#### 奇幻魔法
额外鼓励抽取：
- 势力
- 种族
- 地域
- 魔法规则
- 禁忌 / 契约 / 圣物

#### 古代传统
额外鼓励抽取：
- 门第
- 身份位阶
- 称谓体系
- 家族关系
- 朝堂 / 宗门 / 礼制

#### 现代都市
额外鼓励抽取：
- 组织
- 职位
- 学校 / 公司
- 法律 / 社会规则
- 日常任务与公共空间

### C. Cold Start 初始化时生成“世界画像摘要”
例如在 workbench / diagnostics 中保留：

- 当前画像类型
- 识别依据
- 当前画像带来的主要策略偏置

---

# 6.5 Dream 梦境链

## 当前问题
梦境系统目前没有明显根据世界画像去调节维护策略。

## 建议增强
### A. Dream 的 recall / maintenance proposal 接入 world profile bias
例如：

#### 现代都市
梦境偏向：
- 关系推进
- 日常事件压缩
- 组织角色维护
- 任务/现实状态维护

#### 奇幻魔法
梦境偏向：
- 世界规则强化
- 阵营/种族关系维护
- 地域与势力边界整理
- 能力 / 契约 /禁忌的补全

#### 古代传统
梦境偏向：
- 身份关系再排序
- 礼法与立场巩固
- 家族/门派脉络整理
- 称谓系统和尊卑关系强化

### B. Dream Prompt 根据世界画像选择风格提示
即使不完全换 prompt，也应至少改变 dream reasoning 的重点。

---

# 6.6 结构化抽取链

## 这是世界画像系统最值得深入的地方

当前 `schemaFieldExtensions` 很适合接入抽取链。fileciteturn125file0L1-L1

## 建议做法
### A. 为不同 world profile 增加 schema 扩展字段
例如：

#### urban_modern
- location: district / businessType / publicAccess
- organization: industry / legalStatus

#### ancient_traditional
- organization: rank / allegiance / ritualAuthority
- character_relation: seniority / formalTitle / clanBranch

#### fantasy_magic
- location: regionType / magicalHazard / manaDensity
- organization: factionType / magicalAlignment / tabooScope
- artifact: rarity / contractType / corruptionRisk

### B. 抽取器读取 `mergedFieldExtensions`
在 summary / takeover / cold start / manual extraction 中，统一根据 world profile 扩展 schema 字段集合。

这样画像就不只是“看起来像都市”，而是：

> **都市世界就会真的多抽“商圈 / 行业 / 公共规则”这类字段。**

---

## 7. 建议新增统一服务层

建议新增：

### `world-strategy-service.ts`
职责：

- 读取 world binding
- 在无 binding 时执行 detect + resolve
- 输出统一的 `ResolvedWorldProfile`
- 为各链路提供：
  - retrieval bias
  - summary bias
  - extraction field extensions
  - dream bias
  - injection style
  - diagnostics explanation

这样避免每条链各自零散地读 `worldProfile.primary.xxx`

---

## 8. Workbench / Diagnostics 改造建议

## 8.1 新增“当前世界画像”面板
显示：

- 当前 worldProfileId
- displayName
- confidence
- detectionKeywords 命中原因
- 当前注入风格
- 当前 summary bias
- 当前 preferred schemas
- 当前 field extensions

---

## 8.2 新增“影响说明”
给开发和用户直接解释：

```text
当前画像：现代都市
作用：
- 注入风格：urban_modern
- 召回偏置：event/location/organization/task ↑
- 总结偏置：artifact_rule/race_rule ↓
- 扩展字段：location.district / organization.industry
```

这样世界画像就不再是黑盒。

---

## 9. 状态绑定建议

## 9.1 聊天级 world binding 应成为长期状态
当前世界画像不应只在 prompt 阶段临时识别。

建议规则：

- 新聊天：cold start 时识别并写入 binding
- 旧聊天：takeover 前识别并写入 binding
- 后续 prompt / dream / summary 统一使用 binding
- 只有在用户手动切换或检测置信度明显失真时才重新识别

---

## 9.2 支持手动覆盖
建议允许用户在 workbench 中手动切换：

- 现代都市
- 古代传统
- 奇幻魔法
- 灵异隐秘
- 自定义画像

避免识别错误长期污染整条链路。

---

## 10. 分阶段实施建议

## 阶段一：先让世界画像成为稳定聊天级配置
目标：
- detect -> resolve -> bind
- 所有 prompt 注入统一使用 binding
- diagnostics 中清晰展示当前画像

---

## 阶段二：接入注入和检索偏置
目标：
- retrieval 根据 preferredSchemas / preferredFacets / capabilities 调整权重
- 注入分区优先级根据 world profile 调整

---

## 阶段三：接入 Summary / Takeover / Cold Start 抽取偏置
目标：
- `summaryBias` 真正生效
- `schemaFieldExtensions` 真正进入抽取链
- Takeover/Cold Start 根据画像抽不同重点

---

## 阶段四：接入 Dream 链
目标：
- 梦境召回偏好
- 梦境维护 proposal 偏置
- 梦境 prompt 风格重点

---

## 阶段五：支持手动覆盖与工作台可视化
目标：
- workbench 中可切换世界画像
- 诊断中清楚显示当前策略影响
- 方便调试与运营

## 13. 验收标准

改造完成后，世界画像系统至少要满足以下标准：

### 标准 1
不再只是影响标题风格，而要至少稳定影响：
- retrieval
- summary
- prompt injection

### 标准 2
Takeover 和 Cold Start 能根据 world profile 调整抽取重点。

### 标准 3
Dream 能根据 world profile 调整 recall / maintenance 偏置。

### 标准 4
Workbench 中能看到：
- 当前 world profile
- 识别原因
- 当前偏置效果

### 标准 5
手动切换 world profile 后，系统行为有清晰、可验证的变化。

---

## 14. 最终结论

当前世界画像系统不是没用，而是：

> **它已经具备“策略中枢”的字段设计，但目前只实际发挥了“识别 + 注入风格选择”的部分作用。**

最合理的改造方向不是重做，而是：

1. 保留现有 `world-profile` 结构  
2. 把 `ResolvedWorldProfile` 变成聊天级长期策略配置  
3. 让 retrieval / summary / takeover / cold start / dream 都统一消费它  
4. 让 workbench 能解释它到底在影响什么

---

## 15. 一句话版本

> 把世界画像系统从“识别题材并换一套注入文案”的轻量模板，升级成“驱动召回、总结、抽取、梦境与工作台诊断的聊天级策略中枢”。
