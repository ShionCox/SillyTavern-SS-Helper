# MemoryOS 时间系统精细化改进方案

## 1. 文档目标

本文档用于指导 **SillyTavern-SS-Helper / MemoryOS** 对当前时间系统进行精细化改造，重点解决以下三个核心问题：

1. **如果设定或楼层内容中明确提到了时间，系统应优先记录并结构化保存。**
2. **如果内容中没有明确时间、甚至没有明确事件，系统应自动生成可用的时间定位信息。**
3. **最关键的是让 AI 知道“这条记忆具体是在什么时候”，避免出现两个“半夜/深夜”却无法区分是“昨天半夜”还是“今天半夜”的问题。**

本方案的目标不是追求每条记忆都得到“几点几分”的钟表时间，而是要建立一套：

> **显式时间优先 + 推断时间补足 + 顺序锚点可区分**

的时间定位系统。

---

## 2. 当前问题概述

## 2.1 当前系统已经有时间模型，但展示和落地还不够

从当前代码看，MemoryOS 的时间模型并不算简陋。`MemoryTimeContext` 已经支持：

- `storyTime.absoluteText`
- `storyTime.relativeText`
- `storyTime.normalized.year/month/day/hour/minute/partOfDay`
- `sequenceTime.firstFloor / lastFloor / orderIndex`
- `mode = story_explicit / story_inferred / sequence_fallback` fileciteturn100file0L1-L1

说明底层其实已经具备：

- 明确故事时间
- 推断故事时间
- 系统时序兜底

这三层能力。问题不在“完全没有时间模型”，而在于**现在这套模型没有真正变成 AI 可用的“时间定位系统”。**

---

## 2.2 当前 prompt 里时间信息过于粗糙

目前 prompt 注入时，`PromptAssemblyService` 只会在每条文本前加一个简单时间头：

- `时间：${timeLabelForPrompt}`
- `来源：${timeSourceLabel}`
- 可选 `置信度：...` fileciteturn98file0L1-L1

而 `timeLabelForPrompt` 又来自 `buildTimeLabel()`。  
当前 `buildTimeLabel()` 的逻辑是：

1. `story_explicit`：优先返回 `absoluteText`，没有就返回 `relativeText`
2. `story_inferred`：返回 `推断：xxx`
3. `sequence_fallback`：返回“近期 / 较当前早约 N 层 / 早期内容（约 N 层前）” fileciteturn99file0L1-L1

因此最终给 AI 的经常只是：

- 深夜
- 半夜
- 上午
- 正午
- 近期

这会直接导致：

> **两个“半夜”根本无法区分到底是不是同一天。**

---

## 2.3 当前系统最缺的不是“时间字段”，而是“时间定位能力”

你真正需要的，不是简单把“深夜”换成“23:00”。

你需要的是：

- 这是不是**第 3 天**的深夜？
- 这个“半夜”是在**护城河归家之后**还是在**厨房火锅之后**？
- 这个记忆是在**当前之前 14 层**，还是当前之前 80 层？
- 这个“上午”是和哪个事件处于同一时段？

也就是说，现在系统最大的问题不是“没有时间”，而是：

> **没有把时间变成可区分、可排序、可定位的结构。**

---

## 3. 改造目标

本次改造建议同时满足以下目标：

### 目标 1：显式时间优先保留
只要文本中明确提到了：
- 今天
- 昨晚
- 第三天上午
- 次日清晨
- 晚饭后
- 火锅后
- 护城河回家时

系统都要优先保留，并结构化存储。

---

### 目标 2：没有显式时间时，也必须生成“可区分时间”
即使内容里没有明确时间，也不能只留一个模糊的“半夜”。

系统至少要能补出：
- 第几天
- 属于哪个事件阶段
- 相对当前大约早多少楼层
- 在整个聊天顺序里的位置

---

### 目标 3：让 AI 能区分同名时间段
例如两个“半夜”必须能被区分成：

- 第 2 天半夜（较当前早 64 层）
- 第 3 天半夜（较当前早 8 层）

或者：

- 护城河归家当晚
- 火锅后的厨房深夜

---

## 4. 核心改造思路

本方案建议把时间系统升级为：

# **三层时间定位体系**

## 第 1 层：显式时间层
优先记录文本中明确出现的时间表达。

## 第 2 层：推断时间层
当没有明确时间时，系统根据上下文、前后楼层、事件阶段自动补出一个推断时间。

## 第 3 层：系统顺序层
无论有没有故事时间，都必须保留系统时序：
- 楼层范围
- 顺序索引
- 相对当前距离

这样即使两个记忆都叫“半夜”，AI 也能靠第 2 层和第 3 层区分。

---

## 5. 解决你的三个核心诉求的具体方案

# 5.1 诉求一：有具体时间就记录

## 5.1.1 明确时间抽取优先级

建议系统优先识别以下几类时间：

### A. 显式历法/日期
例如：
- 2026 年 3 月 9 日
- 农历七月十四
- 周一上午

### B. 相对时间
例如：
- 今天
- 昨晚
- 次日
- 第三天上午
- 晚饭后
- 刚刚
- 过了一会儿
- 睡醒后

### C. 时段表达
例如：
- 清晨
- 早晨
- 上午
- 正午
- 午后
- 傍晚
- 夜晚
- 深夜
- 半夜

### D. 事件型时间锚
例如：
- 火锅后
- 回家后
- 教学楼分别前
- 洗完澡后
- 团团撞进中间时

---

## 5.1.2 显式时间结构化建议

现有 `StoryTime` 建议扩展或标准化写入：

```ts
type StoryTime = {
  calendarKind?: CalendarKind;
  absoluteText?: string;
  relativeText?: string;
  normalized?: {
    year?: number;
    month?: number;
    day?: number;
    hour?: number;
    minute?: number;
    partOfDay?: PartOfDay;
  };
  storyDayIndex?: number;
  anchorEventLabel?: string;
  anchorRelation?: 'during' | 'before' | 'after' | 'same_phase';
}
```

其中：

- `storyDayIndex` 用于标明“第几天”
- `anchorEventLabel` 用于标明“挂靠哪个事件”
- `anchorRelation` 用于标明“在这个事件之前/之后/同阶段”

---

## 5.1.3 显式时间的落地原则

### 原则 1：能直接记的绝不丢
如果文本已经说了“第三天上午”，那就不要只保留“上午”。

### 原则 2：能拆成结构就不要只留原文
例如：

```text
第三天上午
```

不应只写：

```json
{ "absoluteText": "第三天上午" }
```

更建议写成：

```json
{
  "absoluteText": "第三天上午",
  "storyDayIndex": 3,
  "normalized": {
    "partOfDay": "morning"
  }
}
```

---

# 5.2 诉求二：没有具体时间时，系统要自动生成

这一步是改造成败的关键。

## 5.2.1 不要再只生成“半夜/深夜”这种裸标签
现在最差的情况就是：

```text
时间：半夜
```

这没有定位能力。

无显式时间时，系统应该自动生成 **复合时间标签**。

---

## 5.2.2 建议生成 4 类补全信息

### A. 故事日序补全
如果上下文能判断这是第几天，就补：

- 第 1 天
- 第 2 天
- 第 3 天

### B. 事件锚点补全
如果能挂靠某个已知事件，就补：

- 护城河归家后
- 火锅后的厨房
- 教学楼分别前
- 团团撞进中间后

### C. 相对阶段补全
如果无法归到明确日期，也应补：

- 当晚稍后
- 次日清晨
- 午后拖成傍晚
- 回家后不久
- 同一段相处的后半程

### D. 系统顺序补全
兜底必须保留：

- firstFloor / lastFloor
- orderIndex
- 相对当前早多少层

---

## 5.2.3 推断规则建议

### 规则 1：优先复用最近明确锚点
如果最近一段已有：

- 第三天上午
- 第三天中午
- 第三天夜晚

那么后续“深夜”优先推断为：

- 第三天深夜

而不是孤零零一个“深夜”。

---

### 规则 2：同场景连续发生时优先视为同一天
例如在同一出租屋、同一段相处、没有明显睡眠/切场时，默认：

- 仍属于同一 storyDayIndex
- 时段允许从 evening -> night -> midnight 微调

---

### 规则 3：出现“睡醒 / 清晨 / 次日 / 第二天”等信号时推进日序
一旦出现明确跨日语义：

- 睡醒后
- 次日
- 第二天
- 清晨
- 天亮了

就推进 `storyDayIndex += 1`

---

### 规则 4：没有任何故事时间信号时，仍必须保留系统顺序定位
比如最终至少生成：

- 推断：较当前早约 18 层的夜间阶段
- 顺序：182-186 层
- 来源：系统时序

这样即使不够优雅，AI 也能区分。

---

# 5.3 诉求三：让 AI 知道“到底是哪一个半夜”

这一步必须通过 **复合时间头** 来解决。

## 5.3.1 当前格式不够
现在只有：

```text
[时间：深夜｜来源：明确故事时间]
```

这不够。fileciteturn98file0L1-L1

---

## 5.3.2 建议升级 PromptTimeMeta

当前 `PromptTimeMeta` 建议扩展为：

```ts
type PromptTimeMetaV2 = {
  primaryLabel: string;        // 第三天深夜
  anchorLabel?: string;        // 火锅后的厨房
  anchorRelation?: string;     // 之后 / 之前 / 同阶段
  sequenceLabel?: string;      // 182-186层
  relativeToNowLabel?: string; // 较当前早14层
  sourceLabel: string;         // 明确故事时间 / 推断时间 / 系统时序
  confidenceLabel?: string;
}
```

---

## 5.3.3 AI 最终看到的时间格式建议

### 完整模式
```text
[时间：第三天深夜｜事件：火锅后的厨房｜顺序：182-186层｜相对当前：14层前｜来源：明确故事时间]
```

### 推断模式
```text
[时间：推断为第三天深夜稍后｜事件：团团撞进中间后｜顺序：188-190层｜来源：推断时间｜置信度：中]
```

### 兜底模式
```text
[时间：未识别具体故事时间｜顺序：较当前早18层｜来源：系统时序]
```

这样即使都是“半夜”，AI 也能区分：

- 第三天深夜，火锅后
- 第三天深夜稍后，团团撞进中间后
- 第二天半夜，护城河归家当晚

---

## 6. 建议引入“事件时间锚账本”

这是时间系统真正变强的关键。

## 6.1 为什么要有事件锚账本
因为很多剧情里根本不会说：
- 2026-03-09 23:48

但会不断出现：

- 护城河归家
- 教学楼分别
- 火锅后的厨房
- 团团撞进中间
- 午后拖成傍晚

这些都是天然时间锚点。

---

## 6.2 建议新增数据结构

```ts
type StoryEventAnchor = {
  eventId: string;
  label: string;
  storyDayIndex?: number;
  partOfDay?: PartOfDay;
  firstFloor: number;
  lastFloor: number;
  previousEventId?: string;
  nextEventId?: string;
  confidence: number;
}
```

---

## 6.3 使用方式

### 生成锚点
当系统检测到明确且重要的事件时，建立事件锚点，例如：

- E1：教学楼分别（第 3 天清晨）
- E2：出租屋谈猫和饭菜（第 3 天上午至中午）
- E3：护城河归家（第 3 天夜晚）
- E4：火锅后的厨房（第 3 天深夜）
- E5：团团撞进中间（第 3 天深夜稍后）

### 记忆挂锚
后续每条记忆不再只写“深夜”，而是挂到：

- `anchorEventId = E4`
- `anchorRelation = same_phase`
- 或 `anchorRelation = after`

这样两个“深夜”立刻就能区分开。

---

## 7. 代码改造建议

## 7.1 修改 `time-types.ts`
建议在 `MemoryTimeContext / StoryTime / PromptTimeMeta` 中增加：

```ts
storyDayIndex?: number;
anchorEventId?: string;
anchorEventLabel?: string;
anchorRelation?: 'during' | 'before' | 'after' | 'same_phase';
relativePhaseLabel?: string;
sequenceLabel?: string;
relativeToNowLabel?: string;
```

现有模型本来就支持：
- `normalized`
- `sequenceTime`
- `story_explicit / story_inferred / sequence_fallback`  
所以加这些是顺势扩展，不是推翻重做。fileciteturn100file0L1-L1

---

## 7.2 修改 `time-ranking.ts`
当前 `buildTimeLabel()` 直接返回 string，这种结构太弱。fileciteturn99file0L1-L1

建议改为：

```ts
buildPromptTimeDisplay(timeCtx, currentMaxFloor): PromptTimeDisplay
```

例如：

```ts
type PromptTimeDisplay = {
  primaryLabel: string;
  anchorLabel?: string;
  anchorRelation?: string;
  sequenceLabel?: string;
  relativeToNowLabel?: string;
}
```

然后 `buildPromptTimeMeta()` 再从这里生成最终注入信息。

---

## 7.3 修改 `prompt-assembly-service.ts`
当前 `prependPromptTimeHeader()` 只会拼：

- 时间
- 来源
- 置信度 fileciteturn98file0L1-L1

建议改成拼接：

- 主时间标签
- 事件锚点
- 顺序区间
- 相对当前距离
- 来源
- 置信度

---

## 7.4 Takeover / Summary / Cold Start 写时间时同步写锚点
当前 `TimeContextSource` 已经区分了：

- `cold_start`
- `takeover_batch`
- `summary_batch`
- `fallback_engine`
- `manual` fileciteturn100file0L1-L1

所以建议：

### Takeover
负责识别和建立历史事件锚点

### Summary
负责把新增聊天挂到已有锚点链上，并在必要时生成新锚点

### Cold Start
负责初始化聊天级时间画像和第一个锚点

---

## 8. Prompt 展示优化建议

## 8.1 顶部增加“小时间线”
建议在 `memory_context` 顶部或 `world_base` 下增加：

```text
## 当前事件时间线
- E1：第3天清晨，教学楼分别
- E2：第3天上午至中午，出租屋谈猫和饭菜
- E3：第3天夜晚，护城河归家
- E4：第3天深夜，火锅后的厨房
- E5：第3天深夜稍后，团团撞进中间
```

这样 AI 在读具体记忆之前，先拿到一个小型 timeline。

---

## 8.2 记忆条目统一使用复合时间头
例如：

```text
- [时间：第3天深夜｜事件：火锅后的厨房｜顺序：182-186层｜来源：明确故事时间] 厨房里的恋人饭桌：...
```

而不是：

```text
- [时间：深夜｜来源：明确故事时间] ...
```

---

## 9. 测试方案

## 9.1 单元测试

### A. 显式时间抽取测试
验证：
- “第三天上午” 能写出 `storyDayIndex = 3`
- “次日清晨” 能推进日序
- “护城河归家后” 能识别为锚点关系

---

### B. 推断时间生成测试
验证：
- 没有显式时间时，系统仍能给出：
  - 推断主标签
  - sequenceLabel
  - relativeToNowLabel

---

### C. 同名时段区分测试
验证：
- 两条都叫“深夜”的记忆
- 最终 prompt 中必须能区分：
  - 不同 dayIndex
  - 或不同 anchorEvent
  - 或不同 sequenceLabel

---

## 9.2 集成测试

### 用例 1：Takeover 历史聊天
导入一段含多个“夜晚/深夜/清晨”的历史聊天，验证：

- 锚点被建立
- 同名时间段可区分
- prompt 中不再只剩“深夜”

### 用例 2：Summary 增量聊天
在已有时间锚点链上新增聊天，验证：

- 新记忆优先挂到已有锚点
- 需要时生成新锚点
- 不会让两个“半夜”混为一谈

### 用例 3：Prompt 输出检查
验证最终注入文本中：

- 至少包含主时间标签
- 有条件时包含事件锚点
- 有 sequenceLabel 或 relativeToNowLabel 兜底

---

## 10. 实施优先级

## 第一优先：改 Prompt 展示
先把：
- 第几天
- 事件锚点
- 顺序区间
展示出来。

这是最快让 AI 变得“不再混淆两个半夜”的方法。

---

## 第二优先：补 storyDayIndex 与 anchorEvent
让底层时间上下文真正能表达：
- 第几天
- 哪个事件阶段

---

## 第三优先：建立事件锚点账本
让整个聊天的时间线形成可复用的锚点链。

---

## 第四优先：统一 Takeover / Summary / Cold Start 的时间写入策略
让三条链都不只是写“上午/深夜”，而是写成统一时间定位结构。

---

## 11. 最终结论

你现在最需要的，不是把“深夜”改成“23:00”，而是：

> **让每条记忆都带上“第几天 + 事件锚点 + 顺序位置”的时间定位结构。**

这样才能真正解决：

- 设定或楼层里有明确时间时能保留
- 没有明确时间时系统能自动补足
- 两个“半夜”不再混淆为同一个时间

---

## 12. 一句话版本

> 不要再让时间系统只输出“深夜/半夜/上午”这种裸标签，而要升级成“显式时间优先、无时间时系统推断、最终用第几天 + 事件锚点 + 楼层顺序来定位记忆”的精细化时间系统。
