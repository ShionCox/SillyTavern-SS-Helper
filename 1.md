# MemoryOS 日志与情境判定优化方案（含 PeroCore 兼容规则包）

> 适用分支：`memoryos-settings-unified`
>
> 目标：
>
> 1. 为当前 MemoryOS 增加一套**全中文、可开关、可追踪整条召回链路**的日志系统。
> 2. 把当前情境识别从“能判定”提升到“**能解释为什么这样判定**”。
> 3. 引入一层 **PeroCore 兼容规则包**，把目前能明确确认的 PeroCore 句式切分、系统事件判定、记忆类型归类、思维簇归类思路迁移到 MemoryOS。
> 4. 让工作台里的诊断能力和日志能力打通，做到“日志里能看流程，界面里能看结果”。

---

## 一、方案结论先说

当前分支已经完成了语境路由、多阶段召回、图扩散、二次补召回、可视化记忆图谱等主体结构，但还缺两块真正能把系统拉到“可调试、可维护、可稳定优化”阶段的能力：

第一块是**日志体系**。现在虽然代码里已经有 `contextRoute`、`reasons`、`diagnostics`、`graph expansion`、`coverage second pass` 这些结构，但它们更多还是“代码内部存在”，还没有形成一套给开发者直接看的中文流水日志。当前版本的检索链条越来越长，如果没有完整日志，后面做调权、查误召回、查漏召回会越来越吃力。

第二块是**规则兼容层**。现在 MemoryOS 已经开始支持 alias、自然语言锚点、recent context bias，但“句式切分”和“情境判定词”的管理还不够模块化。PeroCore 的公开实现里，已经能确认它在检索与记忆阶段用了多意图切分、系统事件前缀识别、记忆类型归类、思维簇归类、以及 NMF/FISTA/DPP 这一整套增强检索链路；这些东西不应该散落在单个函数里，而应该抽成一套独立规则包，变成可维护的兼容层。fileciteturn112file0 fileciteturn104file0 fileciteturn111file0

所以这一版建议不是继续堆功能，而是把下面两件事一次性做完整：

- **做中文日志总线**：把召回每一步的输入、判定、结果、补召回原因、裁剪原因全部打出来。
- **做 PeroCore 兼容规则包**：把已确认规则完整迁进来，并为后续继续补充词表预留扩展点。

## 二、这份方案参考了哪些现有实现

### 1. 你当前分支已经具备的基础

你在最近这轮改动里，已经把这些关键能力接上了：

- `context-router.ts` 已经支持 `actorAliases`、`actorNameToKey`、`locationNameToKey`、`relationAliases`、`worldAliases`、`RecentContextBias`、`reasons`
- `retrieval-orchestrator.ts` 已经把 `contextRoute` 接入种子召回，并额外输出 `diagnostics`
- `graph-expander.ts` 已经支持 `contextRoute` 感知边权重、`hub penalty`、延后 `clamp`
- `coverage-checker.ts` 已经有 facet 子查询思路
- `diversity-pruner.ts` 已经有轻量平衡
- `buildMemoryGraph.ts` 已经开始把图谱边拆分成 `participants`、`location`、`sourceSummary`、`ownerActorRef`、`mentionedActorRef`、`worldKey`、`tag`
- `tabMemoryGraph.ts` 已经支持 `compact / semantic / debug` 模式

这意味着日志和规则兼容层不需要推翻重做，只要接到现有结构上即可。fileciteturn94file0

### 2. PeroCore 公开可确认到的能力

从 PeroCore 公开仓库里，目前能直接确认到四类对这次方案最有价值的东西：

第一类是**检索增强链路**。`retrieval_enhancer.py` 已经明确写出了 NMF、FISTA 稀疏编码残差、共现增益、DPP 多样性采样，并且带了独立参数文件与 Rust 加速回退逻辑。这个思路和你现在的 MemoryOS 召回链方向是一致的。fileciteturn104file0

第二类是**多意图切分规则**。`memory_service.py` 里已经明确使用了按中文标点和连接词拆分多意图的规则，连接词包括：`顺便`、`另外`、`还有`、`对了`、`然后`。它不是只靠 embedding，而是先做句式层切分，再做多段检索。fileciteturn112file0

第三类是**系统事件前缀识别**。`scorer_service.py` 和 `summary.md` 里都明确把 `【管理系统提醒】`、`【系统触发】` 这种前缀当作系统输入，而不是用户自然发言；这一点很适合迁入 MemoryOS，因为你的冷启动、自动总结、观察类输入，本质上也有同样问题。fileciteturn110file0 fileciteturn111file0

第四类是**记忆类型与思维簇归类框架**。`summary.md` 已经定义了 `event / fact / preference / promise / inspiration` 五类记忆类型，以及 `[逻辑推理簇] / [反思簇] / [情感偏好簇] / [人际关系簇] / [计划意图簇] / [创造灵感簇] / [闲聊簇]` 这套归类方式。这不等于你要照抄它的业务语义，但很适合做成 MemoryOS 的兼容规则层，让情境判定与后续摘要/归类更加统一。fileciteturn111file0

## 三、当前 MemoryOS 最需要补的不是算法，而是“看得见的判定过程”

现在的主要问题不是“没有召回”，也不是“没有判定”，而是**开发者看不清楚系统到底怎么想的**。

比如现在你已经有 `contextRoute`，但如果一次输入被判成了 `relationship + event + scene`，开发者很难第一眼知道：

- 到底是哪几个词触发的
- 是哪个 alias 命中了 actorKey
- 为什么 recent context 把 relationship 又抬高了
- 第一轮 seed recall 真正用了什么 query
- graph expansion 哪类边加了分
- second pass 是因为缺 `world` 还是缺 `scene`
- diversity 为什么把某几条高分项延后了

如果这几个问题没有日志，后面你继续加 FISTA、继续加向量接口、继续加图谱边模式，调试成本会直线上升。

所以这一版的核心原则是：

> 先把“判定过程透明化”，再继续做“更高级的判定”。

## 四、日志系统设计：必须全中文，而且按阶段成体系

### 4.1 设计目标

日志系统要满足五个要求：

1. **全中文**：开发时打开日志，不需要在英文和代码变量之间来回翻译。
2. **按阶段输出**：不是一堆散日志，而是明确知道当前处于哪一步。
3. **同时有人类可读文本和结构化字段**：终端能看，后面导出到工作台也能看。
4. **支持级别控制**：默认信息日志简洁，调试时再打开细节。
5. **能和 UI 诊断面板复用同一份数据结构**。

### 4.2 新增日志模块建议

建议新增一个统一模块：

```ts
MemoryOS/src/core/debug/memory-retrieval-logger.ts
```

建议暴露两个入口：

```ts
export interface MemoryDebugLogRecord {
  ts: number
  level: 'debug' | 'info' | 'warn' | 'error'
  stage:
    | 'context'
    | 'seed'
    | 'graph'
    | 'coverage'
    | 'diversity'
    | 'injection'
    | 'workbench'
  title: string
  message: string
  payload?: Record<string, unknown>
}

export function logMemoryDebug(record: MemoryDebugLogRecord): void
export function pushMemoryTrace(chatKey: string, record: MemoryDebugLogRecord): void
```

其中：

- `logMemoryDebug` 负责控制台输出
- `pushMemoryTrace` 负责写入内存 trace 缓冲，供工作台查看

### 4.3 日志输出规范

每条日志都建议统一成这种风格：

```text
[记忆召回][情境判定][信息] 当前输入被判定为：关系 + 事件
[记忆召回][种子召回][信息] 实际使用检索器：词法检索
[记忆召回][图扩散][调试] 关系边获得语境加权，乘数=1.25
[记忆召回][补召回][信息] 当前结果对“世界信息”覆盖不足，触发二次补召回
[记忆召回][多样性裁剪][调试] 关系类占比过高，延后 2 条低分候选
```

要求：

- `title` 一定简短
- `message` 一定是中文完整句
- `payload` 里再放原始字段
- 绝不直接把开发变量名原样打印给用户看作为主信息

### 4.4 具体要加哪些日志

#### A. `context-router.ts`

这里要新增的日志最重要。

建议至少输出以下几条：

1. **开始判定**

```text
当前开始进行情境判定，原始输入长度 42 字。
```

2. **命中的自然语言锚点**

```text
命中角色锚点：艾琳 -> char_erin
命中地点锚点：旧城区桥边 -> bridge_old_city
命中关系锚点：relationship:char_erin:char_mc
```

3. **命中的规则词典**

```text
命中“关系冲突”词典：不信、怀疑、争吵
命中“场景定位”词典：桥、这里、那晚
```

4. **recent context 偏置**

```text
最近场景模式为 relationship_conflict，已对“关系”与“事件”情境追加偏置。
```

5. **最终判定结果**

```text
当前输入被判定为：关系 > 事件 > 场景，置信度 0.78。
```

#### B. `retrieval-orchestrator.ts`

这里要输出完整召回主流程：

1. 当前实际 provider
2. route-aware query 是否改写了原 query
3. boostSchemaIds 是否生效
4. seed 数量
5. expanded 数量
6. second pass 是否触发
7. diversity 剔除数量
8. 最终输出数量

示例：

```text
已生成种子召回查询：原始 query 基础上追加 2 个锚点词。
实际使用检索器：词法检索。
第一轮种子召回完成，共命中 18 条候选。
图扩散完成，候选从 18 条增长到 31 条。
检测到“场景信息”覆盖不足，触发二次补召回。
多样性裁剪完成，最终保留 14 条结果。
```

#### C. `graph-expander.ts`

这里不要只打最终数量，要把“为什么扩散”打出来。

示例：

```text
当前进入图扩散阶段，主情境为 relationship。
关系边命中 topFacet，边权乘数提升为 1.25。
目标节点度数过高，已应用 hub penalty。
图扩散结束，本轮共新增 9 个非种子节点。
```

#### D. `coverage-checker.ts`

重点不是“触发了没有”，而是“为什么触发”。

示例：

```text
检测到当前结果对“世界信息”覆盖不足（覆盖数 0，目标至少 2）。
已生成 world 子查询：禁令 夜间 出城。
二次补召回完成，新增 3 条世界设定候选。
```

#### E. `diversity-pruner.ts`

要把结构平衡的原因说出来：

```text
候选未超上限，但关系类占比达到 71%，触发轻量平衡。
已延后 2 条关系类低分候选，以保留场景与世界条目。
```

#### F. `actor-visible-context-builder.ts`

建议增加最终注入层日志：

```text
当前注入视角角色：char_erin。
本轮共注入 12 条记忆，上下文预算占用约 1830 字。
注入内容包含 clear / blur / distorted 三种记忆呈现层级。
```

### 4.5 设置项

建议在设置里新增：

```ts
retrievalLogEnabled: boolean
retrievalLogLevel: 'info' | 'debug'
retrievalTracePanelEnabled: boolean
```

建议默认：

- `retrievalLogEnabled = true`
- `retrievalLogLevel = 'info'`
- `retrievalTracePanelEnabled = true`

### 4.6 UI 侧联动

工作台新增一个诊断折叠区：

- 当前情境判定
- 命中锚点
- 命中词典
- 第一轮召回 query
- 扩散前后数量变化
- 补召回原因
- 多样性裁剪原因

这样以后看问题，不需要翻控制台就能知道这一轮为什么这么召回。

## 五、PeroCore 兼容规则包：不要散写在函数里，要做成独立文件夹

### 5.1 总体思路

建议新增一个目录：

```text
MemoryOS/src/memory-keywords/perocore/
```

这个目录不是替换你现有的 `memory-keywords/`，而是作为一套“兼容规则包”。

推荐结构：

```text
MemoryOS/src/memory-keywords/perocore/
├── splitters.ts
├── systemPrefixes.ts
├── memoryTypeRules.ts
├── thoughtClusterRules.ts
├── relationshipConflictRules.ts
├── worldQaRules.ts
├── sceneRules.ts
├── conjunctions.ts
├── aliases.ts
└── index.ts
```

### 5.2 这次要完整迁进来的“已确认规则”

#### A. 多意图切分连接词

从 PeroCore 公开代码里，已经明确确认了这些切分连接词：

- `顺便`
- `另外`
- `还有`
- `对了`
- `然后`

以及这些切分标点：

- `， ,`
- `。 .`
- `； ;`
- `！ !`
- `？ ?`
- 换行 `\n`

这一组规则建议直接原样做成：

```ts
export const PEROCORE_INTENT_SPLITTERS = {
  punctuation: ['，', ',', '。', '.', '；', ';', '！', '!', '？', '?', '\n'],
  connectors: ['顺便', '另外', '还有', '对了', '然后'],
}
```

这个规则要用在两处：

1. 检索前的 query 分段
2. 情境判定前的子句归一化

这样你后面不管是词法检索还是 embedding 预留接口，前置子句拆分都一致。fileciteturn112file0

#### B. 系统事件前缀规则

这组规则也建议直接做成独立文件：

- `【管理系统提醒】`
- `【系统触发】`

建议抽成：

```ts
export const PEROCORE_SYSTEM_PREFIXES = [
  '【管理系统提醒】',
  '【系统触发】',
]
```

作用不是只给总结用，而是整个检索链共用：

- 如果是系统事件前缀
- 那么 route 里要把它标成 `system-event`
- 在日志里写明“当前输入属于系统事件，不按普通用户发言处理”
- 后续注入时也避免把它当成“用户本人直接说过的话”

这个规则非常适合你当前的自动总结、冷启动补写、后台观察输入。fileciteturn110file0 fileciteturn111file0

#### C. 记忆类型兼容映射

PeroCore 公开 prompt 里已经明确了五类记忆类型：

- `event`
- `fact`
- `preference`
- `promise`
- `inspiration`

建议在 MemoryOS 里不要直接替代你现有 schema，而是建立“兼容映射”：

```ts
export const PEROCORE_MEMORY_TYPE_TO_MEMORYOS_FACET = {
  event: ['event'],
  fact: ['world', 'interpretation'],
  preference: ['interpretation', 'relationship'],
  promise: ['event', 'relationship'],
  inspiration: ['interpretation', 'event'],
}
```

这层映射的作用不是覆盖当前 schema，而是给 `context-router` 多一层辅助判断：

- 句子如果更像偏好表达，就把 `interpretation` 提高
- 句子如果更像承诺或计划，就把 `event` 和 `relationship` 同时提权
- 句子如果更像事实问答，就把 `world` 提高

fileciteturn111file0

#### D. 思维簇兼容映射

PeroCore 已经明确给出一组“思维簇”分类：

- `[逻辑推理簇]`
- `[反思簇]`
- `[情感偏好簇]`
- `[人际关系簇]`
- `[计划意图簇]`
- `[创造灵感簇]`
- `[闲聊簇]`

这套规则非常适合迁成一层“兼容解释标签”。

建议映射成：

```ts
export const PEROCORE_CLUSTER_TO_ROUTE_HINTS = {
  逻辑推理簇: ['world', 'event'],
  反思簇: ['interpretation'],
  情感偏好簇: ['interpretation', 'relationship'],
  人际关系簇: ['relationship'],
  计划意图簇: ['event'],
  创造灵感簇: ['interpretation', 'event'],
  闲聊簇: [],
}
```

未来无论冷启动、总结、手动整理，MemoryOS 都可以把条目附带上兼容簇标签，后面召回时就能直接提权。

fileciteturn111file0

### 5.3 这次建议顺手扩充的 MemoryOS 规则目录

你现有 `memory-keywords/` 里已经有冲突、修复、场景、设定问答这类方向，所以这次不只是迁 PeroCore，还建议把规则目录整理成三层：

```text
memory-keywords/
├── native/        # MemoryOS 原生规则
├── perocore/      # PeroCore 兼容规则
└── hybrid/        # 混合输出入口
```

然后在 `index.ts` 里统一做：

```ts
export function getActiveKeywordPack(mode: 'native' | 'perocore' | 'hybrid')
```

默认建议：

- `mode = 'hybrid'`

因为对你现在这个插件来说，最稳妥的不是把原生规则全换掉，而是：

- 原生规则负责世界观/角色卡/结构化记忆适配
- PeroCore 兼容规则负责多意图切分、系统前缀、记忆类型框架、思维簇框架

### 5.4 判定流程建议改成“七步”

建议最终情境判定改成下面这条链：

#### 第一步：系统前缀识别

先判断是不是：

- `【管理系统提醒】`
- `【系统触发】`

如果是，就打系统事件标记。

#### 第二步：多意图切分

按 PeroCore 兼容切分规则把 query 分成子句。

#### 第三步：子句级关键词匹配

每个子句单独跑：

- 冲突词
- 修复词
- 场景词
- 世界设定词
- 关系词
- 系统事件词

#### 第四步：自然语言锚点识别

按 alias、displayName、locationName 去命中实体。

#### 第五步：PeroCore 兼容记忆类型推断

如果子句更像：

- 偏好表达
- 承诺/计划
- 创意/灵感
- 事实问答

就给对应 facet 加一层兼容提权。

#### 第六步：recent context 偏置

叠加：

- 当前 sceneMode
- currentLocationKey
- activeRelationPair
- dominantFacetHints

#### 第七步：生成可解释结果

最终 route 不只输出：

- `facets`
- `entityAnchors`
- `topicHints`
- `confidence`

还要输出：

- `subQueries`
- `matchedRulePack`
- `matchedRules`
- `reasons`

建议扩成：

```ts
export interface RetrievalContextRoute {
  facets: RetrievalFacet[]
  entityAnchors: {
    actorKeys: string[]
    locationKeys: string[]
    relationKeys: string[]
    worldKeys: string[]
  }
  topicHints: string[]
  confidence: number
  subQueries?: string[]
  matchedRulePack?: 'native' | 'perocore' | 'hybrid'
  matchedRules?: Array<{
    pack: string
    ruleId: string
    label: string
    matchedText: string[]
  }>
  reasons?: Array<{
    source: 'keyword' | 'pattern' | 'entity' | 'recent-context' | 'perocore-rule'
    detail: string
    weight: number
  }>
}
```

## 六、文件级实施方案

### 6.1 新增文件

#### `MemoryOS/src/core/debug/memory-retrieval-logger.ts`

职责：

- 日志格式统一
- 控制台输出
- trace 缓冲
- 供工作台复用

#### `MemoryOS/src/memory-keywords/perocore/splitters.ts`

## 八、推荐实施顺序

### 第一步：先把日志总线做出来

优先级最高。

原因很简单：

- 没日志，就算你加了更多规则，也看不出到底有没有生效。
- 没日志，后面继续调 graph、coverage、diversity 都会越来越难。

### 第二步：接 PeroCore 兼容切分规则和系统前缀规则

这一层最容易做，而且收益很高：

- 多意图 query 会立刻更稳
- 系统事件输入会立刻更干净

### 第三步：接记忆类型与思维簇兼容层

这一层做完，route 的解释能力会更强，也更适合后面接冷启动和总结。

### 第四步：把 trace 挂到工作台

这一层完成后，你的工作台才真正算“可调试”。

## 十、最后的建议

这一版最重要的不是“把日志加上”这么简单，而是把 MemoryOS 真正推进到**工程可调试阶段**。

你现在这套系统已经不是简单的关键词搜索插件了，而是：

- 有情境判定
- 有 route-aware seed recall
- 有图扩散
- 有二次补召回
- 有多样性裁剪
- 有图谱视图

这种系统如果没有日志，后面会越来越像“黑箱”；但如果这次把中文日志和 PeroCore 兼容规则包一起做进去，后面你再加 embedding、再加更强的残差召回、再加更多图谱边模式，整个工程都会顺很多。

所以我的建议很明确：

> **这一版优先把“中文检索日志 + PeroCore 兼容规则包 + 工作台 trace 面板”一次做完整。**

这样下一版你再继续做算法提升，才不会一直在黑箱里调参。
