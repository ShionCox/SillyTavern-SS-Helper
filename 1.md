# MemoryOS 时间意图融合向量评分实施方案（基于 memoryos-settings-unified 最新代码）

## Summary
- 当前 MemoryOS 已经具备时间系统的基础能力：
  1. 记忆对象可以携带 `timeContext`
  2. 词法检索已经支持时间加分
  3. 混合检索合并阶段已经支持轻量时间重打分
  4. 工作台已经能展示时间信息
- 但目前“时间”还没有真正成为**向量侧与最终排序侧的核心特征**。
- 这会导致一类非常重要的问题还处理得不够好，例如：
  - `XXX最后结果如何`
  - `后来怎么样了`
  - `现在是什么情况`
  - `最早的时候发生了什么`
- 本方案目标是把“时间”从**被动展示字段**升级为**检索排序特征**，并且尽量融入现有主链，而不是另起一套平行系统。

---

## Current Status Review

## 1. 目前已有的能力
### 1.1 时间系统本体已经存在
当前代码里已经有：
- `memory-time/time-types.ts`
- `memory-time/timeline-profile.ts`
- `memory-time/batch-time-assessment.ts`
- `memory-time/fallback-time-engine.ts`
- `memory-time/time-ranking.ts`
- `memory-time/time-format.ts`

说明时间系统已经不是纸面设计。

---

### 1.2 词法检索已支持时间加分
当前 `lexical-provider.ts` 已经会对每个候选读取：
- `candidate.timeContext`
- `currentMaxFloor`
- `computeTimeBoost(query, timeContext, currentMaxFloor)`

然后把 `timeBoost` 融入 `mergeSeedScoreBreakdown()`。

这说明：
> 时间在当前系统中，已经不是纯展示信息，而是部分参与了词法种子评分。

---

### 1.3 混合检索 merge 阶段也有轻量时间加分
当前 `HybridRetrievalService.mergeResults()` 已经会：
- 重新读取候选的 `timeContext`
- 重新计算 `computeTimeBoost()`
- 把时间增益以轻量方式融合进 merge 后总分

这说明：
> hybrid 模式并不是完全忽略时间。

---

## 2. 当前存在的关键问题
### 2.1 时间意图识别过于粗糙
当前 `time-ranking.ts` 只区分两类查询：
- `recent`
- `early`

这不足以覆盖：
- “最后结果如何”
- “后来怎么样”
- “现在关系如何”
- “最终收场是什么”

这些其实不是普通“近期”查询，而是：
- 终局态
- 当前态
- 结果态
- 最新阶段

---

### 2.2 向量主链没有系统吃到时间
当前 `vector_only` 的结果主流程仍然偏向：
- 向量相似度
- 策略路由
- rerank

时间并没有像 lexical 那样成为**系统性的候选评分特征**。

---

### 2.3 rerank 阶段还没有显式理解时间意图
当前 rule rerank / LLMHub rerank 并没有真正区分：
- 这条是最终状态
- 这条是过渡状态
- 这条是早期设定
- 这条是当前状态

这会导致：
- 有时最新但不相关的条目会被抬高
- 有时结果态条目明明更对，却因为文本相似度略低被压下去

---

## Final Goal

## 目标
让系统能够理解并正确处理这些时间型问题：

### 1. 最近态
- 最近发生了什么
- 刚才怎么了
- 最新进展是什么

### 2. 早期态
- 最早是怎样
- 当初发生了什么
- 小时候关系如何

### 3. 最终结果态
- 最后结果如何
- 后来怎么样了
- 结局是什么
- 最终怎样收场

### 4. 当前状态态
- 现在怎么样
- 当前是什么情况
- 目前进展如何

# 一、设计原则

## 原则 1：时间加分必须建立在主题相关性之上
不能因为 query 里有“最后”，就把所有最近条目都顶上来。

正确逻辑应该是：

> 时间意图加分 × 主题相关度 × 条目状态特征

---

## 原则 2：时间不能取代主题相关性
时间是辅助排序特征，不是主相关性。

例如：
- query：`何盈最后结果如何`
- 如果有一条最近的无关事件，只因为“最近”就加分，结果一定会错

所以：
- 先保证候选相关
- 再在相关候选中做时间偏置

---

## 原则 3：结果态 / 当前态 / 早期态要分开
不能把：
- `最近`
- `现在`
- `最后`
- `以前`

都混成同一个时间分。

---

## 2. `early`
适用问题：
- 最早
- 以前
- 当初
- 小时候
- 第一阶段
- 一开始

目标：
- 更偏向最早记忆
- 更偏向起点设定
- 更偏向 firstObserved 内容

---

## 3. `final_outcome`
适用问题：
- 最后结果如何
- 后来怎么样了
- 最终怎样
- 结局是什么
- 收场如何
- 下场如何

目标：
- 更偏向结果态条目
- 更偏向“已完成 / 已解决 / 当前最终状态”
- 更偏向关系的最新有效阶段
- 更偏向任务最终状态

---

## 4. `current_state`
适用问题：
- 现在怎么样
- 当前是什么情况

- 更偏向当前状态型记忆

# 四、向量评分融合方案

## 目标
让向量侧也能理解时间意图，而不是只有 lexical 在做时间加分。

---

## 方案一：对 `vectorResults` 进行时间重打分
### 接入点
`HybridRetrievalService.mapVectorHits()` 之后  
在 `handleVectorOnly()` / `handleHybrid()` 之前，加入统一的时间重打分函数：

```ts
private applyTemporalRescore(
  items: RetrievalResultItem[],
  query: string,
  currentMaxFloor: number
): RetrievalResultItem[]
```

---

## 建议逻辑
### Step 1
解析 query 的 `QueryTimeIntent`

### Step 2
为每个 candidate 计算：
- `timeBoost`
- `stateBoost`
- `outcomeBoost`

### Step 3
按意图融合到 score 中

---

## 时间重打分建议公式
```ts
rescore = baseScore * (1 - temporalWeight) + temporalFeatureScore * temporalWeight
```

其中：
- 普通 query：`temporalWeight = 0`
- `recent / early`：`0.10 ~ 0.12`
- `current_state / final_outcome`：`0.12 ~ 0.18`

---

# 五、状态型 / 结果型条目识别方案

## 原因
“最后结果如何”这种 query 不能只看时间先后，必须识别：

- 这条是不是结果态
- 这条是不是当前状态
- 这条是不是阶段性过渡

---

## 建议新增函数
```ts
export function computeStateLikeBoost(candidate: RetrievalCandidate): number
export function computeOutcomeLikeBoost(candidate: RetrievalCandidate): number
```

---

## 1. 状态型候选加分
适合：
- `relationship`
- `world_global_state`
- `task`
- `summary`
- 包含 `state / status / stage / phase / ongoing`

### 典型字段
- `fields.status`
- `fields.stage`
- `detailPayload.state`
- `detailPayload.lifecycle.status`
- `ongoing === true`

---

## 2. 结果型候选加分
适合：
- `event`
- `task`
- `relationship`
- `summary`

### 典型字段
- `result`
- `outcome`
- `resolution`
- `completed`
- `resolved`
- `结局`
- `最后`
- `收场`

---

## 建议规则
### 如果 query intent = `final_outcome`
- 对包含结果字段的条目加分
- 对已完成/已解决的条目加分
- 对最新阶段 summary 加分

### 如果 query intent = `current_state`
- 对 ongoing / current / latest status 条目加分
- 对世界状态 / 关系状态加分
- 对 still-active 的任务条目加分

---

# 六、检索链具体接入点

## 1. `memory-time/time-ranking.ts`
### 要做的改造
新增：

```ts
resolveQueryTimeIntent(query)
computeTemporalIntentBoost(query, timeCtx, currentMaxFloor, candidate?)
computeStateLikeBoost(candidate)
computeOutcomeLikeBoost(candidate)
buildTemporalExplanation(...)
```

---

## 2. `memory-retrieval/lexical-provider.ts`
### 当前已有
已经在计算 `timeBoost`

### 建议改成
把原本的 `computeTimeBoost()` 升级为：

```ts
const temporal = computeTemporalIntentBoost(...)
const score = mergeSeedScoreBreakdown({
  ...
  timeBoost: temporal.timeBoost
})
```

同时把：
- `intent`
- `stateBoost`
- `outcomeBoost`

写进 `breakdown`

---

## 3. `services/hybrid-retrieval-service.ts`
这是本次最值得改的地方。

### 3.1 `vector_only`
当前 `handleVectorOnly()` 直接：
- vector results -> topK
- 或 rerank

### 建议改为
先：

```ts
const temporallyRescored = this.applyTemporalRescore(vectorResults, query, currentMaxFloor)
```

再进入：
- direct topK
- or rerank

---

### 3.2 `handleHybrid()`
当前 `mergeResults()` 已经做了轻量时间融合，但还不够。

### 建议改为
- merge 前：对 vector 结果做 temporal rescore
- merge 后：按 query intent 再做二次轻量时间偏置

这样 hybrid 排序会更稳。

---

### 3.3 `mergeResults()`
建议把现在固定 8% 的 merge 时间权重改成：

- `recent / early`：0.08 ~ 0.10
- `current_state / final_outcome`：0.12 ~ 0.15
- `none`：0

---

# 七、rerank 层增强方案

## 1. Rule Rerank
建议在 `VectorRerankService` 中新增 temporal features：

### 输入特征
- `queryTimeIntent`
- `candidate.timeContext`
- `candidate 是否状态型`
- `candidate 是否结果型`
- `candidate 是否 ongoing`
- `candidate 是否 completed/resolved`

### 作用
让 rerank 在这些问题上更聪明：
- `最后结果如何`
- `后来怎么样`
- `现在怎样`
- `最早是怎样`

---

## 2. LLMHub Rerank
建议作为第二阶段做。

### 接法
在发送候选给 LLMHub 时，为每条候选补充：
- 时间标签
- 时间模式
- 当前状态提示
- 是否结果态

例如：

```text
候选 1：
标题：……
时间：较当前早约 12 层
类型：关系状态
状态：ongoing
结果字段：无
```

这样 LLMHub 更容易排对。

# 十三、最终结论

## 这个方向有没有搞头
**非常有搞头，而且值得做。**

## 当前最好的切入点
不是继续扩时间模型，而是：

> **把“时间意图”接入向量候选初排、merge 排序和 rerank。**

## 一句话定案
对于“`XXX最后结果如何`”这类问题，正确做法不是简单给最近内容加分，而是：

> **识别 query 的时间意图，再结合时间上下文、状态型特征和结果型特征，对向量与最终排序做时间偏置。**

这样效果会明显比现在只做 recent/early 好得多。
