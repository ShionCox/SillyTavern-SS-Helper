# MemoryOS 统一分层归约收尾修改实施方案

> 适用背景：基于当前 `memoryos-settings-unified` 分支，对近期重构后的 MemoryOS 进行收尾改造。  
> 目标：把“旧聊天接管 / 冷启动 / AI 总结”三条链统一到同一种 **staging → reduce → finalize** 的分层架构上，清理旧链路残留，降低输入/输出上限风险，方便后续维护。

---

# 一、文档目的

这份文档的目标不是继续讨论“要不要这样做”，而是直接给出一份可以执行的修改实施方案。

当前分支已经完成了大量重构工作，但整体仍然存在一个明显的问题：

> **新链路已经搭出来了，但旧链路还没有完全退场。**

表现为：

- 旧聊天接管已经有本地 merge / fallback，但最后仍可能走一次全量 LLM consolidation
- 冷启动已经支持 entityCards，但仍然更像一次性 full extraction
- AI 总结已经做了 lightweight planner + slim mutation context，但 UPDATE 还不是真正 sparse patch
- 很多地方已经具备“分层归约”的雏形，但最终没有真正统一成一个主架构

所以这次的实施方案，核心不是“再加功能”，而是：

> **把当前所有链路统一成同一套可维护、可恢复、可审计、可控输出的主链。**

---

# 二、总体改造目标

## 2.1 总目标

将三条主链统一为以下结构：

```text
原始输入
→ 小批次结构化结果
→ staging 暂存层
→ 本地 reduce / merge
→ 小范围 conflict resolve（必要时才用 LLM）
→ 正式 apply
```

## 2.2 这次改造完成后应达到的状态

### 旧聊天接管
- 不再把所有 `batchResults` 整包重新喂给 LLM 做最终整合
- 每批输出先写 staging
- 最终阶段以本地 merge 为主
- 只有冲突项才交给 LLM

### 冷启动
- 不再坚持“一次性 full document”
- 拆成实体抽取、状态抽取、代码 apply 多阶段
- 支持世界实体与角色实体统一进入账本

### AI 总结
- Planner 保持轻量输入
- Mutation 保持 slim context
- UPDATE 真正变成 sparse patch
- actions 支持 batching，不再一轮要求模型吐太多 mutation

---

# 三、核心原则

## 3.1 LLM 不负责“重建整个最终账本”
LLM 适合做：
- 识别
- 判断
- 提取
- 小范围裁决

LLM 不适合做：
- 全量最终账本重建
- 超大 JSON 一次性输出
- 对几百条对象做全量重写

因此必须把“最终账本”更多交给代码 merge 逻辑，而不是交给模型一次性吐出来。

---

## 3.2 所有中间结果必须可落地、可恢复
每一批、每一步都必须能写入 staging：

- 可恢复
- 可暂停
- 可重试
- 可追踪
- 可调试

不能再依赖：
- 一轮跑到底
- 模型成功输出大对象
- 内存里临时拼接完再全交模型

---

## 3.3 真正的主链必须是 deterministic merge
凡是能通过程序稳定合并的内容，都必须让程序负责：

- compareKey 命中
- actorKey 命中
- relationship key 命中
- task key 命中
- world state key 命中

只有程序无法稳定裁决的冲突，才进入 conflict resolver。

---

## 3.4 “继续生成”只能是兜底，不是主方案
当结构化输出触发模型长度上限时，不应依赖：
- 继续输出
- 接着上轮结果续写
- 把前一轮结果重新塞回模型

这种方案只适合低要求文本草稿，不适合结构化记忆系统的主链。

---

# 四、统一主链架构设计

这次改造建议统一引入三层概念：

## 4.1 Staging 层
所有中间结构化结果先进入 staging。

### staging 用途
- 记录批次输出
- 记录阶段输出
- 记录 role 统计
- 记录 compareKey 命中/冲突
- 作为重试恢复依据
- 作为最终 reduce 的输入

---

## 4.2 Reduce 层
由程序执行本地压缩、去重、合并。

### reduce 的职责
- 批量去重
- actor merge
- entity merge
- relation collapse
- task collapse
- world state collapse
- chapter digest 生成
- 章节归约

---

## 4.3 Finalize 层
Finalize 不再做全量重建，只做：

- 本地 finalize
- 冲突桶裁决
- 正式 apply

也就是说，finalize 分成两个子层：

### Finalize-A：code finalize
程序把 reduce 后的稳定账本整理成最终写入对象。

### Finalize-B：conflict resolver（可选）
只针对 unresolved conflicts 调用 LLM。

---

# 五、旧聊天接管（Takeover）改造方案

这是本次改造的重点。

---

## 5.1 当前问题

旧聊天接管目前的核心问题不是前面分批，而是最后一层职责过重：

- 批次处理已经拆开了
- 本地 fallback merge 也已经做了不少
- 但最后仍把 `batchResults` 重新整包送给模型做 `consolidation`

这会导致：

- 输入上限风险
- 输出上限风险
- JSON 结构不稳定
- 批次越多，最终越危险

---

## 5.2 改造目标

将旧聊天接管改为：

```text
历史消息
→ 分批提取
→ batch staging
→ section reduce
→ final local merge
→ conflict-only resolve
→ apply
```

---

## 5.3 具体改造内容

### 一、保留批次处理
保留现有：
- batch planner
- batch runner
- takeover batch schema

因为批次处理本身没有问题，问题在最后一层。

### 二、每批结果必须写入 staging
新增 takeover session 级 staging 数据结构，例如：

```ts
type TakeoverSessionStaging = {
  takeoverId: string
  status: 'running' | 'paused' | 'failed' | 'completed'
  batchResults: TakeoverBatchResult[]
  sectionDigests: TakeoverSectionDigest[]
  reducedLedger: ReducedTakeoverLedger | null
  unresolvedConflicts: TakeoverConflictBucket[]
  finalResult: TakeoverFinalResult | null
}
```

### 三、把 batchResults 分层归约成 sectionDigests
不要等到最后一步再把所有 batchResults 一次性送给模型。

建议每 5~10 个 batch 生成一个 `sectionDigest`：

```ts
type TakeoverSectionDigest = {
  sectionId: string
  range: { startFloor: number; endFloor: number }
  actorCards: ActorCard[]
  entityCards: EntityCard[]
  stableFacts: string[]
  relationTransitions: RelationTransition[]
  taskTransitions: TaskTransition[]
  worldStateChanges: WorldStateChange[]
  openThreads: string[]
}
```

### 四、finalize 默认只吃 sectionDigests，不吃 raw batchResults
最终整合输入改为：

- activeSnapshot
- sectionDigests
- reducedLedger
- unresolvedConflicts

而不是：
- activeSnapshot
- 全部 batchResults

### 五、finalize 默认走 code merge
在 `takeover-consolidator` 中，默认主链应该是：

1. merge section digests
2. 生成 entity ledger / relation state / task state / world state
3. 检查 unresolved conflicts
4. 若冲突为空，直接 finalize apply
5. 若冲突不为空，再调用 conflict resolver

### 六、LLM 只处理 conflict bucket
新增一个更轻的 conflict schema，例如：

```json
{
  "conflicts": [
    {
      "kind": "organization_merge_decision",
      "left": { "compareKey": "organization:教派:赤月教派", "title": "赤月教派" },
      "right": { "compareKey": "organization:教团:赤月教团", "title": "赤月教团" },
      "question": "这两个对象应视为同一实体还是不同实体？"
    }
  ],
  "resolutions": [
    {
      "kind": "organization_merge_decision",
      "decision": "MERGE",
      "source": "organization:教团:赤月教团",
      "target": "organization:教派:赤月教派",
      "reason": "两者别名与摘要高度一致"
    }
  ]
}
```

### 七、正式写库前只接收 reduced ledger + conflict resolutions
最终 apply 只允许消费：

- `ReducedTakeoverLedger`
- `ConflictResolutionResult`

不再允许消费一个“重新全量生成的 TakeoverConsolidationResult”。

---

## 5.4 文件级修改建议

### `memory-takeover/takeover-consolidator.ts`
#### 保留
- 本地 merge/fallback 逻辑
- dedupe / collapse / merge 一系列 deterministic 逻辑

#### 修改
- 删除或下沉“全量 consolidation LLM 调用”主链
- 改成：
  - `buildReducedLedger()`
  - `collectConflicts()`
  - `resolveConflictsIfNeeded()`
  - `finalizeTakeoverApply()`

#### 建议新增函数
- `buildSectionDigests()`
- `mergeSectionDigests()`
- `collectTakeoverConflicts()`
- `applyTakeoverFinalLedger()`

---

### `memory-takeover/takeover-llm.ts`
#### 保留
- batch 提取相关 task
- batch schema/sample

#### 降级为兼容层
- `TAKEOVER_CONSOLIDATION_SCHEMA`
- `TAKEOVER_CONSOLIDATION_OUTPUT_SAMPLE`

#### 新增
- `TAKEOVER_CONFLICT_RESOLUTION_SCHEMA`
- `TAKEOVER_CONFLICT_RESOLUTION_OUTPUT_SAMPLE`

---

### `memory-takeover/takeover-orchestrator.ts`
#### 修改
- 增加 section digest 归约过程
- 增加 staging 写入与恢复点
- 增加 finalize 模式分支：
  - code finalize
  - conflict resolve + finalize

---

# 六、冷启动（Bootstrap）改造方案

---

## 6.1 当前问题

冷启动当前虽然已经支持：
- actorCards
- entityCards
- worldBase
- relationships
- memoryRecords

但整体仍更像：

> 一次性 full extraction → 一次性 apply

这样在以下场景会遇到问题：

- 世界书条目很多
- 角色卡信息复杂
- 最近事件很多
- 输出文档过大

---

## 6.2 改造目标

将冷启动改成三阶段：

```text
冷启动原始输入
→ 核心实体抽取
→ 状态与关系抽取
→ 本地去重与 apply
```

---

## 6.3 具体改造内容

### Phase 1：核心实体抽取
只要求模型输出：

- identity
- actorCards
- entityCards
- world profile
- worldBase 核心规则

不要在第一轮里同时要求：
- relationships
- memoryRecords
- recent event states

### Phase 2：状态抽取
基于 phase 1 的结果，再要求模型补：

- relationships
- memoryRecords
- recent event state
- current scene / task clues

### Phase 3：代码 apply
程序执行：

- actor/entity compareKey 去重
- worldBase 去重
- relationships merge
- records 写入

---

## 6.4 文件级修改建议

### `memory-bootstrap/bootstrap-orchestrator.ts`
#### 保留
- source bundle 组织
- bootstrap result apply 基础流程

#### 修改
- 拆成：
  - `runBootstrapCorePhase()`
  - `runBootstrapStatePhase()`
  - `applyBootstrapResultPhases()`

#### 新增
- phase 输出缓存
- bootstrap staging 结构

---

### `memory-prompts/prompt-pack.md`
#### 修改
- 冷启动 prompt 拆两段
  - Core Extract
  - State Extract

#### 删除或降级
- “一次性要求输出完整 cold start document”的超大 prompt 要降级为兼容模式，不应继续作为默认主链

---

# 七、AI 总结（Summary）改造方案

---

## 7.1 当前问题

总结链路已经是三条链里最先进的，但仍有两个未收口点：

1. UPDATE 不是真 sparse patch
2. action 数量没有显式 batching

---

## 7.2 改造目标

把总结链统一成：

```text
轻量 planner
→ action planning
→ action batch mutation
→ validate
→ apply
```

---

## 7.3 具体改造内容

### 一、保留 lightweight planner
`buildLightweightPlannerInput()` 方向是对的，继续保留。

### 二、保留 slim mutation context
`buildSlimMutationContext()` 方向是对的，继续保留。

### 三、把 UPDATE 真正改成 sparse patch
现在 patch schema 虽然名字叫 patch，但体积还偏完整对象。

必须改成：

- 只允许输出变更字段
- 未变化字段不应出现
- 嵌套对象也允许局部字段 patch
- patch 内不允许携带整段冗余 payload

### 四、引入 action batching
一轮 mutation 最多处理固定数量动作，例如：

- 8 条
- 10 条
- 或按类型拆批

如果 planner 产出了 24 个 action，则按 3 轮 mutation 跑完，而不是要求一轮输出 24 个完整 mutation。

### 五、按类型分批
在必要时可以分为：

- actor/relationship
- task/world state
- organization/city/nation/location

这样单轮 schema 更小，输出更稳。

---

## 7.4 文件级修改建议

### `memory-summary/summary-orchestrator.ts`
#### 保留
- planner -> mutation 结构

#### 修改
- 增加 `planActionBatches()`
- 按 batch 逐轮 mutation
- 汇总 mutation results 再 apply

---

### `memory-summary/mutation-applier.ts`
#### 保留
- validator / applier 结构

#### 修改
- 适配 sparse patch apply
- 支持多轮 mutation results 聚合 apply

---

### schema builder 相关文件
例如：
- `buildStrictPatchPayloadSchema`
- `schema-resolver`

#### 修改
- 真正实现 sparse patch schema
- 避免 UPDATE 仍然要求整对象字段

---

# 八、统一 staging / reduce / finalize 基础设施

为了方便维护，建议三条链共用同一批基础抽象。

---

## 8.1 新增统一阶段状态模型

```ts
type PipelineRunState = {
  runId: string
  pipelineKind: 'takeover' | 'bootstrap' | 'summary'
  phase: string
  status: 'pending' | 'running' | 'paused' | 'failed' | 'completed'
  createdAt: number
  updatedAt: number
}
```

---

## 8.2 新增统一 staging repository 接口

```ts
interface StagingRepository<T> {
  save(runId: string, payload: T): Promise<void>
  load(runId: string): Promise<T | null>
  append(runId: string, payload: Partial<T>): Promise<void>
  clear(runId: string): Promise<void>
}
```

### 推荐至少有三类 staging
- `TakeoverStagingRepository`
- `BootstrapStagingRepository`
- `SummaryStagingRepository`

---

## 8.3 新增统一 conflict resolver 接口

```ts
interface ConflictResolver<I, O> {
  collect(input: I): ConflictBucket[]
  resolve(conflicts: ConflictBucket[]): Promise<O>
  applyResolution(input: I, resolution: O): I
}
```

三条链都能复用这种模式。

# 十一、实施顺序建议

虽然你希望统一处理，但实际改造仍建议按下面这个顺序推进，这样风险最低。

## 第一步：先收 takeover 最后一层
因为这是当前最危险、最容易爆输入输出上限的一段。

### 完成标准
- 不再默认 full consolidation
- 引入 reduced ledger + conflict resolver
- staging 和 sectionDigest 跑通

---

## 第二步：再收 summary 的 true patch + batching
因为这条链已经最接近完成，补起来成本最低。

### 完成标准
- UPDATE 只输出 sparse patch
- actions 支持分批
- apply 可聚合多轮 mutation

---

## 第三步：最后收 bootstrap phase 化
因为冷启动通常频率没那么高，但也要统一起来，避免后面继续留一条 one-shot 大链。

### 完成标准
- phase 1 / phase 2 跑通
- 支持 entityCards / worldBase / relationships 分步抽取
- apply 支持 phased result

---

# 十三、风险与回滚

## 13.1 风险
### 1. 新旧链共存导致行为不一致
解决：
- 所有 legacy 路径都挂明确 flag
- 默认主链只有新架构

### 2. staging 增加复杂度
解决：
- 先最小实现，仅保留必要状态
- UI 可先不暴露复杂管理功能

### 3. sparse patch apply 容易出错
解决：
- 先为高频类型（actor / organization / task）做 patch apply
- 低频类型可以先走保守 fallback

---

## 13.2 回滚策略
- 保留 legacy consolidate behind flag
- 保留 bootstrap one-shot behind flag
- summary batching 可通过设置关闭，退回单轮 mutation

---

# 十四、最终结论

这次修改的目标，不是继续“补 prompt、补 schema、补限制”，而是统一 MemoryOS 的主架构。

必须彻底收成：

```text
staging
→ reduce
→ conflict-only resolve
→ finalize apply
```

只要这套统一下来：

- 旧聊天接管不会再在最后一步爆掉
- 冷启动不会继续依赖 one-shot full document
- AI 总结不会再因为 mutation 太多而难以稳定输出
- 整个系统后续维护会容易很多

一句话总结这次实施方案：

> **把 LLM 从“最终总装器”降级成“局部裁决器”，把程序提升为真正的主整合链。**
