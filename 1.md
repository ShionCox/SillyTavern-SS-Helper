# RollHelper 第四阶段实施方案：限制无限暗骰与队列滥用

## 文档目标

本阶段聚焦解决你前面提到的两类核心问题：

1. **如何限制“无限暗骰”**
2. **已经暗骰过的内容，不应该再继续暗骰**

这一阶段不再讨论“结果卡泄露”“只绑定最新轮次”“暗骰重投继承”这些前面阶段已经单独处理的主题，而是在它们的基础上，继续把暗骰系统从“能用”收紧到“可控”。

本阶段的目标是让暗骰从一个几乎可以无限堆积、无限刷、无限拖延消费的机制，变成一个：

- 有次数限制
- 有队列上限
- 有过期机制
- 有同类去重
- 有技能/事件白名单
- 有轮次内使用边界

的稳定机制。

---

## 一、当前问题审查

结合当前提交 `37915fe` 之后的代码，暗骰无限化问题主要集中在下面几个点。

---

### 1. `/broll` 目前会无条件往队列里追加

文件：`RollHelper/src/commands/baseRollCommandEvent.ts`

当前 `/broll` 的处理逻辑是：

- 解析输入
- 直接掷骰
- 生成一条 `BlindGuidanceEvent`
- 直接 `push` 到 `pendingBlindGuidanceQueue`
- 保存 metadata

也就是说，目前没有这些限制：

- 没有最大排队数
- 没有“同技能/同标题/同轮次”去重
- 没有“本轮最多几次暗骰”限制
- 没有 TTL 过期时间
- 没有 `consumed` 状态
- 没有“当前无可绑定轮次则拒绝入队”限制

这意味着用户可以：

- 连续 `/broll 察觉`
- 连续 `/broll 调查`
- 连续 `/broll 1d20`
- 一口气塞十几条暗骰进队列

然后等后续 prompt 一次性消费，或者在错误时机被消费。

这就是“无限暗骰”的最直接来源。

---

### 2. 暗骰队列项本身还不够强约束

文件：`RollHelper/src/types/eventDomainEvent.ts`

当前 `BlindGuidanceEvent` 只有这些核心字段：

- `rollId`
- `eventId`
- `eventTitle`
- `skill`
- `diceExpr`
- `total`
- `success`
- `resultGrade`
- `natState`
- `targetLabel`
- `rolledAt`
- `source`
- 可选 `sourceId`
- 可选 `note`

但它缺少很多“约束队列生命周期”所必须的字段，比如：

- `roundId`
- `sourceAssistantMsgId`
- `sourceFloorKey`
- `createdAt`
- `expiresAt`
- `consumed`
- `dedupeKey`
- `origin`
- `boundUserMsgId`（可选）

没有这些字段，后续就很难做：

- 同轮次限制
- 同内容去重
- 只消费一次
- 过期清理
- 精确拒绝旧楼层 / 旧轮次 / 旧来源

---

### 3. Prompt 注入阶段仍然是“整队全部消费”

文件：`RollHelper/src/events/promptEvent.ts`

当前暗骰注入逻辑依然是：

- 读取 `pendingBlindGuidanceQueue`
- `splice(0, queue.length)` 全部拿走
- 构建 `<dice_blind_guidance>`
- 存入 `outboundBlindGuidance`

这意味着它本质上还是一个“漂浮队列”。

如果前面阶段还没把绑定做严，那么第四阶段就更必须补：

- 过期项先清掉
- 非当前轮次先跳过
- 非当前楼层先跳过
- 已消费先跳过
- 队列超限先裁剪

否则即使你做了“每轮最多 1 次暗骰”，旧垃圾项不清掉，最终仍然会形成“看似有限，实际上堆积”的效果。

---

### 4. 事件暗骰虽然受“事件已结算”约束，但没有全局轮次配额约束

文件：`RollHelper/src/events/roundEvent.ts`

当前事件型暗骰相较 `/broll` 已经更安全，因为：

- 事件已经结算后不会再首次掷骰
- 同事件再次走 blind initial 的机会基本会被 existingRecord 拦住

但它仍缺少这些全局约束：

- 本轮暗骰总数上限
- 本轮每个技能暗骰次数上限
- 本轮同对象/同事件类型去重
- 当前轮次已达到限制时，按钮应禁用或拒绝

也就是说：

虽然“一个事件不会无限暗骰”，但“一个轮次里不断生成新事件并全都暗骰”仍然可能发生。

---

### 5. 目前“默认暗骰技能列表”没有真正承担限制职责

当前设置里已经有：

- `defaultBlindSkillsText`

但从当前代码职责看，它更偏向：

- 给交互触发默认暗骰用
- 给 prompt 协议提供建议用

它还没有真正作为“暗骰白名单”统一约束：

- `/broll` 是否允许
- 事件卡是否显示“暗骰检定”按钮
- 交互触发是否允许 blind=1

如果不把它纳入统一限制，暗骰仍会处于“形式上有默认技能，实际谁都能暗骰”的状态。

---

## 二、本阶段要达成的结果

本阶段完成后，暗骰应满足以下规则。

---

### 目标 A：每轮暗骰总数有限

例如：

- 每轮最多 1 次暗骰
- 或每轮最多 2 次暗骰

推荐默认：

- `1`

理由：

- 最稳
- 最容易形成“暗骰是特殊资源”的感觉
- 可以显著抑制刷暗骰

---

### 目标 B：队列长度有限

例如：

- `pendingBlindGuidanceQueue` 最大只保留 5 条

推荐默认：

- `5`

超出后应：

- 拒绝新增
- 或自动丢弃最旧项

推荐策略：

- **优先拒绝新增并给出提示**

因为对暗骰这类“会影响叙事”的东西来说，静默丢弃通常比显式拒绝更危险。

---

### 目标 C：同类暗骰不能短时间重复入队

例如同一轮内：

- `/broll 察觉`
- 再 `/broll 察觉`
- 再 `/broll 察觉`

不应全部进队。

至少要做到：

- 同轮次 + 同技能 + 同目标/标题 的暗骰，只允许保留 1 条

---

### 目标 D：暗骰有过期时间

例如：

- 创建后 3 分钟内没被消费，就自动过期
- 或轮次关闭后立即失效

推荐默认：

- **轮次关闭立即失效**
- 外加 **绝对 TTL 3 分钟**

这样双保险最好。

---

### 目标 E：不是所有技能/事件都允许暗骰

推荐规则：

- 只有命中暗骰白名单技能，才允许显示暗骰入口
- 或事件显式标记 `allowBlind=true` 时才允许

这样能把暗骰收敛到更合理的技能上，例如：

- 察觉
- 调查
- 洞察
- 潜行
- 搜索
- 历史

而不是任何事件都能暗骰。

---

### 目标 F：已经暗骰过的内容，不再重复暗骰

这里分两类：

#### 1）事件型暗骰

- 同一个事件一旦已经暗骰过
- 就不能再作为“首次暗骰”重新进入 blind queue
- 后续如果允许重投，也必须继承 blind，并且仍只算这个事件的一次暗骰链路

#### 2）`/broll` 手动暗骰

- 同轮次、同技能、同 dedupeKey 的暗骰，不允许反复 enqueue

---

## 三、推荐的整体策略

这一阶段我建议采用：

**“白名单 + 配额 + 去重 + TTL + 拒绝提示”**

五层叠加策略。

单独只做某一层都不够稳。

## 四、数据结构改造方案

## 4.1 `BlindGuidanceEvent` 扩展字段

文件：`RollHelper/src/types/eventDomainEvent.ts`

建议把 `BlindGuidanceEvent` 扩展成：

```ts
export interface BlindGuidanceEvent {
  rollId: string;
  eventId: string;
  eventTitle: string;
  skill: string;
  diceExpr: string;
  total: number;
  success: boolean | null;
  resultGrade: EventResultGradeEvent;
  natState: "nat1" | "nat20" | "none";
  targetLabel: string;
  rolledAt: number;
  source: EventRollSourceEvent;

  sourceId?: string;
  note?: string;

  roundId?: string;
  sourceAssistantMsgId?: string;
  sourceFloorKey?: string | null;
  origin?: "slash_broll" | "event_blind" | "interactive_blind";
  createdAt?: number;
  expiresAt?: number | null;
  consumed?: boolean;
  dedupeKey?: string;
}
```

### 每个字段的职责

#### `roundId`
用于统计：

- 本轮暗骰次数
- 同轮去重
- 轮次关闭失效

#### `sourceAssistantMsgId`
用于绑定：

- 来源助手楼层
- swipe/edit/delete 后失效

#### `sourceFloorKey`
用于快速比较：

- 是否仍属于当前楼层

#### `origin`
用于区分来源：

- `/broll`
- 事件暗骰
- 交互触发暗骰

后面做不同限制时很好用。

#### `createdAt / expiresAt`
用于 TTL 清理。

#### `consumed`
用于防止重复消费。

#### `dedupeKey`
用于快速判定“同类暗骰是否已存在”。

## 五、核心实现方案

---

## 5.1 `roundEvent.ts`：补一组统一的暗骰队列工具函数

文件：`RollHelper/src/events/roundEvent.ts`

建议新增以下工具函数。

---

### 5.1.1 清理过期暗骰队列

```ts
function pruneExpiredBlindGuidanceQueueEvent(meta: DiceMetaEvent, now = Date.now()): boolean
```

职责：

- 移除 `consumed=true` 的项
- 移除 `expiresAt <= now` 的项
- 移除其 `roundId` 已不再是 open pendingRound 的项

返回：

- 是否发生了实际清理

---

### 5.1.2 统计当前轮次暗骰次数

```ts
function countBlindRollsInRoundEvent(round: PendingRoundEvent): number
```

统计范围：

- `round.rolls` 里 `visibility === "blind"`
- 不区分来源（事件型、交互型都算）

说明：

`/broll` 如果还没真正写入 `round.rolls`，至少应按 queue 项补统计。

---

### 5.1.3 构建 dedupeKey

```ts
function buildBlindGuidanceDedupKeyEvent(input: {
  roundId?: string;
  skill?: string;
  eventId?: string;
  targetLabel?: string;
  sourceFloorKey?: string | null;
  origin?: string;
}): string
```

推荐策略：

#### 事件型暗骰

优先：

- `event:${roundId}:${eventId}`

#### `/broll`

优先：

- `slash:${roundId}:${skill}:${targetLabel || "_"}`

#### 交互触发暗骰

优先：

- `interactive:${roundId}:${sourceFloorKey}:${sourceId}`

这样能避免不同来源互相覆盖。

---

### 5.1.4 判断是否允许 enqueue 暗骰

```ts
function canEnqueueBlindGuidanceEvent(args: {
  meta: DiceMetaEvent;
  settings: DicePluginSettingsEvent;
  round: PendingRoundEvent | null;
  dedupeKey: string;
  origin: BlindGuidanceEvent["origin"];
  now?: number;
}): { ok: boolean; reason?: string }
```

判断顺序建议：

1. 先清理过期项
2. 没有 open round 就拒绝
3. 当前轮次暗骰数已到上限则拒绝
4. 当前 queue 长度已到上限则拒绝
5. 命中 dedupeKey 则拒绝
6. 否则允许

---

### 5.1.5 安全 enqueue

```ts
function enqueueBlindGuidanceSafeEvent(args: { ... }): { ok: boolean; reason?: string }
```

职责：

- 内部统一设置 `createdAt / expiresAt / dedupeKey / consumed=false`
- 统一 push
- 统一保存前的最终校验

这样 `/broll`、事件暗骰、交互暗骰都走同一套规则。

---

## 5.2 `baseRollCommandEvent.ts`：给 `/broll` 增加真正的限制

这是第四阶段最关键的修改点。

当前 `/broll` 是直接 push 队列。

建议改为：

### 第一步：必须找到当前可绑定轮次

从 `getDiceMetaEvent()` 中读取：

- `pendingRound`

要求：

- 必须存在
- 必须 `status === "open"`
- 必须有 `sourceAssistantMsgIds`

如果没有，则直接拒绝入队：

推荐提示：

`当前没有可绑定的最新轮次，不能把 /broll 注入后续叙事。请先等待新的事件轮次，或使用普通 /roll。`

> 这一点会和第二阶段更自然衔接。

---

### 第二步：命中暗骰白名单才允许 `/broll` 技能模式入队

如果用户输入的是技能名而不是骰式：

- 技能必须命中暗骰白名单
- 否则拒绝作为 blind guidance 入队

推荐实现：

复用 `defaultBlindSkillsText`，把它真正作为“暗骰允许技能列表”之一。

推荐提示：

`技能「xxx」当前不在允许暗骰列表中。`

如果是纯骰式 `/broll 1d20`：

- 可以允许，但建议只在 open round 存在时才入队

---

### 第三步：调用统一的 `canEnqueueBlindGuidanceEvent`

如果被拒绝：

- 控制台明确提示原因
- 不入队
- 仍可选择是否保留本地“仅自己知道的掷骰结果”

推荐第一版：

- **直接拒绝，不入队，也不生成 blind console result card**
- 提示用户为什么被拒绝

这样最清楚。

---

### 第四步：生成完整的 BlindGuidanceEvent

应补齐：

- `roundId`
- `sourceAssistantMsgId`
- `sourceFloorKey`
- `origin: "slash_broll"`
- `createdAt`
- `expiresAt`
- `consumed: false`
- `dedupeKey`

---

## 5.3 `roundEvent.ts`：事件暗骰也要接入限制器

当前事件暗骰 enqueue 是通过 `enqueueBlindGuidanceFromRecordEvent()`。

建议把这里也改成走统一安全 enqueue。

### 要点

- 不是简单 `queue.some(rollId)` 去重
- 而是按 `dedupeKey` + 本轮限制 + queue 限制统一判断

### 推荐 dedupeKey

#### 事件型暗骰

```ts
`event:${record.roundId}:${event.id}`
```

这样同一个事件不会重复 enqueue。

### 达到上限时怎么处理

推荐：

- **允许结算，但不入 blind guidance queue**
- 并在日志中提示：
  - 本轮暗骰引导已达上限，结果不会再进入下一次叙事注入

这样不会把掷骰本身弄坏，但能把叙事注入管住。

---

## 5.4 `promptEvent.ts`：消费前先 prune，再按条件过滤

即使第二阶段已经把绑定关系做得更严，第四阶段仍建议在消费前做一次统一清理。

### 在 `resolvePromptBlindGuidanceInjectionEvent()` 开头增加：

1. prune expired
2. 过滤 `consumed`
3. 过滤 `expiresAt`
4. 过滤不属于当前 open round 的旧项
5. 过滤来源楼层已经失效的项（依赖第二阶段字段）

然后再构建 block。

### 消费方式建议改为：

不是裸 `splice(0, queue.length)`，而是：

- 把可消费项拿出来
- 标记 `consumed = true`
- 再统一清理

这样更利于 debug，也更稳。

---

## 5.5 `renderEvent.ts`：按钮展示也要受限制

这一阶段不仅要改“后台能不能入队”，还要改“前台是不是还给你随便点”。

推荐规则：

### 事件列表中显示“暗骰检定”按钮前，需同时满足：

1. `enableBlindRoll === true`
2. 当前轮次仍 open
3. 本轮暗骰次数未超上限
4. 当前事件技能命中暗骰允许列表，或事件显式 `allowBlind=true`
5. 当前事件尚未存在 blind 记录

如果不满足：

- 不渲染按钮
- 或渲染 disabled 态并给 tooltip

推荐第一版：

- **直接不显示按钮**

这样最简洁。

## 6.1 设置页新增项

建议新增一个分组：

### 暗骰限制

包含：

#### 1）每轮暗骰上限

标题：

`每轮暗骰上限`

说明：

`限制每个未结束轮次最多允许多少次暗骰。达到上限后，新暗骰不会再进入叙事引导队列。`

默认：`1`

---

#### 2）暗骰队列上限

标题：

`暗骰队列上限`

说明：

`限制待注入的暗骰引导最多保留多少条。队列满时将拒绝新增，避免旧暗骰堆积。`

默认：`5`

---

#### 3）暗骰过期时间（秒）

标题：

`暗骰过期时间（秒）`

说明：

`暗骰结果在超过该时长后失效，不再注入后续 prompt。轮次关闭后也会提前失效。`

默认：`180`

---

#### 4）启用暗骰去重

标题：

`启用暗骰去重`

说明：

`同轮次内重复的同类暗骰只保留一次，避免刷同技能或同事件。`

默认：开启

---

## 6.2 白名单文案建议

当前 `defaultBlindSkillsText` 建议文案改成更明确的版本：

### 标题

`允许暗骰的技能列表`

### 描述

`只有命中此列表的技能，才允许显示暗骰按钮或作为 /broll 技能模式写入叙事引导。每行、逗号或竖线分隔。`

这样它就从“默认暗骰技能”升级成“默认 + 允许暗骰技能白名单”。

## 七、推荐的拒绝提示文案

这一阶段体验上非常重要的一点，是**被拒绝时要说明原因**。

推荐统一提示：

### 情况 1：没有 open round

`当前没有可绑定的最新轮次，暗骰不会进入后续叙事。请等待新一轮事件，或改用普通 /roll。`

### 情况 2：本轮上限已满

`本轮暗骰次数已达到上限，新的暗骰不会再进入叙事引导。`

### 情况 3：队列已满

`待处理暗骰队列已满，请先推进对话消费已有暗骰。`

### 情况 4：命中去重

`相同内容的暗骰已存在，本轮无需重复暗骰。`

### 情况 5：技能不在白名单

`技能「xxx」当前不允许作为暗骰使用。`

## 9.1 `/broll` 限制测试

### 用例 1：没有 open round

操作：

- 在没有当前待处理事件轮次时执行 `/broll 察觉`

预期：

- 被拒绝
- 不入 `pendingBlindGuidanceQueue`
- 给出明确提示

---

### 用例 2：本轮第一次 `/broll`

操作：

- 存在 open round
- 执行 `/broll 察觉`

预期：

- 成功入队
- queue 长度 +1
- 补齐 `roundId / createdAt / expiresAt / dedupeKey`

---

### 用例 3：同轮重复 `/broll 察觉`

操作：

- 再执行一次 `/broll 察觉`

预期：

- 命中 dedupe
- 不重复入队
- 给出提示

---

### 用例 4：超出本轮上限

操作：

- `maxBlindRollsPerRound = 1`
- 本轮已经有 1 次暗骰
- 再执行新的 `/broll`

预期：

- 被拒绝
- queue 不增长

---

- 本轮未超上限
- 点击“暗骰检定”

预期：

## 十、完成标准

满足以下条件即可视为第四阶段完成：

- [ ] `/broll` 不再无条件 push 队列
- [ ] 暗骰必须绑定 open round 才能进入叙事引导
- [ ] 每轮暗骰总数存在上限
- [ ] `pendingBlindGuidanceQueue` 存在最大长度
- [ ] 同类暗骰存在 dedupe 机制
- [ ] 暗骰存在 TTL 或轮次关闭失效机制
- [ ] 不允许技能白名单外的内容随便暗骰
- [ ] 同一事件不会重复进入 blind queue
- [ ] 超限 / 去重 / 失效时有明确提示

---

## 十一、这一阶段完成后的效果

完成后，暗骰系统会从现在这种：

- 可以随便 `/broll`
- 可以反复刷同一技能
- 可以积压很多旧暗骰
- 可以拖到后面随机消费

变成：

- **暗骰是有限资源**
- **暗骰必须绑定当前轮次**
- **同类暗骰不会无限堆积**
- **旧暗骰不会无限滞留**
- **不适合暗骰的技能和事件不会乱开口子**

这会明显提升系统的稳定性和玩法感。
