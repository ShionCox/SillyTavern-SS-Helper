# 基于 Commit `be8c704` 的冷启动与 AI 总结实施方案

> 文档定位：本方案只围绕 **冷启动** 与 **AI 总结** 两个方向展开，不包含检索图谱、知识图谱等扩展能力设计。

---

## 1. 本轮实施范围

本轮只做两件事：

1. **冷启动（Cold Start）**
   - 新聊天首次进入时，基于角色卡、世界书、开场上下文，生成一批“基础记忆候选”
   - 用户可选择是否采纳、筛除、补充
   - 目标是建立一个**干净、可控、可解释**的初始记忆底座

2. **AI 总结（AI Summary / Memory Mutation）**
   - 当对话达到设定楼层后，自动对“尚未总结的区间”进行结构化总结
   - 总结结果不是直接覆盖存量记忆，而是以**增删改动作**形式作用于记忆库
   - 目标是让记忆更新从“写一段总结文本”升级为“对结构化记忆做审慎变更”

---

## 2. 本轮核心目标

当前要解决的，不是“记忆功能有没有”，而是“记忆更新是否可靠”。

### 2.1 冷启动阶段的目标

冷启动不是简单让 AI 看一遍世界书然后自由发挥，而是要做到：

- 能区分哪些内容适合进入长期记忆
- 能让用户决定哪些世界书条目参与初始化
- 能把初始化结果落成**结构化候选项**
- 能避免一上来就把模糊设定、临时剧情、语气描述全部塞进长期记忆

### 2.2 AI 总结阶段的目标

AI 总结不能只是输出“本段对话发生了什么”，而要做到：

- 知道当前已有哪些记忆
- 知道应该新增、更新、合并、失效还是不动
- 知道如何以 JSON 动作协议返回结果
- 知道不能因为一句临时玩笑就覆盖长期设定

---

## 3. 当前问题判断

结合你的设计思路，这一轮最需要修的不是 UI，而是**记忆更新协议**。

### 3.1 冷启动问题

当前常见风险：

- 世界书与角色卡信息量很大，但并非全部都该进长期记忆
- AI 容易把叙述性句子、规则性句子、剧情提示混在一起抽取
- 没有分类层时，初始化结果会非常杂
- 没有“候选态”时，冷启动会把脏数据直接写入记忆库

### 3.2 AI 总结问题

当前常见风险：

- 总结直接写结果，没有“先计划再修改”的中间层
- AI 不知道现有记忆长什么样，因此容易重复新增
- AI 看不到已有冲突记录，容易用新一句话错误覆盖旧事实
- 总结只输出自然语言，后续系统无法准确执行增删改

### 3.3 上下文供给问题

最大的技术问题不是“AI 会不会总结”，而是：

**你到底给 AI 看什么，给到什么程度。**

如果每次都把全部记忆喂进去，会出现：

- token 成本失控
- 注意力发散
- 重复率上升
- 模型更容易错把旧无效信息当成当前有效信息

所以本轮方案里，一个关键原则是：

> **AI 不能直接看全库，只能看“与当前总结区间相关的候选存量记忆”。**

## 4. 总体方案概览

本轮把记忆更新拆成两条线：

### 4.1 冷启动主线

```text
识别是否首次聊天
    ↓
弹出冷启动界面
    ↓
自动加载角色卡关联世界书条目
    ↓
用户勾选要参与初始化的条目
    ↓
AI 生成结构化“初始记忆候选”
    ↓
本地规则去重 / 分类 / 校验
    ↓
用户确认后落库
```

### 4.2 AI 总结主线

```text
消息楼层达到阈值
    ↓
定位“未总结区间”
    ↓
召回与该区间相关的已有记忆候选
    ↓
Summary Planner 决定本轮应该动哪些记忆
    ↓
Summary Mutation 输出结构化变更动作 JSON
    ↓
本地校验动作合法性
    ↓
写入记忆库 + 记录变更日志
```

## 5. 冷启动实施设计

## 5.1 冷启动触发条件

建议触发条件如下：

- 当前聊天为新会话
- 当前会话没有完成过冷启动
- 用户未手动关闭冷启动提示
- 当前角色卡存在可读取的角色资料 / 世界书 / 设定条目

建议在会话级状态中记录：

```ts
interface ColdStartState {
  hasStarted: boolean
  hasCompleted: boolean
  selectedLorebookEntryIds: string[]
  generatedAt?: number
  confirmedAt?: number
  dismissedAt?: number
}
```

这样可以区分：

- 已经弹过但没做完
- 已经做过
- 用户只是临时关闭
- 用户更换世界书后是否需要重新初始化

---

## 5.2 冷启动数据输入来源

冷启动不要直接扫全局所有资料，建议只取以下来源：

### 必选来源

- 当前角色卡的基础信息
- 当前聊天的开场消息 / 初始设定消息
- 当前角色关联的世界书条目（由用户选择启用）

### 可选来源

- 用户手动补充的角色背景
- 用户手动输入的“初始关系 / 初始目标 / 初始禁忌”

### 不建议直接纳入冷启动长期记忆的内容

- 纯风格化文案
- 情绪性修辞
- 未被定义为事实的剧情氛围句
- 一次性说明文字
- 明显是作者给模型的控制提示而非世界事实的句子

---

## 5.3 冷启动记忆分类

冷启动阶段，建议先分类，再生成候选。

推荐最少保留以下分类：

```ts
type MemoryType =
  | 'character_profile'   // 人物基础事实
  | 'relationship'        // 关系
  | 'preference'          // 偏好
  | 'world_rule'          // 世界规则
  | 'identity_constraint' // 身份/边界/禁忌
  | 'persistent_goal'     // 长期目标
  | 'initial_state'       // 初始状态
  | 'location_fact'       // 地点事实
  | 'timeline_fact'       // 时间线事实
```

### 分类原则

#### `character_profile`
适合写入：

- 姓名、身份、年龄段、职业、种族、阵营、外貌常量

不适合写入：

- 一时表情、一次动作、某次对话中的临时语气

#### `relationship`
适合写入：

- 谁和谁是什么长期关系
- 主从、亲属、同伴、敌对、师徒、搭档等

不适合写入：

- 一次短时冲突就改写长期关系

#### `preference`
适合写入：

- 长期偏好、稳定喜恶、习惯、口头癖

不适合写入：

- 当前回合一句随口表达

#### `world_rule`
适合写入：

- 世界法则、能力机制、组织规则、硬性背景设定

不适合写入：

- 剧情提示语、作者注释

#### `identity_constraint`
适合写入：

- 不能公开的身份
- 绝不能违背的原则
- 触发禁忌
- 身份保密条件

#### `initial_state`
适合写入：

- 开局所处地点
- 当前身份状态
- 起始持有物 / 起始目标 / 当前任务挂载

这个类型可以允许后续失效或被替换。

## 5.4 冷启动输出形态

冷启动不要直接输出“最终记忆”，而应输出“候选记忆”。

建议结构：

```ts
interface MemoryCandidate {
  id: string
  type: MemoryType
  title: string
  summary: string
  entityKeys: string[]
  confidence: number
  sourceRefs: SourceRef[]
  status: 'candidate'
  reason: string
}
```

其中：

- `title`：一眼能识别的短标题
- `summary`：结构化短摘要
- `entityKeys`：便于后续检索和去重
- `confidence`：给后续自动筛选或排序使用
- `sourceRefs`：告诉系统它来自哪里
- `reason`：解释为什么认为这条应进入长期记忆

### 推荐的冷启动结果机制

- 先生成候选
- 再进行本地去重和合法性检查
- 再由用户确认是否入库

这样能把“AI 抽错了”的风险压住。

---

## 5.5 冷启动去重与校验

冷启动最怕“一次初始化就塞入几十条相似垃圾”。

建议做三层防护：

### 第一层：格式校验

检查字段是否完整：

- `type` 是否合法
- `summary` 是否为空
- `entityKeys` 是否为空
- `confidence` 是否在合法范围
- `sourceRefs` 是否存在

### 第二层：相似项去重

可以按以下特征做近似合并：

- `type` 相同
- `entityKeys` 高度重合
- `summary` 语义相近
- 来源相同或高度重叠

如果发现多个候选只是在措辞上不同，应合并为一条。

### 第三层：禁入规则

以下类型的候选不建议直接进入长期记忆：

- 描述太模糊
- 无主体对象
- 无法定位实体
- 明显是剧情提示而非设定事实
- 明显是一次性临时状态，却被错误标为长期事实

## 6. AI 总结实施设计

## 6.1 总结触发逻辑

AI 总结建议按“楼层区间”触发，而不是每条消息都跑。

建议配置项：

```ts
interface SummarySettings {
  enabled: boolean
  triggerInterval: number      // 多少楼触发一次
  minMessages: number          // 最少消息数
  recentWindowSize: number     // 额外上下文窗口
  plannerModel?: string
  mutationModel?: string
}
```

### 触发条件建议

- 会话消息数达到最小值
- 距上次总结后新增消息数达到阈值
- 当前未在运行总结任务
- 当前区间未被总结过

### 总结区间定位建议

维护一个状态：

```ts
interface SummaryProgress {
  lastSummarizedMessageId?: string
  lastSummarizedIndex: number
  pendingStartIndex: number
  pendingEndIndex: number
}
```

这样每次不是从头总结，而是只总结“未总结区间”。

---

## 6.2 总结流程改为两段式

这是本轮最重要的结构设计之一。

### 第一段：`Summary Planner`

Planner 的职责不是直接改记忆，而是回答三个问题：

1. 本轮是否值得修改记忆
2. 该修改哪些类别
3. 应该召回哪些已有记忆作为候选对照项

Planner 输出应该尽量轻量，例如：

```json
{
  "should_update": true,
  "focus_types": ["relationship", "initial_state", "persistent_goal"],
  "entities": ["user", "character_name", "organization_x"],
  "topics": ["任务进展", "关系变化", "地点切换"],
  "reasons": [
    "本区间出现稳定关系变化",
    "出现明确地点迁移",
    "旧目标已完成，需更新长期目标"
  ]
}
```

### 第二段：`Summary Mutation`

Mutation 才负责真正输出结构化增删改动作。

这样做的好处是：

- Planner 决定“看什么”
- Mutation 决定“怎么改”
- 可以把召回逻辑插在两者之间
- 可以降低“AI 一次性胡乱覆盖”的概率

---

## 6.3 给 AI 的上下文供给策略

这是本方案的关键控制点。

### 绝对不建议

每次总结都把全量记忆库直接喂给 AI。

### 建议给 AI 的内容

只给以下 4 类内容：

1. **未总结对话区间**
   - 本次真正要处理的消息

2. **最近滚动摘要**
   - 上一轮或最近几轮总结出来的短摘要
   - 让模型知道前情，但不用重读全历史

3. **候选已有记忆 Top-K**
   - 通过实体、主题、关键词、时间近邻召回
   - 只给最相关的若干条

4. **记忆动作协议**
   - 明确告诉 AI 只能做哪些动作
   - 以及每种动作适用于什么场景

---

## 6.4 候选已有记忆的“短卡片”机制

为了避免全库塞给 AI，建议引入“候选记忆短卡片”。

```ts
interface MemoryCard {
  id: string
  type: MemoryType
  title: string
  summary: string
  entityKeys: string[]
  status: 'active' | 'invalidated' | 'merged' | 'archived'
  updatedAt: number
  sourceHint?: string
}
```

给 AI 看的是这种短卡片，而不是完整记录。

### 为什么短卡片更合适

- token 低
- 重点清晰
- 模型更容易比对“这是同一条还是新的一条”
- 不会把大量无关元数据灌进去

### 候选召回建议规则

至少结合以下维度：

- 实体重合
- 类型匹配
- 主题相似
- 时间近邻
- 最近被更新过
- 当前区间中频繁出现的关键词

---

## 6.5 AI 总结动作协议

本轮最关键的落地点就是：**别再让 AI 只写总结文本，要让它输出“变更动作”**。

建议支持以下动作：

```ts
type MemoryMutationAction =
  | 'ADD'
  | 'UPDATE'
  | 'MERGE'
  | 'INVALIDATE'
  | 'DELETE'
  | 'NOOP'
```

下面分别说明。

---

### 6.5.1 `ADD`

含义：新增一条全新的记忆记录。

适用场景：

- 当前区间出现了明确且稳定的新事实
- 候选已有记忆中没有高相似对象
- 该内容值得进入长期记忆

示例：

- 新建立的长期关系
- 明确出现的新长期目标
- 新揭示的身份设定

不适用场景：

- 临时情绪
- 一次性动作
- 模糊猜测

---

### 6.5.2 `UPDATE`

含义：更新已有记忆的一部分字段，但保留其身份不变。

适用场景：

- 旧记忆存在，但内容被进一步明确
- 属于同一实体、同一事实槽位的补充或修正

示例：

- 旧记录只写“角色对用户有戒备”，新区间明确成“信任提升但仍保持防备”
- 旧目标状态从“进行中”变成“已完成”

建议 `UPDATE` 用 patch 形式，而不是整条覆盖。

---

### 6.5.3 `MERGE`

含义：将两条或多条重复、近重复的记忆记录合并。

适用场景：

- 历史上已经因为多次总结形成重复项
- 不同措辞描述的是同一事实

示例：

- “她讨厌甜食”
- “她不喜欢过甜的食物”

如果系统判定它们表达的是同一长期偏好，应优先合并。

---

### 6.5.4 `INVALIDATE`

含义：将一条曾经有效的记忆标为失效，而不是直接删除。

适用场景：

- 旧事实曾经成立，但现在不再成立
- 需要保留轨迹用于追溯

示例：

- 旧地点状态不再有效
- 某阶段性任务已经完成
- 某个临时伪装身份已解除

这是非常重要的动作，因为很多状态类信息不应该直接粗暴删除。

---

### 6.5.5 `DELETE`

含义：删除明显错误、噪声型、无价值的记录。

适用场景：

- 明显是误抽取
- 明显重复且已被 `MERGE`
- 明显不该进入长期记忆

注意：

`DELETE` 应比 `INVALIDATE` 更谨慎，最好只用于脏数据清理，而不是剧情状态变化。

---

### 6.5.6 `NOOP`

含义：本轮不需要修改任何长期记忆。

适用场景：

- 当前区间只是闲聊
- 没有形成稳定新事实
- 有变化但不足以进入长期记忆
- 已有记忆已经能覆盖本轮信息

这个动作非常重要，因为它能避免“每次总结都硬写几条”。

---

## 6.6 推荐 JSON 返回结构

建议让 Mutation 输出统一结构：

```json
{
  "version": "1.0",
  "actions": [
    {
      "action": "UPDATE",
      "target_id": "mem_rel_001",
      "type": "relationship",
      "title": "角色与用户的关系状态",
      "reason": "本区间多次出现持续性的信任提升信号，且与既有关系记录高度相关",
      "confidence": 0.84,
      "patch": {
        "summary": "角色对用户的态度由高度戒备转为谨慎信任，但仍保留防备边界",
        "status": "active"
      }
    },
    {
      "action": "ADD",
      "type": "persistent_goal",
      "title": "寻找失踪线索",
      "reason": "该目标在当前区间被明确提出，并具备后续持续性",
      "confidence": 0.78,
      "new_record": {
        "summary": "角色当前形成了持续性目标：追查失踪事件线索",
        "entityKeys": ["character_name", "失踪事件"],
        "status": "active"
      }
    }
  ]
}
```

## 6.7 本地校验层

AI 输出 JSON 不代表就能直接写库。

建议在写库前做本地 Validator：

### 校验点 1：动作合法性

- `action` 是否属于白名单
- `target_id` 是否存在
- `type` 是否合法
- `confidence` 是否在范围内

### 校验点 2：动作与字段匹配

例如：

- `ADD` 必须有 `new_record`
- `UPDATE` 必须有 `target_id` 和 `patch`
- `MERGE` 必须有多个 `source_ids`
- `INVALIDATE` 必须有 `target_id`
- `NOOP` 不应附带无关字段

### 校验点 3：危险动作降级

以下情况建议拒绝或降级：

- 单轮总结试图删除大量记录
- 同一批动作里同时对同一条记录做冲突操作
- 低置信度却试图覆盖核心身份信息
- 没有证据链却修改高优先级世界规则

## 7. 数据结构建议

## 7.1 记忆记录结构

```ts
interface MemoryRecord {
  id: string
  type: MemoryType
  title: string
  summary: string
```

## 9. 给 AI 的 Prompt 设计原则

这一部分不是要你把 prompt 写成很长的自然语言说明，而是要把它写成**协议文档**风格。

推荐原则：

### 9.1 先告诉 AI 它的职责边界

例如：

- 你不是故事续写器
- 你不是普通摘要器
- 你是记忆变更分析器
- 你只能基于给定对话区间与候选记忆做判断
- 你不能虚构未出现的长期事实

### 9.2 先定义动作语义，再给输入

顺序应该是：

1. 你可以做哪些动作
2. 每个动作在什么情况下使用
3. 哪些内容绝不能进入长期记忆
4. 现在给你本轮材料，请输出合法 JSON

这样比先塞大量上下文再让 AI“自由总结”稳定得多。

### 9.3 强制输出模式

建议：

- schema 放在 SYSTEM
- 明确禁止输出解释性废话
- 明确 JSON 不允许多余字段
- 明确 `NOOP` 是合法答案

## 10. 借鉴其他记忆插件时建议吸收的点

这一部分只说“适合融入的设计思想”，不假设你要照搬别人的实现。

### 10.1 借鉴点一：模板先行，而不是自由抽取

优点：

- 能降低 AI 输出漂移
- 更容易控制记忆类型
- 更容易做校验和后处理

### 10.2 借鉴点二：结构化编辑，而不是整段重写

优点：

- 不会因为一句新话把旧记录全部洗掉
- 方便做 `UPDATE / MERGE / INVALIDATE`

### 10.3 借鉴点三：召回候选，而不是喂全库

优点：

- 成本低
- 准确性更高
- 重复率更低

### 10.4 借鉴点四：把 prompt 当作协议，而不是聊天说明

优点：

- 结果更稳定
- 更容易配合 schema
- 更容易扩展成多阶段流程

## 11. 验收标准

本轮是否成功，不是看“跑没跑起来”，而是看质量。

建议至少看以下指标：

### 11.1 冷启动质量

- 候选记忆中，低价值脏数据比例是否下降
- 用户手动删除率是否降低
- 同类重复项是否明显减少

### 11.2 AI 总结质量

- 重复新增率是否下降
- 对已有记忆的错误覆盖率是否下降
- `NOOP` 使用是否合理
- `INVALIDATE` 是否能替代一部分粗暴删除

### 11.3 成本控制

- 每轮总结 token 是否可控
- 候选记忆条数是否稳定在合理范围
- 总结速度是否可接受

### 11.4 可维护性

- 有没有清晰的变更日志
- 能不能回滚
- 能不能定位是哪轮总结引入了脏数据

## 12. 风险与保护措施

## 12.1 风险：AI 误把一次性剧情写成长期事实

保护措施：

- 明确哪些内容不能进入长期记忆
- 低置信度候选只进候选区，不直接入库
- 对 `persistent_goal`、`relationship`、`world_rule` 等高价值类型提高入库门槛

## 12.2 风险：AI 过度删除

保护措施：

- 优先使用 `INVALIDATE`
- `DELETE` 只对脏数据开放
- 一轮中大批量删除必须二次确认或规则拦截

## 12.3 风险：AI 重复新增

保护措施：

- 引入候选记忆短卡片
- 在 Planner 阶段先定位实体和主题
- 在 Mutation 阶段要求说明“为何不是更新已有记录”

## 12.4 风险：不同轮总结互相冲突

保护措施：

- 记录 mutation log
- 对高优先级事实启用保护
- 新冲突先标记待确认，不立即覆盖

## 14. 最终建议

如果只说一句本轮最值得先落的东西，那就是：

> **先把“AI 总结 = 结构化变更动作”这件事做实，再去扩充冷启动和更高级检索。**

因为只要动作协议没定住，后面不管你加多少 UI、多少召回、多少世界书选项，最后都会回到同一个问题：

**AI 输出不可控，记忆库会越积越脏。**

所以你的优先级建议是：

1. 先定动作协议与 JSON schema  
2. 再补候选已有记忆短卡片  
3. 再补冷启动候选流  
4. 最后再做合并、失效、冲突生命周期管理
