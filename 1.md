# MemoryOS 梦境系统一次性收口修改方案

## 目标

本次修改以**一次性处理完当前梦境系统剩余主问题**为目标，不再分阶段打补丁。  
修改完成后，应满足以下结果：

1. **Scheduler / Lock / Session 三条状态链使用同一个真实 dreamId。**
2. **待审批梦境从 Runtime Task Pill / Workbench 打开后，审批结果能够真正落库、应用、拒绝、刷新 UI，而不是只弹窗不生效。**
3. **第三阶段 UI 统一由 `DreamUiStateSnapshot` 驱动运行态、待审批入口、最近结果，不再多处各自推导。**
4. **静默模式 / 手动审批模式语义统一，自动轻梦境的状态展示不再误导。**
5. **Repository / UI / Runtime 的命名与语义一致，减少后续维护歧义。**

---

## 本次必须解决的问题清单

### 问题 1：`activeDreamId` 不是实际 dream session 的 `dreamId`

当前 `DreamSchedulerService` 抢锁时写入的 `dreamId` 是运行占位值：

- `dream_run:${chatKey}:${Date.now()}`

而 `DreamingService.startDreamSession()` 真正创建的会话 ID 是：

- `dream:${chatKey}:${crypto.randomUUID()}`

这会导致：

- `schedulerState.activeDreamId` 无法稳定映射到真实 session
- `DreamUiStateService` 无法可靠定位当前活跃 dream
- 后续待审批、最近结果、pill 点击跳转都只能“猜”当前会话

### 问题 2：待审批 dream 从 Runtime / Workbench 打开审核后，并没有真正应用审批结果

当前链路里：

- `DreamingService.startDreamSession()` 在 manual 触发时会直接拿 `openReviewDialog()` 返回值做写回
- 但 pending dream 通过 Runtime / Workbench 打开时，只是再次调用 `openDreamReviewDialog()` 拿到一个 `DreamReviewDecision`
- **这个 `DreamReviewDecision` 没有进入正式“应用审批”的服务链**
- 结果就是：
  - 用户可以看审批弹窗
  - 也能点通过/拒绝
  - 但很多场景下只是 toast，未真正落库 / 未真正 apply / 未真正更新 approval 状态

这是当前最优先要修的主问题之一。

### 问题 3：第三阶段 UI 虽然有 `DreamUiStateService`，但主界面仍是“双轨状态源”

当前已经有：

- `dream-ui-state-service.ts`
- `dream-notification-service.ts`
- `dream-task-pill.ts`

但 Workbench 和总览区仍然主要直接消费：

- `sessions`
- `maintenanceProposals`
- `qualityReports`
- `schedulerState`

而不是统一吃 `DreamUiStateSnapshot`。

结果：

- 顶部总览、待审批入口、最近结果、task pill 不是同一个状态源
- 某些 phase 逻辑在一个地方成立，在另一个地方不成立
- `resolveUiPhase()` 已支持 `waiting_approval`，但调用方没有把 approval 正确传进去

### 问题 4：`DreamUiStateService.resolveUiPhase()` 的 `approval` 参与不完整

当前 `resolveUiPhase(meta, approval, lock, schedulerState)` 已经支持：

- `waiting_approval`
- `failed`
- `rolled_back`
- `completed`

但 `getSnapshot()` 在调用时没有把当前 active dream 的 approval 真实传进去，而是直接传 `null`。

这会导致：

- 有 pending dream 时，phase 无法准确落到 `waiting_approval`
- pill / runtime 状态卡片的展示粒度不足
- “当前 dream 正在等待审核”无法作为统一活跃状态被准确识别

### 问题 5：`getLatestCompletedSession()` 的语义偏松

当前实现会把：

- `approved`
- 以及某些 `generated`

都视作“已完成”。

这在数据层语义上不够严谨，因为：

- `generated` 可能仍是 `approval.status = pending`
- 这类 dream 不该被视为真正 completed

这会让后续调用者误用 repository 方法。

### 问题 6：静默模式下 `approval.status = 'approved'` 的语义过于乐观

当前 `auto_light` 分支中，即使：

- mutation 全部未应用
- maintenance 也可能一个都没自动落地

仍会写：

- `DreamSessionApprovalRecord.status = 'approved'`
- `approvalMode = 'auto_silent'`

这会造成“系统流程完成”与“内容已真正批准落地”混在一起。

### 问题 7：`runtime-app.ts` 有重复定义的 `syncAuxiliaryUi()`

当前 `MemoryOS` 类里存在重复的 `syncAuxiliaryUi()` 方法定义。  
虽然运行时可能以后一个覆盖前一个，但这属于明显应清理的问题，会增加维护噪音，也容易让后续合并时出问题。

---

# 一次性修改策略

本次不建议继续在现有代码上“点修补”。  
建议采用：

## 总策略

1. **把“pending 审批应用”抽成正式服务能力**
2. **把真实 `dreamId` 提前生成并贯穿 scheduler → lock → session**
3. **让第三阶段顶部状态区统一改吃 `DreamUiStateSnapshot`**
4. **保留现有 Workbench 列表页的数据读取方式，但把“总览态 / 入口态 / 当前态”统一**
5. **补齐 SDK 门面，让 Runtime / Workbench 都通过同一个 API 应用 pending 审批**

---

# 详细修改方案

## 一、统一真实 `dreamId` 链路

### 目标

让以下对象中的 dreamId 完全一致：

- Lock
- SchedulerState.activeDreamId
- DreamSessionMeta.dreamId
- DreamSessionOutput / Approval / Rollback / QualityReport 的 dreamId
- UI snapshot 中 activeTask.dreamId
- pending review dialog 打开后的目标 dreamId

### 修改文件

- `MemoryOS/src/services/dream-scheduler-service.ts`
- `MemoryOS/src/services/dreaming-service.ts`
- `MemoryOS/src/services/dream-types.ts`

### 具体改法

#### 1）修改 `DreamSchedulerJob.execute()` 的上下文，显式传入真实 dreamId

当前：

```ts
execute: (context: {
    holderId: string;
    executionMode: DreamExecutionMode;
    triggerSource: Exclude<DreamTriggerReason, 'manual'>;
}) => Promise<{ ok: boolean; status?: string; reasonCode?: string }>;
```

改为：

```ts
execute: (context: {
    holderId: string;
    executionMode: DreamExecutionMode;
    triggerSource: Exclude<DreamTriggerReason, 'manual'>;
    dreamId: string;
}) => Promise<{ ok: boolean; dreamId?: string; status?: string; reasonCode?: string }>;
```

#### 2）在 scheduler 真正执行前预生成真实 dreamId

在 `processQueue()` 里不要再用：

```ts
dreamId: `dream_run:${job.chatKey}:${Date.now()}`
```

改为统一真实格式，例如：

```ts
const scheduledDreamId = `dream:${job.chatKey}:${crypto.randomUUID()}`
```

然后：

- 抢锁时用这个 `scheduledDreamId`
- 写入 `schedulerState.activeDreamId = scheduledDreamId`
- 传给 `job.execute({ ..., dreamId: scheduledDreamId })`

#### 3）`DreamingService.startDreamSession()` 支持外部传入 `dreamId`

当前 dreamId 在 `startDreamSession()` 内部自行创建。  
改为支持：

```ts
export type DreamExecutionContext = {
    holderId?: string;
    executionMode?: DreamExecutionMode;
    resolvedPlan?: ResolvedDreamExecutionPlan;
    triggerSource?: DreamTriggerReason;
    dreamId?: string;
};
```

然后：

```ts
const dreamId = executionContext?.dreamId?.trim() || `dream:${this.chatKey}:${crypto.randomUUID()}`
```

这样：

- 手动触发仍然可自动生成
- scheduler 自动触发则能注入统一 dreamId

## 二、补上“待审批 dream 的正式应用服务”

### 目标

让以下入口都走同一条正式审批链：

- manual dream 首次审批
- Runtime task pill 打开 pending 审批
- Workbench pending inbox 打开 pending 审批

### 修改文件

- `MemoryOS/src/services/dreaming-service.ts`
- `MemoryOS/src/sdk/memory-sdk.ts`
- `MemoryOS/src/runtime/runtime-app.ts`
- `MemoryOS/src/ui/unifiedMemoryWorkbench.ts`

### 推荐实现方式

新增一个正式方法：

```ts
reviewPendingDreamSession(input: {
    dreamId: string;
    review: DreamReviewDecision;
}): Promise<{
    ok: boolean;
    status?: 'approved' | 'rejected' | 'deferred';
    reasonCode?: string;
}>
```

### 最佳落点

优先建议放在：

- `DreamingService` 内部新增 `applyDreamReviewDecision(...)`
- `startDreamSession()` 与 `reviewPendingDreamSession()` 共同复用这套逻辑

这样能避免：

- manual 审批一套逻辑
- pending 审批另一套逻辑
- 同样的 apply / reject / maintenance merge / rollback metadata 各写一次

### 建议拆分

在 `DreamingService` 内部新增两个私有方法：

#### 1）`applyReviewDecisionToGeneratedSession(...)`

输入：

- `dreamId`
- `meta`
- `output`
- `maintenanceProposals`
- `qualityReport`
- `review`

职责：

- 根据 review 筛选 approved mutations
- 过 quality guard
- apply mutation
- apply maintenance
- reject 其余 pending maintenance
- 更新 approval record
- 更新 meta.status
- merge rollback metadata
- 返回最终状态

#### 2）`rejectGeneratedSession(...)`

职责：

- reject 全部 pending maintenance
- 保存 approval = rejected
- 保存 meta.status = rejected

### SDK 门面新增方法

在 `MemorySDKImpl.chatState` 中新增：

```ts
reviewPendingDreamSession: (input: {
    dreamId: string;
    review: DreamReviewDecision;
}) => Promise<{
    ok: boolean;
    status?: 'approved' | 'rejected' | 'deferred';
    reasonCode?: string;
}>
```

### Runtime / Workbench 改法

#### 当前错误做法

现在 Runtime 和 Workbench 里是：

1. 读出 session
2. 打开 `openDreamReviewDialog()`
3. 得到 `DreamReviewDecision`
4. 只 toast / refresh，不落正式审批

#### 改为

1. 读出 session
2. 打开 `openDreamReviewDialog()`
3. 得到 `DreamReviewDecision`
4. 调 `memory.chatState.reviewPendingDreamSession({ dreamId, review })`
5. 根据结果 toast
6. invalidate snapshot + refresh UI

### 这一步必须覆盖的入口

- `runtime-app.ts` 的 `openPendingDreamReview()`
- `unifiedMemoryWorkbench.ts` 的 `dream-open-pending-review`

## 三、让第三阶段顶部状态统一消费 `DreamUiStateSnapshot`

### 目标

以下展示统一只吃一份 UI snapshot：

- 当前运行态
- 待审批入口
- 最近完成 / 最近失败 / 最近回滚
- task pill
- notification

### 修改文件

- `MemoryOS/src/ui/dream-ui-state-service.ts`
- `MemoryOS/src/ui/workbenchTabs/tabDreamStateOverview.ts`
- `MemoryOS/src/ui/workbenchTabs/tabDream.ts`

### 具体改法

#### 1）扩充 `DreamUiStateSnapshot.activeTask`

建议补充：

```ts
approvalStatus?: DreamApprovalStatus;
outputKind?: DreamSessionOutputKind;
```

#### 2）在 `getSnapshot()` 中真实读取 active dream 的 approval

当前 phase 解析时 approval 传的是 `null`。  
改成：

- 如果有 `activeDreamId`，先 `getDreamSessionById(activeDreamId)`
- 拿到 `session.approval`
- 再传给 `resolveUiPhase(...)`

#### 3）统一 pending inbox 的来源

保留 repository 扫 pending session 的逻辑，但把最终结果统一写入：

```ts
snapshot.inbox.pendingApprovalCount
snapshot.inbox.pendingDreamIds
```

Workbench 顶部入口区不要再自己从 `sessions` 二次推导一遍。

#### 4）顶部三块概览统一改吃 snapshot

把这三块改成直接消费：

- `activeTask`
- `inbox`
- `latestCompleted`
- `latestFailed`
- `latestRolledBack`

而不是继续从 `sessions` 里手工 `find(...)`

### 建议保留的边界

- Workbench 明细列表、session 列表、maintenance 列表仍可直接读 repository 数据
- 但“总览态”和“入口态”必须统一从 snapshot 来

这样兼顾最小改动和状态统一。

## 四、修正 `resolveUiPhase()` 与 active meta 选择逻辑

### 目标

让当前 phase 真正能区分：

- queued
- running
- waiting_approval
- completed
- failed
- rolled_back

### 修改文件

- `MemoryOS/src/ui/dream-ui-state-service.ts`

### 改法

#### 当前问题

现在 activeMeta 的选择依赖：

```ts
metas.find((m) => m.status === 'running' || m.status === 'queued')
```

但：
- meta 很少真正写 `queued`
- 有 `approval.pending` 的 generated dream 也可能应该算当前主要任务入口

#### 建议改为两层逻辑

##### 层 1：优先按 `schedulerState.activeDreamId` 找真实 active session

##### 层 2：如果没有 activeDreamId，则按优先级找：

1. `approval.status === 'pending'`
2. `meta.status === 'running'`
3. 最近 `failed`
4. 最近 `approved/generated(non-pending)`

然后 `resolveUiPhase()` 中：

- `approval.pending` 直接返回 `waiting_approval`
- `meta.running + lock.running` 返回 `running`
- `schedulerState.active && !session ready` 返回 `queued`

## 五、收紧 Repository 语义

### 目标

让 completed / pending / generated 语义清晰，不误导调用者。

### 修改文件

- `MemoryOS/src/services/dream-session-repository.ts`

### 改法

#### 1）废弃或重命名 `getLatestCompletedSession()`

不建议继续保留当前语义模糊版本。建议拆成两个方法：

```ts
getLatestResolvedSession()
getLatestCompletedNonPendingSession()
```

其中：

- `Resolved` = approved / rejected / rolled_back / failed / auto_silent_resolved
- `CompletedNonPending` = approved 或 generated 且 approval 非 pending

#### 2）如果保留旧方法，必须内部收紧

至少不要再把单纯 `generated + pending` 当 completed。

## 六、修正静默模式 approval 语义

### 目标

区分两类状态：

1. **流程已自动处理完**
2. **确实有内容已被批准落地**

### 修改文件

- `MemoryOS/src/services/dream-types.ts`
- `MemoryOS/src/services/dreaming-service.ts`
- `MemoryOS/src/ui/workbenchLocale*`
- `MemoryOS/src/ui/* dream 相关展示文件`

### 推荐方案

保留数据层兼容性最小改法：

#### 方案 A（推荐，兼容最好）

仍允许 `approval.status = 'approved'`，但增加：

```ts
approvalMode = 'auto_silent'
```

并在展示层统一把：

- `approved + approvalMode === auto_silent`

翻译成：

- `已自动处理`

而不是直接显示：

- `已批准`

同时在 summary / recent results / workbench session card 中使用更中性的文案：

- `静默整理完成`
- `自动处理完成`
- `已自动应用低风险维护`
- `仅完成后台整理`

#### 方案 B（更彻底，但改动更大）

扩充 `DreamApprovalStatus`：

```ts
'pending' | 'approved' | 'rejected' | 'auto_resolved'
```

这会影响更多类型和旧分支，不建议这次一次性改。

### 本次建议采用

**方案 A**，用展示语义收紧，不扩大类型面。

## 七、补齐 Runtime / Workbench 的刷新逻辑

### 目标

审批、回滚、应用 maintenance 后，以下都要同步刷新：

- `DreamUiStateService` cache
- task pill
- notification
- workbench 概览
- runtime pending inbox

### 修改文件

- `MemoryOS/src/runtime/runtime-app.ts`
- `MemoryOS/src/ui/unifiedMemoryWorkbench.ts`

### 改法

凡是以下操作成功后：

- `reviewPendingDreamSession`
- `rollbackDreamSession`
- `rollbackDreamMutation`
- `applyDreamMaintenanceProposal`
- `rejectDreamMaintenanceProposal`

都需要调用统一刷新序列：

```ts
this.dreamUiStateService?.invalidateCache()
await this.refreshDreamUiState()
```

Workbench 中也要在动作成功后：

- 重新读取最新 snapshot
- 刷新当前 tab
- 不只更新列表，还要更新顶部概览区

# 验收清单

以下全部通过，才算这次“一次性处理完”：

## A. Scheduler / Session ID 一致性

- 自动 dream 入队后，`schedulerState.activeDreamId` 等于真实 `DreamSessionMeta.dreamId`
- lock 记录里的 `dreamId` 与 session 一致
- UI 当前活跃 dream 能正确定位到实际会话

## B. 待审批真正可用

- Runtime task pill 点击待审批 dream 后，可以真实应用审批
- Workbench 待审批入口打开后，可以真实应用审批
- 审批通过后：
  - approval record 更新
  - meta.status 更新
  - mutation / maintenance 真正落地
  - rollback metadata 正确写入
- 审批拒绝后：
  - approval record 更新
  - meta.status 更新为 rejected
  - 未应用项不会残留 pending

- task pill
