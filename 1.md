# 划词按句数判断逻辑修复实施方案

## 1. 文档目标

本文用于修复 RollHelper 当前“自由划词按句数限制”判断不准确的问题，解决以下现象：

- 设置为“按句数限制，最多 2 句”时
- 玩家选择了一段**明显跨越两句以上语段**的文本
- 系统却仍然允许发起划词检定

本方案的目标不是简单调一个正则，而是把当前“按句数”判断升级为：

> **按“完整句 + 有效残片”的句段数判断**，让结果更符合玩家直觉与产品预期。

---

## 2. 问题现象

当前用户遇到的现象是：

> 选中了  
> `，还有被野兽踩乱的痕迹。” 她略带歉意地看了你一眼，语气温柔得近乎哄慰。 ”`  
> 这种明显跨越多个语段的文本，  
> 在默认“最多 2 句”的配置下，系统仍允许发起划词检定。

这说明：

- 限制逻辑**不是完全失效**
- 但“按句数”的实现方式和用户直觉不一致
- 当前算法把一些**多段文本**误判成了“没有超过 2 句”

---

## 3. 当前实现审查结论

结合当前 `interactiveTriggersEvent.ts` 可知：

### 3.1 默认设置确实是“按句数，最多 2 句”
默认配置中：

- `selectionFallbackLimitMode = "sentence_count"`
- `selectionFallbackMaxSentences = 2`

说明产品默认行为本来就是：

> **自由划词按句数限制，最多允许 2 句。**

---

### 3.2 当前句数统计逻辑非常简单
当前核心函数是：

```ts
countSelectionFallbackSentencesEvent(text: string): number
```

它的实现方式是：

1. 使用正则：
   ```ts
   /[。！？；!?;]+/
   ```
   对文本进行 split
2. 对 split 后结果做：
   - trim
   - 过滤空项
3. 返回剩余段数

也就是说，当前实际做的不是：

> 自然语言句子判断

而是：

> **只按句末标点切段，再统计非空段数**

---

### 3.3 当前算法的根本问题
这个逻辑会导致以下偏差：

#### 问题一：它不会识别“残片”
像下面这种：

- 半句开头
- 引号尾巴
- 句尾残片
- 对白收尾

如果没有被句末标点切成更多段，就可能被低估。

#### 问题二：它不理解引号闭合
例如：

- `。”`
- `！”`
- `？”`
- `。」`

这些在中文里属于**句尾 + 引号闭合**，但当前逻辑只是把句号当切点，并不进一步理解这个句子的边界结构。

#### 问题三：它不判断“是不是完整句”
当前实现不关心：

- 这是完整句
- 还是只是一段残片

只要 split 后非空，就统统按“一个段”处理。

#### 问题四：它统计的是“切段数”，不是“可交互语段数”
这就是当前和产品预期最不一致的地方。

用户真正想要的是：

> 这段选择看起来是不是已经跨了太多句子/语段

而不是：

> 这里面被 `。！？；` 切开了几次

---

## 4. 问题本质

当前问题不是：

- 后续菜单绕过了限制
- 也不是执行链路无视了限制

更准确地说：

> **“按句数”的定义本身过于粗糙。**

当前实现的是：

- **按句末标点切段数**

但产品真正需要的是：

- **按自然语言句段数**
- 其中句段包括：
  - 完整句
  - 有意义的起始残片
  - 有意义的结束残片
  - 被引号包裹的有效对白尾段

---

## 5. 修复目标

修复后应达到以下目标：

### 5.1 更符合用户直觉
当用户选中：

- 一句完整句
- 外加一句完整句
- 再加一个有意义残片

系统应视为已经超过“2 句”限制。

### 5.2 不再被中文引号、对白尾巴轻易绕过
例如：

- `……。”`
- `……？”`
- `……！”`

这类应正确识别句尾，不要误判。

### 5.3 不把纯引号/纯标点当成独立句
例如：

- `”`
- `）`
- `……`
- `。"`

这些单独存在时，不应被算作一个有效句段。

### 5.4 兼容当前设置模型
尽量不引入太多新设置项，优先在现有：

- `sentence_count`
- `selectionFallbackMaxSentences`

框架下直接升级判断逻辑。

---

## 6. 最终修复方向

本方案建议把当前“按句数”逻辑升级为：

> **按“完整句 + 有效残片”的句段统计。**

也就是说：

### 统计总数时，应计入两类内容
#### A. 完整句
由句末标点结束，并带有实际文字内容。

#### B. 有效残片
虽不是完整句，但已经构成明显独立语段，例如：

- 选区开头的对白尾巴
- 选区结尾的有意义残段
- 夹在中间但未闭合的短句残片

---

## 7. 新的句段判断模型

建议新增一个分析函数，例如：

```ts
type SelectionFallbackSentenceAnalysisEvent = {
  completeSentences: number;
  fragmentCount: number;
  totalSegments: number;
  normalizedText: string;
  segments: Array<{
    text: string;
    kind: "complete" | "fragment";
  }>;
};
```

并新增：

```ts
analyzeSelectionFallbackSentencesEvent(text: string): SelectionFallbackSentenceAnalysisEvent
```

---

## 8. 新的判断规则

### 8.1 文本预处理
在分析前，先做以下标准化：

1. 保留原始标点和引号信息
2. 统一空白：
   - 连续空白归一
   - trim 首尾空白
3. 不去掉句末引号
4. 不提前删标点

### 8.2 句尾识别规则升级
当前句尾判断正则：

```ts
/[。！？；!?;]+/
```

建议升级为：

```ts
/[。！？；!?;]+[”’」』】》）)]*/g
```

含义：

- 句末标点后若跟着右引号、右书名号、右括号，也视为句尾的一部分

这样更符合中文文本习惯。

---

## 9. 新的句段统计策略

### 9.1 完整句
如果某段文本：

- 含有效文字
- 由句末标点结束
- 不是纯符号

则记为：

- `kind = "complete"`

并计入：

- `completeSentences += 1`

---

### 9.2 有效残片
如果文本中存在：

- 句首残片
- 句尾残片
- 未被完整句包裹但有实际文字内容的片段

且它不是纯符号/纯引号/纯空白，则记为：

- `kind = "fragment"`

并计入：

- `fragmentCount += 1`

---

### 9.3 最终限制值
最终用于比较的不是：

```ts
completeSentences
```

而是：

```ts
totalSegments = completeSentences + fragmentCount
```

并使用：

```ts
totalSegments <= selectionFallbackMaxSentences
```

来决定是否允许。

---

## 10. 为什么这种方式更符合产品预期

例如下面这种选区：

> `，还有被野兽踩乱的痕迹。” 她略带歉意地看了你一眼，语气温柔得近乎哄慰。 ”`

按新逻辑更可能被拆成：

1. `，还有被野兽踩乱的痕迹。”`  
   - 有效残片 或 完整对白尾句
2. `她略带歉意地看了你一眼，语气温柔得近乎哄慰。`  
   - 完整句
3. `”`  
   - 纯引号，忽略

如果第一段算有效残片，第二段算完整句，则：

- `totalSegments = 2`

如果再有结尾实质残片，则：

- `totalSegments = 3`

这样比当前纯 split 计数更接近用户感知。

---

## 11. 推荐的保守规则

为了避免过度严苛，建议加一条保守判断：

### 纯符号不计入句段
以下内容单独存在时不计入：

- 纯引号
- 纯括号
- 纯句末标点
- 纯省略号
- 纯空白

这样可避免误伤。

## 12. 需要改造的函数

---

## 13. `countSelectionFallbackSentencesEvent(...)`
### 当前状态
当前只负责：

- split
- trim
- count

### 建议
不要继续扩这个函数。  
直接保留为兼容层，或者改成简单代理。

### 推荐做法
新增更高语义的：

```ts
analyzeSelectionFallbackSentencesEvent(...)
```

然后：

```ts
countSelectionFallbackSentencesEvent(...)
```

内部只返回：

```ts
return analyzeSelectionFallbackSentencesEvent(text).totalSegments;
```

这样兼容外层调用。

---

## 14. `isSelectionFallbackTextAllowedEvent(...)`
### 当前状态
在 `sentence_count` 模式下：

- 调 `countSelectionFallbackSentencesEvent(raw)`
- 再与 `selectionFallbackMaxSentences` 比较

### 建议改造
改为：

1. 调用：

```ts
const analysis = analyzeSelectionFallbackSentencesEvent(raw);
```

2. 使用：

```ts
analysis.totalSegments
```

做限制判断

3. 错误提示可升级成更明确：

```ts
当前为按句数限制，检测到 3 个句段，最多允许 2 句。
```

这样对用户更清楚。

---

## 15. `buildSelectionFallbackStatusTextEvent(...)`
### 当前状态
只会显示：

- 规则：按句数最多 2 句

### 建议
保留文案不变即可，避免设置 UI 大改。  
但在 debug 模式或 console 中建议补充：

- 实际检测到的完整句数
- 实际检测到的残片数
- 最终句段总数

这样方便排查。

---

## 16. 推荐新增辅助函数

建议新增以下函数：

### 16.1 去除无意义尾部符号
```ts
trimSelectionFallbackNoiseEvent(text: string): string
```

用途：

- 去掉纯空白、纯引号、纯括号尾部噪声
- 便于判断残片是否有意义

---

### 16.2 判断文本是否为有效残片
```ts
isMeaningfulSelectionFallbackFragmentEvent(text: string): boolean
```

判定：

- 去掉空白和尾部纯符号后仍有实际文字
- 才算有效残片

---

### 16.3 句段分析函数
```ts
analyzeSelectionFallbackSentencesEvent(text: string): SelectionFallbackSentenceAnalysisEvent
```

---

## 28. 最终结论

当前问题不是：

- 后续链路绕过了句数限制
- 或菜单无视了前面的校验

真正问题在于：

> **“按句数”现在实现得太粗，只是按句末标点 split 后统计段数，无法正确识别残片、对白尾巴和引号闭合场景。**

彻底修复的方向应是：

> **把句数限制升级成“完整句 + 有效残片”的句段统计。**

这样既能保持当前设置模型不变，又能显著提升判断准确性。

> **把自由划词的“按句数限制”从简单 split 计数，升级为“完整句 + 有效残片”的句段分析与限制判断。**
