# MemoryOS 梦境系统借鉴改造第三阶段实施方案

> 文档类型：实施方案  
> 适用项目：`ShionCox/SillyTavern-SS-Helper` / `MemoryOS`  
> 阶段目标：完成梦境系统后台运行态 UI、静默反馈与待审批入口整合  
> 文档版本：v1.0  
> 编写日期：2026-04-07

---

## 1. 阶段定位

第一阶段解决的是：

- 持久锁
- 调度硬化
- 跨实例互斥
- 自动触发底座稳定性

第二阶段解决的是：

- `dreamExecutionMode` 正式落地
- 静默模式 / 手动审批模式
- 自动轻梦境 / 自动待审批 / 手动深梦境三档分流

第三阶段开始，重点不再是“能不能跑”和“按什么策略跑”，而是：

> **当梦境系统在后台运行、完成、待审批、失败时，用户怎么感知它。**

这也是整套改造真正从“工程能力”走向“产品体验”的最后一环。

---

## 2. 第三阶段目标

本阶段需要完成五件核心事情：

1. **补足后台运行态 UI**
2. **在静默模式下提供低打扰完成反馈**
3. **在手动审批模式下提供待审批入口与提醒**
4. **统一展示 dream 的生命周期状态**
5. **把第一阶段、第二阶段新增的数据状态真正变成用户可见信息**

---

## 3. 本阶段完成后的最终体验

### 3.1 当 dream 正在后台运行时

用户至少能知道：

- 当前有没有 dream 在跑
- 是自动触发还是手动触发
- 是静默轻梦境还是完整梦境
- 当前大致在哪个阶段
- 是否已经生成待审批结果
- 是否失败、被中止或已完成

也就是说，dream 不再是“黑盒后台任务”。

---

### 3.2 当静默模式下的 dream 完成时

用户不会被强制弹出大面板，但会收到轻量反馈，例如：

- 梦境已完成：整理了 2 条维护建议
- 梦境已完成：自动应用了 1 条低风险维护
- 梦境已完成：未发现可落地更新，仅完成后台诊断

反馈原则：

- 不打断主聊天
- 不弹重型审批框
- 但用户能知道“后台刚刚发生了什么”

---

### 3.3 当手动审批模式下的自动 dream 完成时

用户会得到待审提示，而不是立刻被对话框打断。  
例如：

- 有 1 条梦境结果待审核
- 新增 3 条 dream 提案等待确认
- 点击进入 Dream Workbench / Review 列表

也就是说：

> **自动 dream 变成“可稍后处理的待办”，而不是“当前交互的插入式打断”。**

---

### 3.4 当手动触发 dream 时

用户仍会进入完整深梦境体验，但第三阶段会让整个过程更清楚：

- 运行中可见
- 完成后结果清晰
- 若失败或中断，也有明确状态提示

## 5. 核心设计原则

---

### 原则一：轻重分层，不同模式不同反馈

第三阶段不能一刀切。

#### 静默模式
- 完成时给轻量通知
- 不主动弹大面板
- 结果可追溯，但不强制打断

#### 手动审批模式
- 自动 dream 完成后给待审提醒
- 提供入口，不立刻打断
- 手动 dream 允许进入完整结果页

---

### 原则二：运行态与结果态分开展示

要明确区分：

#### 运行态
- queued
- running
- generating
- maintenance applying
- waiting approval

#### 结果态
- completed
- pending review
- failed
- rolled back
- aborted

这样 UI 才不会混乱。

---

### 原则三：先让用户“知道发生了什么”，再让用户“决定是否深入”

第三阶段不是一上来就给用户塞满信息，而是分层展示：

1. 最表层：状态胶囊 / 通知
2. 第二层：待审数量 / 完成摘要
3. 第三层：进入 workbench / review dialog 看细节

---

### 原则四：状态来源要统一，避免多处拼接“伪状态”

前两阶段已经引入了：

- `DreamSchedulerStateRecord`
- `DreamSessionMetaRecord`
- `DreamSessionOutputRecord`
- `DreamSessionApprovalRecord`
- `DreamLockRecord`
- `executionMode`
- `runProfile`
- `outputKind`
- `approvalMode`

第三阶段必须基于这些已有状态做展示，而不是每个 UI 各自猜测。

---

## 6. 第三阶段 UI 总体方案

推荐采用“三层可见性”结构。

---

### 6.1 第一层：全局轻量状态入口

推荐位置：

- Toolbar 按钮角标
- 顶部状态胶囊

职责：

- 告诉用户当前有没有 dream 相关动态
- 显示最小状态摘要
- 作为进入详情页的统一入口

示例状态：

- 运行中
- 待审批 2
- 刚完成
- 失败
- 已回滚

---

### 6.2 第二层：Dream 状态面板 / Workbench 概览区

推荐位置：

- `dream-workbench`
- `unifiedMemoryWorkbench`
- 梦境 Tab 顶部概览区

职责：

- 展示最近一次 dream 的运行结果
- 展示当前 active dream
- 展示待审批数量
- 展示静默完成摘要
- 展示最近失败/回滚信息

这是第三阶段最核心的 UI 落点。

---

### 6.3 第三层：已有 review dialog / 深度详情

职责：

- 展示 explain、diagnostics、mutation、maintenance proposals

这一层尽量复用现有 `dream-review-dialog.ts`，不要推翻重做。

## 7. 建议新增的状态模型

---

### 7.1 新增 UI 视图模型

建议新增：

`MemoryOS/src/ui/dream-ui-state-service.ts`

定义一个专供前端展示的聚合视图模型：

```ts
export interface DreamUiStateSnapshot {
    chatKey: string;

    activeTask: {
        exists: boolean;
        dreamId?: string;
        triggerReason?: 'manual' | 'generation_ended' | 'idle';
        executionMode?: 'manual_review' | 'silent';
        runProfile?: 'auto_light' | 'auto_review' | 'manual_deep';
        phase?: 'queued' | 'running' | 'generating' | 'post_processing' | 'waiting_approval' | 'applying' | 'releasing';
        startedAt?: number;
        holderId?: string;
    };

    inbox: {
        pendingApprovalCount: number;
        pendingDreamIds: string[];
    };

    latestCompleted?: {
        dreamId: string;
        executionMode?: 'manual_review' | 'silent';
        runProfile?: 'auto_light' | 'auto_review' | 'manual_deep';
        outputKind?: 'full' | 'light';
        completedAt?: number;
        summaryText?: string;
    };

    latestFailed?: {
        dreamId: string;
        failedAt?: number;
        reason?: string;
    };

    latestRolledBack?: {
        dreamId: string;
        rolledBackAt?: number;
    };
}
```

---

### 7.2 为什么需要 UI 聚合层

因为第三阶段的 UI 不应该每个组件自己去拼：

- lock state
- scheduler state
- session meta
- approval
- output

否则会导致：

- 同一状态显示不一致
- 多处复制聚合逻辑
- 后续难以维护

正确做法是：

> **先聚合成统一 UI snapshot，再让多个 UI 组件消费。**

## 10. 静默完成反馈设计

---

### 10.1 为什么静默模式必须有反馈

静默模式不是“完全无感”，而是“低打扰但可感知”。

如果完全没有反馈，用户会不知道：

- dream 是否真的执行了
- 有没有做出变更
- 系统是否只是“看起来有这个功能”

---

### 10.2 推荐的反馈形式

#### 形式 A：轻 toast / 轻通知条
持续 3~5 秒，不打断输入

#### 形式 B：Workbench 顶部 summary banner
在用户下次打开 memory workbench 时能看到

#### 形式 C：Toolbar 状态短暂变化
例如从“梦境”按钮变成“刚完成”

推荐至少同时实现 A + B。

---

### 10.3 静默反馈文案建议

#### 场景 1：只完成诊断
- 梦境已完成：未发现可落地更新，仅完成后台整理

#### 场景 2：自动应用了维护
- 梦境已完成：自动应用了 1 条低风险维护

#### 场景 3：生成了可查看结果但无需审批
- 梦境已完成：整理了 2 条记忆联想，详情可在 Dream Workbench 查看

#### 场景 4：因策略过滤无动作
- 梦境已完成：本轮未形成稳定提案

## 11. 运行阶段可视化设计

---

### 11.1 推荐的阶段枚举

第三阶段建议新增前端专用 phase：

```ts
type DreamUiPhase =
    | 'queued'
    | 'running'
    | 'recalling'
    | 'generating'
    | 'post_processing'
    | 'waiting_approval'
    | 'auto_applying'
    | 'completed'
    | 'failed'
    | 'rolled_back';
```

---

### 11.2 与后端状态的映射原则

#### 调度层提供
- queued
- active
- last decision

#### session 层提供
- generated
- approved
- rejected
- failed
- rolled_back

#### UI 层聚合后输出
更适合用户理解的阶段文案

例如：

- scheduler.active + session.meta.status = running  
  => UI phase = `generating`
- session.approval.status = pending  
  => UI phase = `waiting_approval`
- latestCompleted + outputKind = light  
  => UI phase = `completed`

---

### 11.3 为什么要做 phase 映射

因为直接把数据库状态丢给用户不够友好。  
“generated / approved / rejected”是系统状态，不一定是最佳展示状态。  
第三阶段需要一层用户视角的语义转换。

---

## 12. Workbench 改造方案

---

### 12.1 推荐改造目标

在现有 `dream-workbench.ts` 或 unified memory workbench 中新增三块固定区域：

#### 区域 A：当前梦境状态
展示 active / queued / running 状态

#### 区域 B：待审批
展示 pending 列表

#### 区域 C：最近完成 / 最近失败
展示最近几次重要结果

---

### 12.2 区域 A：当前梦境状态建议内容

- 当前状态：运行中 / 空闲 / 排队中
- 触发方式：manual / generation ended / idle
- 模式：静默 / 手动审批
- 档位：auto_light / auto_review / manual_deep
- 开始时间
- 最近阶段文案
- 可选：取消 / 打开详情

---

### 12.3 区域 B：待审批建议内容

- 待审批数量
- 列表按时间倒序
- 卡片显示摘要
- 一键进入 review dialog
- 一键进入 dream detail 页

---

### 12.4 区域 C：最近结果建议内容

- 最近完成
- 最近失败
- 最近回滚
- 最近静默自动应用摘要

这样用户即使不盯着通知，也能回头查看近期发生了什么。

## 13. 与现有 review dialog 的衔接

---

### 13.1 原则：尽量复用，不重造一套新审批 UI

现有 `dream-review-dialog.ts` 已经很完整，第三阶段不应重写其审批本体。  
重点应该是：

- 怎么从待审批入口打开它
- 怎么从静默结果页跳到它
- 怎么让用户知道“它在等你”

---

### 13.2 推荐新增打开方式

#### 从 pending card 打开
- “打开审核”

#### 从 toolbar badge 打开
- 直接跳转到最新待审批项

#### 从 dream workbench 打开
- 点某一条 pending item

---

### 13.3 推荐保留 review dialog 的职责

- 批准 / 拒绝 / defer
- 看 explain
- 看 diagnostics
- 看 mutation / maintenance

第三阶段不应让 workbench 替代 review dialog，只应让 workbench 成为它的入口和总览层。

## 14. 通知与状态联动方案

---

### 14.1 推荐新增 Dream Notification Service

建议新增：

`MemoryOS/src/ui/dream-notification-service.ts`

职责：

- 根据 `DreamUiStateSnapshot` 决定是否弹出轻通知
- 避免同一次 dream 重复通知
- 对静默完成与待审批提醒做去重

建议至少支持：

- 静默完成通知
- 待审批提醒通知
- 失败通知

---

### 14.2 去重策略建议

为避免反复弹相同通知，建议记录：

```ts
lastNotifiedCompletedDreamId
lastNotifiedPendingDreamId
lastNotifiedFailedDreamId
```

可存在：

- UI service 内存态
- 或轻持久化记录（推荐）

---

### 14.3 通知触发建议

#### 静默完成通知
触发条件：

- runProfile = `auto_light`
- 本次 dream 刚完成
- 尚未通知过

#### 待审批提醒
触发条件：

- approval.status = `pending`
- runProfile = `auto_review`
- 尚未通知过

#### 失败通知
触发条件：

- meta.status = `failed`
- 尚未通知过

## 16. 第三阶段详细实施步骤

---

### 步骤一：新增 Dream UI 状态聚合服务

新增：

- `dream-ui-state-service.ts`

新增：

- `dream-task-pill.ts`

1. 根据 `DreamUiStateSnapshot` 渲染最小状态
2. 支持：
   - running
   - pending review
   - completed
   - failed
3. 点击进入 workbench / latest review

### 步骤三：改造 Dream Workbench 顶部概览区

修改：

- `dream-workbench.ts`
- 或对应 tab

动作：

1. 增加当前 dream 状态卡
2. 增加待审批列表
3. 增加最近完成 / 失败 / 回滚摘要
4. 增加“打开审核”“查看详情”“查看最近结果”按钮

---

### 步骤四：新增轻通知服务

新增：

## 21. 风险与应对

---

### 风险一：通知过多，用户感到吵
应对：

- 静默模式默认只发轻通知
- 所有通知去重
- 可后续追加“关闭 dream 通知”开关（可选扩展）

---

### 风险二：多个 UI 入口显示不一致
应对：

- 强制统一走 `DreamUiStateSnapshot`
- UI 组件不直接各自读 repository 原始数据

---

### 风险三：待审批入口不够明显，用户仍然找不到
应对：

- 至少两处提醒：
  - toolbar badge
  - workbench pending 列表

---

### 风险四：为了做 UI 又把逻辑耦合回 DreamingService
应对：

- DreamingService 只提供状态事实
- UI 聚合、通知、展示全部放在前端服务层

---
