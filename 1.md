# RollHelper 优先级 1 实施方案：先堵暗骰泄露

## 文档目标

本方案只覆盖第一优先级：**先堵暗骰泄露**。

目标不是重做暗骰流程，也不是先处理“只对最新轮次生效”“立即续写”“无限暗骰限制”等后续问题，而是先把当前版本里最危险的点堵住：

1. **暗骰结果卡不应泄露真实结果含义**
2. **摘要不应在关闭开关时泄露暗骰后果**
3. **事件列表中的走向预览默认应隐藏，避免未掷骰前提前剧透**
4. **设置项要和实际逻辑打通，避免存在“字段有了但没生效”的假开关**

---

## 一、当前问题概览

结合当前仓库实现，暗骰相关的泄露主要出在以下几个位置。

### 1. 结果卡泄露

文件：`RollHelper/src/events/renderEvent.ts`

当前 `buildEventRollResultCardEvent()` 虽然在暗骰时隐藏了：

- 总值
- compare
- dc
- 骰面可视化

但仍然继续渲染：

- `outcomeTextHtml`
- `collapsedOutcomePreview`
- `statusImpactHtml`
- `outcomeStatusSummaryHtml`
- `currentStatusesHtml`

这会导致结果卡表面写着“结果已隐藏，仅命运知晓”，但实际上仍把：

- 成功/失败后果
- 获得了什么线索
- 触发了什么状态
- 当前状态变化

间接暴露给用户。

这属于**硬泄露**，优先级最高。

---

### 2. 摘要可能泄露

文件：`RollHelper/src/events/summaryEvent.ts`

当前摘要逻辑中：

- `toSummaryResultSentenceEvent()` 对暗骰会写成“暗骰检定已结算（结果已隐藏）”
- 但如果 `includeOutcomeInSummary=true`，后面仍可能继续输出 `outcomeSentence`

这会形成：

- 表面说“隐藏”
- 实际把“发现暗门”“触发警戒”“误判方向”这类结果继续写进摘要

这属于**软泄露**，但同样会破坏暗骰。

---

### 3. 列表卡走向预览默认过于激进

文件：

- `RollHelper/src/settings/constantsEvent.ts`
- `RollHelper/src/templates/settingsCardHtmlTemplate.ts`
- `RollHelper/src/events/renderEvent.ts`

当前配置里：

- `showOutcomePreviewInListCard: true`

而设置页已经有现成开关：

- `列表卡预览走向`

也就是说，这个能力其实已经存在，只是默认值不合适。

对于暗骰/悬疑/侦查类玩法，列表卡在未掷骰前就展示成功/失败/爆骰的可能走向，会明显降低未知感，甚至提前剧透。

你的要求是：

- 这个能力继续保留在设置里
- **默认隐藏**
- 用户手动开启才显示

这个需求和第一优先级完全一致，可以一并落地。

---

### 4. `blindRevealInSummary` 已定义但主流程未完整接入

文件：

- `RollHelper/src/types/eventDomainEvent.ts`
- `RollHelper/src/settings/storeEvent.ts`
- `RollHelper/src/settings/constantsEvent.ts`
- `RollHelper/src/events/summaryEvent.ts`

当前有设置字段：

- `blindRevealInSummary: boolean`

默认值已经是：

- `false`

但从当前主流程看，摘要生成逻辑并没有把这个开关完整纳入暗骰 outcome 的显示判定里。

这会带来两个问题：

1. 用户以为自己关闭了摘要揭示，实际还有泄露风险
2. 后面继续扩展暗骰能力时，这个字段会越来越像“死开关”

第一优先级里应把它真正接通。

---

## 二、本阶段的改动边界

本阶段只做“堵泄露”，不做下面这些内容：

### 本阶段**要做**

- 暗骰结果卡隐藏所有会暴露结果含义的内容
- 暗骰摘要根据设置决定是否允许显示后果
- 列表卡走向预览默认关闭
- 设置页文案同步调整，明确“可能剧透”
- 为暗骰展示增加统一的“占位描述”策略，保证 UI 不空、不穿帮

### 本阶段**不做**

- 不改暗骰队列结构
- 不改暗骰只绑定最新轮次的逻辑
- 不改“下一轮才生效”的流程
- 不处理无限暗骰限制
- 不处理 blind reroll bug
- 不做立即续写

避免第一优先级改动过宽，影响后续优先级拆分。

---

## 三、实施目标

本阶段完成后，系统需要满足以下结果。

### 目标 A：暗骰结果卡只显示“已暗骰”，不显示真实后果

用户在结果卡上应该看到：

- 这是一次暗骰
- 结果已经由系统记录
- 真实结算不会直接展示

用户**不应该**从结果卡中直接或间接知道：

- 是成功还是失败
- 获得了什么线索
- 是否加了什么状态
- 当前状态是否因此变化
- 是否触发特定 outcome 分支

---

### 目标 B：摘要默认不泄露暗骰后果

当：

- `blindRevealInSummary = false`

则摘要对暗骰只能写成类似：

- 暗骰检定已结算（结果隐藏）
- 暗骰结果将在叙事中体现

绝不能继续拼接：

- 成功走向
- 失败走向
- 爆骰走向
- 状态变化摘要

---

### 目标 C：列表卡走向预览默认关闭

用户初次安装或旧配置迁移后，应该表现为：

- 未开启“列表卡预览走向”时，不显示走向预览块
- 只有用户手动开启后才显示

并且设置文案要明确说明：

- 该功能可能剧透未结算事件后果

---

### 目标 D：隐藏不等于空白

即使暗骰结果被隐藏，界面仍要保持完整、自然，不出现：

- 卡片结构塌陷
- 大量空白区域
- “已隐藏”旁边还留一堆空容器
- 看起来像 bug

所以需要设计统一的占位文案和显示策略。

---

## 四、推荐的最终显示规则

这部分是实现时应统一遵循的规则。

---

## 4.1 事件列表卡（未结算）

### 公共规则

- 走向预览由 `showOutcomePreviewInListCard` 控制
- 默认值改为 `false`

### 当 `showOutcomePreviewInListCard=false`

- 完全不渲染走向预览区域

### 当 `showOutcomePreviewInListCard=true`

- 按现有逻辑显示成功/失败/爆骰预览
- 这仍然是全局行为，不区分公开事件和暗骰事件
- 文案中必须提醒此功能可能剧透

> 说明：
> 由于列表卡阶段还没掷骰，严格来说这里不是“暗骰泄露结果”，而是“事件设计预剧透”。
> 但它确实会破坏未知感，所以和第一优先级一起收口最合适。

---

## 4.2 结果卡（已结算）

### 公开检定

保持现有行为：

- 可显示总值
- 可显示 compare / dc
- 可显示 outcome
- 可显示状态变化
- 可显示当前状态概览

### 暗骰检定

必须隐藏以下内容：

| 区域 | 处理方式 |
|---|---|
| 状态标题 | 不显示“成功/失败”，改为“暗骰已结算” |
| 总值 | 显示“已隐藏” |
| compare/dc | 显示“已隐藏”或“?” |
| 骰面可视化 | 不显示 |
| outcome 预览 | 不显示真实文本 |
| outcome 详情 | 不显示真实文本 |
| 状态影响 | 不显示 |
| 状态变化摘要 | 不显示 |
| 当前状态汇总 | 不显示 |
| 爆骰详细信息 | 不显示真实触发情况 |
| 掷骰拆解 | 不显示真实骰面与修正明细 |

### 暗骰结果卡推荐替代文案

推荐统一成：

- 状态：`暗骰已结算`
- 摘要主文案：`真实结果已隐藏，将通过后续叙事体现`
- 结果核心区：`本次为暗骰，点数、阈值与后果不在卡片中公开`

不要在卡片中出现：

- “成功”
- “失败”
- “大成功”
- “大失败”
- “发现了……”
- “状态变化……”

---

## 4.3 摘要块

### `blindRevealInSummary = false`

暗骰事件摘要应固定为：

- `暗骰检定已结算（结果隐藏）`

如果当前摘要等级是 `balanced/detailed`，也只能保留：

- 事件标题
- 对象
- 技能
- 骰式
- 来源（暗骰检定）
- 时限
- 结果句子：隐藏

不能追加：

- `成功走向：...`
- `失败走向：...`
- `爆骰走向：...`

### `blindRevealInSummary = true`

这时才允许摘要写出 outcome 句子。

但建议在实现上只放开：

- `outcomeSentence`

不要额外放开：

- 状态变化摘要
- 当前状态总览

也就是说即使允许摘要揭示，也尽量只揭示“叙事后果文本”，不要揭示所有系统层细节。

这样安全边界更清楚。

## 5.3 `RollHelper/src/events/renderEvent.ts`

这是第一优先级里的**核心修改文件**。

### 改动目标

统一暗骰结果卡的隐藏策略，彻底切断结果泄露。

---

### 5.3.1 增加统一判断与占位函数

建议新增辅助函数，避免到处散写 `isBlindRecord ? ... : ...`。

### 建议新增函数

#### 1）判定是否为暗骰结果

```ts
function isBlindRollRecordEvent(record: EventRollRecordEvent): boolean {
  return record.visibility === "blind" || record.source === "blind_manual_roll";
}
```

#### 2）返回暗骰通用说明文本

```ts
function getBlindConcealedTextEvent(): string {
  return "真实结果已隐藏，将通过后续叙事体现";
}
```

#### 3）返回暗骰结果卡状态文本

```ts
function getBlindStatusTextEvent(): string {
  return "暗骰已结算";
}
```

#### 4）判断某个展示块是否应隐藏

```ts
function shouldHideBlindOutcomeDetailsEvent(record: EventRollRecordEvent): boolean {
  return isBlindRollRecordEvent(record);
}
```

### 这样做的好处

- 逻辑集中
- 后续第二、三优先级继续改时不容易漏点
- 结果卡模板参数会更清晰

---

### 5.3.2 修改 `buildEventRollResultCardEvent()` 的暗骰分支

### 当前问题

当前 `isBlindRecord` 只影响：

- status
- statusColor
- compareHtml
- dcText
- totalText
- diceVisualBlockHtml
- rollsSummaryHtml

但**没有阻断 outcome / status summary / current statuses** 的渲染。

### 实施方案

在 `buildEventRollResultCardEvent()` 中先计算：

```ts
const isBlindRecord = isBlindRollRecordEvent(record);
const concealText = getBlindConcealedTextEvent();
```

然后把下列字段在暗骰时强制替换。

---

#### A. 状态文本

当前：

```ts
const status = isBlindRecord
  ? "结果已隐藏，仅命运知晓"
  : ...
```

建议改为：

```ts
const status = isBlindRecord
  ? "暗骰已结算"
  : ...
```

原因：

- “仅命运知晓”文艺但不稳定
- “暗骰已结算”更适合做统一状态文案
- 下方再单独放一句说明更清楚

---

#### B. outcome 相关字段全部改占位

当前会计算：

- `resolvedOutcome`
- `outcomeLabel`
- `outcomeText`
- `collapsedOutcomePreview`
- `outcomeStatusSummaryText`
- `currentStatusesSummaryText`

### 暗骰时应改成

```ts
const outcomeLabel = isBlindRecord ? "暗骰结果" : ...
const outcomeText = isBlindRecord ? concealText : ...
const collapsedOutcomePreview = isBlindRecord
  ? {
      text: concealText,
      title: concealText,
      chipClassName: "st-rh-summary-chip-outcome st-rh-summary-chip-concealed",
    }
  : ...
const outcomeStatusSummaryText = isBlindRecord ? "" : ...
const currentStatusesSummaryText = isBlindRecord ? "" : ...
const statusImpactHtml = isBlindRecord ? "" : ...
```

### 关键原则

暗骰时：

- 不展示真实 outcome
- 不展示状态变化摘要
- 不展示当前状态汇总
- 不展示状态影响

即使内部已经结算并写入状态，也不能在结果卡层暴露。

---

#### C. 掷骰拆解区全部隐藏真实内容

当前暗骰虽然在 `rollsSummaryHtml` 里做了“结果已隐藏”，但 `explodeInfoHtml` 和 `modifierBreakdownHtml` 仍可能泄露额外信息。

### 暗骰时应改成

```ts
const rollsSummaryHtml = isBlindRecord
  ? deps.buildRollsSummaryTemplateEvent("结果已隐藏", "隐藏", "")
  : ...

const explodeInfoHtml = isBlindRecord ? "不公开" : ...
const modifierBreakdownHtml = isBlindRecord ? "" : ...
const diceModifierHintHtml = isBlindRecord ? "结果已隐藏" : ...
const dcDescriptionHtml = isBlindRecord ? "" : ...
```

### 原因

即使不显示总值，下面如果还写：

- “已请求，已触发连爆”
- “技能+3 / 状态-2”
- “DC 原因：很容易”

用户仍然能推断结果。

---

#### D. 结果核心区增加固定暗骰说明

建议在 `buildEventRollResultCardTemplateEvent()` 所需参数中，继续使用现有参数，但对暗骰时传入：

- `statusText = 暗骰已结算`
- `totalText = 已隐藏`
- `compareHtml = ?`
- `dcText = 已隐藏`
- `diceVisualBlockHtml = ""`
- `outcomeTextHtml = 真实结果已隐藏，将通过后续叙事体现`

同时建议在模板里复用 `outcomeTextHtml` 作为“说明区”，而不是剧情后果区。

也就是说：

- 公开检定：`outcomeTextHtml` 表示真实剧情后果
- 暗骰检定：`outcomeTextHtml` 表示系统说明文本

这样改动范围最小。

---

### 5.3.3 结果卡模板层避免空白区

如果暗骰时很多字段被置空，可能会出现 UI 空块。

### 建议做法

在 `buildEventRollResultCardTemplateEvent()` 的调用侧确保：

- `outcomeTextHtml` 至少有占位文本
- `outcomeStatusSummaryHtml` 为空时，不渲染条目
- `currentStatusesHtml` 为空时，不渲染条目
- `statusImpactHtml` 为空时，不渲染条目

如果模板本身已经支持空值跳过，则只需要传空。
如果模板会保留空壳，需要补条件判断。

### 推荐视觉效果

暗骰结果卡的走向区只保留一句：

`本次为暗骰，真实结果不会在结果卡中公开，将通过后续叙事体现。`

## 5.4 `RollHelper/src/events/summaryEvent.ts`

这是第一优先级里的第二核心文件。

### 改动目标

让摘要真正受 `blindRevealInSummary` 控制。

---

### 5.4.1 修改 `toSummaryOutcomeSentenceEvent()`

当前这个函数只看 `outcomeKind` 和 `outcomeText`，不看暗骰，不看设置。

### 建议改造

把函数签名改为：

```ts
function toSummaryOutcomeSentenceEvent(
  item: RoundSummaryEventItemEvent,
  settings: DicePluginSettingsEvent
): string
```

然后在函数开头加：

```ts
const isBlind = item.visibility === "blind" || item.resultSource === "blind_manual_roll";
if (isBlind && !settings.blindRevealInSummary) {
  return "";
}
```

### 预期效果

- 暗骰摘要是否展示后果，终于真正受设置控制
- 公开检定逻辑不受影响

---

### 5.4.2 修改 `buildSummaryEventNaturalLineByModeEvent()`

当前它在 `includeOutcomeInSummary` 为真时会无条件拼接 `outcomeSentence`。

### 建议改造

当前：

```ts
const outcomeSentence = includeOutcomeInSummary ? toSummaryOutcomeSentenceEvent(item) : "";
```

改为：

```ts
const outcomeSentence = includeOutcomeInSummary
  ? toSummaryOutcomeSentenceEvent(item, settings)
  : "";
```

因此该函数签名也要改为接收 `settings`：

```ts
function buildSummaryEventNaturalLineByModeEvent(
  item: RoundSummaryEventItemEvent,
  detailMode: SummaryDetailModeEvent,
  includeOutcomeInSummary: boolean,
  settings: DicePluginSettingsEvent
): string
```

并在上层调用链传入。

---

### 5.4.3 保持暗骰结果句子稳定

当前 `toSummaryResultSentenceEvent()` 已经对暗骰返回：

- `暗骰检定已结算（结果已隐藏）`

这个方向是对的，建议保留，只优化为更统一的版本。

### 推荐文案

```ts
return "暗骰检定已结算（结果隐藏，将通过叙事体现）";
```

这样比单纯“结果已隐藏”更自然，也更符合插件实际行为。

---

### 5.4.4 不在摘要中额外输出状态变化

本阶段不建议把 `blindRevealInSummary=true` 扩展成“把状态变化也显示出来”。

### 原因

- 这会让开关语义过重
- 很容易出现“用户只是想看 outcome，却顺便被暴露状态系统细节”
- 第一优先级目标是止血，不是扩展功能

所以本阶段只让它决定：

- **暗骰是否允许输出 outcomeSentence**

其他系统细节继续保持不显示。

---

## 5.5 `RollHelper/src/types/eventDomainEvent.ts`

### 改动目标

本阶段尽量不改 schema。

但如果你希望后续阅读更清楚，可以只补注释，不加字段。

### 建议

给 `blindRevealInSummary` 补一条注释，说明其当前职责：

```ts
blindRevealInSummary: boolean; // 仅控制摘要是否允许揭示暗骰 outcome 文本，不影响结果卡是否隐藏
```

这样后续继续做第二优先级时，不容易把它误用成“结果卡也跟着显示”。

> 这条是可选项，不是必须改动。

## 八、实现时的风险点

---

### 风险 1：把暗骰的状态变化完全隐藏后，用户可能误以为系统没生效

### 处理方式

在暗骰结果卡里固定展示一句说明：

`真实结果已隐藏，将通过后续叙事体现。`

这样用户知道不是没算，而是刻意不公开。

---

### 风险 2：摘要和结果卡规则不一致

例如：

- 结果卡隐藏了
- 摘要却泄露了

### 处理方式

把“暗骰隐藏策略”做成统一判定：

- 结果卡：始终隐藏
- 摘要：由 `blindRevealInSummary` 控制

并在实现文档里固定这条边界，不要混用。

---

### 风险 3：列表预览默认关闭后，部分老用户会觉得功能“没了”

### 处理方式

只改默认值，不主动覆盖老用户已保存设置。

并在设置标题里明确说明这是：

- 一个可选能力
- 默认建议关闭

---

### 风险 4：模板参数为空导致卡片排版异常

### 处理方式

在 `renderEvent.ts` 层保证：

- 暗骰也至少传入占位文本
- 不让模板收到一堆半空字段

必要时在模板层加 `if empty then skip block`。

## 十、完成标准

满足以下条件即可视为第一优先级完成。

### 必须满足

- [ ] 列表卡走向预览默认关闭
- [ ] 设置页文案明确提示“可能剧透”
- [ ] 暗骰结果卡不再显示真实 outcome
- [ ] 暗骰结果卡不再显示状态变化摘要
- [ ] 暗骰结果卡不再显示当前状态汇总
- [ ] 暗骰结果卡不再显示真实爆骰信息与修正拆解
- [ ] `blindRevealInSummary=false` 时，摘要不显示暗骰 outcome
- [ ] `blindRevealInSummary=true` 时，仅摘要允许揭示 outcome，结果卡仍隐藏
- [ ] 公开检定显示不受影响

### 最终效果描述

完成后，暗骰在当前版本中会变成：

- **看得见发生了暗骰**
- **看不见暗骰真实后果**
- **后果只能通过后续叙事体现**

这就为后面的第二优先级打下稳定基础：

- 再去做“只对最新轮次生效”
- 再去做“只消费一次”
- 再去做“立即续写”

不会出现“后续流程还没改，泄露已经先把体验毁掉”的情况。

## 十一、下一阶段衔接建议

当第一优先级完成后，第二优先级建议直接进入：

**“暗骰只能针对于最新对话轮次生效，旧的不能生效”**

因为现在最大的风险点一旦堵住，接下来最该处理的就是：

- 暗骰引导队列必须绑定轮次/楼层
- 旧轮次不能混进新 prompt
- swipe/edit/delete 后旧暗骰不能复活

这会让整个暗骰系统从“能用但会串”变成“可控”。
