# Prompt 精简与 Token 优化实施方案

## 1. 文档目标

本文用于在 **不削弱 RollHelper 功能表达能力** 的前提下，系统性优化当前提示词注入链路的 Token 成本，重点解决：

- `buildDynamicSystemRuleTextEvent()` 过长、重复、解释性 prose 太多
- `trigger_pack`、`rh-trigger`、`status`、`explode`、`advantage` 等规则堆叠在单一大块中
- `<dice_runtime_policy>` 中存在对 AI 生成帮助有限、但消耗 token 的字段
- 提示词更像“说明书”，而不是“协议卡片”
- 需要兼顾：
  - AI 仍能理解各个功能
  - AI 仍能正确填写 JSON
  - AI 仍能理解 `rh-trigger` / `triggerpack` / `outcomes` 的职责边界

本方案不主张简单删规则，而是主张：

> **把“长说明文档”改造成“分层协议 + 按功能开关注入 + 紧凑 schema + 低冗余运行时约束”。**

---

## 2. 当前问题总结

结合当前 `main` 分支中的 `promptEvent.ts`，提示词冗余主要来自以下四类问题。

### 2.1 单一规则块承担了过多职责
当前 `buildDynamicSystemRuleTextEvent()` 同时承担：

- rolljson 协议说明
- triggerpack 协议说明
- outcomes 分支说明
- `rh-trigger` 使用约束
- 爆骰说明
- 优势/劣势说明
- 状态系统说明
- 时间限制说明
- 叙事代价说明
- 各种正例 / 反例 / 禁止项说明

这导致一个问题：

> **规则块变成了“全功能总说明书”，而不是每次生成所需的最小协议。**

---

### 2.2 同一规则被反复表达
当前提示词中，以下概念被多次重复：

- `rh-trigger` 只能写在正文里
- `rh-trigger` 不能写进 `rolljson/outcomes/desc/dc_reason`
- `triggerpack` 通过 `sourceId/sid` 关联
- 暗骰必须写 outcomes
- outcomes 是主来源，blind guidance 只是后续叙事体现
- `!` 与 `explode` 的联动关系
- `kh/kl` 与优势/劣势的关系

这些规则本身没错，但重复表达太多，会显著增加 token 消耗。

---

### 2.3 部分规则在开关关闭时仍然注入
当前明显存在一个优化机会：

#### `triggerpack` 规则
即使交互触发系统不开启，当前核心协议中仍然会出现：

- `triggerpack` schema
- `triggerpack` 的说明

这会带来两个问题：

1. 浪费 token
2. 让 AI 在无需生成 trigger 时，仍被迫理解这块协议

---

### 2.4 runtime policy 中混入了“对 AI 低价值”的字段
当前 `<dice_runtime_policy>` 中有一些字段更偏 UI / 摘要 / 展示，不是生成 JSON 的核心约束，例如：

- `summary_detail`
- `summary_rounds`
- `summary_include_outcome`
- `list_outcome_preview`
- `skill_table_count`
- `skill_table_preview`

这些字段对 AI 是否能正确写 rolljson / triggerpack / rh-trigger 的帮助较弱，却会额外消耗 token。

---

## 3. 优化目标

本次优化后的提示词链路，应达到以下目标。

### 3.1 保留能力表达
AI 仍然必须能正确理解并输出：

- `rolljson`
- `dice_events`
- `outcomes`
- `rh-trigger`
- `trigger_pack`
- `status tags`
- `explode`
- `advantage / disadvantage`
- `timeLimit`
- `dc_reason`

### 3.2 降低重复度
同一条规则尽量只表达一次，不要散落在多个段落中重复说明。

### 3.3 分层注入
按以下层次拆分规则：

1. **最小核心协议**
2. **按功能开关注入的扩展协议**
3. **紧凑的运行时限制块**

### 3.4 让规则更像“协议”，不是“说明文档”
提示词应以：

- schema
- 约束清单
- 最小示例
- 运行时限制

为主，而不是长篇 prose。

### 3.5 支持后续扩展
未来新增功能时，可以继续插入新的扩展块，而不必不断扩充单一巨型规则块。

---

## 4. 总体设计方案

建议把当前提示词拆成三层。

---

## 5. 第一层：最小核心协议块

这是**每轮都应该发送**的最小规则块。

### 5.1 只保留最关键的内容
应当只包含：

1. 只能输出 ` ```rolljson `，禁止 ` ```json `
2. 正文不能直接先写检定结果
3. `rolljson` 的最小 schema
4. `difficulty / compare / checkDice` 的合法取值
5. 暗骰事件必须写 `outcomes`
6. 所有运行时限制以 `<dice_runtime_policy>` 为准

### 5.2 不要包含长解释
不要在核心协议块里写太多：

- 为什么这样设计
- 冗长正例/反例
- 过多叙事型说明

### 5.3 推荐最小核心协议结构
建议压缩成如下风格：

```text
【骰子协议】
1. 只输出 ```rolljson，禁止 ```json。
2. 正文先叙事，不直接宣判检定结果。
3. rolljson:
{
  "type":"dice_events",
  "version":"1",
  "events":[{
    "id":"str",
    "title":"str",
    "difficulty":"easy|normal|hard|extreme",
    "desc":"str",
    "checkDice":"NdM[!][khX|klX][+/-B]",
    "skill":"str",
    "compare":">=|>|<=|<",
    "scope":"protagonist|character|all",
    "target":{"type":"self|scene|supporting|object|other","name":"str?"},
    "outcomes":{"success":"str","failure":"str","explode":"str?"}
  }]
}
4. 暗骰事件必须提供 outcomes。
5. 运行时限制以 <dice_runtime_policy> 为准。
```

### 5.4 核心协议中的示例数量控制
建议：

- 最多 1 个合法示例
- 不再放大段正例 / 反例列表

---

## 6. 第二层：按功能开关注入的扩展协议块

不要把所有功能都堆到一个大规则块里。  
建议改成：**只有功能开启时，才注入对应扩展块。**

---

## 7. 交互触发扩展块（仅 `enableInteractiveTriggers` 开启时）

### 7.1 应包含的内容
只保留这些最重要规则：

1. `rh-trigger` 只能写在正文
2. 不能写进 `rolljson/outcomes/desc/dc_reason`
3. `trigger_pack` 是可选块，不是必出
4. `trigger_pack.sid` 通过 `rh-trigger sourceId` 关联
5. 每轮最多 1~2 个 `trigger_pack`
6. 只给关键情报 trigger 使用
7. `reveal=instant|delayed` 的语义

### 7.2 推荐压缩后的 schema
```json
{
  "type":"trigger_pack",
  "version":"1",
  "defaults":{"dice":"1d20","compare":">="},
  "items":[
    {"sid":"str","skill":"str","difficulty":"easy|normal|hard|extreme","reveal":"instant|delayed","success":"str","failure":"str","explode":"str?"}
  ]
}
```

### 7.3 推荐压缩后的规则文本
建议压缩成这种风格：

```text
【交互触发协议】
1. rh-trigger 只能写在最终剧情正文里，不能写入 rolljson/outcomes/desc/dc_reason。
2. trigger_pack 可选，不是每轮必出。
3. trigger_pack 仅用于少量关键情报 trigger，sid 必须对应 rh-trigger 的 sourceId。
4. 每轮最多 1~2 个 trigger_pack。
5. reveal=instant 表示命中后立即给短反馈；reveal=delayed 表示走后续体现。
6. success/failure/explode 必须简短，尽量 25~45 字。
```

### 7.4 当前可直接删除的冗余说明
建议移除或合并：

- 多处重复强调 trigger 不能塞进 outcomes
- 多处重复强调 trigger 只能写在正文里
- 长篇“关键线索 / 氛围描写 / 被动发现”解释
- 过长的 trigger 策略 prose

保留最关键约束即可。

---

## 8. 状态系统扩展块（仅 `enableStatusSystem` 开启时）

### 8.1 当前问题
当前状态系统说明相对集中，但仍有一点解释性 prose 偏长。

### 8.2 建议保留
仅保留：

- 标签语法
- 正负号约束
- `turns=perm`
- scope / skills 支持方式

### 8.3 推荐压缩格式
```text
【状态标签】
- [APPLY_STATUS:名,整数值,turns=2,skills=A|B 或 scope=all]
- [REMOVE_STATUS:名]
- [CLEAR_STATUS]
- 负面状态必须为负数，正面状态必须为正数。
- turns 默认 1；turns=perm 表示永久。
```

### 8.4 建议删除
- “避免破坏平衡”的长句可压成一句
- 不必每轮都过多解释状态设计理念

---

## 9. 爆骰扩展块（仅 `enableExplodingDice` 开启时）

### 9.1 建议保留
只保留：

- `!` 才允许 `explode`
- 没有 `!` 就不能写 `outcomes.explode`
- AI 自动检定时最多 1 个爆骰事件（如果启用了 auto mode）

### 9.2 推荐压缩格式
```text
【爆骰规则】
1. 只有 checkDice 含 ! 时，才允许写 outcomes.explode。
2. 没有 ! 就必须省略 explode。
3. ! 与 kh/kl 不能同用。
4. 若启用 auto 检定，同一轮最多仅 1 个事件使用 !。
```

### 9.3 建议删除
- 长篇解释“爆骰为何触发”
- 重复说明“系统决定是否爆骰”
- 多段示例

保留一条短正例就够。

---

## 10. 优势/劣势扩展块（仅 `enableAdvantageSystem` 开启时）

### 10.1 建议保留
只保留：

- `2d20kh1 / 2d20kl1`
- 只保留 1 颗结果
- 不要把两次结果相加
- 生成前要检查阈值是否可达

### 10.2 推荐压缩格式
```text
【优势/劣势规则】
1. 可用 advantageState 或 2d20kh1 / 2d20kl1。
2. kh/kl 只保留 1 颗 d20，不是两次结果相加。
3. 写事件前必须检查阈值是否可达，避免理论上必败或必成。
```

### 10.3 建议删除
- 多段重复举例
- 重复说明与 explode 不能同用（可只在 explode 块保留一次）

---

## 11. 时间限制扩展块（仅 `enableTimeLimit` 开启时）

建议只保留一条：

```text
【时间限制】
可填写 timeLimit，且必须不小于 <dice_runtime_policy> 给出的最小时限。
```

---

## 12. 动态 DC 原因扩展块（仅 `enableDynamicDcReason` 开启时）

建议只保留一条：

```text
【难度说明】
可填写 dc_reason，用于解释 difficulty 的叙事依据。
```

---

## 13. 叙事代价扩展块（仅 `enableNarrativeCostEnforcement` 开启时）

### 13.1 建议保留
只保留：

- 成功不要无故追加重罚
- 失败必须有明确代价
- 大失败要显著，但不能把剧情写死

### 13.2 推荐压缩格式
```text
【叙事约束】
1. 成功可以推进，不要无故追加重罚。
2. 失败必须附带明确代价（时间损失、误判、暴露、资源消耗、引发下一风险之一）。
3. 大失败必须显著，但仍保留后续行动空间。
```

---

## 14. 第三层：运行时策略块压缩

当前 `<dice_runtime_policy>` 有价值，但可以更精简。

### 14.1 建议保留的字段
这些字段会直接影响 AI 如何写 JSON，建议保留：

- `apply_scope`
- `round_mode`
- `roll_mode_allowed`
- `round_control_allowed`
- `explode_enabled`
- `ai_auto_explode_event_limit_per_round`
- `advantage_enabled`
- `dynamic_dc_reason_enabled`
- `status_system_enabled`
- `status_tags_allowed`
- `outcome_branches_enabled`
- `explode_outcome_enabled`
- `time_limit_enabled`
- `min_time_limit_seconds`
- `enabled_dice`
- `skill_system_enabled`

### 14.2 建议删除的字段
这些字段更偏展示层 / 摘要层，对 AI 生成协议帮助弱，建议移除：

- `summary_detail`
- `summary_rounds`
- `summary_include_outcome`
- `list_outcome_preview`
- `skill_table_count`
- `skill_table_preview`

### 14.3 为什么可以删
因为这些内容：

- 不影响 AI 是否能写出合法 rolljson
- 不影响 AI 是否能理解 trigger / outcomes / status
- 更像插件自身 UI / 摘要行为，不是协议约束

## 17. 推荐新的总组装流程

建议把 `buildFinalRuleTextEvent()` 改成类似：

```ts
const blocks = [
  buildCoreDiceProtocolBlockEvent(settings),
  settings.enableInteractiveTriggers ? buildInteractiveTriggerProtocolBlockEvent(settings) : "",
  settings.enableStatusSystem ? buildStatusProtocolBlockEvent(settings) : "",
  settings.enableExplodingDice ? buildExplodeProtocolBlockEvent(settings) : "",
  settings.enableAdvantageSystem ? buildAdvantageProtocolBlockEvent(settings) : "",
  settings.enableTimeLimit ? buildTimeLimitProtocolBlockEvent(settings) : "",
  settings.enableDynamicDcReason ? buildDcReasonProtocolBlockEvent(settings) : "",
  settings.enableNarrativeCostEnforcement ? buildNarrativeConstraintProtocolBlockEvent(settings) : "",
  userCustomRuleText,
].filter(Boolean)
```

### 优点
- 各功能只在开启时发
- 后续维护更清晰
- 删除冗余更容易
- 可以单独测试每个协议块

---

## 18. 增加 Prompt Verbosity 模式

建议新增一个设置项：

```ts
promptVerbosityMode: "compact" | "verbose"
```

### 18.1 默认值
建议默认：

- `compact`

### 18.2 compact 模式
用于正常使用，特点：

- 只发最小协议
- 少示例
- 少 prose
- runtime policy 精简

### 18.3 verbose 模式
用于：

- 新模型适配
- 模型总写错格式时
- 调试阶段

特点：

- 保留更多解释
- 保留更多示例
- 更强约束冗余

### 18.4 推荐实现方式
每个协议块函数里可以按 `promptVerbosityMode` 决定：

- 用短版模板
- 还是长版模板

## 24. 分阶段实施顺序

### 第一阶段：快速止血
1. `triggerpack` 规则改为仅在 `enableInteractiveTriggers` 开启时注入
2. 精简 `<dice_runtime_policy>` 删除低价值字段
3. 删除明显重复的长 prose

### 第二阶段：结构重构
1. 拆分 `buildDynamicSystemRuleTextEvent()`
2. 改成协议块拼装
3. 每个功能开关注入自己的扩展块

### 第三阶段：可配置详细度
1. 加 `promptVerbosityMode`
2. 支持 compact / verbose
3. 为不同模型提供切换能力

### 第四阶段：验证
1. 验证 rolljson 格式是否仍稳定
2. 验证 triggerpack 是否仍能正确输出
3. 验证 rh-trigger 是否仍放在正文
4. 对比 token 消耗前后差异

## 25. 验收标准

### 25.1 协议正确性
- [ ] AI 仍能输出合法 ` ```rolljson `
- [ ] AI 仍能正确填写 `dice_events`
- [ ] AI 仍能理解 `outcomes`
- [ ] AI 仍能理解 `rh-trigger`
- [ ] AI 仍能理解 `trigger_pack`
- [ ] AI 仍能理解 `status tags`

### 25.2 冗余减少
- [ ] 同一规则不再跨多个段落重复出现
- [ ] `triggerpack` 不会在交互功能关闭时继续注入
- [ ] runtime policy 只保留高价值字段

### 25.3 token 优化
- [ ] 核心规则块长度明显下降
- [ ] 开关关闭时不会注入无关功能说明
- [ ] compact 模式下 token 成本显著低于当前版本

### 25.4 可维护性
- [ ] 每个功能块能单独调整
- [ ] 后续新增规则不需要继续塞进一个巨型函数
- [ ] verbose / compact 模式能平滑切换

---

## 26. 推荐的最终优化方向

不是把规则删到 AI 看不懂，而是做到：

> **核心规则极简稳定，扩展规则按功能开关注入，运行时约束只保留真正影响生成决策的字段。**

这样可以同时满足：

- AI 仍然理解功能
- JSON 仍然能完整填写
- token 成本下降
- 提示词维护难度下降

---

## 27. 一句话结论

> **把当前“超长总说明书式提示词”改造成“最小核心协议 + 功能扩展块 + 精简运行时策略”，并增加 compact / verbose 两档模式。**
