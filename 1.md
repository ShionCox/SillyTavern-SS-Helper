# MemoryOS「记忆过滤器」模块重构实施计划

> 目标：删除旧「内容拆分台 / ContentLab」实现，重新建设一个更清晰、更好用、可统一接入 MemoryOS 各核心环节的「记忆过滤器」模块。  
> 本计划不考虑向后兼容，旧代码可直接删除或重写。

---

## 1. 重构背景

当前 MemoryOS 已存在一套「内容拆分台」相关逻辑，主要包含：

- `ContentLabSettings`
- `content-tag-registry.ts`
- `content-block-pipeline.ts`
- `content-splitter.ts`
- `content-block-parser.ts`
- `content-block-classifier.ts`
- `tabContentLab.ts`

现有模块的问题不在于底层能力完全缺失，而在于产品定位和使用体验不清晰：

1. **模块名称偏调试工具**  
   “内容拆分台”更像开发者工具，用户很难理解它会影响 MemoryOS 的哪些主流程。

2. **总开关不够明确**  
   虽然已有 `enableContentSplit`，但用户无法直观看出这就是“是否启用记忆过滤”。

3. **生效范围不透明**  
   用户不知道该功能是否影响自动总结、旧聊天接管、梦境召回、向量索引、Prompt 注入等环节。

4. **预览结果不等于最终送模内容**  
   当前主要展示“拆分结果”，但用户真正关心的是 MemoryOS 最终会读取哪些内容。

5. **配置结构偏复杂**  
   XML、分隔符、正则、Markdown、JSONPath 多种模式混合在一个页面里，容易造成理解成本过高。

---

## 2. 新模块定位

新模块名称：

```txt
记忆过滤器 / Memory Filter
```

核心定位：

```txt
控制 MemoryOS 在生成记忆前读取哪些内容。
```

过滤器将聊天楼层内容分为三类：

| 分类 | 说明 | 是否进入记忆 |
|---|---|---|
| 进入记忆 | 正文、剧情、有效对话、可生成长期记忆的内容 | 是 |
| 仅作参考 | 摘要、上下文提示、辅助信息 | 不直接生成记忆，但可作为辅助上下文 |
| 完全排除 | 思考过程、调试信息、指令、作者注释、无关内容 | 否 |

---

## 3. 重构原则

### 3.1 不做旧代码兼容

旧的 `ContentLab` / `ContentSplit` / `ContentTagRegistry` 命名体系全部废弃。

推荐删除或重写：

```txt
MemoryOS/src/config/content-tag-registry.ts
MemoryOS/src/memory-takeover/content-block-pipeline.ts
MemoryOS/src/memory-takeover/content-splitter.ts
MemoryOS/src/memory-takeover/content-block-parser.ts
MemoryOS/src/memory-takeover/content-block-classifier.ts
MemoryOS/src/ui/workbenchTabs/tabContentLab.ts
```

替换为：

```txt
MemoryOS/src/memory-filter/
MemoryOS/src/ui/workbenchTabs/tabMemoryFilter.ts
```

---

### 3.2 统一入口

所有 MemoryOS 需要读取聊天楼层并送给模型的流程，都必须经过同一个入口：

```ts
filterMemoryMessages(messages, settings)
```

不要让自动总结、接管、梦境、向量各自实现过滤逻辑。

---

### 3.3 总开关优先

配置里必须保留并突出：

```ts
enabled: boolean
```

当 `enabled === false` 时：

- 不拆分；
- 不过滤；
- 不改变原始楼层内容；
- 所有链路按原逻辑处理。

---

### 3.4 最终送模预览优先

UI 不能只展示“拆了多少块”，必须展示：

```txt
MemoryOS 最终会读取什么？
```

也就是：

- 进入记忆内容；
- 仅作参考内容；
- 完全排除内容；
- 生效范围；
- 诊断信息。

## 7. 核心服务设计

### 7.1 主入口

```ts
export function filterMemoryMessages<T extends MemoryFilterMessage>(
  messages: T[],
  settings: MemoryFilterSettings = getMemoryFilterSettings(),
): MemoryFilterPreparedResult<T> {
  if (!settings.enabled) {
    return buildDisabledFilterResult(messages, settings);
  }

  const records = messages.map((message, index) =>
    buildMemoryFilterFloorRecord(message, index, settings)
  );

  return assembleMemoryFilterPreparedResult(messages, records, settings);
}
```

---

### 7.2 禁用状态逻辑

当过滤器关闭时：

```ts
function buildDisabledFilterResult<T extends MemoryFilterMessage>(
  messages: T[],
  settings: MemoryFilterSettings,
): MemoryFilterPreparedResult<T> {
  return {
    enabled: false,
    mode: settings.mode,
    messagesForMemory: [...messages],
    contextText: '',
    excludedText: '',
    records: [],
    diagnostics: [],
  };
}
```

---

### 7.3 启用状态逻辑

当过滤器开启时：

1. 根据当前模式拆分楼层；
2. 根据规则分配到三类通道；
3. 清理空白、外层标签、空段；
4. 生成楼层记录；
5. 组装最终结果。

最终模块必须满足：

1. 有明确总开关，可设置是否开启过滤功能。
2. 关闭后不影响原始 MemoryOS 行为。
3. 开启后自动总结、旧聊天接管、梦境召回、向量索引、Prompt 注入都走统一过滤入口。
4. UI 能展示生效范围。
5. UI 能展示最终送模预览。
6. 支持 XML、分隔符、正则、Markdown、JSON 字段五种模式。
7. 支持三通道：进入记忆、仅作参考、完全排除。
8. 支持未知标签策略。
9. 旧 ContentLab 相关代码清理干净。
10. 测试覆盖开启、关闭、未知标签、各核心链路。
