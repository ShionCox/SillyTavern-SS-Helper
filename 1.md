# SillyTavern-SS-Helper 结构化回合总结输入优化方案（基于 `b55a870`）

## 目标

直接缩减 **结构化记忆总结 Planner 阶段** 的输入体积，解决“AI 总结第一部分内容仍然很多”的问题。  
本次方案不讨论抽象思路，下面只写 **应该修改什么内容、替换成什么结构、如何落地**。

---

## 一、先明确本次要改的核心问题

当前 Planner 输入里，真正膨胀的不是“本轮待总结楼层”本身，而是以下几类内容一起堆进去：

1. **当前窗口文本**
   - `window.summaryText`

2. **历史结构化回合总结**
   - `recentSummaryDigest`
   - 这里的多条 `结构化回合总结` 实际上是**历史摘要回灌**
   - 不是当前这次要总结的楼层本体

3. **重复提示信息**
   - `detectedSignals`
   - `plannerHints`

4. **过重的候选记录**
   - `candidateRecords`

5. **Planner 阶段不必携带的全量类型定义**
   - `typeSchemas`

所以本次修改目标很明确：

> 把 Planner 从“吃一大堆历史叙事文本 + 重复辅助数据”  
> 改成“只吃当前窗口事实 + 单条滚动摘要 + 轻量候选卡片”。

---

# 二、直接修改内容

---

## 修改 1：删除 `recentSummaryDigest` 的多条长摘要回灌，改成单条 `rollingDigest`

### 现在的问题

当前输入会把多条历史 `结构化回合总结` 放进 `recentSummaryDigest`，而且每条 `content` 很长，接近重新讲一遍剧情。  
这会导致 Planner 在处理当前窗口前，先吞进去大量历史文本。

### 直接修改内容

### 删除
- 删除 Planner 输入中的：
  - `recentSummaryDigest: Array<{ title, content, ... }>`
- 不再向 Planner 注入多条历史长摘要

### 新增
新增单条轻量滚动摘要：

```json
{
  "rollingDigest": {
    "summary": "上一阶段的核心剧情摘要",
    "openThreads": [
      "未解决问题1",
      "未解决问题2",
      "未解决问题3"
    ],
    "recentDecisions": [
      "最近发生的重要决定1",
      "最近发生的重要决定2"
    ]
  }
}
```

### 字段约束

#### `rollingDigest.summary`
- 最大建议：`220` 个汉字以内
- 只写：
  - 当前长期上下文
  - 人物关系变化
  - 任务推进结果
- 不写完整对白复述
- 不写动作细节流水账

#### `rollingDigest.openThreads`
- 最多 `5` 条
- 每条最多 `30~40` 字
- 只保留：
  - 仍未解决的问题
  - 仍待确认的信息
  - 仍进行中的任务

#### `rollingDigest.recentDecisions`
- 最多 `3` 条
- 只写最近一阶段真正影响后续记忆判断的决定

### 构建逻辑修改

在“历史摘要聚合”模块中，把原来“取最近 N 条结构化回合总结拼起来”的逻辑改成：

1. 只读取最近一次已固化摘要
2. 从中提炼出：
   - `summary`
   - `openThreads`
   - `recentDecisions`
3. 如果没有历史摘要，则返回空对象：

```json
{
  "rollingDigest": {
    "summary": "",
    "openThreads": [],
    "recentDecisions": []
  }
}
```

### 预期效果

这一项通常就是最大减重项。  
它能直接避免 **多条历史结构化回合总结重复进入 Planner**。

## 修改 2：把 `window.summaryText` 改造成 `windowFacts`，不要再给 Planner 喂长叙事文本

### 现在的问题

`window.summaryText` 现在更像“剧情式概述”，仍然偏长。  
Planner 实际要做的是判断“该更新哪些记忆、更新哪些实体、重点是什么”，它并不需要整段文学式复述。

### 直接修改内容

### 替换
把：

```json
{
  "window": {
    "fromTurn": 22,
    "toTurn": 31,
    "summaryText": "很长的一段叙事总结……"
  }
}
```

替换为：

```json
{
  "window": {
    "fromTurn": 22,
    "toTurn": 31,
    "turnCount": 10,
    "windowFacts": [
      "毅毅已支付2500定金",
      "艾莉卡收下定金但未正式接单",
      "艾莉卡要求补充宝石外形和来历信息",
      "毅毅迟到后再次出现",
      "艾莉卡认定原约定已失效，但仍允许补充说明"
    ]
  }
}
```

### `windowFacts` 生成规则

- 最多 `12` 条
- 每条尽量控制在 `15~40` 字
- 只保留：
  - 已发生事件
  - 承诺/拒绝/交易/关系变化
  - 可进入结构化记忆的事实
- 删除：
  - 语气描写
  - 场景修辞
  - 连续动作细节
  - 无法形成记忆价值的对白内容

### 推荐生成策略

当前窗口先经过一次本地压缩，再发给 Planner：

#### 建议输出类型
- 事件事实
- 关系变化
- 任务推进
- 状态变化
- 冲突/拒绝/接受

### 不要保留的内容
- “她看了他一眼”
- “气氛逐渐变得……”
- “语气变冷”
- “沉默了几秒”
- 除非这些内容会直接影响记忆判断，否则全部丢弃

### 预期效果

这一项会把“本轮楼层文本”从剧情摘要，压缩为真正适合 Planner 使用的事实列表。

---

## 修改 3：合并 `detectedSignals` 与 `plannerHints`，二者不能同时保留完整版本

### 现在的问题

这两部分本质上都在描述：

- 哪些类型可能要更新
- 为什么可能需要更新
- 关注重点是什么

现在同时传，会出现明显信息重叠。

### 直接修改内容

### 删除
删除 Planner 输入中的完整 `plannerHints`

### 保留
保留一份极简 `signalPack`：

```json
{
  "signalPack": {
    "candidateTypes": ["task", "relationship", "event"],
    "focusPoints": [
      "是否形成新的委托任务",
      "是否存在关系态度变化",
      "是否有新的长期未解决线索"
    ],
    "shouldUpdate": true
  }
}
```

### 规则

#### `candidateTypes`
- 最多 `5` 个

#### `focusPoints`
- 最多 `5` 条
- 每条最多 `30` 字

#### `shouldUpdate`
- 布尔值即可
- 不要再附带长篇原因解释

### 本地逻辑改法

原来如果是：

- 先检测信号
- 再生成 plannerHints
- 然后两者一起传入 Planner

改成：

- 本地检测信号
- 本地合并成一个 `signalPack`
- 只向 Planner 发 `signalPack`

### 预期效果

减少重复提示，提升 Planner 对输入重点的聚焦。

---

## 修改 4：把 `candidateRecords` 改成 `candidateCards`

### 现在的问题

`candidateRecords` 太重，常常包含：

- `recordId`
- `title`
- `summary`
- `entityKeys`
- `status`
- `updatedAt`
- `sourceHint`

Planner 阶段并不需要读完整记录，它只需要知道：

- 这个候选对象是什么
- 和当前窗口有没有关系
- 是否值得更新

### 直接修改内容

### 替换
把：

```json
{
  "candidateRecords": [
    {
      "recordId": "...",
      "title": "...",
      "summary": "...",
      "entityKeys": ["..."],
      "status": "...",
      "updatedAt": "...",
      "sourceHint": "..."
    }
  ]
}
```

替换为：

```json
{
  "candidateCards": [
    {
      "id": "cand_1",
      "type": "task",
      "brief": "寻找天顶区西城红色宝石，已支付2500定金，尚未正式接单",
      "entities": ["毅毅", "艾莉卡", "红色宝石"],
      "state": "active"
    },
    {
      "id": "cand_2",
      "type": "relationship",
      "brief": "艾莉卡仍对毅毅保持高度戒备，但未彻底拒绝后续合作",
      "entities": ["毅毅", "艾莉卡"],
      "state": "unstable"
    }
  ]
}
```

### 字段保留规则

#### 必留
- `id`
- `type`
- `brief`
- `entities`
- `state`

#### 删除
- `updatedAt`
- `sourceHint`
- 长 `summary`
- 细节化元数据
- 不影响本次判断的调试字段

### 长度限制
- `candidateCards` 最多 `8` 条
- 每条 `brief` 建议不超过 `60` 字

### 预期效果

让 Planner 快速理解“当前有哪些可能要改的记忆对象”，而不是被迫读记录详情。

---

## 修改 5：把全量 `typeSchemas` 从 Planner 阶段移到 Mutation 阶段

### 现在的问题

Planner 阶段的任务是：

- 判断改不改
- 改哪些类型
- 关注哪些实体
- 需要新增、修改还是忽略

它不是最终写库阶段。  
所以没必要在这一步把全量 `typeSchemas` 都发进去。

### 直接修改内容

### 删除
删除 Planner 输入中的：

```json
{
  "typeSchemas": {
    "task": { "editableFields": ... },
    "relationship": { "editableFields": ... },
    "event": { "editableFields": ... },
    ...
  }
}
```

### 替换为
只保留一个轻量类型许可表：

```json
{
  "allowedTypes": [
    "task",
    "relationship",
    "event",
    "profile",
    "world_rule"
  ]
}
```

### 真正的 `typeSchemas` 在哪里加载
把完整 `typeSchemas` 延后到 **Mutation 阶段**：

- 当 Planner 输出：
  - 这次要处理 `task`
  - 要处理 `relationship`

再在 Mutation 阶段按需加载：

```json
{
  "mutationContext": {
    "selectedTypes": ["task", "relationship"],
    "typeSchemas": {
      "task": { "editableFields": ... },
      "relationship": { "editableFields": ... }
    }
  }
}
```

### 预期效果

避免 Planner 阶段承担不属于它的 schema 负担。  
也能让 Mutation 阶段使用更准确的字段约束。

---

## 修改 6：把 Planner 拆成“轻判断”，把“重规则”放到 Mutation

### 现在的问题

当前 Planner 输入太像“半个执行器”，什么都要看，导致输入越来越重。

### 直接修改内容

### Planner 只负责输出：

```json
{
  "plan": {
    "shouldUpdate": true,
    "create": [
      {
        "type": "task",
        "reason": "形成新的委托关系"
      }
    ],
    "update": [
      {
        "candidateId": "cand_2",
        "type": "relationship",
        "reason": "态度发生变化"
      }
    ],
    "ignore": [
      {
        "fact": "普通场景动作，不构成记忆更新"
      }
    ]
  }
}
```

### Planner 不再负责：
- 直接看到完整字段 schema
- 直接处理太多旧摘要详情
- 直接参考大量历史候选记录明细
- 直接做最终字段级 patch

### Mutation 阶段负责：
- 读取 Planner 决策
- 拉取需要操作的候选记录详情
- 拉取所需 `typeSchemas`
- 生成最终增删改 JSON

### 预期效果

模块边界清晰后，Planner 可以稳定保持轻量，不再无限膨胀。

---

## 修改 7：新增统一的输入体积硬限制

### 现在的问题

即使完成上述改造，如果没有体积上限，后续仍可能慢慢失控。

### 直接修改内容

在 Planner 输入组装模块增加硬限制：

### 推荐限制

#### `rollingDigest.summary`
- 最大 `220` 字

#### `rollingDigest.openThreads`
- 最大 `5` 条

#### `windowFacts`
- 最大 `12` 条

#### `signalPack.focusPoints`
- 最大 `5` 条

#### `candidateCards`
- 最大 `8` 条

#### Planner 总文本预算
建议增加一个总预算检查，例如：

- 中文内容总长度超过 `3000~4500` 字时，强制再压缩
- 或按 token 预算控制

### 本地兜底逻辑

如果超限，压缩顺序如下：

1. 先裁掉 `candidateCards` 尾部低优先级项
2. 再裁掉 `rollingDigest.recentDecisions`
3. 再裁掉 `openThreads` 尾部项
4. 最后压缩 `windowFacts` 中低价值事实

### 不要裁掉
- `fromTurn / toTurn`
- `shouldUpdate`
- `candidateTypes`
- 当前窗口中的高价值事实

---

## 修改 8：给旧结构保留兼容层，但新流程一律走轻量结构

### 现在的问题

仓库里已经有旧数据结构和旧摘要产物，不能一次性全打断。

### 直接修改内容

新增兼容转换层：

### 兼容输入
如果读到旧的：

- `recentSummaryDigest`
- `window.summaryText`
- `candidateRecords`
- `plannerHints`
- `typeSchemas`

则在进入 Planner 前统一转换为新格式：

| 旧字段 | 新字段 |
|---|---|
| `recentSummaryDigest` | `rollingDigest` |
| `window.summaryText` | `windowFacts` |
| `candidateRecords` | `candidateCards` |
| `detectedSignals + plannerHints` | `signalPack` |
| `typeSchemas` | `allowedTypes` |

### 注意
兼容层只负责转换，**不允许旧结构直接透传给 Planner**。

也就是说：

- 旧数据可以存在
- 但进入 Planner 前必须压缩成新格式

# 四、建议的模块修改顺序

下面直接按实施顺序写：

## 第一阶段：先减重
1. 删 `recentSummaryDigest` 多条长摘要注入
2. 上 `rollingDigest`
3. 把 `window.summaryText` 改成 `windowFacts`

## 第二阶段：去重
4. 合并 `detectedSignals` 和 `plannerHints`
5. 把 `candidateRecords` 改成 `candidateCards`

## 第三阶段：重新分层
6. 移除 Planner 阶段全量 `typeSchemas`
7. 在 Mutation 阶段按需加载 schema

## 第四阶段：防回退
8. 增加输入体积硬限制
9. 增加旧结构兼容转换层
10. 禁止旧结构直接透传给 Planner

# 五、预期结果

完成以上修改后，效果应该是：

1. **“结构化回合总结”不再以多条长文本形式重复出现在 Planner 输入里**
2. **当前楼层只保留事实，不再保留大量剧情复述**
3. **Planner 只负责做轻判断，不再兼做执行前准备**
4. **Mutation 才加载完整 schema 和详细记录**
5. **后续即使历史摘要变多，也不会再次把 Planner 输入撑爆**

# 七、结论

这次不是“AI 总结楼层设置太大”，而是：

> **历史结构化回合总结被当成长文本重复回灌进 Planner**，  
> 再叠加重复信号和过重候选记录，导致“第一部分内容很多”。

所以正确改法不是单纯调楼层，而是：

> **重做 Planner 输入结构，严格把历史摘要、当前窗口、候选记录、类型定义分层。**
