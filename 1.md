# RollHelper 暗投（暗骰）优化方案

## 1. 文档目标

本文用于整理当前 **RollHelper 暗投系统** 的现状、主要问题、优化方向与分阶段落地方案，目标是把现有“可用”的暗投能力，继续提升为：

- **不泄露**
- **不串轮次**
- **不串楼层**
- **不滥用**
- **可重投但不可变可见性**
- **可追踪但不暴露真实结果**
- **便于后续继续扩展**

本文默认基于当前最新版代码结构展开，重点覆盖以下模块：

- `src/events/roundEvent.ts`
- `src/events/promptEvent.ts`
- `src/events/hooksEvent.ts`
- `src/events/renderEvent.ts`
- `src/events/summaryEvent.ts`
- `src/commands/baseRollCommandEvent.ts`
- `src/templates/eventCardTemplates.ts`
- `src/types/eventDomainEvent.ts`

---

## 2. 当前状态评估

从当前实现来看，暗投系统已经不再是早期的“只有一个 `/broll` 命令”的状态，而是已经形成了完整链路：

### 2.1 已具备的能力

#### 2.1.1 暗投结果隐藏
当前版本已经能在多个层面隐藏暗投真实结果：

- 结果卡不展示真实点数、DC、修正、走向正文
- 事件列表摘要不再拼出暗投的结果细节
- 公开轮次摘要与暗投摘要已经拆分为两套通道
- Prompt 注入层已支持单独的暗投引导块

#### 2.1.2 轮次与楼层绑定
当前暗投引导已开始绑定上下文：

- 绑定 `roundId`
- 绑定 `sourceAssistantMsgId`
- 提取 `sourceFloorKey`
- 在 Prompt 注入时校验当前 open round
- 在楼层失效时同步清理对应暗投引导

#### 2.1.3 暗投队列治理
当前版本已支持一定程度的滥用限制：

- 每轮暗投次数限制
- 队列长度限制
- TTL 过期
- 基础去重
- 非法轮次拒绝入队

#### 2.1.4 暗投重投可见性继承
当前重投逻辑已经基本正确：

- 前端按钮支持 `data-reroll-visibility`
- hooks 层会按 blind/public 分流
- 后端执行时会以历史记录的 visibility 作为最终依据
- 避免“暗投重投后变成公开结果”的问题

#### 2.1.5 暗投历史展示
当前版本已加入暗投历史列表：

- 只记录安全字段
- 不存真实结果
- 提供工具栏入口查看
- 可用于提示“哪些暗投已经发生过”

## 3. 当前仍存在的问题

虽然主链路已经成型，但暗投系统还没有完全打磨完成，仍有几类问题需要继续优化。

---

## 4. 问题一：`/broll` 参数解析过于粗糙

### 4.1 当前问题

当前 `/broll` 的逻辑基本是：

- 如果参数像骰式，则按骰式处理
- 否则整段文本直接视为 `skillName`

这会导致下面这种写法出现问题：

```text
/broll 调查宝箱
/broll 察觉门后动静
/broll 洞察店主表情
```

如果白名单里配置的是：

- 调查
- 察觉
- 洞察

那么：

- `调查宝箱`
- `察觉门后动静`
- `洞察店主表情`

都不会命中白名单，最终会被误判成“不允许暗投”。

### 4.2 风险

这会直接破坏暗投命令的自然输入体验，用户会觉得：

- 白名单不好用
- 暗投像“只能输技能名，不能输目标描述”
- 和事件内交互触发的使用习惯不统一

### 4.3 优化目标

将 `/broll` 从“单字符串解释模式”升级为“技能 + 目标 / 备注”的结构化解析模式。

### 4.4 推荐方案

#### 方案 A：首词视为技能，剩余视为目标
例如：

```text
/broll 调查 宝箱
/broll 察觉 门后动静
/broll 洞察 店主表情
```

解析为：

- `skillName = 调查`
- `targetLabel = 宝箱`

优点：

- 最容易上手
- 对现有命令格式改动最小
- 和白名单机制天然兼容

#### 方案 B：支持命名参数
例如：

```text
/broll skill=调查 target=宝箱
/broll skill=察觉 note=门后似乎有声音
```

优点：

- 更严谨
- 更适合复杂目标说明
- 后续扩展字段更方便

缺点：

- 输入更长
- 对普通用户不如方案 A 自然

### 4.5 最终建议

建议先实现：

1. **默认首词为技能、剩余为目标**
2. 再兼容 `skill=` / `target=` 命名参数
3. 两种写法都允许

### 4.6 落地建议

新增解析函数，例如：

```ts
type ParsedBlindRollInput = {
  diceExpr?: string;
  skillName?: string;
  targetLabel?: string;
  note?: string;
};
```

由专门函数负责：

- 判断是否是骰式
- 判断是否包含命名参数
- 拆出技能名与目标标签

## 5. 问题二：`blindDedupScope` 字段尚未真正发挥作用

### 5.1 当前问题

类型层已经存在：

- `blindDedupScope: "same_round" | "same_floor"`

但当前 dedupe 逻辑主要还是由 `buildBlindGuidanceDedupKeyEvent()` 的固定规则决定。

也就是说：

- 字段已经定义
- 设置层未来可能也会暴露
- 但实际逻辑还没有真正根据这个设置切换去重策略

### 5.2 风险

这会导致两个问题：

#### 5.2.1 设置“看起来可配，实际上没生效”
用户或开发者会误以为：

- same_round
- same_floor

已经生效了，但实际上 dedupeKey 还是固定生成方式。

#### 5.2.2 后续维护时容易造成误判
未来继续改代码时，容易出现：

- UI 一套理解
- 逻辑另一套实现
- 最终行为不符合设置预期

### 5.3 优化目标

让 `blindDedupScope` 真正接入 dedupeKey 生成与入队校验流程。

### 5.4 推荐方案

#### `same_round`
去重范围限定在：

- 同一 `roundId`

适合多数暗投场景。

#### `same_floor`
去重范围限定在：

- 同一 `sourceFloorKey`

适合“同一楼层同一线索不要重复暗投”的玩法。

### 5.5 落地建议

把当前：

```ts
buildBlindGuidanceDedupKeyEvent(...)
```

调整为：

```ts
buildBlindGuidanceDedupKeyEvent(input, settings)
```

然后根据 `settings.blindDedupScope` 决定 key 的组成规则。

例如：

#### round 级去重
```text
slash:{roundId}:{skill}:{targetLabel}
```

#### floor 级去重
```text
slash:{sourceFloorKey}:{skill}:{targetLabel}
```

### 5.6 补充建议

最好不要只改 key 生成，还要在日志里明确打出：

- 当前 dedupeScope
- 当前生成的 dedupeKey
- 是因哪个 existing item 被拒绝

这样后续排查会轻松很多。

## 6. 问题三：暗投历史只有“发生过”，缺少更清晰的上下文信息

### 6.1 当前问题

现在暗投历史已经有了，但主要记录的是：

- 标题
- 技能
- 目标
- 骰式
- 来源
- 时间

这已经足够安全，但在复杂聊天中，仍可能不够用：

- 不知道它对应哪一轮
- 不知道它来自哪个楼层
- 不知道它是事件暗投、命令暗投还是交互暗投时的具体上下文
- 不知道该暗投是否已经被 Prompt 消费过

### 6.2 优化目标

让暗投历史既不暴露真实结果，又能更好辅助排查与理解上下文。

### 6.3 推荐增强字段

在 UI 层可选展示：

- `roundId`（可截断展示）
- `sourceFloorKey`
- `consumed` 状态
- `expired` 状态
- `origin`
- `sourceAssistantMsgId` 的简化版本

### 6.4 UI 展示建议

暗投列表中每条记录可以追加：

- 所属轮次：`round_xxx`
- 来源楼层：`assistant:123`
- 状态：`待注入 / 已消费 / 已过期`
- 类型：`命令暗投 / 事件暗投 / 交互暗投`

### 6.5 这样做的好处

可以在不暴露结果的前提下，让开发者和用户知道：

- 暗投有没有排进队列
- 暗投有没有真正进入下一次叙事
- 是不是因为换楼层被清掉了
- 是不是因为过期了没生效

## 7. 问题四：暗投后的事件卡“直接消失”是否符合预期，需要产品层确认

### 7.1 当前行为

当前事件列表卡中，会把已经是暗投结果的事件直接从可见列表中过滤掉。

也就是：

- 一旦某事件发生了暗投
- 它不会继续在公开事件列表中显示
- 用户只能在“暗投列表”中看到记录

### 7.2 这不一定是 bug

这个行为本身是自洽的，因为它符合一种设计思路：

> 公开事件面板只展示公开信息，暗投后的内容不应继续占据公开位置。

### 7.3 但这是一项产品取舍

另一种同样合理的设计是：

- 原事件仍保留在原位
- 但状态显示为“已暗投 / 结果隐藏”
- 允许用户知道“这个事件已经处理过”

### 7.4 两种方案对比

#### 方案 A：暗投后从公开事件列表移除
优点：

- 更“干净”
- 不会在公开面板留下过多隐藏卡片
- 更符合“暗投不公开”的直觉

缺点：

- 用户容易觉得事件“突然消失了”
- 上下文感较弱
- 需要额外去暗投列表中找

#### 方案 B：原位保留占位卡
优点：

- 上下文连续
- 用户知道事件已被处理
- 更利于长流程跑团

缺点：

- 公开事件面板会混入“隐藏结果”的占位卡
- 视觉上更复杂

### 7.5 建议

建议增加一个设置项：

```ts
blindEventCardVisibilityMode: "remove" | "placeholder"
```

默认先保持当前行为 `remove`，后续再支持 `placeholder`。

### 7.6 Placeholder 模式建议

占位卡上可只显示：

- 标题
- 对象
- 技能
- 状态：已暗投
- 提示：结果将通过后续叙事体现

不显示：

- 点数
- DC
- 修正
- 走向
- 真实结果

## 8. 问题五：暗投 Prompt 注入策略还可以继续增强

### 8.1 当前状态

当前 Prompt 注入已经具备：

- 只消费当前 open round
- 只消费当前楼层还存在的项
- 过期自动清理
- `consumed` 控制

这个方向已经对了。

### 8.2 仍可继续优化的点

#### 8.2.1 支持更明确的“未消费 -> 已消费 -> 已归档”
当前虽然有 `consumed`，但整个生命周期仍比较轻量。

可以考虑把状态细化为：

```ts
type BlindGuidanceState =
  | "queued"
  | "consumed"
  | "expired"
  | "invalidated"
  | "archived";
```

这样调试和 UI 展示都会更清楚。

#### 8.2.2 限制单次 Prompt 注入条数
如果某一轮暗投很多，下一次 prompt 全塞进去可能会导致：

- 提示词膨胀
- 叙事焦点分散
- AI 同时处理太多隐藏信息

建议增加：

```ts
maxBlindGuidanceInjectedPerPrompt
```

例如默认 `2` 或 `3`。

#### 8.2.3 给 AI 的暗投指引可再强化“只体现，不直说”
暗投注入文本可以更明确强调：

- 允许在叙事中体现结果影响
- 禁止直接说“你暗骰成功 / 失败”
- 禁止把隐藏结果翻译成明牌说明

## 9. 问题六：暗投系统缺少更系统的测试覆盖

### 9.1 当前风险

当前暗投链路已经跨越：

- slash command
- round state
- hook lifecycle
- prompt injection
- render
- summary
- reroll

这类链路如果没有系统测试，很容易在后续改动时回归。

### 9.2 建议补的测试类型

#### 9.2.1 单元测试
重点覆盖：

- `buildBlindGuidanceDedupKeyEvent`
- `canEnqueueBlindGuidanceEvent`
- `pruneExpiredBlindGuidanceQueueEvent`
- `resolvePromptBlindGuidanceInjectionEvent`
- `formatRollRecordSummaryEvent`
- `/broll` 参数解析函数

#### 9.2.2 集成测试
重点覆盖：

- 新事件产生 -> 暗投 -> 下一次用户发言 -> Prompt 注入
- swipe / edit / delete 导致楼层变化 -> 对应暗投失效
- 暗投 -> 重投 -> 保持 blind
- 暗投达到上限 -> 拒绝继续入队
- 队列过期 -> Prompt 不再注入

#### 9.2.3 回归测试
重点覆盖以前出过的问题：

- 暗投结果在结果卡泄露
- 暗投结果在事件列表泄露
- 暗投重投后变公开
- 旧楼层暗投串进新楼层 prompt
- `/broll 调查宝箱` 无法通过白名单

## 10. 推荐的分阶段落地顺序

### 第一阶段：命令解析与去重配置补齐
优先级：**最高**

#### 目标
先修最影响实际使用体验的两个问题：

- `/broll` 输入体验
- `blindDedupScope` 真正生效

#### 工作项

1. 新增 `/broll` 输入解析器
2. 支持“技能 + 目标”拆分
3. 支持命名参数写法
4. 将 `blindDedupScope` 接入 dedupeKey 生成
5. 增加详细日志

#### 验收标准

- `/broll 调查 宝箱` 正常通过
- `/broll 调查宝箱` 能合理解析或给出明确提示
- `same_round` / `same_floor` 行为可观察到不同结果
- 队列拒绝日志能看出原因

### 第二阶段：暗投历史与状态可观测性增强
优先级：**高**

#### 目标
让暗投系统不仅“能用”，还“看得清楚”。

#### 工作项

1. 暗投历史补充状态字段
2. UI 增加：
   - 待注入
   - 已消费
   - 已过期
   - 已失效
3. 展示所属轮次 / 楼层简写
4. 为开发模式增加调试信息输出

#### 验收标准

- 能从暗投列表看出每条暗投的生命周期状态
- 能定位暗投为何没有进入叙事
- 不暴露真实结果

### 第三阶段：事件卡保留策略配置化
优先级：**中**

#### 目标
让“暗投后事件卡是否保留”成为可选产品策略。

#### 工作项

1. 新增 `blindEventCardVisibilityMode`
2. 支持：
   - `remove`
   - `placeholder`
3. Placeholder 模式下渲染占位卡

#### 验收标准

- 两种模式都能正常工作
- Placeholder 不泄露结果
- 切换后不影响重投与摘要逻辑

### 第四阶段：Prompt 注入治理增强
优先级：**中**

#### 目标
进一步避免暗投注入过多、过乱。

#### 工作项

1. 新增单次 prompt 最大暗投注入数量
2. 细化暗投生命周期状态
3. 调整 blind guidance 文本模板
4. 增强“只体现、不明说”的系统约束

#### 验收标准

- 多暗投情况下 prompt 不爆炸
- AI 不会把暗投结果直接说破
- 消费状态可追踪

### 第五阶段：测试与回归护栏补齐
优先级：**高，但可插空推进**

#### 目标
防止暗投系统在后续继续迭代时反复回归。

#### 工作项

1. 补单元测试
2. 补集成测试
3. 固化历史 bug 回归用例
4. 输出开发自测清单

#### 验收标准

- 暗投主链路有自动测试
- 历史关键 bug 可被回归用例覆盖
- 后续改动更安全

## 11. 推荐新增设置项

建议未来把以下设置项逐步补齐：

```ts
type DicePluginSettingsEvent = {
  blindDedupScope: "same_round" | "same_floor";
  blindEventCardVisibilityMode?: "remove" | "placeholder";
  maxBlindGuidanceInjectedPerPrompt?: number;
  enableBlindDebugInfo?: boolean;
};
```

### 推荐默认值

- `blindDedupScope = "same_round"`
- `blindEventCardVisibilityMode = "remove"`
- `maxBlindGuidanceInjectedPerPrompt = 2`
- `enableBlindDebugInfo = false`

## 13. 验收清单

下面这份清单可以直接作为开发完成后的手工验收标准。

### 13.1 命令层
- [ ] `/broll 调查` 正常
- [ ] `/broll 调查 宝箱` 正常
- [ ] `/broll skill=调查 target=宝箱` 正常
- [ ] 非白名单技能会被拒绝
- [ ] 无 open round 时会被拒绝

### 13.2 队列层
- [ ] 同轮次重复暗投能按 dedupeScope 正确拦截
- [ ] 不同楼层暗投不会错误互斥
- [ ] 超出每轮上限会拒绝
- [ ] 超出队列上限会拒绝
- [ ] TTL 过期后不会再注入

### 13.3 Prompt 层
- [ ] 只注入当前 open round 的暗投
- [ ] 已删除楼层对应暗投不会再注入
- [ ] 同一 user prompt 不重复消费同一暗投
- [ ] 一次 prompt 注入数量可控

### 13.4 展示层
- [ ] 结果卡不泄露暗投点数
- [ ] 事件列表不泄露暗投结果
- [ ] 摘要区分公开摘要 / 暗投摘要
- [ ] 暗投历史列表不显示真实结果
- [ ] 重投按钮文案与可见性正确

### 13.5 重投层
- [ ] 公开结果重投仍公开
- [ ] 暗投结果重投仍暗投
- [ ] 前端传错 visibility 时后端仍能纠正
- [ ] 重投后只以最新记录作为当前有效结果

## 14. 结论
