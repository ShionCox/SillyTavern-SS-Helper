# MemoryOS 最新 `memoryos-settings-unified` 分支优化与改进实施方案

> 适用对象：`ShionCox/SillyTavern-SS-Helper`  
> 目标分支：`memoryos-settings-unified`  
> 文档目标：基于当前最新分支的实际代码状态，对**语境系统、混合召回系统、可视化记忆图谱**进行一次正式审查，并给出一份**可直接进入开发排期**的详细优化实施方案。  
> 本文不是从零设计，而是围绕当前已经落地的实现进行“二次收敛”。

---

## 1. 当前版本总体评估

从最新分支来看，MemoryOS 已经不再停留在概念验证阶段，而是具备了比较完整的新架构骨架。

当前已经落地的核心能力包括：

- 检索层已经从单一 provider 选择器，升级成**多阶段混合召回编排器**
- 语境系统已经存在，不再只是简单关键词匹配
- 种子召回、结构扩散、缺项补召回、多样性裁剪已经形成主链
- 工作台已经新增“可视化记忆”顶层 TAB
- 记忆图谱已经支持节点点击、右侧详情展示、跳转条目中心、图交互（拖拽、缩放、高亮）
- 遗忘与记忆度已经与角色视角注入层接上

这说明当前版本的性质已经发生变化：

> **它不是“还没做出来”，而是“已经做出来了一版，需要进入精度与结构合理性打磨阶段”。**

换句话说，当前需要的不是推翻重做，而是围绕以下三个方面做**第二轮优化**：

1. **语境系统识别更贴近真实自然语言**
2. **混合召回更贴近真实记忆关系，而不是规则堆叠**
3. **可视化记忆图谱的连线逻辑更符合实际语义，不只是“能连起来”**

---

## 2. 本次优化的核心目标

本次改进目标不是继续加功能，而是提升当前已有能力的**真实性、解释性、可维护性**。

本次优化完成后，应达到以下状态：

- 语境系统能更自然地识别角色名、别名、地点名、关系对象，而不依赖系统内部键名
- 检索语境不只是供 diagnostics 查看，而是能够真正影响第一轮种子召回
- 图扩散不会因为早期 clamp 或过强 hub 节点而失真
- 缺项补召回不只是“facet 补洞”，而更接近真正的“残差补召回”
- 多样性裁剪在候选数量不大时也能主动做结构平衡
- 可视化记忆图谱的边不再过度依赖“共享主角即连线”这种宽松规则
- 角色画像节点不再成为过强吸附中心
- 图谱支持“简洁模式 / 语义模式 / 调试模式”三套边显示逻辑
- 工作台诊断区可以展示本轮语境路由、边类型命中、扩散前后变化、补召回原因

---

## 3. 当前版本的主要问题归类

这次审查下来，当前代码没有“方向性错误”，主要问题集中在**精度、权重、边语义、可解释性**四类。

### 3.1 语境系统问题
当前 `context-router.ts` 已经具备 facet、锚点、topic hints、置信度输出能力，但识别仍偏“规则和内部键名导向”，自然语言适配度还不够。

主要问题：

- 角色锚点识别仍偏向 `actorKey`，对 `displayName`、别名、自然称呼支持不足
- facet 识别仍以规则词典和句式为主，缺乏“最近上下文状态偏置”
- `contextRoute` 已经生成，但对种子召回 query 的构造影响还不够强
- 当前 route 更像“检索诊断描述”，尚未完全成为“检索入口调度器”

### 3.2 混合召回问题
当前 `RetrievalOrchestrator` 主链正确，但内部各阶段仍有“第一版实现”的痕迹。

主要问题：

- `providerId` 回传可能和真实使用 provider 不一致
- lexical provider 的候选文本仍然过于保守
- graph expansion 过早 clamp，导致累积结构增益被压平
- coverage second pass 更像 facet 补洞器，还不够接近真正残差召回
- diversity pruner 在候选数量未超阈值时，可能不会主动做平衡
- 去重阶段存在数组原地 sort 的副作用风险

### 3.3 可视化记忆图谱问题
当前图谱已经可用，但边语义还偏“浏览友好”，还不够“语义真实”。

主要问题：

- `participants` 建边规则过宽，容易让共享主角的大量记忆挤成团块
- `actorRef` 边权重过高，容易让 `actor_profile` 成为星型中心
- `locationKey` 连线合理性高度依赖 locationKey 颗粒度
- `worldKeys` 和 `tags` 当前被整体禁用，虽然干净，但会削弱世界层结构感
- 图谱当前更像“角色中心浏览图”，还不是“严格语义知识图”
- 目前只有一套固定连线策略，不利于区分浏览模式和分析模式

### 3.4 诊断与调试问题
当前系统已经可以跑，但如果后面继续调优，缺少更细的可视化诊断输出会很难调。

主要问题：

- 当前 diagnostics 对 `contextRoute` 的可视化还不够充分
- graphBoost 的来源边类型不可见
- second pass 为什么触发、补了什么，外部不可追踪
- 图谱里每条边为什么存在，当前工作台没有直接展示

---

## 4. 优化总原则

本次优化建议遵循四个原则。

### 4.1 不推翻当前分层
当前分层已经合理：

- `context-router`
- `lexical / embedding provider`
- `graph-expander`
- `coverage-checker`
- `diversity-pruner`
- `actor-visible-context-builder`
- `workbench memory graph`

这一层次不要再打乱。

### 4.2 先提升“真实性”，再提升“高级性”
当前最重要的不是先上完整 FISTA、完整 DPP、完整向量图谱，而是先让：

- 语境识别更像人话
- 召回更像真实记忆联想
- 图谱连线更像真实结构关系

### 4.3 图谱默认要“可读”，高级模式再“可深挖”
默认视图应优先保证：

- 用户一打开就能看懂
- 不会变成一坨毛线球
- 点击有意义

复杂边关系不应该默认全开，而应通过模式切换显示。

### 4.4 diagnostics 必须同步增强
因为这一版进入的是“精度调校阶段”，调参能力比单纯加功能更重要。

## 5. 语境系统优化实施方案

## 5.1 优化目标

让 `context-router.ts` 从“规则语境分类器”升级成“真实检索语境路由器”。

它最终要回答的不是：

- “这句话像不像冲突”

而是：

- “这句话当前想从哪几类记忆里取信息”
- “这句话命中了哪些角色 / 地点 / 关系 / 世界锚点”
- “这句话的 facet 优先级如何”
- “第一轮召回应当怎么偏置”

## 5.2 当前实现的优点

当前 `context-router.ts` 已经做到了：

- 动态构建候选词典
- 关键词信号匹配
- 实体锚点匹配
- facet 打分
- topicHints 提取
- confidence 输出

这说明结构已经对了，不需要重写。

## 5.3 需要改进的地方

### 5.3.1 实体锚点识别要从“键名导向”改成“自然语言导向”

#### 当前问题
当前 `buildContextDictionaryFromCandidates()` 里：

- `actorNames`
- `actorKeys`

本质上都从候选里的 key 级字段构造，显示名和别名还没有真正进入匹配系统。

#### 改进方案
把 `ContextDictionaryRegistry` 扩展为：

```ts
export interface ContextDictionaryRegistry {
  actorKeys: string[];
  actorNames: string[];
  actorAliases: string[];
  locationKeys: string[];
  locationNames: string[];
  relationPairKeys: string[];
  relationAliases: string[];
  worldKeys: string[];
  worldAliases: string[];
}
```

#### 数据来源
- `ActorMemoryProfile.displayName`
- `ActorMemoryProfile.aliases`
- `MemoryEntry.title`
- `MemoryEntry.tags`
- `relationTags.ts`
- `world profile` 中的设定词

#### 目标效果
用户说：

- “艾琳”
- “小艾”
- “那位情报员”

系统都能尽量锚定到同一角色 key。

### 5.3.2 facet 识别加入“最近上下文偏置”

#### 当前问题
现在语境识别基本只看当前这句 query，最近对话状态没有反向影响 facet 分布。

#### 改进方案
在 `routeRetrievalContext()` 增加一个可选参数：

```ts
recentContext?: {
  sceneMode?: string
  currentLocationKey?: string
  activeRelationPair?: string
  dominantFacetHints?: RetrievalFacet[]
}
```

#### 加权逻辑
如果最近上下文已经是：

- `relationship_conflict`
- 且当前 query 有角色锚点

则 `relationship` facet 额外加权。

如果最近场景是：

- `scene_shared_state`
- 且当前 query 包含地点代词“这里/那里/这地方”

则 `scene` facet 额外加权。

#### 目标效果
让语境识别不再是“孤立句子判定”，而是“上下文中的检索意图判定”。

### 5.3.3 `contextRoute` 必须真正进入第一轮 seed recall

#### 当前问题
现在 orchestrator 里虽然先做了 `contextRoute`，但第一轮种子召回仍然基本拿原始 query 去搜。

#### 改进方案
增加 `buildSeedQueryFromRoute()`：

```ts
function buildSeedQueryFromRoute(
  originalQuery: string,
  route: RetrievalContextRoute,
): {
  queryText: string
  candidateTypes?: string[]
  actorBoostKeys?: string[]
  locationBoostKeys?: string[]
  relationBoostKeys?: string[]
}
```

#### 应用方式
- `relationship` 高 → 优先收缩 `candidateTypes`
- `scene` 高 → locationKey 命中项加分
- `world` 高 → world 类 schema 提前提权
- `event` 高 → event / actor_visible_event 提前进入候选范围

#### 目标效果
语境系统不再只是“旁观者”，而是真正影响第一轮召回。

### 5.3.4 语境输出增加“来源解释”

建议给 `RetrievalContextRoute` 增加：

```ts
reasons?: Array<{
  source: 'keyword' | 'pattern' | 'entity' | 'recent-context'
  detail: string
  weight: number
}>
```

这将显著提升 diagnostics 的可调试性。

## 5.4 需要改动的文件

- `MemoryOS/src/memory-retrieval/context-router.ts`
- `MemoryOS/src/memory-keywords/*`
- `MemoryOS/src/memory-world-profile/*`
- `MemoryOS/src/memory-retrieval/retrieval-orchestrator.ts`

## 6. 混合召回系统优化实施方案

## 6.1 总体优化目标

让当前召回从“已经具备多阶段结构”升级成“多阶段结构 + 更可信权重 + 更强解释性”。

## 6.2 `retrieval-orchestrator.ts` 优化

### 问题 1：真实 provider 记录不准确
当前如果 `enableEmbedding = true`，但 embedding 没结果，系统 fallback 到 lexical，最后 `providerId` 仍可能显示初始 provider。

### 解决方案
在 orchestrator 内显式记录 `actualProviderId`：

```ts
let actualProviderId = 'none';

if (query.enableEmbedding) {
  seeds = await this.embeddingProvider.search(...);
  if (seeds.length > 0) actualProviderId = this.embeddingProvider.providerId;
}
if (seeds.length <= 0) {
  seeds = await this.lexicalProvider.search(...);
  if (seeds.length > 0) actualProviderId = this.lexicalProvider.providerId;
}
```

### 目标
让 diagnostics 真实反映“这一轮实际用了谁”。

## 6.3 `lexical-provider.ts` 优化

### 问题 1：候选文本过保守
当前候选文本只用：

- `title`
- `summary`
- `schemaId`

这会让很多结构化信息没有进入第一轮种子召回。

### 改进方案
扩展 `buildCandidateText()`：

拼入：

- `category`
- `tags`
- `relationKeys`
- `locationKey`
- 部分 `detailPayload` 可索引字段
- 适度拼接 `sourceSummaryIds` 中可读标签（若有）

### 注意
不能把整个 `detailPayload` 全扔进去，否则噪音太大。  
只拼：短字段、关键词字段、枚举字段。

### 问题 2：缺少 facet-aware 轻量提权
第一轮 lexical recall 还可以更懂 route。

### 改进方案
在 `search()` 增加可选 route 影响：

- `world` 高：world 类 schema 有小幅加成
- `relationship` 高：有 relationKeys / actorKeys 的候选加成
- `scene` 高：有 locationKey 的候选加成

这不是替代 BM25，而是 route-aware reweight。

## 6.4 `graph-expander.ts` 优化

### 问题 1：过早 clamp
当前扩散时，boost 和累计分数都过早 `clamp01`，会压平多源累积差异。

### 改进方案
中间阶段允许 score > 1，最终输出前再 normalize。

建议改法：

- `applyGraphBoost()` 不 clamp
- `resultMap` 内保存 rawScore
- 最终 `sort` 前统一做 `normalizeScores(items)`

### 目标
保留多个中等结构边叠加后的真实差异。

### 问题 2：扩散边权重现在不受语境影响
当前同样的边，在 relationship query 和 world query 下权重一致。

### 改进方案
graph expansion 接入 `contextRoute`：

```ts
contextRoute?: RetrievalContextRoute
```

#### 加权规则
- 当前 facet 高为 `relationship` → relation 边、actor 边增强
- 当前 facet 高为 `scene` → location 边增强
- 当前 facet 高为 `world` → world 边增强
- 当前 facet 高为 `event` → sourceSummary / participants 边增强

### 目标
扩散不再是静态图，而是“语境感知扩散”。

### 问题 3：hub 节点抑制缺失
现在一些大中心节点可能扩散过强，尤其 actor_profile 一类。

### 改进方案
加入 hub penalty：

```ts
hubPenalty = 1 / Math.sqrt(1 + degree)
```

传播时：

$$
graphBoost = seedScore \times edgeWeight \times decay \times hubPenalty
$$

### 目标
防止强中心节点把图拉成星形吸附。

## 6.5 `coverage-checker.ts` 优化

### 问题 1：当前更像 facet 补洞器，而不是残差补召回器
这没有错，但还可以更好。

### 改进方案
新增“facet 预期强度阈值”和“补召回置信门槛”。

#### 规则
- 只有当该 facet 在 `contextRoute` 中分数较高，才触发补召回
- 若当前结果里已有弱覆盖，则不一定触发

### 再进一步
把第二轮补召回的 query 变成“facet 子查询”，而不是继续用整句原 query。

例如：

- relationship 子查询：角色对 + 冲突主题
- scene 子查询：地点 + 场景词
- world 子查询：规则 / 禁令 / 世界状态词

### 目标
让 second pass 更接近真正“残差补召回”。

### 保留一个未来接口：`fista-lite.ts`
本版不接入完整 FISTA，但要预留接口。

建议：

- 仅在 embedding 开启时启用
- 只对 top 32 / top 64 seeds 做 residual hint
- 输出给 coverage-checker 作为补召回优先级，不直接接管排序

## 6.6 `diversity-pruner.ts` 优化

### 问题 1：候选没超上限时不做主动平衡
这会让“数量少但全同类”的情况被放过。

### 改进方案
即使 `items.length <= maxCandidates`，也至少做一轮 `lightBalanceByCategory()`。

### 问题 2：`relationKeys.sort()` 有副作用
这是明确的小 bug。

### 改进方案
全部改成复制后排序：

```ts
const sortedRelationKeys = [...(candidate.relationKeys ?? [])].sort();
```

### 问题 3：高相似 summary 去重只用 n-gram
后续如果 embedding 开启，建议切换成双模式：

- lexical 模式：n-gram
- embedding 模式：summary embedding sim

## 6.7 diagnostics 增强

在召回链路里新增一个统一输出对象：

```ts
export interface RetrievalDiagnostics {
  contextRoute: RetrievalContextRoute | null
  seedProviderId: string
  seedCount: number
  expandedCount: number
  coverageTriggeredFacets: RetrievalFacet[]
  diversityDroppedIds: string[]
}
```

工作台和诊断中心都可以直接复用。

## 7. 可视化记忆图谱优化实施方案

## 7.1 当前图谱定位重新定义

当前这张图更适合被定义为：

> **“记忆浏览图谱”**

而不是：

> **“严格语义知识图谱”**

这个定位必须先明确，因为它决定我们接下来怎么优化边。

### 当前优点
- 图已经可用
- 不会太炸
- 交互完整
- 节点详情面板做得很好

### 当前问题
- 边语义还不够严格
- 一些边容易形成视觉误导
- 不同用途应该用不同连线模式

## 7.2 当前边逻辑存在的问题

### 7.2.1 `participants` 边过宽
如果很多条目都包含同一个主角，就会形成大团块。

### 改进方案
把 `participants` 边改成“重叠驱动”而不是“任意共享即连”：

#### 新规则
- 参与者交集至少达到 2 个，才建强边
- 只有 1 个共享参与者时，建弱边
- 交集越大，边权越大

建议使用：

$$
participantWeight = \frac{|A \cap B|}{|A \cup B|}
$$

只有当 `participantWeight >= 0.34` 时才建边。

### 7.2.2 `actorRef` 边过强，导致 actor_profile 中心化
当前条目只要提到角色，就直接强连 actor_profile，这很适合浏览，但语义上不够细。

### 改进方案
把 `actorRef` 拆成两类：

- `ownerActorRef`
- `mentionedActorRef`

#### 权重建议
- `ownerActorRef`: 0.55
- `mentionedActorRef`: 0.28

### 目标
避免所有提到主角的条目都被强行吸到角色画像节点上。

### 7.2.3 `locationKey` 连线要分“宽地点”和“细场景”
当前 exact match 本身没有错，但如果 `locationKey` 粒度太粗，图会被地点团块化。

### 改进方案
引入 location granularity：

- `world_region`
- `city_zone`
- `scene_spot`

#### 连线规则
- `scene_spot`：强连
- `city_zone`：弱连
- `world_region`：默认不连，除非开启语义模式

### 7.2.4 `worldKeys` 和 `tags` 不该永久关闭，而应模式化
当前整体去掉了 world/tag 广泛连线，这让图更干净，但削弱了世界层结构感。

### 改进方案
新增图谱显示模式：

#### 模式 1：简洁模式（默认）
- participants（严格版）
- actorRef（分级版）
- location（细粒度优先）
- sourceSummary

#### 模式 2：语义模式
在简洁模式基础上增加：
- worldKeys
- 高置信 tags
- compareKey / mutation lineage

#### 模式 3：调试模式
显示所有边，并展示边类型标记。

### 目标
默认易读，需要分析时再看完整语义。

## 7.3 图谱连线逻辑的推荐重构方案

建议在 `buildMemoryGraph.ts` 里，把建边拆成多个独立步骤：

```ts
buildActorEdges(...)
buildParticipantEdges(...)
buildLocationEdges(...)
buildWorldEdges(...)
buildTagEdges(...)
buildSourceSummaryEdges(...)
buildLineageEdges(...)
```

这样有几个好处：

- 每类边逻辑独立可调
- 模式切换时可以单独开关
- diagnostics 里可以直接显示边构成占比
- 后续修权重更容易

## 7.4 图谱增加“边解释”

右侧详情区的“关联节点”目前已经很好用了，但还可以更进一步。

### 当前问题
现在虽然展示了关联节点和 edgeType，但用户仍然不知道：

- 为什么连线
- 这条边的依据是什么

### 改进方案
在边对象里新增：

```ts
reasons?: string[]
```

例如：

- `共享参与者：艾琳, 男主`
- `同地点：旧城区桥边`
- `引用角色：char_erin`
- `来源同一 summary 批次：summary_102`

然后在关联节点列表 hover 或详情展开时展示。

## 7.5 图布局优化建议

当前 `buildMemoryGraph.ts` 已经有简化版力导向布局，这很好，但还可以增强稳定性。

### 建议增加两点

#### 1. cluster anchor
按大类预先给节点一个初始簇中心：

- 世界类在左上
- 事件类在中部
- 关系类在右侧
- 场景类在左下
- 主观理解类在右下

这样布局更稳，视觉也更容易理解。

#### 2. 布局缓存 key
当前只靠节点顺序和数量维持稳定。  
建议增加图布局缓存键：

```ts
layoutCacheKey = chatKey + ':' + graphMode + ':' + nodeSignature
```

这样在用户切筛选器或重开工作台时，布局不容易跳。

## 7.6 工作台层面新增控制项

在 `tabMemoryGraph.ts` 顶部增加：

- 图谱模式切换：简洁 / 语义 / 调试
- 边过滤：只看强边 / 全部边
- 节点范围：全部 / 当前角色相关 / 当前高记忆度
- 高亮来源：当前 query / 当前角色 / 当前 facet

这样图谱就不仅是展示，而是可分析工具。

## 10. 验收标准

本次优化完成后，至少要满足以下标准：

### 10.1 语境系统
- 支持 displayName / alias / 自然语言称呼识别
- 支持 recent context 偏置
- `contextRoute` 真正影响 seed recall
- diagnostics 可展示 route reasons

### 10.2 混合召回
- providerId 真实准确
- lexical recall 能用更多结构字段
- graph expansion 不再过早 clamp
- coverage second pass 不只是简单补洞
- diversity pruner 即使数量少也能做轻量平衡
- 去重逻辑无副作用

### 10.3 可视化记忆图谱
- participants 边不再形成过宽团块
- actor_profile 不再过度中心化
- location 连线支持粒度控制
- 支持简洁 / 语义 / 调试三种图模式
- 右侧可查看边形成原因
- 当前图谱既可浏览，也可用于调试分析

### 10.4 diagnostics
- 可查看 route、seed、graph、second pass、diversity 全链路信息
- 工作台里可追踪为什么某条记忆被召回
- 图谱中可追踪为什么两个节点连线

## 11. 最终结论

当前最新分支的整体方向是正确的，而且已经完成了最困难的“从旧架构迁移到新架构”的阶段。  
现在真正需要做的，不是继续大规模扩功能，而是做一次**针对真实性、边语义、可解释性和调试能力的系统性收敛**。

这份方案的本质可以概括成一句话：

> **让 MemoryOS 从“已经有语境、召回、图谱”升级为“语境更像人、召回更像记忆、图谱更像真实关系”的第二阶段成型版本。**

如果按本方案执行，下一版将不只是“能工作”，而会开始具备明显的产品成熟度。
