# MemoryOS 梦境系统第三阶段实施方案

## 1. 文档目标

本文档用于指导 **SillyTavern-SS-Helper / MemoryOS** 在 `memoryos-settings-unified` 分支上推进梦境系统第三阶段。
关键代码可以参考.tmp\VCPToolBox项目

如果说第一阶段解决的是“有没有梦境闭环”，第二阶段解决的是“梦境检索为什么更像梦”，那么第三阶段解决的就是：

> 如何让梦境系统真正成为 MemoryOS 的后台维护层、自我整理层和长期优化层。

第三阶段的重点不再只是一次次独立的 dream session，而是：

- 自动做梦
- 梦后整理
- 记忆压缩
- 关系强化
- 影子遗忘调控
- 回流主链
- 工作台运维
- 回滚与审计

也就是说，第三阶段要把梦境系统从一个“功能模块”，推进为 **MemoryOS 的持续性后台能力**。

---

## 2. 第三阶段范围定义

## 2.1 本阶段必须实现

### 核心能力
- 自动触发 dream session
- 增加 dream scheduler / idle scheduler / cooldown 机制
- 梦后自动生成整理任务（但仍遵守审批策略边界）
- 让 dream 写回影响 MemoryOS 主链：
  - retrieval
  - vector indexing
  - compareKey / graph 激活
  - summary candidate
- 新增 dream 工作台
- 增加 rollback 与历史审计能力
- 引入 dream quality self-check
- 形成“梦境 → 主链 → 后续召回”闭环

---

## 2.2 本阶段允许增强但仍建议保守控制的内容

### 可以增强
- 自动触发
- 自动生成整理提案
- 自动排队后台任务
- 低风险 mutation 自动预处理

### 仍建议保守
- 不建议无审批直接 delete 主链 entry
- 不建议自动覆盖高置信硬事实
- 不建议跨 chatKey 自动迁移记忆
- 不建议让 dream 直接替代 takeover / summary 的主职能

---

## 2.3 本阶段不建议做的事

### 不建议直接把 dream 变成回答时强制前置
dream 不是回答时的主链入口，它是后台维护层。  
不要让 dream 和正常 prompt injection 混成同一个同步必经步骤。

### 不建议全自动清库
第三阶段即便开始自动整理，也应以：
- 压缩
- 软归档
- shadow 调整
为主，不要做“自动批量删除”。

---

## 3. 第三阶段总体目标

第三阶段的总体目标可以概括成一句话：

> 让梦境系统成为 MemoryOS 的“夜间维护引擎”和“长期记忆优化器”。

具体来说，第三阶段要完成四件事：

1. **自动触发**：不再仅限手动做梦
2. **自动整理**：梦境后生成主链优化任务
3. **自动回流**：梦境产物反向更新 MemoryOS 检索与图谱
4. **可运营化**：可查看、可审批、可回滚、可诊断

---

## 4. 第三阶段验收标准

满足以下条件即可视为第三阶段完成：

1. dream session 可以按配置自动触发。
2. 自动 dream 有互斥锁、冷却和节流，不会扰乱主链。
3. 每次 dream 后可以生成整理任务：
   - 压缩
   - 关系强化
   - shadow 调整
4. dream mutation 与整理任务可以进入统一工作台。
5. 批准后的 dream 结果会反向影响：
   - 主链检索
   - 向量索引
   - graph activation
   - summary 候选
6. dream 工作台可查看：
   - 历史 session
   - diagnostics
   - 审批状态
   - 已应用变更
   - rollback 入口
7. 系统具备 dream quality self-check。
8. 第三阶段在不破坏第一、第二阶段接口的情况下工作。

---

## 5. 第三阶段核心思路

第三阶段不是简单“让 dream 自动跑”。

真正的目标是把 dream 融入 MemoryOS 的生命周期：

```text
聊天进行中
→ summary / retrieval / takeover / event append 正常运行
→ 满足条件时 dream scheduler 排队
→ dream session 生成整理提案
→ 人工审批 / 半自动流程
→ 更新主链与索引
→ 下一轮 retrieval 变得更稳
```

这意味着第三阶段的 dream，不只是“产生文本”，更要：

- 产生后台任务
- 参与主链维护
- 改善未来召回质量

## 7. 自动做梦调度设计

## 7.1 为什么第三阶段才能做自动触发

自动 dream 一旦做太早，调试和污染风险都很高。

前两阶段已经具备：
- dream session
- wave retrieval
- explain mutation
- diagnostics
- approval

到了第三阶段，才适合接自动化，否则根本不可控。

---

## 7.2 自动触发来源

第三阶段建议支持三类触发源：

### A. generation_ended 触发
在模型回复结束后，检查是否需要入梦。  
`runtime-app.ts` 现在已经监听了 `generation_ended`，这个位置最适合做轻量 dream trigger check，而不是立即强行 dream。fileciteturn38file0L1-L1

建议逻辑：
- 轮次结束
- 当前 chatKey 活跃
- 没有正在进行的 takeover / cold start / summary heavy job
- 满足 cooldown
- 满足阈值后入队

---

### B. idle 触发
当用户在当前 chat 中长时间无操作时，允许 dream scheduler 判断是否入梦。

建议：
- idle 只做“触发资格判断”
- 不直接在页面忙的时候启动重型任务
- 优先排入低优先队列

---

### C. 手动触发
仍然保留。
第三阶段手动触发和自动触发应复用同一个 orchestrator。

---

## 7.3 调度保护机制

### chatKey 级互斥锁
同一个 chatKey 同时只能存在一个 running dream job。

### global 限流
同一浏览器实例同一时刻建议只允许一个 active dream pipeline。

### cooldown
建议：
- 单 chatKey dream cooldown：30~120 分钟
- 每日最大 dream 次数：2~6 次

### 优先级控制
若以下任务正在运行，则 dream 延后：
- takeover
- cold start
- summary batch
- vector rebuild heavy task

---

## 7.4 调度输出结构建议

```ts
type DreamScheduleDecision = {
  shouldTrigger: boolean;
  reasonCodes: string[];
  triggerSource: 'manual' | 'generation_ended' | 'idle';
  blockedBy?: string[];
  suggestedDelayMs?: number;
};
```

## 8. 梦后整理设计

## 8.1 为什么 dream 后必须做整理

如果 dream 只是多写几条 entry，长期来看 MemoryOS 会越来越膨胀。

第三阶段 dream 的真正价值，是在 dream 后自动产生：
- 压缩
- 强化
- 调整
- 归档倾向

这会让 MemoryOS 越用越稳，而不是越用越碎。

---

## 8.2 梦后整理任务类型

### 1）Memory Compression Proposal
目标：
- 多条高重合 entry 合成为一条更稳定的 summary 或 consolidated entry

第二阶段还不做自动 merge。  
第三阶段可以先做“压缩提案”，仍建议审批后应用。

---

### 2）Relationship Reinforcement Proposal
目标：
- 对长期反复出现的 actor / relation 进行强化
- 提高后续检索中这些关系的权重

---

### 3）Shadow Retention Adjustment Proposal
目标：
- 对长时间未激活、且低价值的条目降低显性召回概率
- 不删除，只降级到 shadow 层

---

### 4）Summary Candidate Promotion
目标：
- 将高价值 dream insight 变成后续 summary pipeline 的候选材料
- 让 dream 真正反哺 summary

---

## 8.3 统一的 maintenance proposal 结构建议

```ts
type DreamMaintenanceProposal = {
  proposalId: string;
  dreamId: string;
  proposalType:
    | 'memory_compression'
    | 'relationship_reinforcement'
    | 'shadow_adjustment'
    | 'summary_candidate_promotion';
  confidence: number;
  reason: string;
  sourceEntryIds: string[];
  sourceNodeKeys: string[];
  preview: string;
  payload: Record<string, unknown>;
};
```

## 9. 梦境结果回流主链设计

## 9.1 第三阶段最重要的一步：梦不是终点，回流才是终点

第三阶段 dream 的价值，最终要体现在：

- 未来 retrieval 更准
- 未来 summary 更稳
- 未来角色状态更一致
- 未来 memory graph 更合理

也就是说，dream session 应该是主链的输入，而不是旁路日志。

---

## 9.2 梦回流后要更新什么

### A. 主记忆条目
- 新增 / patch 后写主表

### B. 关系图
- 关系强化后同步更新 graph 层

### C. compareKey / actorKey / topicKey 激活索引
- 让后续 dream / retrieval 都能看到这些变化

### D. vector 索引
- 对新增或被重要 patch 的条目重新入向量索引

### E. summary 候选池
- 把高价值 dream insight 推给 summary pipeline

---

## 9.3 回流顺序建议

建议严格按顺序：

1. apply dream mutation
2. apply maintenance proposal
3. update graph / relation weights
4. update vector index
5. update summary candidates
6. write audit & diagnostics

这样能保证后续链路读取到的是一致状态。

---

## 10. Dream Quality Guard 设计

## 10.1 为什么第三阶段必须有质量守卫

自动 dream 一旦开始，就一定会遇到几个问题：

- 某些 dream 质量很差
- 某些 mutation 过度重复
- 某些提案在硬事实层有风险
- 某些 explain 虽然存在，但其实是空的

所以第三阶段必须给 dream 增加“质量门”。

---

## 10.2 质量检查项目

### A. Explain 完整度
必须检查：
- sourceWave 是否存在
- sourceEntryIds 是否存在
- explanationSteps 是否为空

### B. Mutation 重复率
近期 dream 中重复出现的相同 preview / payload 应降权或阻止自动推进。

### C. Hard Fact 风险
若 mutation 触及：
- 角色身份
- 不可逆硬设定
- 明确事实类 memory entry
则必须强制人工审批并标红。

### D. 幻觉倾向
若提案来源 evidence 太弱、置信度波动太大、sourceEntryIds 过少，应视为高风险。

---

## 10.3 质量守卫输出结构

```ts
type DreamQualityReport = {
  dreamId: string;
  qualityScore: number;
  warnings: string[];
  blockedMutationIds: string[];
  forcedReviewMutationIds: string[];
  createdAt: number;
};
```

## 11. Dream Workbench 设计

## 11.1 为什么第三阶段一定要有工作台

到了第三阶段，dream 不再只是弹一次 dialog 就结束。  
系统会逐渐积累：

- 历史 dream session
- diagnostics
- graph snapshot
- maintenance proposals
- rollback snapshot
- quality reports

如果没有独立工作台，这个系统基本不可运维。

## 12. Rollback 设计

## 12.1 为什么第三阶段必须强化 rollback

第一阶段已经建议写 rollback snapshot。  
第三阶段自动化变多后，rollback 不再是“锦上添花”，而是必要能力。

---

## 12.2 回滚粒度建议

### 粒度一：按 mutation 回滚
回滚某个具体 dream mutation。

### 粒度二：按 dreamId 回滚
回滚一次 dream session 的全部影响。

### 粒度三：按 maintenance proposal 回滚
回滚某次压缩 / 强化 / shadow 调整。

---

## 12.3 回滚执行顺序建议

1. 停止该 dream 相关后台任务
2. 恢复 relation / graph 权重
3. 恢复 entry patch
4. 恢复 summary candidate 状态
5. 恢复 vector index
6. 写 rollback audit

---

## 14. 第三阶段事件流

## 14.1 generation_ended 流程

```text
generation_ended
→ DreamSchedulerService.checkTriggerEligibility
→ 若满足条件则创建 queued dream task
→ 空闲时执行 DreamingService.startDreamSession
→ DreamPostProcessingService 生成 maintenance proposals
→ QualityGuard 打分
→ 工作台 / 审批 / 应用
→ 回流主链
```

这个挂点和你们现有 `generation_ended` 的 summary 触发检查天然兼容，但 dream 应该放在 summary 之后或并列轻量决策，不应阻塞当前对话线程。fileciteturn38file0L1-L1

---

## 14.2 idle 流程

```text
idle timer tick
→ chatKey active?
→ cooldown satisfied?
→ no takeover / no cold start / no heavy summary?
→ scheduler enqueue dream
```

---

## 14.3 手动流程

手动流程仍保留：

```text
toolbar / workbench click
→ DreamingService.startDreamSession
→ 走与自动 dream 相同主链
```

只是 `triggerSource` 不同。

- 硬事实风险项必须被标记

### 用例 5：rollback 可恢复
- 应用某次 dream 影响后能成功回滚

### 用例 6：回流主链后未来召回改变
- 新一轮 retrieval 能读取到 dream 的强化结果

## 17. 风险与控制

## 17.1 风险：自动 dream 过度打扰系统
控制：
- cooldown
- 每日上限
- chatKey 互斥
- 重型任务回避

## 17.2 风险：maintenance 提案过多
控制：
- 每次 dream 限制 maintenance proposal 上限
- quality guard 筛掉低质量项

## 17.3 风险：dream 回流导致主链漂移
控制：
- 对高风险 mutation 强制审批
- 优先强化、压缩、shadow 调整
- 尽量避免自动改硬事实

## 17.4 风险：工作台复杂度过高
控制：
- 默认展示摘要
- 深度 diagnostics 放展开层
- rollback 入口集中在 session 级别

## 18. 开发顺序建议

建议严格按下面顺序推进：

### 第 1 步：做调度器
先把：
- scheduler
- cooldown
- idle / generation_ended eligibility
做出来。

### 第 2 步：做 dream 后整理
让 dream 不再止于 mutation，而能产生 maintenance proposals。

### 第 3 步：做质量守卫
在自动化增强前先加安全阀。

### 第 4 步：做工作台
把 session、proposal、rollback 全部集中展示。

### 第 5 步：做回滚
最后把闭环补齐。

## 20. 最终结论

第三阶段真正要完成的，不是“梦境系统更复杂”，而是：

- 梦境可自动触发
- 梦后能整理主链
- 结果能回流主链
- 主链能因 dream 变得更稳
- 全链路可观测、可审批、可回滚

也就是说，第三阶段的目标是：

> 把 Dream Pipeline v2 升级为真正参与 MemoryOS 生命周期的 Dream Maintenance Pipeline v3。
