# 基于 Commit `4763c72` 的继续优化方案（参考 PeroCore）

## 目标

在保留 `4763c72` 已经引入的 **轻量 Planner 输入** 前提下，继续优化以下问题：

1. `windowFacts` 仍然混入了过多对白碎片和氛围描写
2. `rollingDigest` 目前仍然是“长摘要截断”，不是高质量滚动摘要
3. `signalPack` 仍然偏粗糙，容易把低价值信号带进 Planner
4. `candidateCards` 仍然有“条目太多、条目不够代表、条目不够聚焦”的问题
5. 压缩后的输入虽然更轻，但“信息提纯”还不够，导致 Planner 质量波动

本方案直接写 **该改什么、按什么原则改、建议如何落地**。

# 一、先明确：`4763c72` 的方向是对的，但压缩器质量还不够

这次 `4763c72` 已经做对了三件事：

- 不再把完整 `plannerResult.context` 直接发给 Planner
- 引入了 `LightweightPlannerInput`
- 把输入改成：
  - `windowFacts`
  - `rollingDigest`
  - `signalPack`
  - `candidateCards`
  - `allowedTypes`

这条路必须保留，不建议回退。

当前真正的问题不是“轻量输入思路错了”，而是：

> **轻量输入的生成器还不够聪明。**

也就是说，下一步应该继续优化：

- `extractPlannerWindowFacts()`
- `buildRollingDigest()`
- `buildSignalPack()`
- `buildPlannerCandidateCards()`

而不是重新回到“把大量原始上下文塞给 Planner”。

# 二、参考 PeroCore 可以学什么

PeroCore 真正值得借的，不是把算法名搬过来，而是它在“旧信息压缩”和“高价值信息提纯”上的思路。

## 2.1 PeroCore 的核心启发 1：旧记忆不是一直原样回灌，而是会被整合压缩

PeroCore 的 `ReflectionService` 会把：

- 低重要性陈旧事件
- 多条相近记忆
- 冗余碎片

整合成一条总结性记忆，再把旧事件归档。  
这说明它不是把“过去所有摘要”都重新喂回 Prompt，而是不断把旧内容压缩成**更短、更稳定、更陈述性**的表示。

### 可以直接借用的原则
你的 `rollingDigest` 不应该来自：

- “最近一条摘要直接截断前 220 字”

而应该来自：

- “最近一阶段的稳定长期状态压缩结果”

也就是说：

> **滚动摘要应该是“状态陈述”，不是“剧情摘录”。**

---

## 2.2 PeroCore 的核心启发 2：摘要必须短，而且抓核心，不要剧情复述

PeroCore 的 scorer prompt 对摘要有非常强的约束：

- 30～50 字
- 只抓核心事实
- 不写说明书式长文本
- 不做大段剧情回放

这说明一点：

> **进入记忆系统的文本，不应该像故事，而应该像高密度陈述。**

### 可以直接借用的原则
你这边无论是：

- `windowFacts`
- `rollingDigest.summary`
- `candidateCards.brief`

都应该朝这个方向收：

- 高密度
- 短句
- 事实导向
- 少修辞
- 少语气描写

---

## 2.3 PeroCore 的核心启发 3：维护层会主动清理、合并、归档旧记忆

PeroCore 不是只会“越存越多”，而是会在反思维护中做：

- 记忆合并
- 偏好提取
- 错误记忆清理
- 低重要性事件压缩
- 重复社交日报清理

这说明它实际上是在维护一个：

> **越来越适合检索，而不是越来越适合回放剧情** 的记忆库。

### 可以直接借用的原则
你这边的 `recentSummaryDigest` 历史产物不能无限累积并原样回灌。  
应该增加一个“摘要净化层”，把旧的结构化回合总结再加工成：

- 稳定背景
- 未解决问题
- 最近决定
- 当前关系基调
- 当前任务状态

而不是继续保留大量戏剧性原文。

# 三、继续优化的核心原则

后面的改法都围绕下面这四条。

## 原则 1：Planner 只看“判断所需信息”
Planner 的职责是：

- 是否需要更新
- 关注哪些类型
- 关注哪些实体
- 主题是什么

它不需要看：

- 长对白
- 长剧情
- 长摘要
- 大量旧记录详情
- 完整 schema 字段说明

## 原则 2：原文只作“证据”，不作“主上下文”
如果一定要保留原始语言风格，只能保留极少量证据片段。  
主输入必须是：

- 事实帧
- 状态摘要
- 候选短卡片

## 原则 3：压缩不是简单截断，而是语义提纯
现在最大的问题是很多地方在“截断”，而不是“提纯”。

例如：

- 长摘要截 220 字，不等于高质量滚动摘要
- 长记录 brief 截 60 字，不等于高质量候选卡片
- 长剧情分句，不等于高质量事实帧

## 原则 4：越旧的信息越要结构化，越新的信息才允许保留少量语言痕迹
也就是：

- 当前窗口：允许保留极少量对话痕迹
- 历史摘要：必须变成状态摘要
- 候选记录：必须是卡片化陈述
- Mutation 阶段：才去取详细内容

## 修改 1：重写 `extractPlannerWindowFacts()`，不要再按简单句号拆分

### 当前问题
现在的做法是：

- 把 `summaryText` 按 `。！？\n` 分句
- 取长度在一定范围内的句子

这会导致两类坏结果：

1. 把戏剧性对白片段也当成“事实帧”
2. 把真正的重要事实和氛围描述放在同一层级

例如这些句子就不应该直接进入 `windowFacts`：

- 她低低笑了一声
- 门只开着一条危险的缝
- 那双异色瞳冷静得近乎残忍

它们可能是氛围证据，但不是“结构化记忆事实”。

### 直接修改内容

把 `extractPlannerWindowFacts()` 改成“两段式”：

#### 第一步：事件句筛选
只保留含有高信号动词或状态变化的句子。  
建议优先命中以下词：

- 接受 / 拒绝 / 收下 / 要求 / 允许 / 禁止
- 约定 / 作废 / 取消 / 推迟 / 确认
- 交付 / 支付 / 委托 / 交易 / 说明
- 前往 / 出发 / 返回 / 到达
- 怀疑 / 信任 / 警告 / 威胁 / 试探

#### 第二步：事实重写
不要原样保留句子，而是压成短事实。

例如把：

- “她把钱袋收进门后，门却依旧只开着那条危险的缝”

重写为：

- “艾莉卡收下定金，但仍未放松戒备”

把：

- “答复你：不。至少不是现在，不是这种做法”

重写为：

- “艾莉卡拒绝立刻出发的提议”

### 推荐输出格式
每条都用“主语 + 行为/状态 + 结果”的句型：

- 艾莉卡收下定金，但未正式接单
- 艾莉卡拒绝立即前往天顶区
- 艾莉卡要求毅毅先交代全部已知信息
- 当前交涉仍在门口继续进行
- 双方关系维持高度不信任状态

### 建议增加本地规则表
新增一个轻量规则表，例如：

```ts
const FACT_PRIORITY_RULES = {
  accept: ['收下', '接受', '同意'],
  reject: ['拒绝', '不', '不能', '不会'],
  demand: ['要求', '条件', '必须', '先'],
  stateChange: ['作废', '失效', '继续', '暂停'],
  transaction: ['定金', '交易', '委托', '报酬'],
};
```

然后优先抽取这些句子。

## 修改 2：重写 `buildRollingDigest()`，不要再从旧摘要里“扫关键词截残句”

### 当前问题
现在的 `buildRollingDigest()` 本质上是：

- 取最近一条历史摘要
- 截前 220 字作为 summary
- 用关键词扫句子，找 openThreads / recentDecisions

这会产生很多问题：

- `summary` 仍然像长对白摘要
- `openThreads` 可能是半句话
- `recentDecisions` 经常不是“决策”，只是碰巧含关键词

### 直接修改内容

把 `rollingDigest` 改成**状态聚合**，不要从原文直接切残句。

### 新结构建议

```ts
rollingDigest: {
  stableContext: string
  taskState: string[]
  relationState: string[]
  unresolvedQuestions: string[]
}
```

### 字段含义

#### `stableContext`
只写当前长期盘面，1 句即可。  
例如：

- “毅毅与艾莉卡围绕红色宝石委托持续交涉，但合作尚未成立。”

#### `taskState`
只写任务当前状态，最多 3 条。  
例如：

- “已支付2500定金”
- “委托未正式确认”
- “缺少宝石来历与具体位置信息”

#### `relationState`
只写关系当前基调，最多 2 条。  
例如：

- “艾莉卡仍高度戒备毅毅”
- “薇拉的名字只能换来有限克制”

#### `unresolvedQuestions`
只写未解决问题，最多 3 条。  
例如：

- “红色宝石具体来历不明”
- “天顶区西城的精确位置不明”
- “委托人掌握的信息真实性不足”

### 构建逻辑
不要从长摘要里做关键词切句，而是：

1. 从最近一次 summary snapshot 中提取 task / relation / world / unresolved
2. 没有结构字段时，再从文本中二次归纳
3. 输出时只保留短陈述句，不保留台词碎片

### 推荐改法
如果你现在暂时没有单独字段，就先做一个“归类压缩器”：

- 含“定金/交易/委托/接单” → `taskState`
- 含“怀疑/信任/戒备/克制” → `relationState`
- 含“未知/不明/尚未/缺少/无法确认” → `unresolvedQuestions`

而不是直接保留原句。

## 修改 3：增强 `buildSignalPack()`，不要直接把 `plannerHints.reasons` 当 focusPoints

### 当前问题
现在 `focusPoints` 直接来源于 `plannerHints.reasons`，这会导致：

- 信息太泛
- 不够统一
- 不一定真的是本轮需要关注的焦点
- 容易把“解释性文本”重新塞回 Planner

### 直接修改内容

让 `signalPack` 改成完全本地产生，不再直接搬运 `plannerHints.reasons`。

### 新结构建议

```ts
signalPack: {
  candidateTypes: string[]
  focusPoints: string[]
  evidenceSignals: string[]
  shouldUpdate: boolean
}
```

### 含义
#### `candidateTypes`
保留，但只保留最高优先的 3～5 个

#### `focusPoints`
只写“本轮需要模型判断的焦点”，例如：

- “委托是否进入正式成立阶段”
- “拒绝立即出发是否构成任务状态变化”
- “关系是否从试探转为明确拒绝”

#### `evidenceSignals`
只写本地检测到的高价值证据标签，例如：

- “支付定金”
- “明确拒绝”
- “提出条件”
- “要求补充信息”

这样模型更容易理解为什么会关注这些点。

### 推荐判断逻辑
`shouldUpdate=true` 只有在以下任一条件成立时触发：

- 出现新任务 / 新承诺 / 新约定
- 原任务状态发生推进或失效
- 明确关系变化
- 新地点/组织/规则被确认
- 长期未决问题出现新阶段变化

否则就让 Planner 有机会返回 `false`。

## 修改 4：重写 `buildPlannerCandidateCards()`，不要简单从完整 summary 截断

### 当前问题
现在 `candidateCards.brief` 是从 `rec.summary` 截 60 字。  
这虽然变短了，但本质上仍然是：

- 长记录摘要的一小段残片

这样会出现两个问题：

1. 重要事实可能被截断
2. 每条卡片都像“没说完”

### 直接修改内容

给 `candidateCards` 做“类型化卡片模板”。

### 示例模板

#### task
```ts
brief = `任务：${目标}；状态：${当前状态}`
```

#### relationship
```ts
brief = `${角色A}对${角色B}：${当前关系基调}`
```

#### event
```ts
brief = `事件：${发生了什么}；结果：${当前结果}`
```

#### location
```ts
brief = `地点：${地点名}；作用：${当前用途/约束}`
```

#### world_global_state
```ts
brief = `世界状态：${当前阶段性现实}`
```

### 推荐额外字段
比现在多加一个很值钱的字段：

```ts
whyRelevant: string[]
```

例如：

```json
{
  "id": "cand_4",
  "type": "relationship",
  "brief": "艾莉卡对毅毅：高度戒备，但未彻底切断交涉",
  "entities": ["erika_duskshade", "user"],
  "state": "active",
  "whyRelevant": ["命中核心角色", "命中关系变化", "当前窗口多次出现拒绝与条件"]
}
```

这样 Planner 更容易理解为什么这张卡片会出现。

### 数量策略
当前上限 8 条还行，但建议加“类型代表性去重”：

- 同一类型相似卡片过多时，只保留代表项
- 对关系类尤其重要
- 不要让与当前窗口无关的 `relationship` 充斥候选区

例如你这次样例里：

- `cand_7`
- `cand_8`

这类和当前交涉主线关系不够强的候选，其实优先级应该更低。

## 修改 5：给 Planner 增加“证据片段槽”，不要让对白完全消失

### 当前问题
完全改成事实帧后，有时候会丢掉一些“语义温度”：

- 是试探性拒绝
- 还是绝对拒绝
- 是高压戒备
- 还是表面冷淡

这在关系变化判断时有时很重要。

### 直接修改内容

不要把原文大段塞回去，而是增加一个**极小证据槽**：

```ts
evidenceSnippets: string[]
```

### 规则
- 最多 `3` 条
- 每条最多 `20~35` 字
- 只保留当前窗口里最能体现关系/态度/状态变化的原句

例如：

- “答复你：不。至少不是现在，不是这种做法。”
- “把你知道的全吐干净——哪怕是碎片。”
- “这不代表我接单。”

### 作用
这样你既不会重新把剧情全文塞回去，  
又能保留判断细微变化所需的原文证据。

这点是对当前轻量输入非常值钱的补强。

## 修改 6：增加“旧摘要净化器”，把历史结构化回合总结转成可复用状态块

### 当前问题
现在 `recentSummaryDigest` 的来源本身就很脏，所以即使只取一条最新摘要，也会把老问题带进来。

### 直接修改内容

增加一个历史摘要净化步骤：

```ts
normalizeSummarySnapshot(snapshot) => {
  stableContext,
  taskState,
  relationState,
  unresolvedQuestions
}
```

### 功能
每次产生 summary snapshot 后，不仅保存原始 summary，还同步生成一个净化版：

- `normalizedSummary`

以后给 Planner 的 `rollingDigest` 一律优先读 `normalizedSummary`，  
而不是读原始 `content` 再现场压缩。

### 好处
这样旧摘要只需要净化一次，后面每轮都能复用高质量滚动摘要。

这点非常接近 PeroCore 的“维护层持续压缩旧记忆”的思路。

## 修改 7：把 `allowedTypes` 再裁一层，不要把所有能处理的类型都给 Planner

### 当前问题
现在 `allowedTypes` 是从 `typeSchemas` 全量拉出来的。  
虽然比直接发完整 schema 轻很多，但仍然可能把很多与本轮无关的类型带进去。

### 直接修改内容

给 `allowedTypes` 增加一个“上下文裁剪层”：

### 裁剪规则
优先保留：

- 当前窗口命中的类型
- `candidateCards` 覆盖到的类型
- `signalPack.candidateTypes`

交集优先，最多 `5` 个。

### 举例
如果当前窗口只涉及：

- task
- relationship
- event

那就不要再发：

- world_core_setting
- world_hard_rule
- organization
- location

除非它们在当前窗口中真的被强命中。

这样能继续减轻 Planner 的选择空间。

# 六、建议的修改顺序
