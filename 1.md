# MemoryOS 工作台新增“向量实验室”Tab 设计方案（基于 memoryos-settings-unified / fe489d8）

## Summary
- 本方案基于当前 `memoryos-settings-unified` 分支 **Commit `fe489d8`**。
- 当前工作台已经在“诊断中心 / preview”中补充了不少**召回结果级**诊断信息，包括：
  - `finalProviderId`
  - `seedProviderId`
  - `vectorHitCount`
  - `mergeUsed`
  - `rerankUsed`
  - `rerankSource`
  - `strategyDecision`
- 这些信息已经足够说明“**这次召回发生了什么**”。
- 但工作台里还没有一个页面可以回答：
  1. **有哪些内容被向量化了**
  2. **当前向量索引状态是否健康**
  3. **vector_only / hybrid 实际召回效果如何**
  4. **LLMHub rerank / 规则 rerank 的差别是什么**
  5. **如何手动重建、清理、刷新向量索引**
- 因此建议在工作台中新增一个独立 Tab：
  **向量实验室**

---

## Why This Tab Is Needed

### 当前“诊断中心”解决的问题
当前 preview / recall 解释页已经适合看：
- 本次 query 是怎么命中的
- 最终链路是什么
- 是否走了融合
- 是否走了重排
- 是规则重排还是 LLMHub 重排

也就是说，它适合看：

> **一次召回结果的解释**

---

### 但当前还缺少的问题
你现在的向量系统已经有这些模块：
- `EmbeddingService`
- `VectorStoreAdapterService`
- `VectorStrategyRouter`
- `VectorRerankService`
- `LLMHubRerankService`
- `HybridRetrievalService`
- `vector-index-service`

但工作台没有地方可以直接看：
- 向量文档列表
- embedding 状态
- 向量索引是否完整
- recall stats
- 手动测试 query 的 vector 命中
- vector_only / hybrid 对比
- 手动 rebuild / clear / refresh

所以现在最缺的是：

> **向量系统本体的可视化与可测试入口**

---

## Proposed Tab Name

### 推荐名称
**向量实验室**

### 不建议的名称
- 向量诊断
- 检索实验
- 语义召回

### 原因
“向量实验室”更准确，因为它同时包含：
- 浏览
- 测试
- 诊断
- 运维操作

不是单纯看日志。

---

# 一、Tab 的定位

## 核心定位
“向量实验室”不是替代 preview，也不是替代条目编辑页。  
它的职责是：

1. **向量资产浏览**
2. **向量召回测试**
3. **向量链运行状态诊断**
4. **向量索引维护操作**

---

## 和现有 Tab 的边界

### `preview / 诊断中心`
负责看：
- 当前 prompt 召回解释
- recall explanation
- why this query matched these entries

### `vectors / 向量实验室`
负责看：
- 向量文档本体
- 索引状态
- query 测试
- vector_only / hybrid 对比
- rerank 效果
- rebuild / clear / refresh

---

# 二、页面结构规划

## 总体建议
不要把“向量实验室”做成单一大列表。  
建议做成 **四区块布局**：

### A. 顶部：向量总览
### B. 左侧：向量文档列表
### C. 右侧：文档详情
### D. 底部：召回测试台

---

## A. 顶部：向量总览

### 目标
让用户一进来就知道：
- 向量系统有没有启用
- 是否可用
- 是否已建索引
- 当前运行模式是什么

### 建议展示字段
#### 运行时状态
- runtime ready
- embedding available
- embedding unavailable reason
- vector store available

#### 当前配置
- retrievalMode
- vectorEnableStrategyRouting
- vectorEnableRerank
- vectorEnableLLMHubRerank
- vectorEmbeddingModel
- vectorEmbeddingVersion

#### 当前聊天统计
- 向量文档总数
- ready 数
- pending 数
- failed 数
- vector index 数
- recall stats 数

### 展示形式建议
用 2~3 行统计卡片：
- 状态卡片
- 配置卡片
- 数量卡片

---

## B. 左侧：向量文档列表

### 目标
直接查看“有哪些内容已经被向量化”。

### 列表建议字段
- `vectorDocId`
- `sourceKind`
- `sourceId`
- `title`
- `compareKey`
- `schemaId`
- `embeddingStatus`
- `embeddingModel`
- `embeddingVersion`
- `embeddingDim`
- `updatedAt`

### 建议过滤器
#### 基础过滤
- sourceKind
  - `entry`
  - `relationship`
  - `actor`
  - `summary`
- embeddingStatus
  - `pending`
  - `ready`
  - `failed`

#### 文本过滤
- title / text contains
- compareKey
- sourceId

#### 结构过滤
- actorKey
- worldKey
- relationKey
- schemaId

### 建议交互
- 单击列表项：右侧显示详情
- 双击列表项：滚动定位到详情
- 支持按更新时间倒序

---

## C. 右侧：向量文档详情

### 目标
查看选中文档的向量化内容和索引状态。

### 建议展示内容
#### 基本信息
- `vectorDocId`
- `sourceKind`
- `sourceId`
- `chatKey`
- `schemaId`
- `compareKey`

#### 向量状态
- `embeddingStatus`
- `embeddingModel`
- `embeddingVersion`
- `embeddingDim`
- `lastError`

#### 结构标签
- actorKeys
- relationKeys
- worldKeys
- locationKey

#### 文本内容
- title
- text（完整或可折叠）
- 更新时间

#### 索引信息
- 是否已写入 index
- 最近 recall 次数
- 最近 recall 时间
- 最近 recall 模式

### 建议操作按钮
- 重新 embedding 当前文档
- 删除当前文档索引
- 打开来源对象
- 复制 vectorDocId / sourceId

---

## D. 底部：召回测试台

### 目标
人工测试向量召回链是否工作正常，以及效果如何。

### 输入区建议
#### Query 输入
- query 文本框
- 可选 query context 文本框（高级模式）

#### 模式选择
- `lexical_only`
- `vector_only`
- `hybrid`

#### 开关项
- 启用 strategy routing
- 启用 rerank
- 启用 LLMHub rerank
- 启用 graph expansion

#### 参数项
- topK
- deepWindow
- finalTopK
- rerankWindow

#### 可选过滤
- sourceKind
- schemaId
- actorKey
- worldKey

---

### 输出区建议拆成三块

#### 1. Query 诊断
展示：
- finalProviderId
- seedProviderId
- strategyDecision
- vectorHitCount
- mergeUsed
- rerankUsed
- rerankSource
- reasonCodes

#### 2. 原始命中区
展示：
- lexical seeds
- vector hits
- merged candidates
- rerank 前窗口

#### 3. 最终结果区
展示最终 top3~5：
- title
- sourceKind
- sourceId
- compareKey
- summary
- source label
- score / breakdown（能给就给）

---

# 三、页面功能边界

## 应该做的
- 看向量文档
- 看索引状态
- 看 embedding 状态
- 做 query 测试
- 对比 vector_only / hybrid
- 看 rerank 来源
- 手动 rebuild / clear / refresh

## 不应该做的
- 不编辑业务主数据正文
- 不承载 actor / relationship 正文编辑
- 不代替 preview 的 prompt 解释职责
- 不让 UI 自己直接拼底层服务调用

---

# 四、数据结构改造建议

## 1. 扩展 `WorkbenchView`
当前定义在：
- `MemoryOS/src/ui/workbenchTabs/shared.ts`

### 当前
```ts
export type WorkbenchView =
  'entries' | 'types' | 'actors' | 'world-entities' | 'preview' | 'memory-graph' | 'takeover';
```

### 建议改为
```ts
export type WorkbenchView =
  'entries' | 'types' | 'actors' | 'world-entities' | 'preview' | 'memory-graph' | 'takeover' | 'vectors';
```

---

## 2. 扩展 `WorkbenchState`
建议新增：

```ts
vectorQuery: string;
vectorMode: 'lexical_only' | 'vector_only' | 'hybrid';
vectorSourceKindFilter: string;
vectorStatusFilter: string;
vectorSchemaFilter: string;
vectorActorFilter: string;
vectorSelectedDocId: string;
vectorEnableRerankTest: boolean;
vectorEnableLLMHubRerankTest: boolean;
vectorEnableGraphExpansionTest: boolean;
vectorTopKTest: string;
vectorDeepWindowTest: string;
vectorFinalTopKTest: string;
vectorTestResultExpanded: boolean;
vectorLoading: boolean;
vectorTestRunning: boolean;
```

### 原则
测试台状态不要复用 preview 的状态字段，避免互相污染。

---

## 3. 扩展 `WorkbenchSnapshot`
建议新增一块：

```ts
vectorSnapshot: {
  runtimeReady: boolean;
  embeddingAvailable: boolean;
  embeddingUnavailableReason?: string;
  vectorStoreAvailable: boolean;
  retrievalMode: string;
  embeddingModel?: string;
  embeddingVersion?: string;
  vectorEnableStrategyRouting?: boolean;
  vectorEnableRerank?: boolean;
  vectorEnableLLMHubRerank?: boolean;
  documentCount: number;
  readyCount: number;
  pendingCount: number;
  failedCount: number;
  indexCount: number;
  recallStatCount: number;
  documents: DBMemoryVectorDocument[];
  recallStats: DBMemoryVectorRecallStat[];
};
```

### 原则
不要让 Tab 自己临时乱调底层服务。  
统一通过 snapshot 提供基础数据。

# 五、SDK / Diagnostics 接口建议

## 目标
让 UI 通过统一 MemorySDK 拿向量数据，而不是直接碰底层 service。

---

## 建议新增接口
放到现有 `memory.unifiedMemory.diagnostics` 或相邻能力中。

### 1. 获取向量运行时状态
```ts
getVectorRuntimeStatus(): Promise<{
  runtimeReady: boolean
  embeddingAvailable: boolean
  embeddingUnavailableReason?: string
  vectorStoreAvailable: boolean
  retrievalMode: string
  embeddingModel?: string
  embeddingVersion?: string
}>
```

### 2. 列出向量文档
```ts
listVectorDocuments(input?: {
  sourceKind?: string
  status?: string
  schemaId?: string
  actorKey?: string
  query?: string
}): Promise<DBMemoryVectorDocument[]>
```

### 3. 列出向量统计
```ts
getVectorIndexStats(): Promise<{
  documentCount: number
  readyCount: number
  pendingCount: number
  failedCount: number
  indexCount: number
  recallStatCount: number
}>
```

### 4. 测试向量召回
```ts
testVectorRetrieval(input: {
  query: string
  retrievalMode: 'lexical_only' | 'vector_only' | 'hybrid'
  topK?: number
  deepWindow?: number
  finalTopK?: number
  enableRerank?: boolean
  enableLLMHubRerank?: boolean
  enableGraphExpansion?: boolean
  filters?: {
    sourceKind?: string
    schemaId?: string
    actorKey?: string
    worldKey?: string
  }
}): Promise<{
  diagnostics: RetrievalOutputDiagnostics
  items: RetrievalResultItem[]
  lexicalSeeds?: RetrievalResultItem[]
  vectorHits?: VectorSearchHit[]
  mergedItems?: RetrievalResultItem[]
}>
```

### 5. 运维操作
```ts
rebuildAllVectorDocuments(chatKey?: string): Promise<void>
rebuildAllEmbeddings(chatKey?: string): Promise<void>
clearVectorIndex(chatKey?: string): Promise<void>
clearVectorRecallStats(chatKey?: string): Promise<void>
reindexVectorDocument(vectorDocId: string): Promise<void>
```

# 十、最终建议

## 最推荐的做法
- **单独新增一个 Tab**
- 名字就叫：**向量实验室**
- 不要继续把这些东西塞到 preview 里
- 先做 P0 最小可用版
- 后面再慢慢补 P1 / P2

---

## 一句话结论
你现在这个分支已经有足够完整的向量链和诊断字段了，**非常适合在工作台加一个“向量实验室”Tab**。  
它最合理的定位就是：

> **向量资产浏览 + 召回测试台 + 运维操作区**

而不是单纯再加几行诊断文本。
