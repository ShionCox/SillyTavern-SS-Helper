# 原格式获取链路与 ROLLJSON 清理问题彻底修复实施方案

## 1. 文档目标

本文用于彻底解决当前 RollHelper 在以下场景中的核心问题：

- 复制原格式按钮拿不到 ROLLJSON
- 原始文本快照有时存在、有时丢失
- 同一条助手消息在不同模块读取到的“原文”不一致
- 最终清理后只能拿到正文，拿不到原始控制块
- 流式阶段、最终处理阶段、聊天切换阶段对原文的处理职责混乱

本方案不是局部补丁，而是要把整条链路重构为：

> **单一真相源 + 明确捕获时机 + 明确清理时机 + 统一读取入口**

---

## 2. 当前问题定性

当前问题不是 `buildAssistantRawSourceCopyButtonEvent()` 本身有 bug。

这个函数只是创建按钮节点，不负责原文解析、缓存或读取。

真正的问题在于：

### 2.1 原始文本存在两套来源
当前仓库里同时存在两类“原始文本”：

#### A. 插件自己持久化的快照
来自：

- `messageSanitizerEvent.ts`
  - `rememberAssistantOriginalSourceTextEvent(...)`
  - `getAssistantOriginalSourceTextEvent(...)`

保存位置：

- `message.extra`
- 或 `swipe_info[swipeId].extra`

#### B. 宿主侧原文 getter
来自：

- `promptEvent.ts`
  - `getAssistantOriginalSourceTextFromHostEvent(...)`

这导致：

- 捕获时有时走宿主
- 保存时写插件 extra
- 复制按钮读取时走插件快照
- 对账 / 恢复 / 构建版本 ID 时又常常走宿主 getter

结果就是：

> **系统没有统一的原文真相源。**

---

### 2.2 宿主 getter 存在危险降级逻辑
`getAssistantOriginalSourceTextFromHostEvent(...)` 当前有类似逻辑：

```ts
if (originalText.endsWith(`\n${preferredText}`) || originalText.endsWith(preferredText)) {
  return preferredText;
}
```

这会导致一种危险情况：

- `originalText` 其实包含 ROLLJSON / 事件控制块
- `preferredText` 是已清理的正文
- 只要宿主原文以正文结尾
- 函数就直接返回正文，丢弃原文中的控制块

也就是说：

> **宿主 getter 现在可能主动把“带 ROLLJSON 的原文”降级成“纯正文”。**

---

### 2.3 清理阶段会放大这个问题
当前真正执行清理的不是每个流式 token，而是：

- `message_received`
- `generation_ended`
- `message_swiped`
- `message_edited`
- `message_deleted`
- `chat_changed / chat_reset / chat_started ...`

这些阶段会调用：

- `handleGenerationEndedEvent()`
- `sanitizeCurrentChatEventBlocksEvent()`
- `sanitizeAssistantMessageArtifactsEvent(...)`

如果在进入清理前，没有稳定保住真正 raw source，那么：

- 清理后消息正文只剩纯文本
- 后续再次读取时只能拿到清理后的内容
- 复制按钮自然也拿不到 ROLLJSON

---

## 3. 根因总结

本问题的根因不是“流式每 token 都在清理”。

真正根因有三条：

### 根因一：没有单一原文真相源
插件快照和宿主 getter 并存，且不同模块使用不一致。

### 根因二：宿主 getter 会误降级
宿主 getter 把包含控制块的原文错误降级成正文。

### 根因三：清理链路不是幂等安全的
一旦某条消息没保住 raw source，后续聊天扫描和再次清理会把问题不断放大。

---

## 4. 修复目标

彻底修复后，系统应满足以下目标：

### 4.1 真相源唯一
所有“原格式 / 原始文本 / 带 ROLLJSON 的原文”只认一处：

> **插件自己持久化的 raw snapshot**

### 4.2 宿主 getter 只做输入候选
宿主侧原文只用于：

- 第一次捕获时的候选来源
- 绝不直接作为复制 / 恢复 / 对账的最终来源

### 4.3 清理只影响正文，不影响原始快照
清理后：

- UI 中显示的消息正文可以去掉控制块
- 但插件持久化的原始快照必须完整保留

### 4.4 复制按钮只读稳定快照
复制原格式时：

- 只读稳定 raw snapshot
- 不再直接依赖宿主 getter 的即时结果

### 4.5 所有模块统一使用同一个稳定读取函数
包括：

- 复制按钮
- 对账恢复
- 构建助手消息版本 ID
- 事件包解析
- 未来的导出 / 调试 / 回放

---

## 5. 最终架构方案

推荐采用下面这套结构。

### 5.1 定义三种文本概念

#### A. `hostOriginalText`
来自宿主的原始文本候选  
用途：**仅用于捕获输入**

#### B. `stableOriginalSourceText`
插件持久化后的稳定原始快照  
用途：**系统唯一真相源**

#### C. `displayText`
清理控制块后的正文  
用途：**UI 渲染**

### 5.2 明确职责分离

| 文本类型 | 来源 | 是否含 ROLLJSON | 是否可被清理 | 用途 |
|---|---|---:|---:|---|
| `hostOriginalText` | 宿主 getter | 可能有 | 不直接清理 | 候选输入 |
| `stableOriginalSourceText` | 插件 extra/swipe_info.extra | 应保留 | 否 | 复制、恢复、对账 |
| `displayText` | message.mes / content / text | 不应有 | 是 | 展示 |

---

## 6. 核心改造原则

### 原则一：插件快照是唯一真相源
以后所有需要“原格式”的场景，必须统一读取：

- `getStableAssistantOriginalSourceTextEvent(...)`

而不是：

- 有时读 `getAssistantOriginalSourceTextEvent(...)`
- 有时读 `getAssistantOriginalSourceTextFromHostEvent(...)`

### 原则二：宿主 getter只做捕获
`getAssistantOriginalSourceTextFromHostEvent(...)` 以后只负责：

- 尝试从宿主拿到候选原文
- 不再承担最终读取职责

### 原则三：禁止危险降级
如果文本中命中：

- ROLLJSON
- 事件 envelope
- `rh-trigger`
- 任意 RollHelper 控制块

那么绝不能自动降级成 `preferredText`

### 原则四：所有清理前必须先稳定保留快照
在任何可能清理正文的链路执行前，必须先：

- 检查当前 stable snapshot 是否存在
- 如不存在或当前候选更丰富，则先补写 snapshot
- 然后再清理 displayText

---

## 7. 新增 / 重构的关键函数设计

## 8. 新增统一稳定读取函数

建议在 `messageSanitizerEvent.ts` 中新增：

```ts
export function getStableAssistantOriginalSourceTextEvent(
  message: TavernMessageEvent | undefined,
  deps?: {
    getHostOriginalSourceTextEvent?: (message: TavernMessageEvent | undefined) => string;
    getPreferredAssistantSourceTextEvent?: (message: TavernMessageEvent | undefined) => string;
    getMessageTextEvent?: (message: TavernMessageEvent | undefined) => string;
    parseEventEnvelopesEvent?: (text: string) => {
      events: DiceEventSpecEvent[];
      ranges: Array<{ start: number; end: number }>;
    };
  }
): string
```

### 规则

1. 先读插件快照：
   - `getAssistantOriginalSourceTextEvent(message)`
2. 如果快照存在且有效：
   - 直接返回
3. 否则尝试读宿主候选原文：
   - `getHostOriginalSourceTextEvent(message)`
4. 如果宿主候选更“丰富”：
   - 先写入插件快照
   - 再返回
5. 否则 fallback：
   - `preferredText`
   - 或 `messageText`

### 关键点
这将成为：

> **系统唯一合法的原始文本读取入口**

---

## 9. 新增“原文丰富度判定”函数

建议新增：

```ts
function scoreAssistantOriginalSourceRichnessEvent(
  text: string,
  deps: {
    parseEventEnvelopesEvent?: (text: string) => {
      events: DiceEventSpecEvent[];
      ranges: Array<{ start: number; end: number }>;
    };
  }
): number
```

### 建议评分规则

| 条件 | 分值 |
|---|---:|
| 非空文本 | 1 |
| 含 `rh-trigger` | +10 |
| 含事件 envelope / ranges | +20 |
| 含 ROLLJSON / 控制块 | +20 |
| 文本更长 | +少量补分 |

### 用途
用于比较：

- 当前 stable snapshot
- 当前 hostOriginalText
- 当前 preferredText

谁更值得保留。

这样就不要再写那种：

```ts
endsWith(preferredText) => return preferredText
```

这种粗暴逻辑。

---

## 10. 新增“是否允许降级”为正文的严格判断

建议新增：

```ts
function canDowngradeAssistantOriginalSourceToPreferredTextEvent(
  originalText: string,
  preferredText: string,
  deps: {
    parseEventEnvelopesEvent?: (text: string) => {
      events: DiceEventSpecEvent[];
      ranges: Array<{ start: number; end: number }>;
    };
  }
): boolean
```

### 只有满足下面全部条件，才允许降级
- `originalText` 不含 `rh-trigger`
- `originalText` 不含事件 envelope
- `originalText` 不含 ROLLJSON / RollHelper 控制块
- `originalText` 和 `preferredText` 只有宿主包裹差异
- `preferredText` 不是更“贫”的版本

### 最终建议
更稳的做法其实是：

> **直接不降级。**

也就是宿主 getter 只负责返回原始候选，不再做这种裁剪。

---

## 11. 新增“确保快照已稳定保存”函数

建议新增：

```ts
export function ensureAssistantOriginalSnapshotPersistedEvent(
  message: TavernMessageEvent,
  deps: {
    getHostOriginalSourceTextEvent?: (message: TavernMessageEvent | undefined) => string;
    getPreferredAssistantSourceTextEvent?: (message: TavernMessageEvent | undefined) => string;
    getMessageTextEvent?: (message: TavernMessageEvent | undefined) => string;
    parseEventEnvelopesEvent?: (text: string) => {
      events: DiceEventSpecEvent[];
      ranges: Array<{ start: number; end: number }>;
    };
  }
): boolean
```

### 作用
在任何清理动作之前调用，保证：

- 如果当前宿主候选比插件快照更丰富
- 就优先更新 stable snapshot

这样即使后面把正文清干净，快照仍然保住。

## 12. 文件级改造方案

## 13. `messageSanitizerEvent.ts` 改造方案

### 13.1 保留现有快照存储结构
保留：

- `RH_ORIGINAL_SOURCE_KEY_Event`
- `RH_ORIGINAL_SOURCE_ENABLED_KEY_Event`

### 13.2 新增统一稳定 getter
新增：

- `getStableAssistantOriginalSourceTextEvent(...)`

### 13.3 新增丰富度评分函数
新增：

- `scoreAssistantOriginalSourceRichnessEvent(...)`

### 13.4 新增快照持久化保障函数
新增：

- `ensureAssistantOriginalSnapshotPersistedEvent(...)`

### 13.5 调整 `rememberAssistantOriginalSnapshotEvent(...)`
改成：

- 始终基于评分和富文本特征来决定是否覆盖
- 不再只靠是否“看起来是 rich text”

### 13.6 调整 `sanitizeAssistantMessageArtifactsEvent(...)`
在真正执行：

- `removeRangesEvent(...)`
- `stripRollJsonBlocks(...)`
- `sanitizeMessageInteractiveTriggersEvent(...)`

之前，先调用：

- `ensureAssistantOriginalSnapshotPersistedEvent(...)`

这样正文清理前一定先把快照落好。

---

## 14. `promptEvent.ts` 改造方案

### 14.1 重定义 `getAssistantOriginalSourceTextFromHostEvent(...)`
把它改成：

> **仅返回宿主候选原文，不做最终真相判断**

### 14.2 删除或收缩危险逻辑
重点去掉这类逻辑：

```ts
if (originalText.endsWith(`\n${preferredText}`) || originalText.endsWith(preferredText)) {
  return preferredText;
}
```

### 14.3 推荐新函数命名
为了减少歧义，建议直接改名：

```ts
getAssistantOriginalSourceCandidateFromHostEvent(...)
```

而不是继续叫：

```ts
getAssistantOriginalSourceTextFromHostEvent(...)
```

因为现在这个名字太像“最终原文 getter”，实际不该承担这个职责。

---

## 15. `hooksEvent.ts` 改造方案

### 15.1 复制按钮读取逻辑统一改用稳定 getter
点击复制按钮时，当前逻辑是：

- 先读 `getAssistantOriginalSourceTextEvent(...)`
- 没有再 fallback `getMessageTextSafe(...)`

建议改成：

```ts
const copyText = getStableAssistantOriginalSourceTextEvent(message, deps).trim()
  || getMessageTextSafe(message).trim();
```

### 15.2 `resolveAssistantEnvelopeEvent(...)` 读取链统一
这里目前会混用：

- `getAssistantOriginalSourceTextEvent`
- `getPreferredAssistantSourceTextEvent`
- `getMessageTextEvent`

建议统一收口成：

- 第一候选始终为 `getStableAssistantOriginalSourceTextEvent(...)`
- 其他作为兜底

### 15.3 `captureLatestAssistantOriginalSnapshotEvent(...)`
继续保留，但改为：

- 仅负责在流式阶段尽早补写 stable snapshot
- 不直接依赖宿主 getter 做最终判断

### 15.4 `handleGenerationEndedEvent(...)`
在 `rememberAssistantOriginalSourceTextEvent(...)` 之前，先调用统一的快照保障函数。

---

## 16. `eventRuntime.hooks.ts` 改造方案

### 16.1 不再把宿主 getter当成“原文 getter”到处传
当前 wiring 里传来传去的是：

- `getAssistantOriginalSourceTextFromHostModuleEvent`

建议调整为两种依赖明确分开：

#### A. `getHostOriginalSourceCandidateEvent`
只负责宿主候选读取

#### B. `getStableAssistantOriginalSourceTextEvent`
只负责稳定原文读取

### 16.2 所有“读取原文”的地方统一改用稳定 getter
包括：

- `buildAssistantMessageIdEvent`
- `handleGenerationEndedEvent`
- `reconcilePendingRoundWithCurrentChatEvent`
- `sanitizeAssistantMessageEventBlocksEvent`
- `registerEventHooksEvent`

## 17. 清理执行时机调整方案

## 18. 流式阶段
### 当前判断
流式阶段不应该清理正文，这个方向是对的。

### 维持规则
`stream_token_received` 时只做：

- `ensureAssistantOriginalSnapshotPersistedEvent(...)`

不要做：

- `sanitizeAssistantMessageArtifactsEvent(...)`

---

## 19. 最终处理阶段
### 保持主处理时机
仍以：

- `message_received`

为主，  
`generation_ended` 作为兜底。

### 关键新增规则
最终清理前必须先执行：

- `ensureAssistantOriginalSnapshotPersistedEvent(...)`

然后才能：

- 去掉 ROLLJSON
- 去掉控制块
- 去掉 `rh-trigger` 标记展示痕迹

---

## 20. 聊天切换 / 编辑 / swipe 阶段
### 当前风险
这些阶段会触发全聊天扫描清理。

### 建议改造
在全聊天扫描前，对每条 assistant message 执行：

- 如果 stable snapshot 已存在：直接跳过 raw 捕获
- 如果不存在：先尝试从宿主候选补写 snapshot
- 再做清理

### 目标
确保：

> **任何后续全量清理都不会把“尚未保存原文”的消息直接洗成只剩正文。**

---

## 21. 推荐的最终读取规则

全系统以后统一使用下面的优先级：

### 规则
1. **stable snapshot**
2. host original candidate
3. preferred text
4. display text

### 原则
- 复制原格式：只看 1 为主
- 恢复 / 对账：1 为主，2 为辅
- UI 展示：4
- 原文捕获：2 -> 写入 1

## 28. 最终结论

这个问题不能靠“修复制按钮”解决。

因为根因不在按钮，而在于：

- 原始文本真相源不统一
- 宿主 getter 误降级
- 清理时机虽不在每个流式 token，但会放大丢失问题

彻底修复的唯一正确方向是：

> **把插件持久化的 raw snapshot 设为唯一真相源，宿主原文只做候选输入；**
>
> **所有清理前先保快照，所有复制 / 恢复 / 对账统一读稳定 getter。**

---

## 29. 一句话方案定义

> **先统一原文真相源，再约束清理时机，最后统一读取入口；不要让宿主 getter 再决定什么叫“原格式”。**
