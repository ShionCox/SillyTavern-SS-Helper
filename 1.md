我看了这个分支的核心链路，重点过了这些文件：`runtime-app.ts`、`prompt-injection-pipeline.ts`、`memory-sdk.ts`、`unified-memory-manager.ts`、`compare-key.ts`、`entity-schema.ts`、`graph-edge-builder.ts`、`summary-orchestrator.ts`、`LLMHub/src/schema/strict-json-schema.ts`。

先说结论：

**方向是对的，已经明显不是“老式摘要插件”了，而是在往“结构化记忆平台”走。** 这和你 PDF 里定的方向是一致的：**结构化记忆做主数据，冷启动 / 总结 / 召回分层处理**。

但现在最大的优化点不是“再加功能”，而是这四件事：

## 1）先拆“大总管类”，现在职责还是太挤了

你现在有两个明显的“过胖类”：

* `MemorySDKImpl`
* `UnifiedMemoryManager`

它们都在同时做很多事：

* 数据 CRUD
* 总结触发
* 提示词组装
* 遗忘计算
* 旧聊天接管
* 世界画像绑定
* 审计记录
* 测试导出

这会带来一个问题：
**功能虽然都能挂进去，但后面会越来越难改，改一处容易牵一片。**

我建议优先拆成这几层：

* `EntryRepository`：只管 entry / actor / roleMemory / summary / audit 的数据库读写
* `SummaryService`：只管总结、planner、mutation apply
* `PromptAssemblyService`：只管召回、遗忘、注入文本组装
* `TakeoverService`：只管旧聊天接管
* `GraphService`：只管图谱节点和边
* `CompareKeyService`：只管 compareKey 生成、解析、匹配

**一句话：SDK 只做门面，不要继续往里塞业务。**

---

## 2）你现在的“稀疏 patch”设计，代码上有点打架，这是最该先修的

你在 `summary-orchestrator.ts` 里，思路是对的：

* 想让 AI 只返回变化字段
* 不要完整重写旧对象
* 做 sparse patch

这个方向很好。

但现在实现上有个明显风险：

你前面明明在强调 **payload 只输出变化字段**，
后面又有一层 `ensureStrictRequired(...)` 递归把对象属性都补成 required。

这样很容易变成：

**你嘴上让 AI 做 patch，实际上 schema 又在逼它补全完整对象。**

这会直接带来两个问题：

1. AI 更容易“把旧内容整份重写一遍”
2. `NOOP / ADD / DELETE` 这类动作也会被迫带很多没必要字段

这个我会列成 **P0**。

### 建议你改成：

* 顶层只要求最小必填：

  * `action`
  * `targetKind`
* `candidateId / compareKey / payload / reasonCodes`
  不要全局一刀切 required
* 按 action 分开约束：

  * `ADD`：允许没有 `candidateId`
  * `NOOP`：允许没有 `payload`
  * `DELETE / INVALIDATE`：可以只要定位信息
  * `UPDATE / MERGE`：才要求 patch payload

**也就是：不要用一个通用 schema 硬套所有 action。**

---

## 3）Prompt 注入现在有重复计算，实际运行会浪费一次检索

这个点很实际。

你的 `runPromptReadyInjectionPipeline()` 里先做了一次 preview，
然后 `inject()` 里又会重新调用 `buildPromptAssembly()`。

结果就是：

**一次 prompt ready，可能做两次完整召回 / 组装。**

如果后面数据量上来，或者接了 embedding，这里会越来越贵。

### 建议：

把它改成一个统一流程：

* 第一步：只构建一次 `PromptAssemblySnapshot`
* 第二步：

  * preview 直接用这个 snapshot
  * inject 也直接用这个 snapshot
* 不要 preview 一次、inject 再来一次

**这个优化非常值，属于“改动不算大，但立刻见效”。**

---

## 4）图谱层还没完全切到“结构化主源”，还带着旧 entries 思维

这个和你前面问的图谱问题是同一类。

你现在 `graph-edge-builder.ts` 还是：

* 基于 `MemoryEntry`
* 用标题 / alias / fields 去猜边
* 边的 source/target 还是 `entryId`

这说明图谱现在还是偏“从条目反推关系”，而不是“从实体状态直接出图”。

但你新分支的目标已经不是这样了。
按你 PDF 的方向，应该是：

* **结构化实体 / 关系 / 世界状态 才是主数据**
* 图谱应该直接吃这些主数据，而不是再从 `entries` 里二次猜测。

### 我建议：

图谱分两层：

* **主图谱层**

  * 节点：actor / organization / city / nation / location / task / event
  * 边：relationship / located_in / belongs_to / occurs_at / related_to
* **兼容层**

  * 从旧 `entries` 推临时边，只做 fallback

也就是说：

**图谱主源改成结构化结果，`entries` 不再当主构图源。**

## 5）`compareKey` 现在还不够强，后面会卡住你

你这次已经开始把 `compareKey` 提到核心位置了，这是对的。

但我看下来，当前版本还不够稳，主要有几个问题：

### 问题一：标题归一化太弱

现在基本只是去空格，强度不够。
像这些情况容易出问题：

* 大小写
* 标点差异
* 全半角
* 别名
* 中英文混排
* 同名不同区域实体

### 问题二：`parseCompareKey()` 太通用

比如关系这种：

* `relationship:source:target`

你现在这种通用解析，语义其实丢了一半。
后面一旦复杂一点，就不够用了。

### 问题三：默认 fallback 太容易冲突

有些类型最后会退回到：

* `type:title`

这对图谱、合并、旧聊天接管来说都不太稳。

### 建议：

把 `compareKey` 正式升级成协议：

* 分类型生成器
* 分类型解析器
* 分类型 matcher
* 支持 schema version
* 支持 alias canonical name
* 区分：

  * `entityKey`：内部稳定主键
  * `compareKey`：跨流程比对键

**不要把 compareKey 同时当“唯一主键”和“模糊匹配键”。**

## 6）总结、接管、冷启动三条写入链路，建议统一成一个 mutation 落盘入口

现在你这三条链路虽然方向一致了，但写入逻辑还是有点散：

* 冷启动一套 apply
* 总结一套 apply
* takeover 里又有很多手工 `saveEntry(...)`

这会导致后面同一个字段规则改了，要改三处。

### 建议：

统一成一个入口：

* `applyLedgerMutationBatch(mutations, context)`

无论来源是：

* cold start
* summary
* takeover
* manual edit

最后都走同一套：

* compareKey 匹配
* action 决策
* save/update/invalidate
* role binding
* audit
* history

**这样规则才不会漂。**

## 7）遗忘机制现在“可用”，但还不够像你想要的“角色化记忆”

你现在的 `shouldForget()` 是可运行的，优点是简单。

但它的问题也明显：

* 很依赖当前 query
* 同一条记忆会因为问法不同突然消失
* 更像“检索扰动”，不像“角色记忆衰减”

这和你文档里想做的：

* clear
* blur
* distorted
* perActor retention

还差一层。

### 建议：

把遗忘拆成两步：

#### 第一步：先算 retention stage

基于：

* importance
* rehearsalCount
* recency
* actor memoryStat
* relation sensitivity

算出：

* clear
* blur
* distorted

#### 第二步：再决定渲染方式

不是简单“忘了/没忘”，而是：

* clear：正常输出
* blur：模糊输出
* distorted：错误理解输出

**这样才更接近你原本想做的“角色记忆人格系统”。**

## 8）数据查询要开始考虑性能了，不然后面 prompt 阶段会越来越重

现在一些核心读法还是偏“先全量取，再内存过滤”，比如：

* `listEntries()`
* `buildPromptAssembly()`
* `readTakeoverExistingKnownEntities()`

现在数据少还好，后面一多就会卡。

### 建议：

优先做这几个：

* 给常用筛选加索引
* 常用集合做 cache
* prompt 阶段尽量不要全量扫 entries
* actor / relationship / world / recent event 分开索引
* compareKey 建单独映射表

**总结一句：召回链路要从“全表扫描型”变成“索引命中型”。**

## 9）楼层数 / 回合数的定义，要尽快统一

现在总结触发里，`messageFloorCount` 有时更像“消息数”，
但你整体设计里又经常说“assistant 楼层”“回合窗口”。

这个如果不统一，后面会出现：

* UI 显示一个数
* summary 实际按另一个数触发
* takeover 又按第三种规则切批次

### 建议直接定一个标准：

* `turn`: 一对 user+assistant
* `message`: 单条消息
* `assistantFloor`: assistant 消息序号
* `summaryWindow`: 按哪一种切，必须固定

我建议你整个系统统一成：

* **触发看 assistantFloor**
* **窗口切分按 message index**
* **展示给用户也明确写 assistant 楼层**

这次把 strict schema 兼容处理放到 LLMHub，是对的。
这个分层跟你 PDF 里说的也一致。

但还可以再补两点：

### 建议补：

现在已经有雏形了，但还可以再独立一点，不要让 MemoryOS 自己猜太多 provider 细节。
