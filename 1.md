# 基于 Commit b1999ea 的流式检测降载与 Finalize 事件重构实施方案

## 1. 目标

本方案用于解决 RollHelper 在最新 `b1999ea` 提交后仍存在的这类问题：

- 流式传输过程中反复检测，资源消耗过高
- `rolljson / triggerjson` 在未闭合时就被持续尝试解析
- 控制台持续出现“JSON 解析失败 / 修复失败 / 已隐藏代码块”类日志
- 流式阶段与最终落地阶段职责混杂，导致链路不稳定

本次重构的核心目标是：

> **流式阶段只做轻量快照缓存，不做解析；最终只在消息落稳后统一 sanitize、解析、DOM 同步与 autoRoll。**

---

## 2. 审查结论

结合 `b1999ea` 当前实现，问题已经不是“最终清理顺序”本身，而是：

> **虽然 finalize 已经后移，但流式阶段仍然会间接碰到解析链。**

也就是说，你现在已经做了：

- `MESSAGE_RECEIVED` 作为主要 finalize
- `GENERATION_ENDED` 作为慢速兜底
- `autoRoll` 放到 DOM 同步后

这些方向本身是对的。

但当前仍有一个关键缺口：

> **`STREAM_TOKEN_RECEIVED` 阶段仍然会间接进入“稳定原文选择 / 消息版本计算 / envelope 判断”这条重链。**

这会导致在 AI 还没输出完时：

- `rolljson` 还没闭合
- `triggerjson` 还没闭合
- `trigger_pack` 还没完整
- `rh-trigger` 还可能只输出了一半

系统却已经开始尝试：

- 判断 source candidate
- 计算稳定 messageId
- 做 envelope 检测
- 做 JSON 修复尝试
- 记录失败日志

最终形成“每来一个 token 就检测一轮”的高频无效消耗。

---

## 3. 现状问题拆解

## 3.1 现在的主问题不是 finalize，而是 stream_snapshot 仍然太重

当前 `hooksEvent.ts` 里已经把事件分成：

- `stream_snapshot`
- `message_received_finalize`
- `generation_ended_finalize`

但 `stream_snapshot` 仍然走进了统一楼层生命周期体系的一部分。

虽然它表面上只是想“提前保留原文快照”，但从当前报错链路可以看出，流式阶段仍然会间接碰到：

- `getStableAssistantOriginalSourceTextEvent(...)`
- `buildAssistantMessageIdEvent(...)`
- `parseEventEnvelopesEvent(...)`

这说明现在的流式阶段并不轻。

---

## 3.2 解析失败在流式阶段其实不该算异常

流式阶段拿到的是“未完成文本”。

此时出现这些情况是正常的：

- fence 未闭合
- JSON 未闭合
- `events` 数组未闭合
- `trigger_pack.items` 未闭合
- 引号、括号、花括号只输出一半

所以如果流式阶段还在做 JSON 解析，那么：

> **解析失败本来就是常态，不应该被当成告警。**

现在控制台刷屏，本质不是 parser 太笨，而是“触发时机不对”。

---

## 3.3 `buildAssistantMessageIdEvent(...)` 不适合流式阶段参与

当前 `buildAssistantMessageIdEvent(...)` 会为了生成稳定版本标识，构建“稳定版本文本”。

而“稳定版本文本”为了排除控制块，通常会继续碰：

- 原文候选选择
- envelope 解析
- 控制块剥离

这在 finalize 阶段是合理的，但在 stream token 阶段不合理。

因为流式阶段文本还没稳定，版本哈希本来就不应参与。

---

## 4. 参考 SillyTavern 事件能力

根据 SillyTavern `public/scripts/events.js` 当前事件定义，已经明确存在这些事件：

- `STREAM_TOKEN_RECEIVED`
- `MESSAGE_RECEIVED`
- `GENERATION_STARTED`
- `GENERATION_ENDED`
- `MESSAGE_UPDATED`

这意味着宿主已经提供了足够清晰的分层时机：

### 适合流式缓存的事件
- `STREAM_TOKEN_RECEIVED`

### 适合最终处理的事件
- `MESSAGE_RECEIVED`

### 适合慢速兜底的事件
- `GENERATION_ENDED`

### 适合会话初始化的事件
- `GENERATION_STARTED`

所以 RollHelper 不需要在 `STREAM_TOKEN_RECEIVED` 上做解析型处理，只需要做轻量缓存即可。

---

## 5. 本次重构总原则

## 原则一：流式阶段禁止解析
`STREAM_TOKEN_RECEIVED` 不再允许触发：

- rolljson 解析
- triggerjson 解析
- trigger_pack 解析
- rh-trigger metadata 提取
- messageId 稳定哈希构建
- JSON 修复尝试
- 失败 warn 日志

---

## 原则二：流式阶段只缓存“最新原始文本”
流式阶段的唯一目标是：

> **把当前这次生成会话的最新原始文本记下来。**

它不负责理解内容，不负责做业务处理。

---

## 原则三：最终只在消息落稳后处理
真正的业务处理只放到：

1. `MESSAGE_RECEIVED`
2. `GENERATION_ENDED` 兜底

在这两个阶段再统一执行：

- sanitize
- event parse
- trigger metadata parse
- DOM 同步
- merge event
- autoRoll

---

## 原则四：`GENERATION_STARTED` 明确开启新会话
每次生成开始时，必须开启一个新的 assistant generation session，并清空本次会话缓存。

---

## 原则五：parser 告警仅允许出现在 finalize 阶段
流式阶段遇到不完整 JSON，不应输出 warn。

---

## 6. 推荐的新链路

## 6.1 新链路总览

```text
GENERATION_STARTED
  -> 开启生成会话
  -> 重置本次会话缓存

STREAM_TOKEN_RECEIVED
  -> 仅记录最新流式原文
  -> 不做解析

MESSAGE_RECEIVED
  -> 主 finalize
  -> sanitize + parse + DOM sync + autoRoll

GENERATION_ENDED
  -> 若本次会话尚未 finalize，则做兜底 finalize
```

---

## 6.2 阶段 A：生成开始

### 目标
建立“这次回复”的独立运行上下文。

### 应做的事
1. 新建或更新当前 assistant generation session
2. 记录：
   - sessionKey
   - targetIndex（若能拿到）
   - swipeId
   - startedAt
3. 清空本次会话的流式缓存：
   - `latestStreamText = ""`
   - `latestStreamUpdatedAt = 0`
   - `finalized = false`

### 禁止做的事
- 不解析
- 不 sanitize
- 不生成 messageId
- 不触发 DOM 更新
- 不写 trigger metadata

---

## 6.3 阶段 B：流式传输中

### 目标
仅记录当前最新的原始流内容。

### 应做的事
1. 定位当前生成会话
2. 只更新：
   - `latestStreamText`
   - `latestStreamUpdatedAt`
3. 可选记录：
   - 最近一次 token 数
   - 最近一次消息长度

### 禁止做的事
- 不调用 `rebuildAssistantFloorLifecycleEvent(...)`
- 不调用 `finalizeAssistantFloorDataEvent(...)`
- 不调用 `sanitizeAssistantMessageArtifactsOnceEvent(...)`
- 不调用 `getStableAssistantOriginalSourceTextEvent(...)`
- 不调用 `buildAssistantMessageIdEvent(...)`
- 不调用 `parseEventEnvelopesEvent(...)`

---

## 6.4 阶段 C：消息落稳后主 finalize

### 推荐以 `MESSAGE_RECEIVED` 为主入口

### 目标
在宿主消息已经基本落稳时，统一完成业务处理。

### 应做的事
1. 根据当前 message 对象读取最终文本
2. 结合 session buffer 决定最终源文本候选
3. 执行单次 sanitize：
   - rolljson 清理
   - triggerjson 清理
   - trigger_pack 提取
   - rh-trigger 提取
4. 写回 message 文本与 metadata
5. 立即同步当前消息 DOM
6. 接管 event
7. 执行 autoRoll
8. 保存 chat / metadata
9. 标记 `session.finalized = true`

---

## 6.5 阶段 D：`GENERATION_ENDED` 兜底 finalize

### 目标
处理少数 `MESSAGE_RECEIVED` 时序异常的情况。

### 应做的事
1. 检查当前 session 是否已经 finalize
2. 若已 finalize，直接跳过
3. 若未 finalize，补跑一次 finalize
4. 完成后标记 finalized

### 这样做的价值
避免：
- 双重 finalize
- 双重 autoRoll
- 双重清理
- 双重日志

---

## 7. 代码级重构建议

## 7.1 `hooksEvent.ts`：把流式处理从统一重建链路中拆出去

### 当前问题
现在 `STREAM_TOKEN_RECEIVED` 还是走：

- `rebuildAssistantFloorLifecycleEvent(...)`

虽然 reason 是 `stream_snapshot`，但它仍然挂在统一生命周期函数里，后续极易再被扩展时误带重逻辑。

### 建议改法
新增一个专门的流式轻量函数，例如：

```ts
captureAssistantStreamSnapshotEvent(args)
```

### 它只负责
- 找到当前 target
- 开启 session（如果需要）
- 更新 session.latestStreamText
- 更新 session.latestStreamUpdatedAt

### 明确禁止
- parse
- sanitize
- finalize
- buildAssistantMessageId

---

## 7.2 `hooksEvent.ts`：增加 `GENERATION_STARTED` 监听

### 需要新增
监听：

- `types.GENERATION_STARTED`
- `"generation_started"`

### 进入：
```ts
beginAssistantGenerationSessionEvent(...)
resetAssistantGenerationStreamBufferEvent(...)
```

### 目的
不要等到 stream token 到来时才“猜测是否新会话”，而是在生成开始时明确初始化。

---

## 7.3 `hooksEvent.ts`：保留 `MESSAGE_RECEIVED` 为主 finalize，缩短延迟后可继续保留
你现在在 `MESSAGE_RECEIVED` 上延迟 30ms 再 finalize，这个思路可以保留。

### 建议
继续让它作为主 finalize 入口，不要把主处理再迁回 `GENERATION_ENDED`。

---

## 7.4 `hooksEvent.ts`：`GENERATION_ENDED` 只做兜底，不做主处理
当前 800ms 慢兜底可以保留，但建议显式写清楚：

- 仅在 `session.finalized === false` 时运行
- 不要再次开始新 session
- 不要重复清理

---

## 7.5 `messageSanitizerEvent.ts`：禁止被流式阶段调用
### 要求
把这几个函数的使用场景收紧到 finalize 阶段：

- `sanitizeAssistantMessageArtifactsOnceEvent(...)`
- `sanitizeAssistantMessageArtifactsEvent(...)`

### 原因
它们本身就是“最终清理函数”，不适合流式中间态。

---

## 7.6 `buildAssistantMessageIdEvent(...)`：只允许 finalize 阶段调用
### 原因
它依赖“稳定版本文本”，而稳定版本文本往往需要剥离控制块。

### 结论
不允许在 `STREAM_TOKEN_RECEIVED` 链路里触发。

---

## 8. 新增会话缓存结构建议

建议新增一个更明确的流式会话结构：

```ts
type AssistantGenerationLiveStateEvent = {
  sessionKey: string;
  runtimeKey: string;
  swipeId: number;
  targetIndex: number | null;
  latestStreamText: string;
  latestStreamUpdatedAt: number;
  finalized: boolean;
};
```

---

## 9. 新增函数建议

## 9.1 `openAssistantGenerationLiveSessionEvent(...)`
职责：

- 创建或重置当前生成会话
- 标记 finalized=false
- 清空流式缓存

---

## 9.2 `captureAssistantStreamSnapshotEvent(...)`
职责：

- 接收 `STREAM_TOKEN_RECEIVED`
- 只更新 `latestStreamText`
- 不做任何解析

---

## 9.3 `resolveFinalizeSourceTextEvent(...)`
职责：

在 finalize 阶段统一决定最终源文本优先级。

### 推荐优先级
1. 当前 message 最终文本
2. session.latestStreamText
3. host original source
4. 已持久化原始快照

### 目的
让 finalize 阶段再做一次“最终选择”，而不是在流式阶段不断选。

---

## 9.4 `finalizeAssistantGenerationIfNeededEvent(...)`
职责：

- 检查当前 session 是否已 finalize
- 若没有则执行 finalize
- 若已经 finalize 则直接跳过

---

## 10. 日志系统优化建议

## 10.1 流式阶段禁止 warn
流式阶段只允许：

- debug
- trace 级别内部统计

不允许：

- JSON 解析失败 warn
- 修复失败 warn
- 已隐藏代码块 warn

---

## 10.2 finalize 阶段才输出真正解析异常
只有在以下阶段，解析失败才是有意义的：

- `MESSAGE_RECEIVED finalize`
- `GENERATION_ENDED finalize fallback`

因为这时文本应该已经是完整的。

---

## 10.3 增加“阶段来源”日志字段
建议统一日志附带：

- `phase=stream`
- `phase=finalize_main`
- `phase=finalize_fallback`

这样后续排查能很快看出是哪一层出的日志。

---

## 11. 推荐的新伪代码

```ts
function onGenerationStarted(...) {
  openAssistantGenerationLiveSessionEvent(...)
}

function onStreamTokenReceived(...) {
  captureAssistantStreamSnapshotEvent(...)
}

async function onMessageReceived(...) {
  await finalizeAssistantGenerationIfNeededEvent({
    phase: "finalize_main",
  })
}

async function onGenerationEnded(...) {
  await finalizeAssistantGenerationIfNeededEvent({
    phase: "finalize_fallback",
  })
}
```

---

## 12. 分阶段实施顺序

## 第一阶段：先把流式阶段彻底减重
1. 新增 `GENERATION_STARTED` 监听
2. 新增 `captureAssistantStreamSnapshotEvent(...)`
3. 移除 `STREAM_TOKEN_RECEIVED -> rebuildAssistantFloorLifecycleEvent(...)`
4. 确保流式阶段不再碰 parser

### 验收结果
- 控制台不再每个 token 刷 warn
- CPU 占用明显下降
- 流式期间不再做 sanitize

---

## 第二阶段：收紧 finalize 入口
1. `MESSAGE_RECEIVED` 作为主 finalize
2. `GENERATION_ENDED` 只做兜底
3. finalize 必须有 session 去重

### 验收结果
- 不再双重 finalize
- 不再双重 autoRoll
- 不再重复清理

---

## 第三阶段：统一 finalize 源文本选择
1. 新增 `resolveFinalizeSourceTextEvent(...)`
2. 流式阶段不再做 source candidate 选择
3. 所有稳定选择放到 finalize 阶段执行

### 验收结果
- 解析链只在最终文本阶段运行
- messageId 计算更稳定

---

## 第四阶段：日志与测试完善
1. 区分 stream / finalize_main / finalize_fallback 日志
2. finalize 失败才输出 warn
3. 加回归测试

## 15. 最终结论

当前 `b1999ea` 的方向已经比之前更接近正确结构，但还差最后一刀：

> **必须把流式阶段从“统一生命周期重建链”里彻底降级成“纯缓存层”。**

真正的问题不在 parser 本身，而在于：

> **你现在还让未完成的文本在流式阶段间接进入了解析链。**

所以最优修复方向不是继续补 parser，而是：

1. `GENERATION_STARTED` 开新会话
2. `STREAM_TOKEN_RECEIVED` 只缓存最新流文本
3. `MESSAGE_RECEIVED` 做主 finalize
4. `GENERATION_ENDED` 做去重兜底 finalize

---

## 16. 一句话方案定义

> **把 `STREAM_TOKEN_RECEIVED` 从解析入口改成纯快照缓存入口，把所有 sanitize / event parse / trigger parse / messageId 计算统一延后到 `MESSAGE_RECEIVED`，并让 `GENERATION_ENDED` 只做一次性兜底 finalize。**
