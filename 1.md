# MemoryOS 第一阶段详细改造方案（支持独立向量链路的向量预留版）

## Summary
- 本阶段**不直接上完整向量系统**，目标是先把 MemoryOS 改造成“**向量就绪架构**”。
- 当前 MemoryOS 已经具备：
  1. `RetrievalOrchestrator` 检索编排骨架
  2. `LexicalRetrievalProvider` 规则型召回
  3. `EmbeddingRetrievalProvider` 预留接口
  4. `enableEmbedding` 设置开关
- 但目前真正的 embedding 检索**尚未落地**：
  `EmbeddingRetrievalProvider.search()` 仍是占位实现，直接返回空数组。
- 所以本阶段的正确目标不是“马上补向量索引”，而是：
  1. 收口检索主链
  2. 把查询配置和过滤协议正式化
  3. 支持**向量链路独立开启**
  4. 增加秘书层 / 蒸馏层
  5. 增加维护治理层
  6. 给后续向量召回和混合检索预留稳定接口

---

## Current Assessment

### 当前 MemoryOS 已有基础
- `PromptAssemblyService` 已经把 `enableEmbedding` 传给检索编排层。
- `RetrievalOrchestrator` 已具备：
  - context routing
  - seed retrieval
  - graph expansion
  - coverage second pass
  - diversity prune
- `LexicalRetrievalProvider` 已经实现：
  - BM25
  - n-gram similarity
  - edit similarity
  - memory weight
  - recency weight
- `EmbeddingRetrievalProvider` 还只是占位类。
- 设置层已有：
  - `enableEmbedding`
  - `retrievalRulePack`
  - `retrievalLogEnabled`
  - `retrievalTracePanelEnabled`

### 当前最明显的问题
1. 检索虽然有 orchestrator，但**还不是全系统唯一入口**
2. `enableEmbedding` 虽然存在，但现在**不会产生真实 embedding 结果**
3. 候选构建、召回过滤、召回解释、图扩展，虽然已经有雏形，但**还没有彻底协议化**
4. 原始对话进入记忆系统前，缺少一个像 PeroCore （https://github.com/YoKONCy/PeroCore）那样的“秘书层”
5. 缺少明确的维护治理层，后续一旦引入向量，脏数据、重复数据、低价值数据会一起被向量化
6. 目前设置层还是 `enableEmbedding:boolean` 思维，不利于后续做：
   - `lexical_only`
   - `vector_only`
   - `hybrid`

---

## 本阶段总目标
本阶段完成后，MemoryOS 应该具备以下能力：

1. **统一召回总入口**
   - Prompt、Takeover、Workbench、Graph Flashback 都通过同一套召回主链工作
2. **正式 RecallConfig 协议**
   - 后续 lexical / vector / graph 都用同一套查询配置
3. **正式 PayloadFilter 协议**
   - 后续向量召回可以直接做预过滤
4. **独立向量链路开关**
   - 向量功能可以单独开启，不依赖 lexical
5. **秘书层 / 蒸馏层**
   - 原始对话先清洗、压缩、提炼，再进入长期记忆和召回层
6. **维护治理层**
   - 为后续向量系统提供干净、稳定、可维护的数据底座
7. **预留 Query Context Builder**
   - 为第二阶段 embedding 查询上下文构造直接留好接口

# 一、第一阶段的改造原则

## 1. 不改主业务目标，只收口主链
本阶段不推翻现有：
- actor registry
- relationship registry
- entry 主链
- takeover 主流程
- prompt 注入工作流

本阶段做的是：
- 检索层收口
- 独立向量链路模式设计
- 写入前蒸馏
- 维护层补齐
- 接口前置设计

## 2. 不做“半套向量”
本阶段不建议做：
- 简陋 embedding 表先糊一个
- 直接把所有 entry 全量向量化
- 没有 filter / config / maintenance 就先堆 embedding

因为那样会把第二阶段做死。

## 3. 先借 PeroCore 的“闭环调度”，再为 TriviumDB 风格检索铺路
本阶段主要吸收：
- PeroCore 的秘书层 / 维护层 / 预处理链思路
- TriviumDB 的 SearchConfig / payload_filter / search_hybrid / dry-run 思路

## 4. 第一阶段必须支持“向量单独走一条链”
不是简单保留：
- `enableEmbedding: boolean`

而是要升级成：
- `lexical_only`
- `vector_only`
- `hybrid`

这样后续可以：
- 单独测试向量效果
- 单独排查向量问题
- 独立比较 lexical 和 vector 质量
- 在不动主链结构的前提下自由切换

---

# 二、第一阶段的核心改造项

## A. 统一召回主链：正式引入 `MemoryRetrievalService`

### 目标
把当前召回正式收口为全系统统一入口，不再让不同模块自己拼检索逻辑。

### 当前基础
已有 `RetrievalOrchestrator`，但建议升级为更明确的服务边界。

### 建议新增服务
- `MemoryOS/src/services/memory-retrieval-service.ts`

### 建议职责
1. 接收统一查询输入
2. 归一化 query / context
3. 应用 payload filter
4. 根据 `retrievalMode` 决定走哪条链
5. 调用 lexical provider
6. 调用 embedding provider（当前阶段可继续返回空，但接口必须完整）
7. 合并 graph expansion / coverage / diversity
8. 输出：
   - items
   - diagnostics
   - explanation trace

### 对外建议接口
```ts
searchHybrid(input: MemoryRetrievalInput): Promise<MemoryRetrievalOutput>
searchForPrompt(input: PromptRecallInput): Promise<MemoryRetrievalOutput>
searchForTakeover(input: TakeoverRecallInput): Promise<MemoryRetrievalOutput>
searchForWorkbench(input: WorkbenchRecallInput): Promise<MemoryRetrievalOutput>
```

### 为什么要先做这个
因为第二阶段一旦接入真实 embedding，不能再让 Prompt、Takeover、Workbench 各自走自己的检索逻辑。

---

## B. 正式设计 `RecallConfig`

### 目标
让召回行为变成“查询级配置”，而不是系统全局死开关。

### 建议新增类型
- `MemoryOS/src/memory-retrieval/recall-config.ts`

### 建议结构
```ts
export type RetrievalMode = 'lexical_only' | 'vector_only' | 'hybrid'

export type RecallConfig = {
  retrievalMode: RetrievalMode
  topK: number
  minScore: number
  expandDepth: number
  enableHybridText: boolean
  enableGraphExpansion: boolean
  enableDiversity: boolean
  payloadFilter?: PayloadFilter
  actorScope?: string[]
  schemaScope?: string[]
  relationScope?: string[]
  worldScope?: string[]
}
```

### 第一阶段必须支持的字段
- `retrievalMode`
- `topK`
- `minScore`
- `expandDepth`
- `enableHybridText`
- `enableGraphExpansion`
- `payloadFilter`

### 第一阶段可以先不做复杂的
- `teleportAlpha`
- `diversityWeight`
- `candidateBudgetProfile`
- `queryRouteWeight`

这些可以等第二阶段向量系统时再加。

---

## C. 用 `retrievalMode` 替代单纯的 `enableEmbedding`

### 目标
正式支持“向量功能单独开启并单独走链路”。

### 建议新增设置项
把：
- `enableEmbedding: boolean`

升级为：
```ts
retrievalMode: 'lexical_only' | 'vector_only' | 'hybrid'
```

### 三种模式的语义
#### 1. `lexical_only`
- 只走词法检索链
- 不调用 embedding provider
- 适合当前默认稳定模式

#### 2. `vector_only`
- 只走向量检索链
- lexical provider 不参与 seed
- 适合测试向量质量、做 A/B 实验、排查语义召回问题

#### 3. `hybrid`
- lexical + vector 同时参与
- 后续做融合排序
- 适合正式上线后的综合模式

### 第一阶段的实现原则
当前即便 embedding provider 还没真正接上，也要先把这三态模式设计好。

### 第一阶段下的预期行为
- `lexical_only`
  - 正常走现有 lexical provider
- `vector_only`
  - 当前若 embedding provider 无真实实现，应返回“provider unavailable”诊断，而不是悄悄混回 lexical
- `hybrid`
  - 当前可先 lexical 生效，embedding provider 为空时记录 diagnostics

### 为什么必须这样设计
因为如果继续只保留 `enableEmbedding:boolean`，后续根本没法优雅支持：
- 向量单独链路
- lexical / vector 对比实验
- 明确的 fallback 行为
- 线上灰度切换

---

## D. 正式设计 `PayloadFilter`

### 目标
把当前 MemoryOS 已有的候选字段，转成统一过滤协议，为后续向量检索预过滤铺路。

### 当前基础
`RetrievalCandidate` 已经包含：
- `actorKeys`
- `relationKeys`
- `worldKeys`
- `locationKey`
- `schemaId`
- `compareKey`
等字段。

### 建议新增类型
- `MemoryOS/src/memory-retrieval/payload-filter.ts`

### 建议结构
```ts
export type PayloadFilter = {
  chatKey?: string
  actorKeys?: string[]
  schemaIds?: string[]
  relationKeys?: string[]
  worldKeys?: string[]
  locationKeys?: string[]
  compareKeys?: string[]
}
```

### 第一阶段建议先支持
1. `chatKey`
2. `schemaIds`
3. `actorKeys`
4. `worldKeys`

### 为什么这一步很重要
因为第二阶段向量检索不能先全库搜完，再回来过滤。  
一定要让“先过滤，再召回”成为标准协议。

---

## E. 召回主链拆成三条明确模式链

### 目标
让当前检索系统从“是否启用 embedding”升级成“明确存在三条模式链”。

### 建议逻辑分层

#### 1. Lexical Chain
负责：
- tokenize
- BM25
- n-gram
- edit similarity
- memory weight
- recency weight
- lexical explanation

#### 2. Vector Chain
第一阶段先做协议壳，负责：
- query context input
- embedding query contract
- vector provider capability check
- payload filter contract
- vector explanation shell

#### 3. Hybrid Fusion Chain
负责：
- lexical results
- vector results
- graph expansion
- coverage
- diversity
- final rerank

### 第一阶段的关键要求
不要只在 orchestrator 里 `if enableEmbedding` 分支一下就算完。  
而是要明确做成：

```ts
switch (retrievalMode) {
  case 'lexical_only':
    ...
  case 'vector_only':
    ...
  case 'hybrid':
    ...
}
```

---

## F. 补“秘书层 / 蒸馏层”：`MemoryScoringService`

### 目标
参考 PeroCore 的 `ScorerService`，把原始对话先处理成更干净的召回文本和结构化变更候选。

### 建议新增服务
- `MemoryOS/src/services/memory-scoring-service.ts`

### 职责
1. 接收原始对话批次
2. 清理噪音
3. 对长批次自动拆分
4. 提炼摘要候选
5. 提取结构化线索
6. 输出：
   - distillation text
   - summary candidate
   - actor / relationship / entity clues
   - mutation suggestions
   - audit trace

### 第一阶段不要求
- 直接生成 embedding
- 直接写向量索引

### 第一阶段就应该支持的输入来源
1. normal summary pipeline
2. takeover old chat batch
3. future manual import batch

### 要清洗的典型噪音
- thinking / monologue 片段
- tool 调用细碎结果
- 超长系统注入块
- 重复样板文本
- 过度结构化标记内容

### 为什么这一步必须先做
如果不先做蒸馏，第二阶段向量系统只会把脏内容更稳定地“放大”。

---

## G. Query Context Builder：为 embedding 查询预留接口

### 目标
让未来查询向量的构建方式先协议化。

### 建议新增服务
- `MemoryOS/src/services/query-context-builder.ts`

### 建议输出结构
```ts
export type QueryContextBundle = {
  queryText: string
  recentUserText: string
  recentAssistantText: string
  recentToolText: string
  mergedContextText: string
  actorBiasKeys: string[]
  relationBiasKeys: string[]
  worldBiasKeys: string[]
}
```

### 第一阶段先做什么
- 只做文本组装和上下文规范化
- 不真正生成 embedding

### 第二阶段会怎么用
- `EmbeddingService.encode(QueryContextBundle.mergedContextText)`
- 或 user / assistant / tool 分开编码再加权融合

### 当前阶段的好处
先把协议定死，第二阶段接 embedding 时不用再改上游。

---

## H. 图扩展保持轻量，但补“热点降权”

### 目标
参考 TriviumDB 的图扩散思想，但只做轻量版。

### 当前已有
`RetrievalOrchestrator` 已经在做 `expandFromSeeds()`。

### 当前阶段建议新增
1. 最近高频命中节点轻惩罚
2. 高入度基础节点轻惩罚
3. 可配置 `expandDepth=0/1/2`

### 第一阶段先不做
- FISTA
- DPP 数学版
- PPR 复杂阻尼
- 不应期完整模型

### 为什么
当前最该先解决的是：
- 检索稳定
- 结果不僵化
- 召回不被热门节点垄断

先做轻量版就够。

---

## I. 新增维护治理层：`MemoryMaintenanceService`

### 目标
在真正上向量前，先让长期记忆库进入“可维护状态”。

### 建议新增服务
- `MemoryOS/src/services/memory-maintenance-service.ts`

### 第一阶段建议职责
1. 重复候选检测
2. compareKey 冲突检测
3. 低价值旧条目压缩
4. 检索访问日志统计
5. 检索 trace 归档

### 第一阶段建议不要做太重的
- 梦境
- 周报
- 自动人格演化
- 全量智能改写

### 最值得先做的三件事
1. 去重
2. 压缩
3. 留痕

### 为什么
因为第二阶段一旦上向量，坏数据会被更高效地召回。  
先治理，再上向量，才不会越做越脏。

# 四、当前代码层的详细改造建议

## 1. `prompt-assembly-service.ts`
### 当前问题
- 直接构建候选、组织查询、调用 orchestrator，职责还是偏多
- 后续一旦加入真正向量召回，继续堆这里会越来越重

### 第一阶段建议
- 只保留：
  - prompt 组装
  - visible context 渲染
  - prompt snapshot 输出
- 把：
  - query context 构建
  - recall config 构建
  - retrieval 调用
  全部委托给 `MemoryRetrievalService`

### 目标
让它重新变回：
**“Prompt 注入层”**
而不是：
**“半个检索控制器”**

---

## 2. `retrieval-orchestrator.ts`
### 当前问题
- 方向对，但现在更像内部流程类，不够像正式的总入口
- embedding provider 目前是空实现，导致 orchestrator 的向量路径只是逻辑占位

### 第一阶段建议
- 保留 orchestrator 作为内部执行器
- 在其外层新增 `MemoryRetrievalService`
- orchestrator 专心做：
  - seed retrieval
  - graph expansion
  - coverage
  - diversity
- 不直接承载业务入口语义

### 还要新增
- 支持 `retrievalMode`
- `vector_only` 时不要悄悄回退 lexical
- `hybrid` 时才允许 lexical + vector 融合

---

## 3. `embedding-provider.ts`
### 当前问题
- 目前就是占位类，返回空数组

### 第一阶段建议
- 不急着实现真实向量检索
- 但要先补“正式接口”
- 建议增加：
  - provider capability
  - pending status
  - unsupported reason
  - dry-run trace
  - mode awareness

### 目标
让它从“空类”升级成“可接后端的正式 provider 协议壳”。

### 第一阶段建议接口语义
```ts
search(query, candidates): Promise<RetrievalResultItem[]>
isAvailable(): boolean
getUnavailableReason(): string | null
supportsMode(mode: RetrievalMode): boolean
```

---

## 4. `lexical-provider.ts`
### 当前问题
- 当前逻辑可用，但还缺正式 filter 接口
- 当前只是按 candidateTypes 过滤，未来不够

### 第一阶段建议
- 增加 payload filter 支持
- 增加 retrieval explanation 输出
- 把 candidate text 构建提炼成可复用模块

### 原因
后续 vector / hybrid 检索也需要统一 explanation 输出。

---

## 5. `settings/store.ts`
### 当前问题
- 已经有 `enableEmbedding`，但还不够表达“向量就绪阶段”的需求

### 第一阶段建议新增设置项
```ts
retrievalMode: 'lexical_only' | 'vector_only' | 'hybrid'
retrievalUseUnifiedService: boolean
retrievalDefaultTopK: number
retrievalDefaultExpandDepth: number
retrievalEnablePayloadFilter: boolean
retrievalEnableGraphPenalty: boolean
retrievalEnableQueryContextBuilder: boolean
maintenanceAutoCompressEnabled: boolean
maintenanceDuplicateCheckEnabled: boolean
scoringServiceEnabled: boolean
```

### 第一阶段建议处理方式
- 保留旧 `enableEmbedding` 一小段时间仅做兼容映射
- 但新主链只认 `retrievalMode`

### 原则
先把设置能力补齐，后续第二阶段接向量时不会再拆设置协议。

# 六、第一阶段的验收标准

## 检索主链
1. Prompt、Workbench、Takeover 检索都通过统一入口
2. `RecallConfig` 能控制 topK / expandDepth / filter / graph 开关
3. `PayloadFilter` 已能作用在 lexical 路径上
4. retrieval diagnostics 输出结构统一

## 独立向量链路
5. 系统支持：
   - `lexical_only`
   - `vector_only`
   - `hybrid`
6. `vector_only` 模式下，当前即使 provider 为空实现，也能明确返回“不可用原因”
7. `hybrid` 模式下，能区分 lexical / vector / fusion 各自来源
8. 不再依赖单一 `enableEmbedding:boolean` 决定检索链路

## 向量预留
9. `EmbeddingRetrievalProvider` 不再只是“空壳类”，而是正式 provider 协议
10. QueryContextBuilder 能输出未来 embedding 所需统一查询结构
11. 上游调用不再直接依赖“是否已有 embedding 实现”

## 秘书层
12. takeover / summary 的输入可先过 `MemoryScoringService`
13. 噪音片段能被清洗
14. 长批次可拆分

## 维护层
15. compareKey 冲突有检测能力
16. 重复候选有识别逻辑

# 八、风险与规避

## 风险 1：第一阶段做太多，反而拖慢主线
### 规避
- 第一阶段不碰底层向量存储
- 不做 embedding 批处理队列
- 不做大规模 schema 重构
- 只做协议和服务收口

## 风险 2：秘书层和现有 summary / takeover 主链打架
### 规避
- 第一阶段秘书层先只做“前处理”
- 不直接替代 mutation apply
- 先作为 takeover / summary 的前置清洗器

## 风险 3：检索接口改太大，影响现有 Prompt 注入
### 规避
- `PromptAssemblyService` 只改调用入口，不改最终 visible context 渲染协议
- 保持现有 snapshot 输出结构兼容

## 风险 4：`vector_only` 做得像假开关
### 规避
- 明确规定：`vector_only` 下不允许偷偷走 lexical fallback
- provider 不可用时必须显式输出 unavailable diagnostics
- 这样这个模式才真的能用于测试和灰度

# 九、第一阶段完成后的状态定义

第一阶段完成后，MemoryOS 应该满足：

1. **没有真实向量检索，也能先提升召回质量**
2. **上层业务已经不依赖“是否有向量实现”**
3. **向量相关接口已经正式化**
4. **独立向量链路模式已经协议化**
5. **未来第二阶段只需要补：**
   - EmbeddingService
   - VectorDocumentBuilder
   - VectorStoreAdapter
   - Hybrid ranking
6. **不会再因为“先上向量再补协议”导致架构返工**

---

## Final Recommendation
- 第一阶段不是“做一点向量”，而是：
  **把 MemoryOS 改造成可承接完整向量系统、并支持独立向量链路的上层框架。**
- 最重要的五件事是：
  1. 统一召回服务
  2. 正式 RecallConfig / PayloadFilter
  3. 正式 RetrievalMode 三态化
  4. 增加秘书层
  5. 增加维护治理层
- 只要这五件做稳了，第二阶段向量系统就不是“另起炉灶”，而是自然接入。
