# rolljson/triggerjson 清理与 autoRoll 时序重构实施方案

## 1. 文档目标

本文用于彻底重构 RollHelper 当前这条关键链路：

> **AI 回复完成 → 清理 rolljson / triggerjson / rh-trigger → 接管事件 → autoRoll → 投骰动画 → 页面增强**

当前这条链路已经出现两个明显问题：

- AI 回复结束后，**投骰动画先出现**，但 `rolljson` / `triggerjson` 还没在页面上清干净
- 偶发出现**清理不完全**，例如 `triggerjson` 仍残留，必须手动刷新页面才彻底清掉

本方案的目标不是补单点判断，而是把整条链路重构为：

> **先完成数据清理，再立即同步当前消息 DOM，确认页面已干净，最后才允许 autoRoll 与投骰动画启动。**

---

## 2. 当前问题现象

用户当前看到的现象主要有两类：

### 2.1 动画早于清理
在 AI 回复结束时，页面上仍可短暂看到：

- `rolljson`
- `triggerjson`
- 甚至部分 `rh-trigger` 残留

但投骰动画已经开始播放。

这会让玩家感觉：

- 消息还没清理完
- 系统就已经开始结算
- 整个过程像是“半成品就进入下一步”

---

### 2.2 清理不完全
偶发情况下：

- `rolljson` 被清了
- 但 `triggerjson` 还残留
- 或触发标记相关 DOM 没完全同步

最终表现是：

- 要手动刷新页面
- 页面重建后才真正干净

这说明：

> **当前不是单纯“没清理”，而是“清理时序与 DOM 同步链路不稳定”。**

---

## 3. 根因结论

经过当前主链路审查，可以确认问题不是单一函数失效，而是以下三个问题叠加：

### 3.1 数据层清理早于页面层同步，但 autoRoll 启动又早于页面层同步完成
当前主流程中，消息文本与 metadata 的清理发生在数据层，但页面层真正“看起来干净”的更新，要等到后面的统一 hydrate。

与此同时，`autoRoll` 已经在数据层阶段启动，并且会直接进入 3D 投骰动画。

于是形成当前顺序：

1. 数据层开始清理
2. merge 事件
3. autoRoll 启动
4. 动画播放
5. 页面层 hydrate 才开始
6. 用户这时才看到消息真正干净

这就是“动画先出来，清理后完成”的根因。

---

### 3.2 当前消息正文没有“立即重绘”
当前文本清理主要是：

- 改 message 对象里的文本
- 再靠后续全局 hydrate 去补页面层

也就是说：

> **数据已经变了，不代表当前 `.mes_text` 立刻按最新文本重渲染了。**

这会造成：

- 页面上短暂仍显示旧 HTML
- 只能靠后续隐藏 code block、MutationObserver、刷新等机制补救
- 如果某一步没踩准，就出现残留

---

### 3.3 rolljson 与 triggerjson 清理仍是两段式
当前清理链路本质上仍是：

#### 第一段
- 解析并移除事件 envelope / rolljson

#### 第二段
- 再解析并清理 triggerjson / trigger_pack / rh-trigger

这会带来一个典型风险：

> **中间态会暴露给 UI。**

例如：

- 第一段已经把 `rolljson` 去掉
- 第二段还没把 `triggerjson` 清完
- autoRoll 已经开始
- 页面上就会短暂显示半清理状态

---

## 4. 当前链路为什么不稳

当前 finalize 阶段职责过重，且层次不清：

### 当前 `finalizeAssistantFloorDataEvent(...)` 同时在做：
- 原文快照补写
- assistantMsgId / floorKey 计算
- event envelope 解析
- 消息正文清理
- trigger metadata 清理
- 旧楼层失效
- merge 新事件
- autoRoll
- metadata 保存

而页面层的：

- hide code blocks
- trigger DOM 重建
- raw source 按钮增强
- widget 刷新

却在函数外部的统一 hydrate 里晚一步处理。

所以这条链天然就会形成：

> **“数据先动，页面后动，但动画又夹在中间”**

这正是当前不稳定感的来源。

---

## 5. 重构目标

本次重构后应达成以下目标：

### 5.1 清理一定先于动画
任何情况下：

- 页面上不应再可见 `rolljson`
- 页面上不应再可见 `triggerjson`
- 当前助手消息正文必须已经是最终干净状态

然后才允许 autoRoll 和投骰动画启动。

### 5.2 当前消息 DOM 要立即同步
不是只改数据，而是要对“刚刚完成的这条 assistant 消息”立即做局部 DOM 重绘与增强。

### 5.3 rolljson 与 triggerjson 清理要收敛为单次产物
避免：

- 多次改文本
- 分两段 setText
- 中间态暴露给 UI

### 5.4 finalize 阶段拆层
明确分成：

1. 数据清理
2. 当前消息 DOM 立即同步
3. 事件接管与 autoRoll
4. 全局 UI 收尾

### 5.5 整条链路幂等
即使 `message_received` 和 `generation_ended` 都触发，也不能：

- 重复清理
- 重复动画
- 重复插入按钮
- 重复增强 trigger
- 重复 merge 事件

---

## 6. 推荐的新主链路

建议把当前主链路彻底改成下面这个顺序。

---

## 7. 阶段 A：数据层单次清理

### 目标
把当前助手消息从“原始 AI 回复”变成“最终可展示正文 + 完整 metadata”。

### 应做的事
1. 读取稳定原文候选
2. 一次性解析：
   - rolljson / event envelope
   - triggerjson / trigger_pack
   - rh-trigger markup
3. 生成统一清理结果：
   - `cleanText`
   - `events`
   - `triggers`
   - `triggerPack`
   - `shouldEndRound`
   - `changedFlags`
4. 只做一次：
   - `setMessageText`
   - metadata 写入

### 结果
到阶段 A 结束时：

- message 数据层文本已经是最终干净文本
- trigger metadata 已经齐全
- event 列表已抽取
- 页面层还没动，但数据层已经是最终状态

---

## 8. 阶段 B：当前消息 DOM 立即同步

### 目标
在 autoRoll 开始之前，让用户当前看到的这条消息已经干净。

### 必须新增一个局部同步步骤
建议新增函数，例如：

```ts
hydrateAssistantMessageDomImmediatelyEvent(target, deps)
```

### 它只负责当前这条助手消息
而不是全局整页。

### 需要做的事
1. 根据最新 message text 重绘当前 `.mes_text`
2. 在当前消息节点内立即：
   - 隐藏 / 删除事件代码块
   - 隐藏 / 删除 triggerjson 残留
   - 恢复 trigger 点击区
   - 恢复 raw-source 按钮
3. 确保这一条消息在视觉上已经是最终版

### 关键要求
这个步骤必须发生在：

- `mergeEventsIntoPendingRoundEvent(...)`
- `autoRollEventsByAiModeEvent(...)`

之前。

---

## 9. 阶段 C：事件接管与 autoRoll

### 目标
只有在消息已经视觉上干净之后，才开始系统结算。

### 应做的事
1. 旧楼层失效
2. merge 新事件到 pendingRound
3. 根据策略执行 autoRoll
4. 若需要，启动 3D 投骰动画

### 此时用户看到的页面应当已经满足
- 不再显示 `rolljson`
- 不再显示 `triggerjson`
- 当前 assistant 消息已是最终正文
- trigger 热区已可交互

---

## 10. 阶段 D：全局 UI 收尾

### 目标
处理非关键的全局刷新。

### 可以做的事
- refresh widgets
- refresh countdown
- sweep timeout failures
- 全局 trigger 增强兜底
- 全局 raw-source 按钮增强兜底

### 注意
这一步应该是“兜底收尾”，而不是“真正让当前消息变干净的关键步骤”。

也就是说：

> **当前消息的清理完成，不应依赖这一步。**

---

## 12. 新增统一清理产物函数

建议新增：

```ts
type AssistantMessageSanitizeResultEvent = {
  cleanText: string;
  events: DiceEventSpecEvent[];
  triggerPack: TriggerPackEvent | null;
  triggers: InteractiveTriggerEvent[];
  shouldEndRound: boolean;
  hasRollArtifacts: boolean;
  hasTriggerArtifacts: boolean;
  changedText: boolean;
  changedMetadata: boolean;
};
```

以及：

```ts
sanitizeAssistantMessageArtifactsOnceEvent(
  message: TavernMessageEvent,
  options: ...
): AssistantMessageSanitizeResultEvent
```

### 职责
一次性完成：

- rolljson 清理
- triggerjson 清理
- rh-trigger 提取
- metadata 生成

### 好处
避免当前这种：

- 先一段 removeRanges
- 再一段 trigger sanitize
- 多次 setText
- 中间态被 UI 看见

---

## 13. 不建议继续沿用“边清边写”的两段模式

当前问题之一就是：

- 第一段写回了 cleanText
- 第二段又继续改
- 中间态可能暴露

### 建议改成
#### 旧逻辑
- parse A
- setText
- parse B
- setText
- 再去 UI hydrate

#### 新逻辑
- parse A + parse B
- 产出统一 result
- setText 一次
- 写 metadata 一次
- 立即同步 DOM

---

## 14. 新增当前消息局部 DOM 同步函数

建议新增：

```ts
hydrateAssistantMessageDomImmediatelyEvent(args: {
  target: AssistantMessageTargetEvent;
  deps: ...
}): boolean
```

### 职责
只同步当前 assistant 消息对应的 DOM。

### 建议行为
1. 定位当前消息 `.mes[mesid=...]`
2. 重新写入最新 `.mes_text`
3. 调当前消息范围内的：
   - `hideEventCodeBlocksInMessageEvent(...)`
   - `enhanceInteractiveTriggersInMessageEvent(...)`
   - `enhanceAssistantRawSourceButtonInMessageEvent(...)`

### 注意
这里建议做“消息级增强”，不要再全靠全局扫描。

---

## 15. 新增 finalize 前置清理屏障

建议新增一个显式步骤，例如：

```ts
await finalizeAssistantMessageCleanupBarrierEvent(...)
```

### 作用
确保：

- 数据层清理完成
- 当前消息 DOM 已同步完成

只有 barrier 完成后，才允许继续：

- `mergeEventsIntoPendingRoundEvent(...)`
- `autoRollEventsByAiModeEvent(...)`

### 这是本次重构最关键的“顺序保证”
没有这个屏障，问题还会反复出现。

---

## 16. 对 `finalizeAssistantFloorDataEvent(...)` 的重构建议

当前这个函数职责太重，建议拆成两半。

### 建议拆分成：

#### 1. `prepareAssistantFloorSanitizedDataEvent(...)`
负责：
- 原文快照
- 统一清理产物
- 文本写回
- trigger metadata 写回

#### 2. `attachAssistantFloorEventsAndAutoRollEvent(...)`
负责：
- 旧楼层失效
- merge 新事件
- autoRoll
- metadata 保存

### 中间强制插入：
#### 3. `hydrateAssistantMessageDomImmediatelyEvent(...)`

最终顺序变成：

```ts
prepareAssistantFloorSanitizedDataEvent(...)
hydrateAssistantMessageDomImmediatelyEvent(...)
attachAssistantFloorEventsAndAutoRollEvent(...)
hydrateAssistantFloorUiEvent(...)
```

---

## 17. 对 `hydrateAssistantFloorUiEvent(...)` 的定位调整

当前它承担了太多“当前消息真正变干净”的职责。

### 建议重构后把它降级为：
- 全局兜底刷新
- 补扫遗漏节点
- 刷 widget / countdown / toolbar

而不是继续把它当成：

> 清理消息正文完成的关键步骤

---

## 18. 为什么这能解决“清理不完全”

因为现在的“清理不完全”本质是：

- 数据层已变
- 但当前页面没有立刻按最新数据重绘
- 只能靠后面的全局 sweep/hide/enhance 补
- 某一步没赶上，就残留

### 改成新链路后
当前消息在 autoRoll 前就已经：

- setText 完整写回
- 当前消息 DOM 局部重绘
- trigger 和按钮在当前消息范围内重建

这会大幅降低：

- 半清理状态
- 残留 triggerjson
- 刷新后才干净

这些问题。

## 24. 局部 DOM 重绘相关文件 / 宿主桥接层

### 需要做的事
你现在缺的是一个“消息级立即重绘”能力。

建议补一个统一桥接，例如：


或：


### 需要保证

## 25. 推荐的新主流程伪代码

```ts
async function finalizeAssistantFloorLifecycleEvent(target, deps) {
  // A. 数据层统一清理
  const prepared = prepareAssistantFloorSanitizedDataEvent(target, deps);

  // B. 当前消息 DOM 立即同步
  await hydrateAssistantMessageDomImmediatelyEvent({ target, deps });

  // C. 再做事件接管与 autoRoll
  const attached = await attachAssistantFloorEventsAndAutoRollEvent(prepared, deps);

  // D. 最后做全局 UI 收尾
  hydrateAssistantFloorUiEvent(deps);

  return attached;
}
```

## 26. 推荐的关键约束

### 约束一
**没有经过“当前消息 DOM 已同步”确认，不允许启动 autoRoll。**

### 约束二
**当前消息正文清理必须是单次最终产物，不允许分两段写回。**

### 约束三
**全局 hydrate 只能做兜底，不能再承担当前消息清理完成的关键责任。**

### 第一阶段：先修顺序
1. 把 autoRoll 启动时机后移

1. rolljson / triggerjson 清理合并为单次产物

### 第三阶段：补消息级重绘能力
1. 增加局部消息 DOM 重绘桥接

### 第四阶段：测试与回归

2. 带 rolljson + triggerjson 混合回复

## 35. 最终结论

当前问题不是：

- 单纯哪个清理函数没执行
- 也不是只有 triggerjson 某个正则没命中

真正的问题是：

> **rolljson/triggerjson 清理、当前消息 DOM 同步、autoRoll 启动、投骰动画播放，这几个步骤的先后顺序没有被正确锁定。**

目前系统是：

> **数据层清理先做，autoRoll 也很快开始，但页面层真正变干净要等后面的统一 hydrate。**

这就必然导致：

- 动画先于页面清理完成
- 偶发残留
- 刷新后才彻底干净

彻底解决方向应当是：

> **先统一清理产物，再立即同步当前消息 DOM，确认页面已干净后，才允许 autoRoll 和动画启动。**

---

## 36. 一句话方案定义

> **把 `rolljson/triggerjson` 清理与 `autoRoll` 彻底解耦：先完成单次清理与当前消息 DOM 同步，再启动事件接管和投骰动画，全局 hydrate 只做兜底收尾。**
