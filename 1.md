# MemoryOS 统一分层归约改造方案（旧聊天接管 / 冷启动 / AI总结）

> 基于当前 `memoryos-settings-unified` 分支现状整理，目标是统一解决：
>
> - 最终阶段输入/输出过大导致模型触发上限
> - 单轮结构化输出过长导致 JSON 不稳定
> - 批次数增多后最终整合输入持续膨胀
> - 冷启动、旧聊天接管、AI总结三条链路各自为战，维护成本高
>
> 本方案核心思想：
>
> **统一采用「分批提取 → 暂存落库 → 分层归约 → 小范围冲突裁决 → 正式应用」架构。**

---

## 1. 背景与问题定义

### 1.1 当前旧聊天接管的主要风险

当前 `takeover-consolidator.ts` 已经先用代码对批次结果做了大量本地合并：

- `dedupeFacts`
- `mergeActorCards`
- `mergeRelationships`
- `mergeEntityCards`
- `collapseEntityTransitions`
- `collapseRelations`
- `collapseTasks`
- `collapseWorldState`

但在生成 `fallback` 之后，仍然会把 **全部 `batchResults` + `activeSnapshot` 再整包送给 LLM** 执行 `memory_takeover_consolidation`，要求模型输出完整 `MemoryTakeoverConsolidationResult`。这会带来两个问题：

1. **输入大小随批次数线性增长**，批次越多，最终整合越容易超过输入上限。
2. **输出对象过大**，角色、关系、实体、任务、世界状态、章节索引一起输出，极易触发输出上限或 JSON 结构不完整。

### 1.2 当前冷启动的主要风险

`bootstrap-orchestrator.ts` 目前采用的是 **一次性结构化生成整份 `ColdStartDocument`** 的方式。输入包含：

- characterCard 多字段
- semantic prompt 相关内容
- user persona
- worldbooks entries
- recent events
- 其它来源文本

只要世界书或设定较大，就会出现：

- 一次输入过大
- 一次输出过大
- 结构化结果抖动严重

### 1.3 当前 AI 总结的风险与现状

`summary-orchestrator.ts` 相比前两者已经开始做“瘦身”处理：

- 先做 planner 阶段
- second stage 用 `buildSlimMutationContext()` 缩短上下文
- 把 `summaryText` 抽成 `windowFacts`
- 把候选记录裁成 `slimCandidates`

这条链路方向是对的，但仍存在问题：

- 一次 mutation action 太多时，输出依然可能超长
- `UPDATE` / `MERGE` 语义上是 patch，实际上 payload 仍接近完整对象
- 没有统一的预算与分批框架

### 1.4 当前设置层的局限

`settings/store.ts` 已经有一些窗口与字符上限控制，例如：

- `summarySecondStageRollingDigestMaxChars`
- `summarySecondStageCandidateSummaryMaxChars`
- `takeoverDefaultBatchSize`

但这些控制仍然是**局部、零散的**，缺少统一的：

- 输入预算
- 输出预算
- 单阶段最大动作数
- 分层归约阈值
- 冲突裁决阈值
- 暂存保留策略

---

## 2. 改造目标

### 2.1 总体目标

统一三条链路的处理模型：

```text
原始输入
→ 批处理结构化提取
→ 暂存层落库
→ 分层归约
→ 冲突检测
→ 小范围 LLM 冲突裁决
→ 正式应用到记忆库
```

### 2.2 设计原则

1. **能代码归约的绝不交给 LLM。**
2. **LLM 只做识别与裁决，不做全量最终账本重建。**
3. **任何中间阶段都要可中断、可恢复、可重试。**
4. **任何长链路都必须引入暂存层。**
5. **任何最终步骤都不能直接读取所有原始批结果。**
6. **任何“继续生成下一轮”只能作为兜底，不得作为主流程。**

---

## 3. 统一架构设计

## 3.1 新的统一层次

建议引入四层：

### A. Extraction Layer（提取层）

负责把原始输入按小批次转成结构化增量结果。

输出形式示例：

- actor 增量
- entity 增量
- relation transition 增量
- task transition 增量
- world change 增量
- local digest / batch digest

### B. Staging Layer（暂存层）

负责存放所有批次结果与中间归约结果。

要求：

- 可按 jobId / pipelineId 查询
- 可按 phase / batch / section 查询
- 可恢复中断状态
- 可重新归约
- 可标记冲突状态

### C. Reduce Layer（归约层）

负责 deterministic merge：

- 按 compareKey / actorKey / targetKey 合并
- 生成 sectionDigest
- 生成 domain ledger
- 标记 unresolved conflicts

### D. Resolve & Apply Layer（裁决与应用层）

- LLM 仅处理 unresolved conflict buckets
- 应用层再写正式记忆库
- 记录 mutation history / snapshot / trace

---

## 3.2 统一的处理单位

无论是旧聊天接管、冷启动还是总结，都统一抽象成：

### `PipelineJob`

字段建议：

- `jobId`
- `jobType`: `takeover | cold_start | summary`
- `status`: `pending | running | paused | failed | completed`
- `phase`: `extract | reduce | resolve | apply`
- `sourceMeta`
- `createdAt`
- `updatedAt`
- `errorCode`
- `errorMessage`
- `resumeToken`

### `PipelineBatch`

字段建议：

- `jobId`
- `batchId`
- `phase`
- `range`
- `inputDigest`
- `outputDigest`
- `status`
- `retryCount`
- `startedAt`
- `finishedAt`

### `PipelineLedger`

按领域维护增量账本：

- actor ledger
- entity ledger
- relation ledger
- task ledger
- world ledger
- digest ledger

## 4. 暂存层设计

## 4.1 为什么必须有暂存层

如果每批结果直接写正式记忆表，会产生以下问题：

1. 中间失败后正式数据已被污染。
2. 后续批次可能推翻前面批次，回滚困难。
3. merge / invalidate / supersede 逻辑会变得混乱。
4. 冷启动与旧聊天接管都存在“大量一次性导入”的天然需求，不适合直接写主库。

因此，建议：

> **每批写入 staging，而不是直接写正式记忆库。**

---

## 4.2 暂存层建议的数据结构

建议新增以下存储对象（名称可按项目风格调整）：

### 4.2.1 `pipeline_job`

记录整条流水线任务。

### 4.2.2 `pipeline_batch_result`

记录每个批次的结构化输出原文与摘要。

字段建议：

- `jobId`
- `batchId`
- `domain`
- `sourceRange`
- `summary`
- `rawStructuredResult`
- `normalizedStructuredResult`
- `tokenEstimateIn`
- `tokenEstimateOut`
- `status`

### 4.2.3 `pipeline_domain_ledger`

记录领域级归约账本。

字段建议：

- `jobId`
- `domain`: `actor | entity | relationship | task | world | digest`
- `ledgerKey`
- `canonicalRecord`
- `sourceBatchIds`
- `conflictState`: `none | suspected | unresolved | resolved`
- `updatedAt`

### 4.2.4 `pipeline_section_digest`

处理中层 digest，例如：

- 旧聊天接管每 5~10 个 batch 归约成 1 个 section digest
- 冷启动可按来源分 section
- summary 可按窗口聚合成 mini digest

字段建议：

- `jobId`
- `sectionId`
- `batchIds`
- `summary`
- `actors`
- `entities`
- `relationships`
- `tasks`
- `worldChanges`
- `unresolvedConflicts`

### 4.2.5 `pipeline_conflict_bucket`

记录待 LLM 裁决的问题集合。

字段建议：

- `jobId`
- `bucketId`
- `domain`
- `conflictType`
- `records`
- `resolutionStatus`
- `resolutionResult`

---

## 4.3 暂存层要求

### 必须满足

- 幂等写入
- 可按 jobId 重放 reduce
- 可按 batchId 重试单批
- 可按 domain 局部重算
- 可查询 unresolved conflict

### 推荐满足

- 记录 token 估算
- 记录模型名 / schema 版本
- 记录 prompt section 版本
- 记录原始输入摘要与 hash

## 5. 统一预算控制设计

## 5.1 新增统一预算模型

建议引入 `PipelineBudgetPolicy`：

```ts
export interface PipelineBudgetPolicy {
  maxInputCharsPerBatch: number;
  maxOutputItemsPerBatch: number;
  maxActionsPerMutation: number;
  maxSectionBatchCount: number;
  maxConflictBucketSize: number;
  maxSectionDigestChars: number;
  maxRollingDigestChars: number;
  maxCandidateSummaryChars: number;
  maxFinalizerItemsPerDomain: number;
}
```

### 目的

统一替代“每个模块自己截一点”的零散做法。

---

## 5.2 推荐默认值

初始建议：

- `maxInputCharsPerBatch`: 12000 ~ 20000
- `maxOutputItemsPerBatch`: 20
- `maxActionsPerMutation`: 8 ~ 12
- `maxSectionBatchCount`: 5
- `maxConflictBucketSize`: 10
- `maxSectionDigestChars`: 2000
- `maxRollingDigestChars`: 1200
- `maxCandidateSummaryChars`: 300
- `maxFinalizerItemsPerDomain`: 50

这些值应可在设置层覆盖。

---

## 5.3 设置层修改建议

在 `MemoryOS/src/settings/store.ts` 中新增统一预算设置项：

### 新增字段建议

- `pipelineBudgetEnabled`
- `pipelineMaxInputCharsPerBatch`
- `pipelineMaxOutputItemsPerBatch`
- `pipelineMaxActionsPerMutation`
- `pipelineMaxSectionBatchCount`
- `pipelineMaxConflictBucketSize`
- `pipelineMaxSectionDigestChars`
- `pipelineMaxFinalizerItemsPerDomain`
- `pipelineStagingRetentionDays`
- `pipelineResolveOnlyUnresolvedConflicts`

### 原有字段处理建议

保留以下旧字段做兼容：

- `summarySecondStageRollingDigestMaxChars`
- `summarySecondStageCandidateSummaryMaxChars`
- `takeoverDefaultBatchSize`

但逐步迁移为：

- UI 上显示为“高级兼容参数”
- 内部优先取新预算设置
- 未设置时回退旧字段

## 6. 旧聊天接管改造方案

## 6.1 当前问题总结

当前旧聊天接管最大的风险点不在批处理，而在**最终 consolidation 阶段仍是全量 LLM 整合**。

### 现状

- 前面已经做 batch 提取
- 本地已经有 fallback merge
- 但最后依然全量送 LLM 输出完整 consolidation result

### 本质问题

> 让模型承担了“全量最终账本输出”职责。

---

## 6.2 目标状态

改造后流程应为：

```text
旧聊天原始楼层
→ 按 batch 提取结构化结果
→ 每批写 staging
→ 按 section 做中层 digest
→ 本地 reduce 形成领域 ledger
→ 仅把 unresolved conflict bucket 送 LLM
→ 得到 conflict resolutions
→ 本地 finalize 最终 consolidation
→ apply 到正式记忆库
```

---

## 6.3 详细改造步骤

### 6.3.1 批处理结果立即落暂存层

每个 batch 完成后立即落：

- `pipeline_batch_result`
- 对应的 actor / entity / relation / task / world ledger 增量
- batch digest

### 6.3.2 引入 section digest

例如：

- 每 5 个 batch 形成 1 个 section
- 每个 section 做 deterministic reduce
- 输出：
  - `sectionActors`
  - `sectionEntities`
  - `sectionRelationships`
  - `sectionTasks`
  - `sectionWorldChanges`
  - `sectionConflictHints`

### 6.3.3 最终阶段不再把全部 batchResults 送 LLM

替换为：

- 只读取 section digests
- 只读取 domain ledgers
- 只提取 unresolved conflicts

### 6.3.4 LLM 只做 conflict resolution

定义新的任务：

- `memory_takeover_actor_conflict_resolve`
- `memory_takeover_entity_conflict_resolve`
- `memory_takeover_relation_conflict_resolve`
- `memory_takeover_world_conflict_resolve`

输入只包含：

- 冲突记录
- 周边必要上下文
- 候选 canonical 版本
- 来源 batch 摘要

输出仅包含：

- 保留哪条
- 是否 merge
- 哪些字段覆盖
- 是否 invalidated
- 原因码

### 6.3.5 最终 consolidation 改成本地 finalize

`takeover-consolidator.ts` 最终只负责：

- 读取 domain ledger
- 应用 resolution patch
- 输出稳定 final consolidation result

不再要求模型输出完整 final object。

---

## 6.4 需要新增/调整的文件建议

### 需要修改

- `MemoryOS/src/memory-takeover/takeover-consolidator.ts`
- `MemoryOS/src/memory-takeover/takeover-llm.ts`
- 旧聊天接管的批调度/编排文件（若当前另有 orchestrator，应一并调整）

### 建议新增

- `MemoryOS/src/pipeline/pipeline-job-store.ts`
- `MemoryOS/src/pipeline/pipeline-ledger-store.ts`
- `MemoryOS/src/pipeline/pipeline-conflict-store.ts`
- `MemoryOS/src/memory-takeover/takeover-section-reducer.ts`
- `MemoryOS/src/memory-takeover/takeover-conflict-resolver.ts`
- `MemoryOS/src/memory-takeover/takeover-finalizer.ts`

---

## 6.5 `takeover-consolidator.ts` 具体修改建议

### 当前职责过重

当前文件同时承担了：

- 本地 fallback merge
- LLM 全量 consolidation 请求
- structured result fallback 合并

### 调整后职责

改成只做：

1. 读取 section digest 与 domain ledger
2. 读取 resolution patch
3. 应用 patch
4. 输出 final result
5. 生成统计信息

### 删除/弱化内容

- 删除“把 `batchResults` 全量发给 `memory_takeover_consolidation`”的逻辑
- 保留 deterministic merge 工具函数
- 把 LLM 调用迁出到专门的 `conflict-resolver`

---

## 6.6 takeover 的兜底机制

如果 conflict resolver 失败：

- 允许保守使用 deterministic ledger 输出
- 对 unresolved 项打 `needs_review` 或 `low_confidence`
- 整体任务可完成，但附带 diagnostics

这样就不会因最后一轮 LLM 超限而整条链失败。

## 7. 冷启动改造方案

## 7.1 当前问题总结

冷启动当前是“一次吃完 sourceBundle，一次吐完整 ColdStartDocument”。

风险点：

- 设定长时输入超限
- 输出结构大时容易截断
- 单轮同时做 identity / actor / entity / relationship / world / memory record，职责过重

---

## 7.2 目标状态

```text
sourceBundle
→ source segmentation
→ phase 1: 核心角色/实体/世界规则提取
→ 暂存
→ phase 2: 关系/任务/近期事件提取
→ 暂存
→ phase 3: reduce + conflict resolve
→ candidate build
→ 用户确认/自动确认
→ apply
```

---

## 7.3 冷启动分阶段建议

### Phase 1：核心骨架提取

只提：

- identity
- actorCards
- entityCards
- worldBase core setting
- world hard rules
- world profile

### Phase 2：关系与近期状态提取

只提：

- relationships
- memoryRecords
- 当前世界状态
- 近期事件影响

### Phase 3：reduce + resolve

- 去重 actor cards
- 合并 entity cards
- 识别冲突 relationship
- 生成 candidate list

---

## 7.4 冷启动为什么也要 staging

冷启动不是一次性 UI 功能而已，它本质上是“批量知识导入”。

如果没有 staging：

- 用户取消时无法恢复
- worldbook 很大时无法断点续跑
- 不利于后续增量冷启动

建议冷启动和旧聊天接管共用一套 `PipelineJob` / staging 基础设施。

---

## 7.5 需要修改的文件建议

### 需要修改

- `MemoryOS/src/memory-bootstrap/bootstrap-orchestrator.ts`

### 建议新增

- `MemoryOS/src/memory-bootstrap/bootstrap-source-segmenter.ts`
- `MemoryOS/src/memory-bootstrap/bootstrap-phase-runner.ts`
- `MemoryOS/src/memory-bootstrap/bootstrap-reducer.ts`
- `MemoryOS/src/memory-bootstrap/bootstrap-conflict-resolver.ts`
- `MemoryOS/src/memory-bootstrap/bootstrap-finalizer.ts`

---

## 7.6 `bootstrap-orchestrator.ts` 具体修改建议

### 当前职责

- 构建整份 cold start payload
- 一次 runTask 获取完整 `ColdStartDocument`
- 再构建 candidates
- 再 apply

### 调整后职责

改成总控编排器：

1. source segmentation
2. phase runner 执行多个结构化子任务
3. 暂存结果
4. reduce
5. resolve
6. build candidates
7. apply selected candidates

也就是说：

> 让 orchestrator 只负责调度，不直接承载最大体积的结构化输出。

## 8. AI 总结改造方案

## 8.1 当前优点

AI 总结目前已经有较好的两阶段结构：

- planner
- mutation

并且 second stage 已有瘦身逻辑。

这部分不需要推翻重来，而是要纳入统一架构。

---

## 8.2 当前仍需解决的问题

### 问题 1：单轮 action 太多

当一个窗口里变更多时，`actions` 数组会很长。

### 问题 2：payload 仍然偏重

虽然语义上是 patch，但 schema 仍要求很多字段，导致输出体积偏大。

### 问题 3：没有批次拆分与预算兜底

如果 planner 决定范围很大，mutation 阶段仍有超限风险。

---

## 8.3 目标状态

```text
summary window
→ planner
→ mutation chunk planner
→ 多轮小 mutation batch
→ staging snapshot
→ deterministic apply reduce
→ applySummarySnapshot
```

---

## 8.4 改造重点

### 8.4.1 引入 `mutation batch planner`

planner 完成后，不要直接进入一次大 mutation。

而是先根据：

- focus types
- candidate count
- action estimate
- payload weight

拆分成多个 mutation batches。

示例：

- batch 1: actor / relationship
- batch 2: task / world
- batch 3: entity

### 8.4.2 限制每轮最大 action 数

建议设置：

- 一轮最多 8~12 个 action

超过则自动切 batch。

### 8.4.3 UPDATE / MERGE 真正 patch 化

现在 `buildStrictPatchPayloadSchema()` 与 create payload 几乎一致，导致 patch 体积并没有真正下降。

建议改造为：

- patch 允许字段稀疏输出
- 内部再由 applier 与 existing record merge
- schema 仍保持封闭，但 `payload` 只包含本轮需要改动的字段

### 8.4.4 mutation 结果先落 staging snapshot

不要立即一次性 apply 全部结果。

建议先写：

- `summary_mutation_batch_result`
- `summary_mutation_staging_snapshot`

等全部 batch 完成后，再统一 `applySummarySnapshot`。

---

## 8.5 需要修改的文件建议

### 需要修改

- `MemoryOS/src/memory-summary/summary-orchestrator.ts`
- `MemoryOS/src/memory-summary/mutation-applier.ts`

### 建议新增

- `MemoryOS/src/memory-summary/mutation-batch-planner.ts`
- `MemoryOS/src/memory-summary/mutation-staging-store.ts`
- `MemoryOS/src/memory-summary/mutation-batch-runner.ts`
- `MemoryOS/src/memory-summary/mutation-finalizer.ts`

---

## 8.6 `mutation-applier.ts` 修改建议

### 当前职责

- 直接遍历完整 action 数组
- 立刻生成 entryUpserts / refreshBindings
- 一次性 applySummarySnapshot

### 调整后职责

分成两部分：

#### A. `mutation-patch-applier`

只负责：

- 把 action payload 与 existing record merge
- 生成标准 upsert patch

#### B. `mutation-finalizer`

只负责：

- 汇总多个 mutation batch 结果
- 去重 refreshBindings
- 统一 applySummarySnapshot

这样更适合多 batch 执行。

## 9. LLM 调用策略统一改造

## 9.1 统一原则

### LLM 应该做的

- 小批结构化提取
- 冲突识别
- 冲突裁决
- 分类与排序

### LLM 不应该做的

- 整条链最终全量账本重建
- 一次性输出几十上百条对象
- 重复输出已经在本地可确定的 canonical state

---

## 9.2 统一的任务类型设计

建议把大任务拆成三类：

### 1. Extract Task

输入小片段，输出小型结构化增量。

### 2. Resolve Task

输入 conflict bucket，输出 resolution patch。

### 3. Optional Digest Task

当某段文本过大时，用来生成短 digest，而不是直接要求最终对象。

---

## 9.3 不推荐的做法

### 不推荐 1：输出超限后继续下一轮

原因：

- JSON 续写脆弱
- 重复输出难去重
- 输入会越来越大
- 幂等性差

### 不推荐 2：批次直接写正式主库

原因：

- 会污染正式状态
- 回滚困难
- merge / invalidate 复杂度大增

---

## 10. 统一冲突处理机制

## 10.1 冲突类型建议

### Actor 冲突

- 同 actorKey 不同 displayName
- 同 displayName 疑似不同人
- traits / identityFacts 相互冲突

### Entity 冲突

- 同 compareKey 多版本
- 同 title 不同 compareKey
- fields 冲突

### Relationship 冲突

- 同 pair 不同 relationTag
- 数值状态冲突
- summary 冲突

### Task 冲突

- 同 task 不同 stage
- 已完成与进行中冲突

### World 冲突

- 同 key 多个 state
- world state 与 entity state 重叠

---

## 10.2 冲突桶组织方式

建议一个 bucket 只处理：

- 同一 domain
- 同一 conflict type
- 不超过固定条数

例如：

- `entity/title_collision/001`
- `relationship/state_divergence/003`

这样能显著缩小 resolve 阶段输入与输出。

---

## 10.3 冲突裁决输出格式建议

输出不要是完整对象，而应是 resolution patch：

```json
{
  "bucketId": "entity/title_collision/001",
  "resolutions": [
    {
      "action": "merge",
      "primaryKey": "city:red_harbor",
      "secondaryKeys": ["city:red-harbour"],
      "fieldOverrides": {
        "title": "赤港",
        "fields.aliases": ["红港"]
      },
      "reasonCodes": ["same_place_different_spelling"]
    }
  ]
}
```

## 11. 正式应用层改造

## 11.1 统一 apply 原则

所有链路最终都应走：

- staging → normalized canonical patch → apply

而不是：

- LLM 原始结果 → 直接正式落库

---

## 11.2 apply 阶段应具备的能力

- 幂等 upsert
- invalidate
- merge into primary
- refresh binding 去重
- summary snapshot 统一写入
- mutation history 记录
- diagnostics 输出

## 13. 幂等、恢复、重试设计

## 13.1 幂等要求

### 每个 batch 必须可重跑

使用：

- `jobId + batchId + inputDigest`

作为幂等键。

### 每个 reduce 必须可重算

因为 reduce 来源于 staging，不应依赖某次内存态。

### 每个 apply 必须避免重复写入

通过：

- existing entryId
- compareKey
- sourceLabel + summaryId
- lifecycle markers

防止重复 apply。

---

## 13.2 恢复逻辑

任务中断时：

- 若停在 extract：从未完成 batch 继续
- 若停在 reduce：重新从 staging 计算
- 若停在 resolve：从 unresolved bucket 继续
- 若停在 apply：根据 applied marker 补齐剩余 patch

---

## 13.3 重试策略

建议：

- extract 失败可自动重试 1~2 次
- resolve 失败自动降级为 deterministic fallback
- apply 失败不自动覆盖，要求基于 staging 重放

---

## 14. 兼容性与迁移策略

## 14.1 分阶段落地，不要一次重写

建议分三期：

### 第一期：抽基础设施

先新增：

- pipeline job / staging / ledger / conflict store
- unified budget policy
- diagnostics 输出结构

暂时不大改业务逻辑。

### 第二期：旧聊天接管先落地

因为它最容易撞上输入/输出上限，优先收益最大。

### 第三期：冷启动与 AI 总结接入同框架

- 冷启动改分阶段
- summary 接 mutation batch planner

---

## 14.2 兼容旧配置

- 保留旧设置字段
- 新增设置优先
- 旧链路入口不变
- UI 操作尽量不变
- diagnostics 中标识当前是否使用 unified pipeline

## 17. 验收标准

## 17.1 功能标准

### 旧聊天接管

- 批次数增加时，最终阶段输入不再随 batchResults 线性膨胀
- 最终阶段不再要求 LLM 输出完整 consolidation object
- 单个 unresolved bucket 失败时仍可产出 fallback result

### 冷启动

- 大 worldbook 不再要求一轮吃完
- 冷启动中断后可恢复
- phase 结果可单独重试

### AI 总结

- 单轮 mutation 可切 batch
- action 数超预算时自动拆分
- patch payload 明显减小

---

## 17.2 性能标准

建议以以下目标为验收参考：

- 旧聊天接管最终阶段输入体积降低 60% 以上
- AI 总结第二阶段平均输出体积降低 30% 以上
- 冷启动长设定场景失败率显著下降
- 结构化 JSON 截断/不完整问题显著减少

## 第 1 步：抽通用 pipeline 基础设施

先做：

## 第 2 步：改旧聊天接管

## 19. 最终结论

当前最核心的问题不是“模型偶尔输出太短”，而是：

> **旧聊天接管、冷启动、AI总结里，仍存在让模型承担过重最终输出职责的设计。**

统一改造后的正确方向是：

### 一句话版本

**批次先落暂存层，本地先归约，模型只做小范围冲突裁决，最后再正式应用。**

### 这样做的直接收益

- 不再依赖“输出超限后继续生成”
- 最终阶段输入体积不再随历史批次数无限膨胀
- JSON 结构化输出更稳定
- 任务中断后可恢复
- 三条链路架构统一，后续维护成本更低
