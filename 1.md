## 4. 这次 Commit 之后仍然存在的问题

虽然方向很好，但从这次 commit 的实现看，还有几类问题没有完全收住。

## 4.1 严格 Schema 已经上了，但 Payload 字段仍然偏“宽松”
`buildLooseFieldSchema()` 现在的规则是：

- `relationTag` 用 enum
- 数值字段用 `number`
- 数组字段用 `string[]`
- 其他大部分字段默认还是 `string` fileciteturn145file0

### 这会导致什么
第二阶段虽然已经比以前严格很多，但还没到“真正字段级约束很强”的程度。

比如：

- `summary` 和 `title` 都是 string，合理
- 但某些字段如果本质上更适合 enum / object / typed patch，现在还是被放宽成 string
- `payload` 的 required 目前是“所有 properties 全 required”，虽然适合 strict schema，但对 patch 型 UPDATE 还不够自然

### 也就是说
当前版本更像：

> **从“过松”走到了“中等严格”**

但还没有走到“适合最终长期维护”的强约束版本。

---

## 4.2 `NOOP` / `DELETE` / `INVALIDATE` 还没完全拉开语义
从 `buildStrictActionBranch()` 看，目前所有 action 分支都统一要求：

- `candidateId`
- `compareKey`
- `payload`
- `reasonCodes`

即使是：

- `NOOP`
- `DELETE`  
也还是同一套 required 逻辑，只是给了空对象 schema fileciteturn145file0

### 这在 schema 层会带来一个问题
动作协议现在虽然更规范了，但不同动作之间的“字段语义差异”还不够清晰。

例如：

- `NOOP` 其实不一定需要 `candidateId + compareKey + payload`
- `DELETE` 和 `INVALIDATE` 的证据需求也应该不同
- `ADD` 更应该强调 `new_record`
- `UPDATE` 更应该强调 `patch`

现在还是偏“统一动作壳 + payload 装内容”。

---

## 4.3 `typeSchemas` 仍然决定了很多字段，但还没解决“提示词太大”的根问题
这次 commit 已经让第二阶段 schema 更严格了，但它并没有直接解决：

- `mutationContext` 本身太重
- `candidateRecords` 可能仍然很长
- `recentSummaryDigest` 和 `window` 可能仍然存在重复
- Planner 阶段和 Mutation 阶段职责仍可能混得不够开

也就是说：

> **这次 commit 主要在提升“结构化约束”，不是在做“上下文瘦身”。**

这很好，但下一步就该补瘦身了。

---

## 4.4 诊断中心现在更丰富了，但还差“阶段摘要”和“体积统计”
你这次已经把 trace、summary、mutation history、entry updates 都接进来了，这很好。  
但如果后面你要优化提示词体积，当前面板里还缺两个非常值钱的信息：

### 1. 各阶段输入体积
例如：

- Planner 输入字符数
- Mutation 输入字符数
- schema 字符数
- candidateCards 字符数
- rollingDigest 字符数

### 2. 阶段摘要
例如：

- 本轮总结是否真的走了 Mutation
- Planner 返回 `should_update = false` 还是 `true`
- 最终 action 分布
- 被 validator 拒绝了多少条

这些信息现在还没有被清楚聚合出来。

## 5. 基于 `0773eb6` 的后续优化方向

下面这部分就是这次 commit 之后最适合继续推进的一套优化方案。

## 5.1 方向一：把“严格 Schema”继续升级成“动作语义型 Schema”

### 当前状态
现在已经能根据 `typeSchemas` 动态构建严格 schema，这是非常好的第一步。

### 下一步建议
不要只按 `targetKind` 分支，还要按 `action` 分支做更强约束。

例如：

#### `ADD`
建议要求：

- 不需要 `candidateId`
- 不需要 `compareKey`
- 必须有 `payload`
- `payload` 必须包含核心创建字段

#### `UPDATE`
建议要求：

- 必须有 `candidateId` 或 `compareKey`
- `payload` 更适合 patch 语义
- 不需要所有字段都出现

#### `MERGE`
建议要求：

- `candidateId` 允许数组或 source list
- `payload` 更偏合并后摘要

#### `INVALIDATE`
建议要求：

- 必须有目标定位字段
- `payload` 至少要有原因或替代关系

#### `NOOP`
建议要求：

- 不需要 payload
- 可以只保留 `reasonCodes`

### 目标
让模型在 schema 层就更难“用错动作”。

---

## 5.2 方向二：给总结第二阶段做“字段级 patch 模式”
现在 `buildStrictMutationPayloadSchema()` 会把可写字段直接变成一个严格对象，且 required = 所有字段名。fileciteturn145file0

### 问题
这对 `ADD` 比较友好，对 `UPDATE` 不够友好。  
因为 UPDATE 本质上应该允许：

> 只更新少数字段，而不是要求 AI 每次都补齐所有字段。

### 建议做法
把 payload 拆两类：

#### `ADD`
使用完整对象 schema

#### `UPDATE / MERGE`
使用 patch schema，例如：

```ts
{
  type: 'object',
  additionalProperties: false,
  properties: {
    patch: {
      type: 'object',
      additionalProperties: false,
      properties: ...
    }
  },
  required: ['patch']
}
```

### 目标
避免模型在 UPDATE 时为了满足 schema 被迫“补全没变的字段”。

---

## 5.3 方向三：继续瘦身 Mutation 阶段上下文
这次 commit 解决的是“结构化约束”，但上下文体积问题还没动。

### 下一步建议
Mutation 阶段继续做这 4 件事：

#### 1. `window.summaryText` 改成 `windowFacts`
不要长剧情文本，改成事实帧。

#### 2. `recentSummaryDigest` 改成单条 `rollingDigest`
不要堆多条历史长摘要。

#### 3. `candidateRecords` 改成短卡片
Mutation 阶段仍然可以比 Planner 详细，但也不应该给完整大对象。

#### 4. `typeSchemas` 只发本轮 focus types
如果本轮只动 `relationship` 和 `event`，就别再把 `world_rule`、`organization`、`location` 全发进去。

## 7. 针对“提示词有点多”的专项优化建议

虽然这个 commit 本身不是专门瘦身的，但下一步最该接的是这块。

## 7.1 Planner / Mutation 输入职责彻底分离
现在最应该坚持的原则是：

### Planner 只看
- 事实帧
- 单条滚动摘要
- 候选短卡片
- 允许处理的类型

### Mutation 才看
- 精选旧记录
- 严格 schema
- 动作协议
- 必要 patch 范围

不要让 Mutation 的 schema 逻辑跑到 Planner 去。

---

## 7.2 让 `strictSummarySchema` 真正变成“精简后的必要 schema”
你现在已经把它做成动态 schema，这是非常好的基础。  
下一步要做的是：

- 不是所有可写字段都发
- 而是 Planner 判定 focus types 后，再进一步裁掉本轮不需要的字段

也就是说：

> 先按 type 裁  
> 再按 action 裁  
> 最后按本轮 focus 裁

这样 schema 会继续明显变小。

---

## 7.3 上下文去重
你现在下一轮最该砍掉的内容还是：

- 多条历史摘要重复
- 长剧情式 `summaryText`
- 候选记录过长
- Planner / Mutation 规则重叠

这部分和 `0773eb6` 的方向并不冲突，反而更适合在这次“结构化约束已经更强”的基础上继续推进。
