# SillyTavern-SS-Helper / MemoryOS 一次性整合版实施方案（语境系统 + 混合召回 + 记忆可视化 TAB）

> 目标分支：`memoryos-settings-unified`  
> 目标：在**一个版本内**完成以下三类能力的整合落地：
>
> 1. **语境系统**：让检索不再只是“文本匹配”，而是先识别当前输入的语境、facet、实体锚点与主题，再驱动召回。  
> 2. **混合召回系统**：把当前已有的 lexical / embedding provider 升级成完整的多阶段检索管线：**种子召回 → 结构扩散 → 缺项补召回 → 多样性裁剪 → 角色视角渲染**。  
> 3. **记忆工作台新增可视化记忆 TAB**：在记忆工作台中新增一个类似“记忆图谱/记忆网络”的可视化视图，可以点击单个记忆节点查看内容，并与条目中心联动。

---

## 1. 版本目标与总体原则

这一版不是“小修小补”，而是一次完整的中层架构升级。  
升级完成后，MemoryOS 的行为将从“有结构化存储，但检索与展示仍偏线性”提升为：

- 检索阶段会先理解**当前在聊什么**
- 再根据角色视角与世界模板决定**该找哪类记忆**
- 通过种子召回和结构扩散联想到**真正相关的上下文**
- 最后把结果以**角色视角 + 遗忘程度 + 预算约束**的形式输出给提示词系统
- 同时在工作台里新增一个**可视化记忆网络**，帮助查看当前聊天数据库中的记忆节点、连接关系与记忆度

这一版有两个核心原则：

### 1.1 不推翻当前架构，只在现有骨架上升级
当前分支已经具备几个很重要的基础：

- `memory-retrieval/retrieval-orchestrator.ts` 已经有 lexical / embedding provider 切换
- `memory-retrieval/lexical-provider.ts` 已有 BM25 + n-gram + 编辑距离 + memoryPercent 混合打分
- `memory-retention/forgetting-engine.ts` 已有 `memoryPercent -> clear / blur / distorted`
- `memory-injection/actor-visible-context-builder.ts` 已有角色视角渲染
- `ui/unifiedMemoryWorkbench.ts` 已有完整工作台框架
- `ui/workbenchTabs/actorTabs/relationshipGraph.ts` 已有现成的图渲染能力（节点、边、缩放、拖拽、聚焦、点击）

所以这一版不需要推翻重做，而是**在现有模块上扩容与编排**。

### 1.2 不照搬 PeroCore 的运行形态，只借它真正有价值的检索思想
PeroCore 真正值得借鉴的，不是它的 Python/Rust 技术栈本身，而是它的这条检索思路：

> **种子召回 → 图扩散/联想 → 弱信号补召回 → 去冗余/多样性 → 最终注入**

这一版直接把这条思路落进 MemoryOS 的 TS 结构化体系中。

## 2. 本版最终要交付的功能效果

本版完成后，应当能达到下面这些效果。

### 2.1 语境系统效果
当用户输入一句自然语言时，系统不会只靠简单关键词找相似记忆，而是先输出一个“检索语境路由”对象，例如：

- 当前更偏 `relationship`
- 同时有 `event`
- 还带有 `scene`
- 命中角色：艾琳 / 男主
- 命中地点：旧城区桥边

1. 用 lexical / embedding 找第一批种子
2. 按结构关系向周边扩散
3. 检查 query 想要的 facet 有没有缺失，必要时补召回
4. 去掉重复、挤压、同质化结果
5. 按预算输出给 Prompt 注入层

- 节点大小 / 亮度表示记忆权重和记忆度
- 边表示结构关系
- 支持拖拽、缩放、聚焦
- 点击某个记忆节点后，右侧面板显示该记忆的完整内容
- 右侧还可以直接跳转到“条目中心”进行编辑

## 3. 本版最关键的算法设计

这一部分是整版升级的核心，必须讲清楚。

---

## 3.1 语境系统（Context System）

### 3.1.1 为什么要做语境系统

当前 MemoryOS 如果只做字符串相似度，最多只能回答：

- “这句话和哪几条记忆像”

但在真实对话里，系统真正要知道的是：

- 这句话现在是在谈**世界规则**、**关系冲突**、**某次事件**，还是**某个场景**
- 这句话命中了哪些角色、地点、关系、世界状态
- 这一轮检索应该优先找哪类记忆

所以需要新增一层：**检索语境路由层**。

它不是“状态机标签”，也不是“聊天情绪分类器”，而是专门服务于**记忆检索**的。

### 3.1.2 语境系统输出什么

新增一个统一结果对象：

```ts
export interface RetrievalContextRoute {
  facets: Array<'world' | 'scene' | 'relationship' | 'event' | 'interpretation'>;
  entityAnchors: {
    actorKeys: string[];
    locationKeys: string[];
    relationKeys: string[];
    worldKeys: string[];
  };
  topicHints: string[];
  confidence: number;
}
```

### 3.1.3 语境系统怎么识别

这版不依赖 AI，不依赖昂贵 embedding，也不依赖重模型。  
第一版采用**规则 + 实体识别 + 主题词典 + 句式模式**的混合方案。

#### 第一步：关键词与句式匹配
由 `memory-keywords/` 提供不同语境的词典和模式：

- `conflict.ts`
- `repair.ts`
- `scene-transition.ts`
- `setting-qa.ts`
- `plot-progress.ts`

#### 第二步：实体锚点匹配
从当前聊天数据库动态整理实体索引：

- 角色名 / actorKey / displayName / aliases
- 地点名
- 组织名
- 世界状态名
- 稳定 relation pair key

#### 第三步：语境打分
对 5 类 facet 分别打分：

- `world`
- `scene`
- `relationship`
- `event`
- `interpretation`

例如用户输入：

> “你为什么在桥上那晚还是不信我？”

会命中：

- relationship：高
- event：高
- scene：中
- world：低
- interpretation：中

最终输出：

```jsonc
{
  "facets": ["relationship", "event", "scene"],
  "entityAnchors": {
    "actorKeys": ["char_erin", "char_mc"],
    "locationKeys": ["bridge_old_city"],
    "relationKeys": ["relationship:char_erin:char_mc"],
    "worldKeys": []
  },
  "topicHints": ["trust_conflict", "bridge_argument", "doubt"],
  "confidence": 0.84
}
```

### 3.1.4 为什么这层比“纯状态分类”更值钱

因为它直接决定后续：

- provider 的 query 怎么构造
- 哪些候选类型要优先
- 扩散时哪些边更强
- 哪些 facet 缺了要补
- 图可视化时哪些节点优先高亮

所以语境系统是这一版混合检索的真正入口。

## 3.2 种子召回（Seed Retrieval）

### 3.2.1 作用
从全量候选记忆中，先召回第一批与 query 最直接相关的“种子记忆”。

### 3.2.2 第一版主方法
第一版默认不开 embedding 时，主方法使用：

- **BM25** 作为主召回分
- **n-gram** 作为近似表达增强
- **编辑距离** 作为短文本修正
- **memoryPercent** 作为记忆权重
- **recencyWeight** 作为时间增强

### 3.2.3 评分公式

$$
seedScore =
0.55 \cdot bm25 +
0.15 \cdot ngram +
0.08 \cdot edit +
0.12 \cdot memoryWeight +
0.10 \cdot recencyWeight
$$

其中：

- `memoryWeight = memoryPercent / 100`
- `recencyWeight` 根据 `updatedAt` 的时间衰减计算
- 若启用 embedding，则再额外加入 `embeddingScore`

### 3.2.4 这版为什么不做纯 embedding 起步
因为当前架构已经明确：

- embedding 是可开关项
- 默认必须允许 lexical fallback
- 插件不应强依赖本地模型服务

所以种子召回第一版仍以 lexical 为主，这是最稳妥的做法。

---

## 3.3 结构扩散（Graph Expansion）

### 3.3.1 作用
第一轮种子只能拿到“文本上最像的记忆”，但很多真正该给模型看的内容并不会直接命中 query。  
所以需要从种子节点沿着记忆之间的结构关系扩散一层，把相关的上下文带出来。

### 3.3.2 结构图怎么构建
不引入重型图数据库，也不在数据库里额外维护一大堆图表。  
本版采用**运行时轻量邻接图**。

#### 节点
每个 `MemoryEntry` 都是一个节点。

#### 边
如果两条记忆满足任意一个条件，就建立边：

- 相同 `actorKeys`
- 相同 `relationKeys`
- 相同 `participantActorKeys`
- 相同 `locationKey`
- 相同 `sourceSummaryIds`
- 相同 `tags`
- 相同 `worldKeys`

这些边不需要数据库存储，可在构建候选池时实时生成。

### 3.3.3 扩散公式

设种子节点分数为 `seedScore`，则它给相邻节点传播的增益为：

$$
graphBoost = seedScore \cdot edgeWeight \cdot decay
$$

第一版：

- 只扩散 1 层
- `decay = 0.65`

边权重建议：

- same relation：`0.22`
- same participants：`0.16`
- same location：`0.12`
- same sourceSummary：`0.10`
- same tags：`0.06`
- same category：`0.04`

最终得分：

$$
finalScore = seedScore + graphBoost
$$

### 3.3.4 这样做的价值
这一步就是把 PeroCore 那种“种子 + 联想”能力搬到现在这套结构化系统里。  
检索从单轮排序升级成了一种可解释的“结构联想”。

## 3.4 缺项补召回（Coverage Second Pass）

### 3.4.1 为什么要补召回
第一轮种子和扩散做完后，不代表 query 的关键 facet 都覆盖到了。

比如 query 是：

> “为什么这个地方晚上不能出城？”

结果第一轮只召回到了某次事件，却没召回世界规则。  
这时候需要再做一轮“缺什么补什么”。

### 3.4.2 第一版做规则版，不做完整 FISTA

#### 为什么不在这一版直接做完整 FISTA
PeroCore 的 FISTA 残差很强，但它依赖：

- 稳定 embedding 空间
- 主题/基底矩阵
- 较完整的向量候选池
- 更偏服务端的调参与调试环境

而当前这版 MemoryOS：

- embedding 仍是可选项
- 第一版需要保证 lexical fallback 独立可跑
- 插件环境更重视可解释与易调试

所以这一版**不直接做完整数学 FISTA 残差**，而是做**规则版残差替代**。  
目标和 FISTA 一样：检查第一轮结果有没有漏掉 query 真正需要的内容。  
只是实现上更稳、更适合插件。

### 3.4.3 缺项补召回怎么做

1. `context-router` 先给出 query 想覆盖的 facet  
2. 第一轮召回后统计当前结果已覆盖的 facet  
3. 如果某个高权重 facet 缺失，就执行第二轮定向召回

例如：

- 缺 `world`：补 `world_core_setting`, `world_hard_rule`, `world_global_state`
- 缺 `scene`：补 `scene_shared_state`, `location`
- 缺 `relationship`：补 `relationship`, `actor_private_interpretation`
- 缺 `event`：补 `event`, `actor_visible_event`

### 3.4.4 这一步本质上是什么
它本质上就是一个**规则版 FISTA-lite**：

- 第一轮候选负责解释 query
- 如果解释不完整，就再补弱信号
- 但不引入复杂的数学优化链条

---

## 3.5 多样性裁剪（Diversity Pruning）

### 3.5.1 为什么要裁剪
如果直接把高分记忆全塞进提示词，常见问题是：

- 全是关系类
- 全是同一次事件的变体
- 多条高度重复 summary
- 世界信息和场景信息被挤掉

所以最终输出前必须做多样性裁剪。

### 3.5.2 第一版做规则版，不做 DPP 数学版
PeroCore 用 DPP 的核心目的是“不要全是一类”。  
你这里第一版直接做规则版即可。

#### 第一步：去重
按这些维度去重：

- `entryId`
- `compareKey`
- `schemaId + relationPair`
- summary 高相似的同类事件

#### 第二步：类别平衡
尽量保持下面几类都有代表：

- 世界基础
- 场景共享
- 关系
- 事件
- 主观理解

#### 第三步：预算截断
不做死板条数限制，而是做预算式控制：

$$
usedChars += title.length + summary.length
$$

接近 prompt 预算就停。

---

## 3.6 记忆度与遗忘渲染统一联动

当前 `memoryPercent` 已经存在，遗忘引擎也已经可用。  
本版要做的是让 `memoryPercent` 不只影响“渲染成 clear / blur / distorted”，还要同时影响：

- seed 检索分
- graph 扩散初始能量
- 最终节点亮度和大小
- prompt 注入时的叙事方式

这意味着：

> 同一个记忆度值，同时驱动“检索排序、可视化表现、提示词叙事”。

这样整个系统会更统一。

```ts

  -> seed recall
  -> graph expansion
  -> coverage second pass
  -> diversity prune
  -> return
```

### 大致伪代码
```ts
const route = routeRetrievalContext(query.query, candidates, dictionaries);

const covered = await applyCoverageSecondPass(...);
const pruned = pruneForDiversity(...);
return { providerId, contextRoute: route, items: pruned };
```

## 5. 记忆工作台“可视化记忆”TAB 的交互方案

这一部分是产品体验层面的明确设计。

### 5.1 进入 TAB 后看到什么

#### 顶部统计
- 节点总数
- 边总数
- 当前筛选类型
- 当前视角角色
- 当前高亮 query

#### 左侧主图
显示记忆网络：

- 节点聚类
- 节点间结构边
- 可缩放
- 可拖动
- 可点击

#### 右侧详情
点击节点后显示：

- 记忆标题
- entryType
- category
- tags
- summary
- detail
- memoryPercent
- updatedAt
- 相关节点
- 跳转编辑按钮

### 5.2 节点视觉规则

#### 颜色
- 世界基础：金色
- 事件：橙色
- 关系：粉色
- 场景：青绿色
- 主观理解：紫色

#### 大小
按：

$$
nodeSize = 12 + 20 \times (0.6 \cdot importance + 0.4 \cdot memoryPercent)
$$

#### 发光
按 `memoryPercent`：

- 高记忆度：亮
- 低记忆度：暗

#### 透明度
被过滤掉的低亮度显示，当前高亮链路高亮显示。

### 5.3 点击节点的行为

点击节点后：

1. 高亮当前节点与一阶邻居
2. 边透明度提升
3. 其他无关节点降透明
4. 右侧详情刷新
5. 同步 `selectedEntryId`
6. 可以一键跳到条目中心编辑该记忆

### 5.4 和当前关系图的关系

这版不是替换当前角色关系图，而是新增一个顶层 **可视化记忆** TAB。  
关系图继续存在于 `角色档案` 子视图中。  
这样两者职责分明：

- **关系图**：看角色与角色的关系
- **记忆图**：看记忆与记忆的结构关系

## 9. 风险与边界说明

### 9.1 本版不做完整数学 FISTA
原因不是 FISTA 没价值，而是当前插件阶段更适合先用规则版残差补召回。  
本版会在架构上保留未来接入 `fista-lite.ts` 的位置，但不会把它放进当前主链。

### 9.2 本版不引入重型图数据库
图谱全部采用运行时轻量索引，不引入独立 graph storage。

### 9.3 本版 embedding 仍然可开关
embedding 不是强依赖，默认关闭也能正常工作。

### 9.4 本版记忆图是工作台图，不是 prompt 图
也就是说，可视化图主要用于人看和调试，不直接决定 prompt 内容。  
prompt 内容仍由 retrieval + prune + injection 结果决定。

## 10. 本版最终结论

这一版的本质不是“再加一个图形界面”，而是一次真正意义上的 MemoryOS 中层升级：

- **检索从单轮匹配升级为多阶段混合召回**
- **语境系统正式成为检索入口**
- **记忆度同时影响排序、渲染与可视化表现**
- **工作台第一次具备“图方式查看记忆”的能力**
- **整套能力一次性打通，不拆成多个松散版本**

如果严格按本方案实施，这一版上线后，MemoryOS 的核心能力将从“可编辑的结构化记忆数据库”，升级为：

> **可理解语境、可联想召回、可图形浏览、可角色视角注入的长期记忆系统**
