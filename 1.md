# MemoryOS Prompt ID / KEY 压缩改造实施方案

## 1. 文档目标

本文档用于指导 **SillyTavern-SS-Helper / MemoryOS** 对当前 Dream / Summary / Maintenance 等 AI Prompt 中大量重复、过长的 `ID / KEY` 进行压缩改造，以降低 Token 消耗，同时不破坏现有主链：

- 仓储主键
- compareKey / entityKey
- mutation history
- audit
- rollback
- summary / dream / maintenance
- vector refresh
- diagnostics / workbench

本方案的目标不是“重做整个底层主键体系”，而是优先解决：

> **模型输入层把内部长 ID / 长 KEY 原样暴露给 LLM，导致 prompt token 浪费严重。**

---

## 2. 现状问题

当前测试 prompt 中，存在大量非常长且重复的字段，例如：

- `chatKey`
- `dreamId`
- `entryId`
- `relationship:*`
- `nodeKey`
- `edgeKey`

并且这些字段在：

- Runtime Context
- Recall Bundle
- Diagnostics Summary
- Graph Snapshot Summary

中被反复展开。

这不是个别样本问题，而是当前 Dream Prompt 结构本身就会出现的现象。你贴出的测试文本里可以清楚看到：

- `entry:tavern_...`
- `relationship:tavern_...`
- `entry:entry:tavern_...`
- `co_occurrence:entry:entry:...->topic:...`

这些字段非常长，而且在多个区块中重复出现。fileciteturn66file0

同时，当前 `dreaming-service.ts` 的 `buildDreamPrompt()` 也确实会把：

- `dreamId`
- `entryId`
- `actorKeys`
- `relationKeys`
- `activatedNodeKeys`

等字段直接拼进 Prompt，说明 token 浪费不是错觉，而是当前实现的直接结果。fileciteturn61file0

---

## 3. 结论先行

### 3.1 不建议先改整个底层 ID / KEY 系统
当前问题首先是 **Prompt 传输层问题**，不是数据库主键设计的第一优先问题。

### 3.2 最优先方案：增加 Prompt Alias / Prompt DTO 压缩层
推荐方案：

- 底层真实 ID 保持不变
- Prompt 发送给模型前进行 alias 压缩
- 模型返回后再做 alias 解码
- 程序内部仍使用真实 ID 执行 mutation、audit、rollback

### 3.3 只有当内部结构本身也明显冗余时，才考虑局部瘦身
例如：
- `entry:entry:...`
- `relation:relationship:...`
- `co_occurrence:entry:entry:...`

这种内部 nodeKey / edgeKey 也可以做格式瘦身，但不建议第一步就重构全局主键体系。

---

## 4. 为什么不建议直接改底层主键体系

当前长 ID 不只是给 LLM 用，它们还承担：

- 仓储主键
- compareKey / entityKey
- mutation history
- 审计记录
- rollback 定位
- summary / dream / maintenance 之间的引用
- vector 索引与刷新
- diagnostics / workbench 定位

如果直接重构底层 ID 体系，会牵动：

- `EntryRepository`
- `SummaryService`
- `DreamingService`
- `DreamMutationApplier`
- `DreamRollbackService`
- `DreamSessionRepository`
- compareKey 索引
- vector index refresh
- workbench / diagnostics / rollback UI

这类改动风险非常大，而当前要解决的问题，主要集中在：

> **Prompt 输入给模型时过于“泄露内部结构”，导致 token 冗余。**

所以正确顺序应该是：

1. 先做 Prompt 压缩层
2. 再观察是否还有必要优化内部 KEY 格式
3. 最后才考虑底层 ID 体系重构

---

## 5. 总体改造方向

## 5.1 双层引用体系

建议引入两层体系：

### A. 内部真实引用层
继续保留当前真实：
- `chatKey`
- `dreamId`
- `entryId`
- `relationshipId`
- `nodeKey`
- `edgeKey`

用于：
- 仓储
- audit
- rollback
- vector refresh
- diagnostics 持久化

### B. Prompt 别名引用层
仅给模型看的短引用：

- `C1`：chat
- `D1`：dream session
- `E1 ~ En`：entry
- `R1 ~ Rn`：relationship
- `N1 ~ Nn`：node
- `S1 ~ Sn`：summary

模型只处理这些短引用。  
程序负责在调用前编码、调用后解码。

---

## 5.2 统一原则

### 原则 1：只压缩 Prompt 视图，不压缩数据库主键
### 原则 2：Prompt 中不再暴露完整真实 ID
### 原则 3：模型输出只允许使用 alias 引用
### 原则 4：程序必须能完全反解 alias
### 原则 5：回滚、审计、history 仍基于真实 ID

---

## 6. 改造目标

本次改造要达到以下结果：

1. Dream Prompt Token 显著下降
2. Summary Prompt / Maintenance Prompt 也能复用同一压缩层
3. 模型依然能稳定引用正确条目
4. mutation 写回不受影响
5. rollback 不受影响
6. diagnostics / workbench 仍能展示真实对象
7. 不要求立刻改变数据库主键体系

---

## 7. 需要压缩的字段范围

## 7.1 优先压缩对象

### Runtime Context
- `chatKey`
- `dreamId`

### Recall Bundle
- `entryId`
- `relationKeys`
- `actorKeys`（可部分保留短名）
- `updatedAt`（多数情况下可裁剪）
- `source`（如果已经分组，可省）

### Diagnostics Summary
- `seedEntryIds`
- `activatedNodeKeys`
- `bridgeNodeKeys`

### Graph Snapshot Summary
- `nodeKey`
- `edgeKey`
- `fromNodeKey`
- `toNodeKey`
- `evidenceEntryIds`

---

## 7.2 不建议直接传给模型的字段

以下字段大多数时候对 LLM 价值不高，但很占 token：

- 完整 `chatKey`
- 完整 `dreamId`
- 完整 `relationship:*`
- 完整 `nodeKey`
- 完整 `edgeKey`
- 原始毫秒 `updatedAt`
- 大量 `activatedEdges` 明细
- 完整 `evidenceEntryIds`

这些应优先转成：

- alias
- 摘要
- top-n bridge summary
- top activated topics / actors / entries

---

## 8. Prompt Alias 设计

## 8.1 Alias 编码规则

建议编码如下：

```ts
type PromptAliasMap = {
  chat: Map<string, string>;
  dream: Map<string, string>;
  entry: Map<string, string>;
  relationship: Map<string, string>;
  node: Map<string, string>;
  summary: Map<string, string>;
};
```

反向映射：

```ts
type PromptAliasReverseMap = {
  C1: string;
  D1: string;
  E1: string;
  R1: string;
  N1: string;
  S1: string;
};
```

---

## 8.2 Alias 示例

原始：

```json
{
  "entryId": "entry:tavern_...:9fc25924-eff3-41e0-a037-24f33deff02a"
}
```

压缩后：

```json
{
  "entryId": "E1"
}
```

模型输出：

```json
{
  "sourceEntryIds": ["E1", "E3"]
}
```

程序再解码回真实 entryId。

---

## 8.3 Alias 分配原则

### 原则 1：当前 session 内稳定
同一 dream session 中，`E1` 必须始终映射同一 entry。

### 原则 2：只在当前 prompt 上下文内有效
不同 session 不要求跨轮复用同一 alias。

### 原则 3：按出现顺序分配
便于调试与对照。

---

## 9. Prompt DTO 压缩设计

## 9.1 不直接把仓储对象原样拼进 Prompt
必须新增“Prompt DTO 构建层”。

### 当前问题
现在 `buildDreamPrompt()` 直接把 recall / diagnostics 的内部对象拼进去。fileciteturn61file0

### 改造目标
改成：

- 内部真实对象
- -> 压缩 DTO
- -> Prompt JSON

---

## 9.2 Dream Prompt DTO 建议结构

建议 Dream Prompt 使用压缩后的 DTO：

```ts
type DreamPromptDTO = {
  runtime: {
    chatRef: string;
    dreamRef: string;
    triggerReason: string;
    promptInfo: {
      promptVersion: string;
      stylePreset: string;
      schemaVersion: string;
    };
    qualityConstraints: {
      maxHighlights: number;
      maxMutations: number;
      weakInferenceOnly: boolean;
      requireExplain: boolean;
    };
  };
  recall: {
    recentHits: PromptRecallHitDTO[];
    midHits: PromptRecallHitDTO[];
    deepHits: PromptRecallHitDTO[];
  };
  diagnostics: {
    topBridgeNodes: PromptBridgeNodeDTO[];
    waveHints: PromptWaveHintDTO[];
  };
  graphSummary?: {
    topTopics: string[];
    topActors: string[];
    topEntries: string[];
  };
};
```

其中：

```ts
type PromptRecallHitDTO = {
  entryRef: string;
  title: string;
  summary: string;
  score: number;
  actors?: string[];
  relations?: string[];
  tags?: string[];
};
```

---

## 9.3 Diagnostics 压缩策略

### 不发完整 activatedNodeKeys 列表
改成：
- top activated actors
- top activated topics
- top activated entries

### 不发完整 edges
改成：
- bridge summary
- top co-occurrence hints

例如：

```json
{
  "graphSummary": {
    "topActors": ["user", "seraphina"],
    "topTopics": ["event", "world", "location"],
    "topEntryRefs": ["E1", "E2", "E4"]
  }
}
```

而不是发一大坨 `activatedEdges`。

---

## 10. 内部 KEY / ID 局部瘦身建议

虽然不建议立刻改全局 ID 体系，但有些内部 KEY 本身就冗余，值得单独优化。

## 10.1 可优化项

### A. `entry:entry:xxx`
改成：
- `entry:xxx`

### B. `relation:relationship:xxx`
改成：
- `relation:xxx`

### C. `co_occurrence:entry:xxx->topic:yyy`
改成：
- `co:entry:xxx->topic:yyy`

### D. `summary:summary-snapshot:xxx`
改成：
- `summary:xxx`

---

## 10.2 注意事项
这类优化建议限定在：

- graph nodeKey
- graph edgeKey
- diagnostics 对象键

不要第一步就动：
- `entryId` 主键
- `relationshipId` 主键
- `compareKey`
- `chatKey`

---

## 11. 代码改造建议

## 11.1 新增文件

### `MemoryOS/src/services/prompt-reference-service.ts`
职责：
- 生成 alias
- 保存本轮映射
- 提供 encode / decode

### `MemoryOS/src/services/dream-prompt-dto-service.ts`
职责：
- 把真实 dream recall / diagnostics / graph snapshot 压缩成 Prompt DTO
- 自动裁剪大字段
- 去除重复字段

### `MemoryOS/src/services/summary-prompt-dto-service.ts`
职责：
- Summary Prompt 也统一走 DTO 压缩层

### `MemoryOS/src/types/prompt-alias.ts`
定义 alias map、DTO、decode 结果等类型。

---

## 11.2 修改文件

### `MemoryOS/src/services/dreaming-service.ts`
当前 `buildDreamPrompt()` 应改造为：

- 先从真实 recall / diagnostics 生成 `DreamPromptDTO`
- 再把 DTO 序列化为 Prompt

不再直接拼真实内部 ID。fileciteturn61file0

### `MemoryOS/src/services/dream-wave-recall-service.ts`
不一定要改算法本身，但应为 Prompt DTO 提供更简洁的输出接口：

- top activated nodes
- top bridge nodes
- summary-friendly diagnostics  
而不是让 Prompt 层直接消费全量 graph 信息。fileciteturn62file0

### `MemoryOS/src/services/dream-prompt-service.ts`
如果已有 prompt service，应接入 alias / dto 压缩层。

### `MemoryOS/src/services/summary-service.ts`
让 summary 也能复用 prompt alias 机制，避免重复出现长 entryId / compareKey。

### `MemoryOS/src/services/dream-mutation-translator.ts`
若 dream 输出使用 alias 引用，则 translator 必须支持：
- `E1 -> entryId`
- `R1 -> relationshipId`
- `N1 -> nodeKey`

### `MemoryOS/src/services/unified-memory-mutation-service.ts`
执行层继续使用真实 ID，不接受未解码 alias。

## 12. Prompt 改造要求

## 12.1 Dream Prompt
Dream Prompt 中：

### 当前
- 直接引用真实 `entryId`
- 直接引用真实 `nodeKey`
- `sourceEntryIds` 返回真实 ID

### 改造后
- 使用 `entryRef`
- 使用 `nodeRef`
- `sourceEntryIds` 替换为 `sourceEntryRefs`

例如：

```json
{
  "sourceEntryRefs": ["E1", "E4"]
}
```

---

## 12.2 Summary Prompt
Summary 若需要引用既有条目，也应使用 alias，而不是直接输出完整 entryId / compareKey。

## 12.3 Maintenance Prompt
Maintenance proposal 若引用：
- entry
- relationship
- summary candidate

也应统一走 alias 体系。

## 12.4 非写操作 Prompt
纯 narrative / 风格层 Prompt 不一定需要改，只要不引用结构化对象即可。

---

## 13. 输出解码设计

模型返回后必须经过：

1. JSON 校验
2. alias 解码
3. schema normalize
4. 执行前二次校验

建议增加：

- `decodeDreamPromptOutput(...)`
- `decodeSummaryPromptOutput(...)`

### 校验点
- alias 是否存在
- alias 是否越界
- alias 是否跨类型误用
- 空 alias 是否拒绝

---

## 15. 分阶段实施建议

## 15.1 第一阶段：只做 Dream Prompt Alias
范围：
- Dream Prompt
- Dream 输出解码
- Dream mutation 写回保持不变

目标：
- 先验证这条链可行
- 先拿到 token 优化收益

## 15.2 第二阶段：扩展到 Summary / Maintenance
范围：
- Summary Prompt
- Maintenance Prompt
- Unified mutation 兼容

目标：
- 所有会生成结构化引用的 Prompt 复用同一套 alias 机制

## 15.3 第三阶段：局部瘦身内部 nodeKey / edgeKey
范围：
- graph nodeKey
- graph edgeKey
- diagnostics 对象键

目标：
- 进一步减小内部对象体积
- 但不动数据库主键

## 15.4 第四阶段：评估是否有必要重构底层 ID 体系
只有当你们发现：

- 不只是 Prompt 浪费
- 连仓储、日志、UI、调试都被长 ID 拖累

再考虑底层主键体系重构。

---

## 16. 风险与控制

## 16.1 风险：alias 解码错误导致写错条目
控制：
- 强制 decode 校验
- 未映射 alias 一律拒绝
- 类型错位一律拒绝

## 16.2 风险：Prompt 变短但信息不够
控制：
- Recall 核心摘要保留
- graph 只裁剪冗余，不裁掉关键 bridge summary

## 16.3 风险：summary / dream / maintenance 各自搞一套 alias
控制：
- 用统一 `PromptReferenceService`
- 不允许三套并行实现

## 16.4 风险：workbench / diagnostics 也被 alias 化，影响调试
控制：
- 只压缩 Prompt 视图
- diagnostics 持久化与 UI 仍保留真实 ID

## 19. 最终结论

你现在遇到的问题，核心不是“数据库 ID 太长”，而是：

> **Prompt 把内部真实 ID / KEY 原样、重复、过量地暴露给模型。**

因此最合理的改造方向不是先重构整套底层主键体系，而是：

1. 增加 Prompt Alias 层  
2. 增加 Prompt DTO 压缩层  
3. 裁剪 diagnostics / graph snapshot  
4. 保留真实 ID 用于执行、审计、回滚  
5. 用测试确保 alias 化后仍能正常变更记忆

---

## 20. 一句话版本

> 优先做“给模型看的短引用别名层”，而不是先改数据库主键；先把 Prompt 中重复出现的长 ID / 长 KEY 压缩掉，再通过解码层恢复真实对象，从而在不破坏 MemoryOS 主流程的前提下显著降低 token 消耗。
