# MemoryOS 向量链闭环修复与 LLMHub 重排序实施方案

## Summary
- 本方案基于当前最新 `memoryos-settings-unified` 分支审查结果整理。
- 目标不是重新推翻现有阶段，而是：
  1. 修掉当前向量链已经暴露出的关键闭环问题
  2. 把设置层真正贯穿到运行时
  3. 为重排序增加“**默认规则重排 / 可选 LLMHub 模型重排**”双链路
- 本次方案的核心原则：
  - `lexical_only` 不启用向量策略路由与向量重排
  - `vector_only / hybrid` 才启用向量增强链
  - `rerank` 默认走本地规则链
  - 用户可在 MemoryOS 设置中开启“LLMHub 模型重排”
  - 开启后仅在向量深路径中走 LLMHub；关闭则保持当前默认规则重排

---

## Current Findings

### 1. 第二阶段主体已经真正落地
当前分支已经不是“只建了接口”：

- 已有统一检索入口 `MemoryRetrievalService`
- 已有 `HybridRetrievalService`
- 已有 `EmbeddingService`
- 已有 `VectorStoreAdapterService`
- 已有 `VectorStrategyRouter`
- 已有 `VectorRerankService`
- 已有 `vector-index-service`
- 已有 `vector-runtime`

说明第二阶段主骨架已经成型，不需要回退重做。

---

### 2. 当前还存在的关键问题
虽然主体已落地，但现在至少还有以下几个实质问题：

#### P0-1：设置已统一，但很多没有真正驱动运行时
目前设置里已经有：
- `vectorTopK`
- `vectorDeepWindow`
- `vectorFinalTopK`
- `vectorEnableStrategyRouting`
- `vectorEnableRerank`
- `vectorRerankWindow`
- `vectorEmbeddingModel`
- `vectorEmbeddingVersion`

但这些设置并没有完整贯穿到：
- `HybridRetrievalService`
- `VectorStrategyRouter`
- `VectorRerankService`
- `EmbeddingService`

当前更接近“设置层统一了”，还不是“运行时完全由设置驱动”。

---

#### P0-2：持久化后的向量索引重新加载会丢失来源信息
当前 `memory_vector_index` 只保存：
- `vectorDocId`
- `chatKey`
- `dim`
- `model`
- `version`
- `vector`
- `updatedAt`

但没有保存：
- `sourceKind`
- `sourceId`

结果是：
- 运行时新建的向量索引可以查
- 页面刷新后从 DB 回载时，向量命中可能无法映射回候选记录

这是当前最优先需要修的向量链闭环问题。

---

#### P0-3：向量检索存在重复执行风险
当前流程里：
- `MemoryRetrievalService` 先调用 `orchestrator.retrieve()`
- 然后在 `vector_only / hybrid` 下又进入 `HybridRetrievalService`

而 `RetrievalOrchestrator` 自己在 `vector_only / hybrid` 下已经可能调用一次 `EmbeddingRetrievalProvider`

这会导致：
- query embedding 可能被做两次
- vector search 可能被做两次
- latency 变高
- diagnostics 变乱

---

#### P1-1：向量文档状态字段没有写满
当前 `DBMemoryVectorDocument` 已支持：
- `embeddingModel`
- `embeddingVersion`
- `embeddingDim`
- `embeddingStatus`
- `lastError`

但实际写入时主要只更新了：
- `embeddingStatus`
- `embeddingVersion`

没有完整回填：
- `embeddingModel`
- `embeddingDim`
- `lastError`

这会影响：
- 故障排查
- 索引重建审计
- UI 状态展示
- 模型升级判断

---

#### P1-2：部分 repository 向量接口语义不完整
例如：
- `listVectorDocuments(sourceKind)` 在只有 `sourceKind` 没有 `sourceId` 时，当前实现会返回空
- 这会让按类型查看向量文档的接口行为异常

---

#### P1-3：自动索引与 retrieval mode 绑定过紧
当前 `vectorAutoIndexOnWrite` 只有在：
- runtime ready
- 且 `retrievalMode in {vector_only, hybrid}`

时才生效。

这会带来一个问题：
- 用户长期在 `lexical_only` 下使用时不会建立向量索引
- 后续切到 `hybrid` 后，历史数据不会自动补齐索引
- 必须手动 rebuild，体验不够直观

---

## Core Goal
本次实施的最终目标是：

1. 修复向量持久化与回载闭环
2. 让设置层完整驱动运行时
3. 消除向量检索重复执行
4. 补齐向量文档状态字段
5. 新增可配置的 LLMHub 模型重排序
6. 保持默认链路安全：
   - 不开 LLMHub rerank 时仍可用
   - 不支持 rerank 资源时仍可回退规则重排

---

# 一、总体设计原则

## 1. 重排序仍然只属于向量链
本方案继续保持当前方向：

### lexical_only
- 不走向量策略路由
- 不走向量重排序
- 不走 LLMHub rerank

### vector_only
- 走向量策略路由
- 走向量召回
- 深路径时可选规则重排或 LLMHub 重排

### hybrid
- 走 lexical + vector 融合
- 深路径时可选规则重排或 LLMHub 重排

---

## 2. LLMHub 重排序是“可选增强”，不是默认硬依赖
本方案明确：

- 默认仍走 `VectorRerankService` 规则重排
- 用户在 MemoryOS 设置中开启后，才使用 LLMHub 模型 rerank
- 若 LLMHub rerank 不可用：
  - 自动回退到规则重排
  - 不允许直接让整条链失败

---

## 3. LLMHub 重排序只建议用于深路径
不建议所有 query 都走模型重排。

推荐策略：

### fast path
- 不走 LLMHub rerank
- 直接规则裁切 / 融合后出结果

### deep path
- 候选窗口扩大
- 可启用 LLMHub 模型重排
- 最终压缩到 top3~5

---

# 二、设置层改造方案

## 新增设置项
在 `MemoryOS/src/settings/store.ts` 中新增以下字段：

```ts
/** 是否启用 LLMHub 模型重排序 */
vectorEnableLLMHubRerank: boolean

/** LLMHub 重排序资源提示（可选） */
vectorLLMHubRerankResource: string

/** LLMHub 重排序模型提示（可选） */
vectorLLMHubRerankModel: string

/** LLMHub 重排序触发最小候选数 */
vectorLLMHubRerankMinCandidates: number

/** LLMHub 重排序最大候选数 */
vectorLLMHubRerankMaxCandidates: number

/** LLMHub 重排序失败时是否回退规则重排 */
vectorLLMHubRerankFallbackToRule: boolean
```

---

## 建议默认值
```ts
vectorEnableLLMHubRerank: false,
vectorLLMHubRerankResource: '',
vectorLLMHubRerankModel: '',
vectorLLMHubRerankMinCandidates: 8,
vectorLLMHubRerankMaxCandidates: 12,
vectorLLMHubRerankFallbackToRule: true,
```

---

## 设置语义
### `vectorEnableLLMHubRerank`
- 总开关
- 关闭：只走当前默认规则重排
- 开启：允许深路径使用 LLMHub rerank

### `vectorLLMHubRerankResource`
- 指定 rerank 资源
- 不填时走 LLMHub 默认路由

### `vectorLLMHubRerankModel`
- 指定 rerank 模型
- 不填时交给 LLMHub 自动选

### `vectorLLMHubRerankMinCandidates`
- 候选太少时没有必要走模型
- 小于此值直接跳过 LLMHub rerank

### `vectorLLMHubRerankMaxCandidates`
- 送给 LLMHub 的最大候选数
- 避免输入过长、速度太慢

### `vectorLLMHubRerankFallbackToRule`
- 若 LLMHub 失败，是否自动回退规则重排
- 建议默认 true

---

# 三、运行时设置贯通方案

## 1. `EmbeddingService` 绑定设置
### 目标
让这些设置真正生效：
- `vectorEmbeddingModel`
- `vectorEmbeddingVersion`

### 修改建议
在 `MemoryOS/src/services/embedding-service.ts` 中增加：

- 从 `readMemoryOSSettings()` 读取：
  - `vectorEmbeddingModel`
  - `vectorEmbeddingVersion`
- 编码时自动把 model 作为 routeHint 传给 LLMHub embed
- 启动时 / 设置变更时调用 `setVersion()`

### 结果
Embedding 运行时不再是“有接口但没吃到设置”。

---

## 2. `VectorStrategyRouter` 绑定设置
### 目标
让这些设置真正驱动策略路由：
- `vectorTopK`
- `vectorDeepWindow`
- `vectorFinalTopK`
- `vectorEnableStrategyRouting`
- `vectorEnableRerank`
- `vectorRerankWindow`

### 修改建议
在 `MemoryOS/src/runtime/vector-runtime.ts` 初始化 `HybridRetrievalService` 时，把 settings 传进去：

```ts
new HybridRetrievalService({
  embeddingService,
  vectorStore,
  strategyRouterConfig: {
    enabled: settings.vectorEnableStrategyRouting,
    rerankEnabled: settings.vectorEnableRerank,
    fastCandidateWindow: settings.vectorTopK,
    fastFinalTopK: Math.min(settings.vectorTopK, settings.vectorFinalTopK),
    deepCandidateWindow: settings.vectorDeepWindow,
    deepFinalTopK: settings.vectorFinalTopK,
  },
})
```

### 结果
路由器不再只吃硬编码默认值。

## 3. `VectorRerankService` 绑定设置
### 目标
让 rerank 相关参数可配置。

### 修改建议
`VectorRerankService` 增加 config：
```ts
interface VectorRerankServiceConfig {
  defaultWindow?: number
  useLLMHubRerank?: boolean
}
```

并在 `HybridRetrievalService` 内部从 settings 读取：
- `vectorEnableRerank`
- `vectorRerankWindow`
- `vectorEnableLLMHubRerank`

# 五、向量检索重复执行修复方案

## 问题
当前：
- orchestrator 可能先跑一次 vector seed
- `HybridRetrievalService` 又会再跑一次 vector search

---

## 目标
做到单次 query 只走一条明确的向量链。

---

## 推荐改法
### 方案：让 orchestrator 只负责“基础 lexical 编排”
职责收窄为：
- context routing
- lexical seeds
- graph expansion
- coverage
- diversity

### 向量相关全部收进 `HybridRetrievalService`
负责：
- embedding
- vector search
- strategy route
- rerank
- hybrid merge

---

## 模式行为
### lexical_only
- 只走 orchestrator

### vector_only
- 不再让 orchestrator 先跑 embedding provider
- 直接由 `HybridRetrievalService` 主导

### hybrid
- orchestrator 只产 lexical 结果
- `HybridRetrievalService` 接管 vector 结果并融合

---

## 预期收益
- 降低延迟
- diagnostics 更清楚
- 向量链职责更干净

---

# 六、向量文档状态补齐方案

## 问题
当前向量文档记录没有完整落：
- `embeddingModel`
- `embeddingDim`
- `lastError`

---

## 修改目标
在 `vector-index-service.ts` 中补齐以下写入逻辑：

### 编码前
```ts
embeddingStatus = 'pending'
embeddingVersion = currentVersion
```

### 编码成功后
```ts
embeddingStatus = 'ready'
embeddingModel = encodeResult.model
embeddingDim = encodeResult.dim
lastError = undefined
```

### 编码失败后
```ts
embeddingStatus = 'failed'
lastError = encodeResult.error
```

---

## 收益
- UI 可直接展示向量状态
- 排障更方便
- 模型升级判断更可靠

# 七、LLMHub 模型重排序实施方案

## 设计目标
在当前已有 `VectorRerankService` 基础上，新增可选的 `LLMHubRerankService`：

- 默认关闭
- 在设置里开启后生效
- 只在 `vector_only / hybrid` 的深路径使用
- 失败时可回退规则重排

---

## 新增服务
建议新增文件：

```text
MemoryOS/src/services/llmhub-rerank-service.ts
```

---

## 建议接口
```ts
export interface LLMHubRerankInput {
  query: string
  mode: 'vector_only' | 'hybrid'
  candidates: RetrievalResultItem[]
  finalTopK: number
  resource?: string
  model?: string
}

export interface LLMHubRerankResult {
  ok: boolean
  items: RetrievalResultItem[]
  reasonCodes: string[]
  providerResource?: string
  providerModel?: string
  fallbackUsed?: boolean
  error?: string
}
```

---

## 调用方式
通过 `window.STX.llm.rerank(...)` 调用 LLMHub：

```ts
await llm.rerank({
  consumer: 'stx_memory_os',
  taskId: 'memory_vector_rerank',
  taskDescription: '记忆向量结果重排序',
  query,
  docs,
  topK: finalTopK,
  routeHint: {
    resource,
    model,
  },
  enqueue: {
    displayMode: 'silent',
  },
})
```

---

## 候选输入建议
不要直接把整条 `RetrievalResultItem` 原样喂给 LLMHub。  
建议先把候选压成短文本：

```ts
type RerankDoc = {
  candidateId: string
  text: string
}
```

`text` 建议由：
- title
- summary
- compareKey
- actorKeys / relationKeys / worldKeys
- 必要的 detail 摘要

拼成短文本，控制长度。

---

## 建议候选窗口
### fast path
- 不走 LLMHub rerank

### deep path
- 先从 merged results 取前 `8~12` 条
- 再交给 LLMHub rerank
- 输出 top `3~5`

---

## 失败回退策略
如果出现以下情况：
- `window.STX.llm` 不可用
- 路由不到 rerank 资源
- rerank 请求失败
- 返回结果为空
- 映射 candidate 失败

则按设置决定：

### `vectorLLMHubRerankFallbackToRule = true`
- 自动回退 `VectorRerankService`

### `vectorLLMHubRerankFallbackToRule = false`
- 直接返回 merge 后裁切结果

---

# 八、HybridRetrievalService 改造方案

## 当前问题
当前 `HybridRetrievalService` 只支持：
- vector search
- merge
- rule rerank

还不支持：
- LLMHub rerank
- 设置驱动的 rerank 选择

---

## 修改目标
在 `handleVectorOnly()` 和 `handleHybrid()` 中加入：

### 决策顺序
1. 如果 `decision.rerankEnabled === false`
   - 直接返回 fast path 结果

2. 如果 `decision.rerankEnabled === true`
   - 若设置启用 `vectorEnableLLMHubRerank`
     - 尝试 LLMHub rerank
   - 若关闭或失败
     - 回退 `VectorRerankService`

---

## 建议伪代码
```ts
if (!decision.rerankEnabled) {
  return fastPathResult
}

if (settings.vectorEnableLLMHubRerank && candidateCount >= settings.vectorLLMHubRerankMinCandidates) {
  const llmhubResult = await llmhubRerankService.rerank(...)
  if (llmhubResult.ok) {
    return llmhubResult
  }
  if (!settings.vectorLLMHubRerankFallbackToRule) {
    return directMergedSlice
  }
}

return ruleRerankService.rerank(...)
```

1. 在 `vector_only` 下执行 query
2. 确认 embedding 只执行一次
3. 确认 vector search 只执行一次

## D. LLMHub rerank
1. 关闭 `vectorEnableLLMHubRerank`
   - 只走规则重排
2. 开启 `vectorEnableLLMHubRerank`
   - deep path 走模型重排
3. 模型 rerank 失败
   - 自动回退规则重排
4. fast path
   - 不触发模型重排

1. LLMHub 未初始化
2. LLMHub 无 rerank 资源
3. rerank 请求失败
4. docs 映射异常
5. 全部能回退到规则重排或直接裁切

# 十三、最终建议

## 结论
当前 `memoryos-settings-unified` 分支已经进入“可以打磨闭环”的阶段，不需要再推翻重做。

真正最重要的不是继续堆新能力，而是先完成这四件事：

1. 修持久化向量索引 metadata 丢失
2. 让 settings 真正驱动 runtime
3. 消除向量检索双重执行
4. 补齐向量文档状态

做完这四件事后，再加：

5. **LLMHub 模型重排序可选开关**

这样结果最稳。

---

## 一句话版本
- **默认：规则重排**
- **可选：LLMHub 模型重排**
- **触发条件：只在向量深路径**
- **失败策略：自动回退规则重排**

这会比“直接把所有 rerank 都改成模型”更稳，也更适合你当前这条分支的现状。
