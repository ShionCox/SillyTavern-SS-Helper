# MemoryOS 旧聊天内容分块识别、标签维护与控制台可视化完整实施方案

## Summary
- 本方案解决你当前旧聊天接管中的一个核心问题：**单层消息里混杂正文、思考、总结、表格更新、注释、其他插件标签**，导致：
  1. 正文抽取被污染
  2. 角色识别误判或漏判
  3. 接管楼层“看起来很多，实际送模很少”
  4. 用户无法直观看到系统到底把哪些内容当正文处理了
- 本方案的目标不是只“做一个标签白名单”，而是建立一套完整系统：
  1. **单文件标签维护系统**
  2. **每层消息强制纳入批次，不再整层遗漏**
  3. **分块识别与正文/辅助/排除分类**
  4. **记忆控制台新增专用 Tab，用于编辑规则与预览拆分结果**
  5. **左侧编辑、右侧预览**
  6. **支持先选一个楼层预览处理，同时显示原始内容**
- 最终要达成的是：
  > **每一层都进入接管批次，但每一层内部的不同内容块会被有控制地分类处理，而不是整层被丢弃或整层被错误当正文。**

---

## Core Problem

## 1. 单层消息的信息密度极不均匀
当前旧聊天里，常见一层消息同时包含：
- `<game>` 正文
- `<summary>` 总结
- `<tableEdit>` 表格更新
- `<details>` 注释/留言
- `<think>` / `<think_nya~>` 思考和规划
- 无标签的 meta 分析文字

这意味着：

### 楼层数 != 正文体积
同样是 1 层：
- 有的层只有一句话
- 有的层可能包含几千字正文 + 总结 + tableEdit + details + 思考

所以如果系统直接按“整层”处理，必然会出问题。

---

## 2. 当前问题不只是“筛选过度”，而是缺少“层内分块系统”
现在最大的缺口不是“有没有过滤”，而是：

> **没有把一层消息拆成 block，再按 block 类型决定如何参与接管。**

这会导致两个极端：
- **整层一起送**：正文被 think / tableEdit / summary 污染
- **整层过滤掉**：这一层真正有价值的正文也一起丢失

---

## 3. 你现在最想解决的是这两个目标同时成立
### 目标 A
**每一层都要进入批次，不然遗漏太多。**

### 目标 B
**每一层内部不能所有内容都当正文。**

所以真正需要的不是“按层保留或丢弃”，而是：

> **按层进入批次，按块做分类。**

---

# Final Design Decision

## 最终定案
本方案建议采用：

### **“层必入批次 + 层内分块 + 标签注册表 + 预览式调试”**

也就是：

1. **每个选中的楼层都必须保留到 takeover batch manifest 中**
2. 每层先拆成多个 `blocks`
3. 每个 block 根据标签规则和分类规则进入三种通道：
   - `primary_story`
   - `hint_only`
   - `excluded`
4. 真正送给 AI 的正文上下文只由 `primary_story` 构成
5. `hint_only` 作为辅助上下文/弱线索
6. `excluded` 不参与主抽取，但会保留在可视化预览中
7. 用户可以在控制台里维护标签规则，并实时预览某一层会怎么被拆分

# 二、必须满足的核心原则

## 原则 1：每一层都进入批次
无论这一层是否最终没有可用正文，都必须保留一条 `RawFloorRecord`。

### 为什么
因为否则你会永远分不清：
- 是“这层没被选中”
- 还是“这层被过滤掉了”
- 还是“这层只有辅助信息”

### 正确做法
批次里要保留：
- floor 原始编号
- 原始文本
- 拆分后的 blocks
- 每个 block 的分类结果
- 最终是否进入 primary / hint / excluded

---

## 原则 2：层内按 block 分类，不按整层一刀切
### 错误做法
整层：
- 要么全进正文
- 要么全丢弃

### 正确做法
一层里可能同时包含：

- `primary_story`
- `hint_only`
- `excluded`

三类 block。

---

## 原则 3：标签维护集中在一个文件
你要求“在一个文件内单独做标签维护”，这个是对的。  
我建议建立一个单独配置文件作为全局唯一来源。

---

## 原则 4：控制台必须能看见系统怎么拆
这套东西不能黑盒运行。  
必须让用户能看到：

- 原始内容
- 分块结果
- 标签识别结果
- 最终归类结果
- 哪些进正文
- 哪些只做 hint
- 哪些被排除

---

## 1. 先做 block parser
把一层消息切成：

- 标签块
- 普通文本块

### 输入
一层原始消息文本

### 输出
```ts
type ParsedContentBlock = {
  blockId: string;
  floor: number;
  rawTagName?: string;
  rawText: string;
  startOffset: number;
  endOffset: number;
};
```

---

## 2. 再做 classifier
根据：
- 标签注册表
- 规则判断

给每个 block 一个最终分类：

```ts
type ClassifiedContentBlock = ParsedContentBlock & {
  resolvedKind: ContentBlockKind;
  includeInPrimaryExtraction: boolean;
  includeAsHint: boolean;
  allowActorPromotion: boolean;
  allowRelationPromotion: boolean;
  reasonCodes: string[];
};
```

---

## 3. 分类逻辑
### 先走标签注册表
如果 block 有 tag 且命中 registry，就直接采用。

### 再走规则分类
如果没有 tag 或 tag 未命中，就做规则判定：

#### 判断为正文的特征
- 连续叙事
- 动作描写
- 对话
- 场景推进
- 情绪描写

#### 判断为思考/meta 的特征
- “当前剧情”
- “本次输出计划”
- “第一段、第二段”
- “Let’s write”
- “特别注意”
- “用户要求”
- “剧情推演”

#### 判断为工具层的特征
- `insertRow(`
- `updateRow(`
- `deleteRow(`
- patch 指令
- 结构化注释

# 五、楼层批次保留机制（这是关键）

## 目标
你要求“每一层都要加入到批里面处理”，这个要落成明确机制。

---

## 建议新增结构
```ts
type RawFloorRecord = {
  floor: number;
  originalText: string;
  originalRole: 'user' | 'assistant' | 'system' | 'tool' | 'unknown';
  includedInBatch: true;
  parsedBlocks: ClassifiedContentBlock[];
  hasPrimaryStory: boolean;
  hasHintOnly: boolean;
  hasExcludedOnly: boolean;
};
```

---

## 这意味着什么
### 不管这一层最终有没有正文
都必须有一条 `RawFloorRecord`

所以不会再出现：
- “这层是不是压根没进批次？”
这种不透明问题。

---

## 真正送模时怎么处理
### 送模 payload 不再直接基于“原始整层文本”
而是基于 `RawFloorRecord.parsedBlocks`

### 送模内容拆成两部分
#### primary stream
只拼：
- `includeInPrimaryExtraction = true` 的 blocks

#### hint stream
拼：
- `includeAsHint = true` 但不在 primary 的 blocks

#### excluded stream
不送主抽取，但保留在预览和审计中

---

## 这样就满足了你的两个要求
### 要求 1
每层都进批次  
——满足，因为每层都有 `RawFloorRecord`

### 要求 2
不要整层污染正文  
——满足，因为真正送模的是 block 级 primary/hint

# 十一、AI 兜底分类（可选但建议）

## 目标
处理未知标签和无标签复杂内容。

### 原则
AI 不负责直接抽取记忆，只负责：
- 这块像不像正文
- 这块像不像 summary
- 这块像不像 think/meta/tool

### 输出
```ts
type AIBlockClassificationResult = {
  resolvedKind: ContentBlockKind;
  confidence: number;
  reason: string;
}
```

---

## 何时启用
只在：
- 标签注册表没命中
- 规则分类器置信度不足
- 用户开启 AI 兜底分类

时启用。

---

# 十二、送模链路修改方案

## 当前错误方向
按整层直接送。

## 新方案
### Step 1
选中楼层范围内，每层都生成 `RawFloorRecord`

### Step 2
每层拆 block 并分类

### Step 3
把所有楼层都写入 batch manifest

### Step 4
真正给 AI 的抽取文本按 block 通道组装：

- `primary_story` -> 主正文
- `hint_only` -> 辅助上下文
- `excluded` -> 不进入正文抽取

### Step 5
batch result 保留 floor manifest，便于回查

---

## 重要保证
### 每层都进入 batch
即使这一层最后没有 primary block，也会在 batch manifest 中显示：
- `hasExcludedOnly = true`
或
- `hasHintOnly = true`

这样不会再“消失”。

# 十六、最终建议

## 这套系统的关键不是“怎么过滤”
而是：

> **怎么让每一层都保留下来，同时让每层内部的内容被正确分类。**

你要求的几个点其实是一整套系统，必须一起上：

1. **单文件标签维护**
2. **控制台专用 Tab**
3. **左编辑右预览**
4. **单层预览 + 原始内容显示**
5. **每层强制进入 batch**
6. **真正送模按 block 分类而不是按整层**

---

## 一句话结论
最优方案不是“手动白名单”也不是“全交给 AI”，而是：

> **单文件标签注册表 + 层内分块 + 控制台可视化预览 + 每层批次保留机制**

这样才能既不漏层，也不让插件格式污染正文抽取。
