# MemoryOS 第二阶段详细改造方案（基于 85736ef，向量链专属重排序版）

## Summary
- 本方案建立在 **Commit `85736ef`** 已完成的第一阶段之上。
- 当前第一阶段已经具备：
  1. `retrievalMode = lexical_only | vector_only | hybrid`
  2. `MemoryRetrievalService` 统一检索入口
  3. `RecallConfig`
  4. `PayloadFilter`
  5. `EmbeddingRetrievalProvider` 正式协议壳
  6. `RetrievalOrchestrator` 三态模式支持
- 第二阶段不再讨论“如何为向量做预留”，而是直接进入：
  **完整向量系统落地**
- 本阶段新增一个明确原则：
  **策略路由与重排序只属于向量链路。**
  也就是说：
  - `lexical_only` 不做向量策略路由
  - `lexical_only` 不做向量重排序
  - 只有 `vector_only` / `hybrid` 才会启用：
    - query complexity routing
    - vector candidate window
    - rerank stage

## Core Goal
第二阶段完成后，MemoryOS 应该具备以下能力：

1. **真正可用的 `vector_only`**
   - 独立完成 query -> embedding -> vector search -> optional rerank
2. **真正可用的 `hybrid`**
   - lexical + vector 融合召回
   - 复杂 query 下触发 rerank
3. **向量链专属策略路由**
   - 只有向量开启时，才走复杂度判断与候选放大
4. **向量链专属重排序**
   - rerank 不属于 lexical_only
   - rerank 不作为全局默认阶段
5. **独立的向量文档层**
6. **独立的 embedding 服务层**
7. **独立的向量存储适配层**
8. **可重建、可升级、可维护的完整向量系统**

# 一、第二阶段的核心原则

## 1. 重排序只属于向量链
本阶段明确规定：

### `lexical_only`
- 不启用 vector strategy routing
- 不启用 rerank
- 保持当前第一阶段的快速 lexical 主链

### `vector_only`
- 启用 vector strategy routing
- 启用 vector candidate widening
- 可按 query 复杂度决定是否 rerank

### `hybrid`
- 启用 vector strategy routing
- 启用 vector candidate widening
- 融合 lexical + vector 后，可按 query 复杂度决定是否 rerank

也就是说：

> **rerank 不是全局检索阶段，而是“向量链增强阶段”。**

---

## 2. lexical_only 保持快路径
当前 lexical provider 本身已经不弱，已有：
- BM25
- n-gram similarity
- edit similarity
- memory weight
- recency weight

所以 `lexical_only` 的定位应该保持：

- 快
- 稳
- 成本低
- 适合精确实体命中

不要为了“功能统一”把重排序硬塞进 lexical_only。

---

## 3. 默认推荐上线模式仍然是 `hybrid`
虽然 rerank 只属于向量链，但正式上线建议默认还是：

- `hybrid`

原因很简单：

### lexical 擅长
- 人名
- compareKey
- relation tag
- schema 精确命中
- 短 query / 专有名词

### vector 擅长
- 语义近似
- 模糊表达
- 长 query
- “说法不同但意思相近”

所以生产模式最合理的是：
- **lexical 负责“准”**
- **vector 负责“广”**
- **rerank 负责“最后选对”**

但 lexical_only 仍然保留，适合：
- 调试
- 回退
- 快速模式
- 无 embedding 环境

# 二、第二阶段总体架构

## 建议分为五层

### A. `EmbeddingService`
负责：
- 文本转向量
- 批量编码
- 模型信息管理
- 错误重试
- 版本管理

### B. `VectorDocumentBuilder`
负责：
- 把 MemoryOS 主表对象转换成“可向量化文档”

### C. `VectorStoreAdapter`
负责：
- 向量 upsert
- 向量 search
- 删除
- 重建

### D. `VectorStrategyRouter`
负责：
- 判断 query 简单 / 复杂
- 决定向量链是否：
  - 直接 topK
  - 扩大窗口
  - 启用 rerank

### E. `VectorRerankService`
负责：
- 对向量或融合候选做最终重排序
- 输出 top3~5

## 4. `VectorStrategyRouter`

### 新增文件
```text
MemoryOS/src/services/vector-strategy-router.ts
```

### 目标
把你刚才提到的“简单问题直接向量 TOP3，复杂问题 TOP30 -> TOP3”做成正式策略层。

### 职责
输入：
- query
- query context bundle
- retrieval mode
- actor / relation / world anchor diagnostics

输出：
```ts
type VectorStrategyDecision = {
  route: 'fast_vector' | 'deep_vector'
  candidateWindow: number
  finalTopK: number
  rerankEnabled: boolean
  reasonCodes: string[]
}
```

### 建议规则
#### 简单 query
触发 `fast_vector`
- query 短
- 实体锚点明确
- facet 单一
- actor / relation / world scope 清晰

输出建议：
- `candidateWindow = 3 ~ 5`
- `finalTopK = 3`
- `rerankEnabled = false`

#### 复杂 query
触发 `deep_vector`
- query 长
- 多实体
- 多 facet
- 含场景/关系混合理解需求
- route confidence 不够集中

输出建议：
- `candidateWindow = 20 ~ 30`
- `finalTopK = 3 ~ 5`
- `rerankEnabled = true`

### 重要原则
这个策略路由**只在向量开启时生效**：
- `vector_only`
- `hybrid`

`lexical_only` 一律跳过。

---

## 5. `VectorRerankService`

### 新增文件
```text
MemoryOS/src/services/vector-rerank-service.ts
```

### 目标
在向量链中，对候选做最终重排序。

### 输入建议
```ts
type VectorRerankInput = {
  query: string
  queryContextText: string
  mode: 'vector_only' | 'hybrid'
  candidates: RetrievalResultItem[]
  finalTopK: number
}
```

### 输出建议
```ts
type VectorRerankResult = {
  items: RetrievalResultItem[]
  used: boolean
  reasonCodes: string[]
}
```

### 第一版原则
先不要做太复杂模型。  
第一版建议做“规则增强型 rerank”：

- vectorScore
- lexicalScore（如果 hybrid）
- graphBoost
- actor / relation / world anchor consistency
- recencyWeight
- memoryWeight

### 关键点
`VectorRerankService` 的职责不是“代替召回”，而是：

> **从较大的候选窗口里，选出最适合 prompt 的最终 few-shot 记忆。**

# 五、第二阶段的核心链路

## A. 写入链
### 流程
1. 主表更新
2. `VectorDocumentBuilder` 构建文档
3. upsert `memory_vector_documents`
4. 标记 `embeddingStatus = pending`
5. `EmbeddingService` 生成向量
6. 写入 `memory_vector_index`
7. 标记 `embeddingStatus = ready`

### 适用来源
- entry 更新
- relationship 更新
- actor 更新
- summary 生成
- takeover 完成

---

## B. 查询链（vector_only）
### 流程
1. `QueryContextBuilder` 生成 `QueryContextBundle`
2. `VectorStrategyRouter` 判断：
   - `fast_vector`
   - `deep_vector`
3. `EmbeddingService.encodeOne(mergedContextText)`
4. `VectorStoreAdapter.search(topK = candidateWindow)`
5. 如果 `rerankEnabled = false`
   - 直接取 top3~5
6. 如果 `rerankEnabled = true`
   - `VectorRerankService` 做 TOP30 -> TOP3/5
7. 输出最终结果

### 关键原则
- 不允许 lexical fallback
- 不允许全局强制 rerank
- rerank 是否开启，交给 `VectorStrategyRouter`

---

## C. 查询链（hybrid）
### 流程
1. lexical chain 跑 seeds
2. `QueryContextBuilder` 生成上下文
3. `VectorStrategyRouter` 判断 fast / deep
4. vector chain 跑 seeds
5. merge lexical + vector
6. graph expansion
7. 如果是 `fast_vector`
   - 可直接融合后出结果
8. 如果是 `deep_vector`
   - 进入 `VectorRerankService`
9. 输出 top3~5

### 关键原则
- 重排序虽然作用于“融合结果”，但仍然属于**向量增强阶段**
- 只有因为向量开启了，才会触发这一步

---

## D. 查询链（lexical_only）
### 流程
维持第一阶段现状：
1. lexical seed
2. graph expansion（如配置开启）
3. coverage
4. diversity
5. final

### 明确规定
- 不走 `VectorStrategyRouter`
- 不走 `VectorRerankService`
- 不引入 vector candidate window

---

# 六、策略路由与重排序的正式规则

## 1. 快路径：`fast_vector`
### 适用
- query 短
- 单实体
- 单场景
- facet 明确
- 语义歧义低

### 行为
#### vector_only
- vector top3~5
- 不 rerank
- 直接 final top3

#### hybrid
- lexical top3~5
- vector top3~5
- 快速融合
- 不 rerank
- final top3

### 目标
让简单问题不被重排序拖慢。

---

## 2. 深路径：`deep_vector`
### 适用
- query 长
- 多实体
- 多跳关系
- 多 facet
- 需要上下文理解

### 行为
#### vector_only
- vector top20~30
- rerank
- final top3~5

#### hybrid
- lexical top10~15
- vector top10~15
- merge
- graph expansion
- rerank
- final top3~5

### 目标
让复杂问题不是“先天只看原始向量分数”。

---

## 3. 重排序评分建议
### 第一版建议
```ts
rerankScore =
  vectorScore * vectorWeight +
  lexicalScore * lexicalWeight +
  graphBoost * graphWeight +
  anchorConsistency * anchorWeight +
  recencyWeight * recencyWeightFactor +
  memoryWeight * memoryWeightFactor
```

### 推荐默认值
#### vector_only
- vectorWeight = 0.70
- graphWeight = 0.15
- anchorWeight = 0.10
- recencyWeightFactor = 0.03
- memoryWeightFactor = 0.02

#### hybrid
- vectorWeight = 0.40
- lexicalWeight = 0.35
- graphWeight = 0.10
- anchorWeight = 0.10
- recencyWeightFactor = 0.03
- memoryWeightFactor = 0.02

### 为什么这样设计
- vector_only 下，向量分仍是主导
- hybrid 下，不让 lexical 被完全压掉
- rerank 是在“融合后细排”，不是纯改分游戏

4. hybrid 下 rerank 能兼顾 lexical/vector 分数

## E. RetrievalMode
1. lexical_only 只走 lexical
2. vector_only 只走 vector
3. hybrid 走融合链
4. vector_only unavailable 不 fallback lexical

## F. Rebuild / Maintenance
1. 清空向量索引后可重建
2. 主表变更后可增量刷新
3. embeddingVersion 更新后可重新生成索引

# 十三、风险与规避

## 风险 1：把 rerank 做成全局阶段
### 规避
- 明确规定 rerank 只属于向量链
- lexical_only 不得启用

## 风险 2：vector_only 做得像假模式
### 规避
- 明确规定 vector_only 不允许 lexical fallback
- provider unavailable 直接显式失败

## 风险 3：简单 query 也强行 TOP30 -> TOP3
### 规避
- 必须引入 `VectorStrategyRouter`
- 简单 query 走 fast 路径，不强制 rerank

## 风险 4：混合模式里 lexical 被完全压掉
### 规避
- hybrid rerank 必须保留 lexicalScore 参与
- 不能把 hybrid 变相做成“伪 vector_only”

## Final Recommendation
- 这版第二阶段的核心不是“把 embedding 接上”而已，而是：
  **在你已经完成的第一阶段骨架上，正式把向量链做成独立、完整、可重建、可重排的能力层。**
- 最重要的七件事是：
  1. `EmbeddingService`
  2. `VectorDocumentBuilder`
  3. `VectorStoreAdapter`
  4. `VectorStrategyRouter`
  5. `VectorRerankService`
  6. `vector_only / hybrid` 真正跑通
  7. 重建与维护能力补齐
- 这样做完以后，MemoryOS 的向量能力不是“开了个 embedding 开关”，而是真正有独立链路、独立优化和独立维护能力。
