# MemoryOS 可写目标、Key、Patch 路径链路优化实施方案

---

## 1. 当前问题总结

你这次 commit 已经把提示词质量、sourceEvidence、confidence、memoryValue、quality guard 这些方向做起来了，整体方向是对的。

但现在仍存在一个关键问题：

```text
AI 要 UPDATE / MERGE / INVALIDATE / DELETE 时，它必须知道：
1. 应该更新哪条旧记忆；
2. 那条旧记忆的稳定 key 是什么；
3. 它能写哪些 patch path；
4. 当前旧值是什么；
5. patch 写完后由谁负责合并；
6. 如果新增，entityKey / compareKey 谁生成。
```

当前链路虽然有候选记忆和 ref 压缩，但这些信息还没有统一组织成“可写目标协议”。

因此 AI 很容易出现：

```text
- 只写 entityKey，但没有 candidateId / targetId，导致 UPDATE 无法定位；
- 生成了 compareKey，但没有真正绑定旧记录；
- patch 写了不在白名单里的字段；
- 把完整旧 state 重新输出；
- 不知道某个字段当前是什么，导致 patch 等于旧值；
- ADD 时自己编 entityKey，后期重复或冲突；
- Dream proposedMutations 里让 AI 写 payload.entryId，容易写错真实 ID；
- Takeover 阶段输出 entityKey / compareKey，但缺少统一解码和冲突闭环。
```

---

## 2. 审查到的现有能力

### 2.1 Summary 已经有候选检索与短引用雏形

当前 summary 链路会：

```text
runSummaryOrchestrator()
  → buildSummaryMutationContext()
  → resolveCandidateRecords()
  → buildSlimMutationContext()
  → SummaryPromptDTOService.build()
  → runSummaryMutationBatch()
  → decodeSummaryPromptDocument()
  → applyPatchDiffGuard()
  → validateSummaryMutationDocument()
  → applySummaryMutation()
```

其中 `SummaryPromptDTOService` 已经会把真实 `candidateId / entryId` 压缩成 prompt 内的 `candidateRef / entryRef`，并保存映射：

```ts
candidateRefToCandidateId: Map<string, string>
entryRefToEntryId: Map<string, string>
```

这是很好的基础。

### 2.2 Summary Applier 真正需要的是 candidateId / targetId

当前 `applySummaryMutation()` 执行 UPDATE 时，本质上靠：

```ts
const candidate = action.candidateId
  ? candidateIdMap.get(action.candidateId)
  : null;

const resolvedRecordId = String(
  action.targetId ?? candidate?.recordId ?? ''
).trim();
```

所以：

```text
UPDATE / INVALIDATE / DELETE 的真实落点不是 entityKey，
而是 targetId 或 candidateId → recordId。
```

这点非常重要。

### 2.3 Validator 已经要求 UPDATE 必须有 target

当前 validator 也已经判断：

```text
UPDATE 必须有 targetId 或 candidateId
INVALIDATE 必须有 targetId 或 candidateId
DELETE 必须有 targetId 或 candidateId
```

否则会出现：

```text
update_requires_target
invalidate_requires_target
delete_requires_target
```

这说明系统层已经知道“更新必须有目标”，但 prompt 层还没有让 AI 稳定选择目标。

### 2.4 Dream 已经有 entryRef / nodeRef / relationshipRef

Dream 链路里，`DreamPromptDTOService` 会生成：

```text
entryRef
nodeRef
relationshipRef
```

并保存：

```ts
entryRefToEntryId
nodeRefToNodeKey
relationshipRefToRelationshipKey
```

但是当前 dream output schema 仍要求 proposedMutations 里存在：

```json
"payload": {
  "entryId": "string",
  "relationshipId": "string",
  "compareKey": "string",
  "entityKey": "string"
}
```

这会把 AI 引向“自己写真实 ID / key”的方向，风险较高。

---

## 3. 总体解决方案：Writable Target Manifest

建议新增一套全系统通用的协议：

```text
Writable Target Manifest
```

中文可以叫：

```text
可写目标清单
```

它的目标是：

```text
AI 只知道短引用；
系统知道真实 key；
AI 只写 patch；
系统负责解码、补 key、校验路径、应用 patch。
```

核心原则：

```text
1. AI 不直接写真实 entryId / recordId / relationshipId；
2. AI 不直接决定数据库主键；
3. AI 只从 manifest.patchTargets 里选择 targetRef；
4. UPDATE / MERGE / INVALIDATE / DELETE 必须引用 targetRef；
5. ADD 不引用 targetRef，只输出 keySeed + newRecord；
6. entityKey / compareKey 由系统优先生成或补齐；
7. patch 只能写 target.editablePaths 允许的路径；
8. patch 只写变化字段；
9. 系统 decode targetRef 后再进入 validator / diff guard / applier。
```

## 4. 推荐数据结构

### 4.1 新增 PromptReferenceKind

当前已有：

```ts
export type PromptReferenceKind =
  | 'chat'
  | 'dream'
  | 'entry'
  | 'relationship'
  | 'node'
  | 'summary';
```

建议扩展：

```ts
export type PromptReferenceKind =
  | 'chat'
  | 'dream'
  | 'entry'
  | 'relationship'
  | 'node'
  | 'summary'
  | 'target';
```

并在 `PromptReferenceService` 中增加前缀：

```ts
const PROMPT_REF_PREFIX: Record<PromptReferenceKind, string> = {
  chat: 'C',
  dream: 'D',
  entry: 'E',
  relationship: 'R',
  node: 'N',
  summary: 'S',
  target: 'T',
};
```

这样统一使用：

```text
T1 / T2 / T3
```

表示本轮可写目标。

### 4.2 新增通用 Manifest 类型

建议新建文件：

```text
MemoryOS/src/services/writable-target-manifest-service.ts
```

类型：

```ts
export type WritableTargetAction =
  | 'UPDATE'
  | 'MERGE'
  | 'INVALIDATE'
  | 'DELETE'
  | 'RELATIONSHIP_PATCH'
  | 'ENTRY_PATCH';

export interface WritableTargetManifestItem {
  /** Prompt 内短引用，例如 T1。 */
  targetRef: string;

  /** Prompt 内候选短引用，例如 S1。 */
  candidateRef?: string;

  /** Prompt 内 entry 短引用，例如 E1。 */
  entryRef?: string;

  /** Prompt 内 relationship 短引用，例如 R1。 */
  relationshipRef?: string;

  /** 真实内部 id，不进入 prompt，可留在 context 映射中。 */
  candidateId?: string;
  entryId?: string;
  relationshipId?: string;

  targetKind: string;
  title: string;
  status: string;

  /** 系统已有稳定键，给 AI 参考，但不让 AI 编。 */
  entityKey?: string;
  compareKey?: string;
  matchKeys?: string[];

  /** 旧值摘要，只放可写字段相关内容，不给完整 detailPayload。 */
  current: Record<string, unknown>;

  /** 当前 target 允许 patch 的路径。 */
  editablePaths: string[];

  /** 推荐动作，例如 UPDATE 优先、MERGE 谨慎等。 */
  allowedActions: WritableTargetAction[];

  /** 针对当前 target 的提示。 */
  operationHints: string[];
}

export interface WritableTargetManifest {
  version: 'writable-target-manifest.v1';
  rules: string[];
  patchTargets: WritableTargetManifestItem[];
}

export interface WritableTargetManifestBuildResult {
  manifest: WritableTargetManifest;
  targetRefToCandidateId: Map<string, string>;
  targetRefToEntryId: Map<string, string>;
  targetRefToRelationshipId: Map<string, string>;
  targetRefToItem: Map<string, WritableTargetManifestItem>;
}
```

## 5. Summary 链路实施方案

### 5.1 扩展 SummaryMutationAction

当前 `SummaryMutationAction` 已经有：

```ts
targetId?: string;
sourceIds?: string[];
candidateId?: string;
entityKey?: string;
compareKey?: string;
patch?: Record<string, unknown>;
newRecord?: Record<string, unknown>;
```

建议增加 prompt-only 字段：

```ts
targetRef?: string;
sourceRefs?: string[];

keySeed?: {
  kind?: string;
  title?: string;
  qualifier?: string;
  participants?: string[];
};
```

完整建议：

```ts
export interface SummaryMutationAction {
  action: SummaryMutationActionType;
  targetKind: string;

  /**
   * Prompt 内可写目标短引用，例如 T1。
   * AI 输出，系统 decode。
   */
  targetRef?: string;

  /**
   * MERGE 时的来源目标短引用。
   */
  sourceRefs?: string[];

  /**
   * 真实落库目标，decode 后由系统填充。
   */
  targetId?: string;
  sourceIds?: string[];
  candidateId?: string;

  /**
   * 新增时的 key 种子，系统据此生成 entityKey / compareKey。
   */
  keySeed?: {
    kind?: string;
    title?: string;
    qualifier?: string;
    participants?: string[];
  };

  entityKey?: string;
  compareKey?: string;
  matchKeys?: string[];

  patch?: Record<string, unknown>;
  newRecord?: Record<string, unknown>;
}
```

### 5.3 SummaryCandidateRecord 补 current

当前 `SummaryCandidateRecord` 有：

```ts

recordId
targetKind
schemaId
title
summary

compareKey

```

建议增加：

```ts
current?: {
  summary?: string;
  fields?: Record<string, unknown>;
  bindings?: Record<string, unknown>;
  state?: string;
  lifecycle?: Record<string, unknown>;
```

在 `resolveCandidateRecords()` 里由 `entry.detailPayload` 提取：

```ts
function buildCandidateCurrent(entry: MemoryEntry): SummaryCandidateRecord['current'] {
  const payload = toRecord(entry.detailPayload);
  return {
    summary: entry.summary,
    fields: pickWritableFields(toRecord(payload.fields)),
    bindings: toRecord(payload.bindings),
```

只放简化状态，不要把完整 payload 全塞给 AI。

### 5.5 改造 Summary 输出 schema

建议让 AI 输出：

```json
{
  "action": "UPDATE",
  "targetKind": "task",
  "targetRef": "T1",
  "confidence": 0.86,
  "memoryValue": "high",
  "sourceEvidence": {
    "type": "story_dialogue",
    "brief": "艾琳确认北城门仍是撤离路线，{{user}}也决定同行。",
    "turnRefs": [108, 109]
  },
  "patch": {
    "summary": "任务已从确认情报推进为正式撤离执行。",
    "fields": {
      "status": "进行中",
      "goal": "确保密使安全离城",
      "lastChange": "确认北城门撤离路线并开始执行"
    }
  },
  "reasonCodes": [
    "target_ref_selected",
    "task_progressed",
    "update_existing_preferred"
  ]

```

不要再让 AI 必填 `entityKey`。  
`entityKey / compareKey` 可以由系统从 manifest 补齐。

建议修改 prompt 中这条：

```text
ADD 必须输出 entityKey，且 entityKey 必须是稳定内部键；
```

改成：

```text
ADD 不要编造 entityKey；ADD 必须输出 keySeed，系统将根据 keySeed 生成 entityKey 与 compareKey。
UPDATE / MERGE / INVALIDATE / DELETE 不要编造 targetId；必须选择 patchTargetManifest 中的 targetRef。
```


### 5.6 decode 阶段映射 targetRef

建议新增：

```ts
function decodeWritableTargetRefs(
  document: SummaryMutationDocument,
  manifest: WritableTargetManifestBuildResult,
): SummaryMutationDocument {
  return {
    ...document,
    actions: document.actions.map((action) => decodeWritableTargetAction(action, manifest)),
  };
}
```

具体：

```ts
function decodeWritableTargetAction(
  action: SummaryMutationAction,
  manifest: WritableTargetManifestBuildResult,
): SummaryMutationAction {
  const targetRef = String(action.targetRef ?? '').trim();

  if (!targetRef) {
    return action;
  }

  const item = manifest.targetRefToItem.get(targetRef);
  const candidateId = manifest.targetRefToCandidateId.get(targetRef);
  const targetId = manifest.targetRefToEntryId.get(targetRef);
  const relationshipId = manifest.targetRefToRelationshipId.get(targetRef);

  if (!item) {
    return {
      ...action,
      action: 'NOOP',
      patch: undefined,
      newRecord: undefined,
      reason: `无法解析 targetRef: ${targetRef}`,
      reasonCodes: [
        ...(action.reasonCodes ?? []),
        'target_ref_decode_failed',
      ],
    };
  }

  return {
    ...action,
    candidateId: action.candidateId ?? candidateId,
    targetId: action.targetId ?? targetId ?? relationshipId,
    entityKey: action.entityKey ?? item.entityKey,
    compareKey: action.compareKey ?? item.compareKey,
    matchKeys: action.matchKeys?.length ? action.matchKeys : item.matchKeys,
    reasonCodes: [
      ...(action.reasonCodes ?? []),
      'target_ref_decoded',
    ],
  };
}
```

### 5.7 decode 顺序调整

建议 Summary 阶段顺序改成：

```text
LLM raw output
  ↓
decode targetRef/sourceRefs
  ↓
system fill entityKey/compareKey
  ↓
normalize patch/newRecord
  ↓
patch path validator
  ↓
diff guard
  ↓
mutation validator
  ↓
applier
```

当前链路里 `decodeSummaryPromptDocument()` 后接 `validateSummaryPatch()`、`normalizeSummaryPatch()`、`applyPatchDiffGuard()`、`validateSummaryMutationDocument()`，这个顺序基本合理。需要补的是 decode 时能拿到 `targetRefToEntryId` 和 manifest item。

## 7. Takeover 链路实施方案

Takeover 是批量接管旧聊天，重点不是单次 patch，而是：

```text
稳定 key 生成
重复实体归并
冲突解决
批次间 key 延续
```

你现在 TAKEOVER_BATCH_SYSTEM 已经强调：

```text
复用 knownContext / knownEntities
保持 compareKey、entityKey、bindings 稳定
```

但建议继续升级成：

```text
Known Entity Manifest
```


```text

系统生成 actorKey / entityKey / compareKey / matchKeys。
```

### 8.2 user / actor 规则保持现状


## 9. Validator 增强方案

### 9.1 新增 targetRef 校验

```ts
function validateTargetRefContract(
  action: SummaryMutationAction,
  manifest: WritableTargetManifestBuildResult,
): string[] {
  const errors: string[] = [];
  const actionName = String(action.action ?? '').toUpperCase();
  const targetRef = String(action.targetRef ?? '').trim();

  if (['UPDATE', 'MERGE', 'INVALIDATE', 'DELETE'].includes(actionName)) {
    if (!targetRef) {
      errors.push(`${actionName.toLowerCase()}_requires_target_ref`);
      return errors;
    }

    if (!manifest.targetRefToItem.has(targetRef)) {
      errors.push(`target_ref_not_found:${targetRef}`);
      return errors;
    }
  }

  if (actionName === 'ADD' && targetRef) {
    errors.push('add_should_not_have_target_ref');
  }

  return errors;
}
```

### 9.2 新增 patch path 校验

```ts
function validatePatchPathsAgainstTarget(
  action: SummaryMutationAction,
  manifest: WritableTargetManifestBuildResult,
): string[] {
  const errors: string[] = [];
  const targetRef = String(action.targetRef ?? '').trim();
  const target = manifest.targetRefToItem.get(targetRef);
  if (!target) return errors;

  const patch = action.patch ?? {};
  const paths = flattenPaths(patch);

  for (const path of paths) {
    if (!target.editablePaths.includes(path)) {
      errors.push(`patch_path_not_allowed:${targetRef}:${path}`);
    }
  }

  return errors;
}
```

### 9.3 新增 sourceRefs 校验

MERGE 的 sourceRefs 也必须来自 manifest：

```ts
function validateMergeSourceRefs(
  action: SummaryMutationAction,
  manifest: WritableTargetManifestBuildResult,
): string[] {
  if (action.action !== 'MERGE') return [];
  const errors: string[] = [];

  for (const ref of action.sourceRefs ?? []) {
    if (!manifest.targetRefToItem.has(ref)) {
      errors.push(`merge_source_ref_not_found:${ref}`);
    }
  }

  return errors;
}
```

### 9.4 ADD keySeed 校验

```ts
function validateAddKeySeed(action: SummaryMutationAction): string[] {
  if (action.action !== 'ADD') return [];

  const errors: string[] = [];
  const title = String(action.keySeed?.title ?? '').trim();

  if (!title) {
    errors.push('add_requires_key_seed_title');
  }

  if (action.entityKey) {
    errors.push('add_should_not_generate_entity_key_directly');
  }

  return errors;
}
```

## 11. Key 生成策略

### 11.1 AI 不负责生成真实 key

AI 可以输出：

```json
"keySeed": {
  "kind": "task",
  "title": "护送密使离开王都",
  "qualifier": "王都"
}
```

系统负责：

```text
compareKey = buildCompareKey(kind, title, fields)
entityKey = buildEntityKey(kind, compareKey, title)
matchKeys = buildMatchKeys(kind, title, aliases)
```

### 11.2 新增 KeyResolver

建议新文件：

```text
MemoryOS/src/core/memory-key-resolver.ts
```

示例：

```ts
export interface MemoryKeySeed {
  kind: string;
  title: string;
  qualifier?: string;
  participants?: string[];
  aliases?: string[];
  fields?: Record<string, unknown>;
}

export interface ResolvedMemoryKeys {
  entityKey: string;
  compareKey: string;
  matchKeys: string[];
}

export function resolveMemoryKeysFromSeed(seed: MemoryKeySeed): ResolvedMemoryKeys {
  const kind = normalizeKind(seed.kind);
  const title = normalizeTitle(seed.title);
  const fields = {
    ...(seed.fields ?? {}),
    qualifier: seed.qualifier,
    participants: seed.participants,
    aliases: seed.aliases,
  };

  const compareKey = buildCompareKey(kind, title, fields);
  const entityKey = buildEntityKey(kind, compareKey, title);
  const matchKeys = buildMatchKeys(kind, title, seed.aliases ?? []);

  return { entityKey, compareKey, matchKeys };
}
```

### 11.3 UPDATE 直接沿用 Manifest 的 key

对于 UPDATE：

```text
entityKey = manifest.target.entityKey
compareKey = manifest.target.compareKey
matchKeys = manifest.target.matchKeys
```

不要使用 AI 输出的 key 覆盖系统 key。AI 输出的 key 最多当诊断。

## 12. Prompt 修改建议

### 12.1 SUMMARY_SYSTEM 加 Patch Target 规则

加入：

```md
## Patch Target 规则

你会收到 patchTargetManifest，其中包含本次允许更新的已有记忆目标。

- UPDATE、MERGE、INVALIDATE、DELETE 必须选择一个 targetRef。
- targetRef 必须来自 patchTargetManifest.patchTargets。
- 不要猜测 targetRef。
- 如果没有合适 targetRef，但信息值得保存，使用 ADD。
- 如果信息已被某个 targetRef 完整覆盖，使用 NOOP。
- 如果信息是对某个 targetRef 的阶段推进、状态变化、关系变化或细节补充，使用 UPDATE。
- patch 只能包含该 targetRef.editablePaths 中允许的字段。
- patch 只写变化字段，不要重复 current 中未变化的内容。
- 不要输出真实 entryId、recordId 或数据库 key。
- 你可以引用 targetRef，但真实 key 由系统解析。
```

### 12.2 ADD 规则替换

把：

```text
ADD 必须输出 entityKey，且 entityKey 必须是稳定内部键；
```

改成：

```text
ADD 不要编造 entityKey、compareKey 或真实 ID。
ADD 必须输出 keySeed 和 newRecord。
系统会根据 keySeed 生成 entityKey、compareKey 与 matchKeys。
```

### 12.3 UPDATE 规则明确

```md
UPDATE：
- 必须使用 targetRef；
- 必须使用 patch；
- patch 只能写 target.editablePaths 允许的路径；
- patch 不要重复 current 中已经存在且未变化的值；
- 不要输出 targetId、entryId、recordId；
- 不要编造 entityKey 或 compareKey。
```

### 12.4 MERGE 规则明确

```md
MERGE：
- 必须使用 targetRef 作为主目标；
- sourceRefs 必须来自 patchTargetManifest.patchTargets；
- patch 只写主目标合并后的增量字段；
- 不要自行输出 sourceIds 或真实 entryId。
```

## 13. 一次对话轮次内的最终流程

保留你现在的两阶段 summary planner + mutation 也可以。

重点是 mutation 阶段必须一次完成：

```text
当前窗口
  ↓
本地检索候选旧记忆
  ↓
构建 patchTargetManifest
  ↓
一次 LLM mutation 调用：
    - 选择 targetRef
    - 决定 ADD / UPDATE / MERGE / INVALIDATE / DELETE / NOOP
    - 输出 patch / newRecord / keySeed
  ↓
系统 decode targetRef
  ↓
系统补 entityKey / compareKey / targetId / candidateId
  ↓
patch path validator
  ↓
patch diff guard
  ↓
mutation validator
  ↓
applier
```

## 17. 风险点

### 17.1 Manifest 太大

解决：

```text
只给 top 3~8 个可写目标
只给 current 的可写字段摘要
长文本裁剪
同类型最多 2 个候选
```

### 17.2 targetRef 与 candidateId 混用

解决：

```text
prompt 内统一只用 targetRef
candidateRef / entryRef 仅用于 explain
进入 applier 前必须 decode 成 candidateId / targetId
```

### 17.3 AI 仍输出真实 ID

解决：

```text
validator 直接报错或丢弃：
- 出现 entry_ / rel_ / uuid / raw record id
- action 中出现 targetId
- payload 中出现 entryId
```

例外：系统 decode 后内部补上的 targetId 不走这个检查。

### 17.4 新增 key 重复

解决：

```text
ADD 使用 keySeed
系统生成 compareKey 后查重
若 compareKey 命中已有 target，则 ADD 自动降级为 UPDATE/MERGE/NOOP
```

## 18. 最终推荐落地顺序

### 阶段一：Summary 主链解决

```text
1. 新增 targetRef/sourceRefs/keySeed 到 SummaryMutationAction；
2. buildSlimMutationContext 增加 patchTargetManifest；
3. Summary prompt 改成 targetRef + patch；
4. decode targetRef → candidateId / targetId；
5. validator 增加 targetRef 与 patch path 校验；
6. ADD 改成 keySeed，系统生成 entityKey / compareKey。
```

### 阶段二：Dream 对齐

```text
1. DreamPromptDTO 加 writableTargets；
2. dream output schema 去掉 AI 直接写 entryId / relationshipId；
3. entry_patch / relationship_patch 使用 targetRef；
4. entry_create 使用 keySeed；
5. dream mutation translator decode targetRef。
```

### 阶段三：Takeover 对齐

```text
1. Takeover 输入加 knownEntityManifest；
2. 批次输出支持 targetRef + patch；
3. entityKey / compareKey 由系统生成和归并；
4. 冲突解决也使用 targetRef。
```

### 阶段四：统一基础设施

```text
1. WritableTargetManifestService；
2. WritablePathRegistry；
3. MemoryKeyResolver；
4. TargetRefValidator；
5. PromptAliasSnapshot 扩展 target refs；
6. UI 显示写入目标和 patch diff。
```

## 19. 一句话结论

当前 MemoryOS 已经具备“候选检索、短引用、patch、validator、diff guard”的基础，但还缺一个统一协议把它们串起来。

最好的解决方案是：

```text
所有 AI 可写操作都统一改成：
AI 选择 targetRef + 输出 patch / keySeed
系统 decode targetRef + 补齐真实 key + 校验 patch path + 应用 patch
```

这能在一次对话轮次内解决 AI 不知道 key / target / patch path 的问题，同时避免 AI 猜真实数据库 ID。
