我再看了这版，能删的、该收口的、以及我觉得像 bug 的点，主要有这几类。

## 优先处理

### 1. 设置页里还留着一整套“旧逻辑表维护台”

现在 settings 面板的 relation tab 里，刷新时还会跑：

* `populateEntityTypes()`
* `renderLogicTable(logicTableSelect.value)`

也就是设置页自己还在维护一套逻辑表 UI 

但你新的记录编辑器已经把“逻辑维护 / 诊断维护 / 原始库表”完整承接了，而且逻辑维护能力明显更完整，包括：

* `listLogicTables`
* `getLogicTableView`
* `listBackfillCandidates`
* `promoteDerivedRow`
* `repairTable`
* 行级 alias / tombstone / restore / merge / inline edit

这些都已经在 `recordEditorNext.ts` 里成型了 

### 建议

把 settings 页 relation tab 里的这套逻辑表维护链路删掉，settings 页只保留：

* 关系/状态摘要
* 跳转“打开记录编辑器”

### 可以删/收口的对象

大概率包括这些旧入口：

* `logicTableEntitySelectId`
* `logicTableRefreshBtnId`
* `logicTableContainerId`
* relation tab 中和 `populateEntityTypes / renderLogicTable / executeLogicRowAction / executeLogicTableRepair / renderLogicDetailDrawer` 相关的绑定和 DOM 责任 

这个是我认为**最值得先删**的一块，因为它已经和 `recordEditorNext` 形成双轨。

---

### 2. `worldInfo.getLogicTable()` 已经明确是 legacy 兼容口

`SDK/stx.d.ts` 里已经直接把它标成 deprecated：

> 新编辑器请改用 `logicTable.getLogicTableView()`、`logicTable.listLogicTables()` 或 `rows.listTableRows()` 

### 建议

检查仓库里是否还有任何新 UI、新 facade、新维护动作在调它。
如果没有外部调用依赖了，就可以删实现；至少也应该：

* 保留接口一段时间
* 实现内部直接转发到新接口
* 并在注释里写“准备移除版本”

这块不一定立刻删接口，但**很适合删掉旧实现逻辑**。

---

### 3. `recordEditor.ts` 现在只是转发壳

这个文件现在只剩一句：

* `export { openRecordEditor } from './recordEditorNext';` 

### 判断

这不算 bug，但它是典型“过渡壳”。

### 建议

如果外部 import 已经全都能直接改到 `recordEditorNext`，可以删。
如果你还想保兼容，那就留着，但最好加一句注释说明它是兼容转发，不然以后容易被当成“还有旧版编辑器”。

---

## 明显冗余

### 4. 一批工具函数在 `index.ts` 和 `recordEditorNext.ts` 重复了

我至少看到这些重复：

* `escapeHtml`
* `parseLooseValue`
* `normalizeLookup`
* 一些格式化逻辑

两边都有独立实现  

### 风险

这类重复最容易造成“一个地方修了，另一个地方忘了”，尤其是：

* 松散值解析
* 字段展示
* sourceRef 文本拼接

### 建议

抽到一个共享 util，比如：

* `ui/editorUtils.ts`
* 或 `ui/shared/formatters.ts`

这个不一定删文件，但很值得做一次归并。

---

### 5. 逻辑表维护动作在 settings 和 record editor 各写了一遍

现在 settings 里有：

* `executeGlobalLogicRepair`
* `executeLogicTableRepair`
* `executeLogicRowAction`

record editor 里也有完整一套：

* repair
* promote
* alias
* merge
* restore
* tombstone
* inline edit  

### 判断

这不是简单“多一个入口”，而是**维护逻辑重复实现**。

### 建议

保留一处真正的维护入口就够了：

* settings：只显示摘要和跳转
* record editor：负责所有维护动作

---

## 我觉得像 bug / 易出问题的点

### 6. settings relation tab 和 record editor 同时维护逻辑表，状态可能不一致

因为两边都在：

* 读 `logicTableView`
* 读候选
* 执行动作
* 各自 refresh

这会导致典型问题：

* 一边操作后另一边 UI 还是旧状态
* 选中的 tableKey / rowId 心智不一致
* 用户不知道“去哪边改才算正式入口”

这更像**结构性 bug**，不是语法 bug。根因就是第 1 点的双轨。

---

### 7. settings 页太胖了，`index.ts` 已经成了“大一统控制器”

`MemoryOS/src/ui/index.ts` 现在承载了：

* settings 页面基础渲染
* experience 刷新
* tuning
* record filter
* AI diagnostics
* template
* audit
* worldInfo writeback
* logic table maintenance
* editor action
* task presentation
* bus inspector

而且这些都集中在一个文件里 

### 风险

这类文件未来特别容易出：

* 监听器重复绑定
* 某段 UI 改动误伤别的面板
* 隐式状态互相污染

### 建议

哪怕不立刻拆完，也建议先把这几块抽出去：

* `logicTableSettingsPanel`
* `editorActions`
* `candidateSourceDialogs`
* `templatePanel`
* `auditPanel`

---

### 8. `recordEditorNext.ts` 里也开始变得很大

它现在已经同时负责：

* 聊天列表
* maintenance view
* diagnostics view
* raw table view
* inline raw editing
* logic maintenance actions
* context menu
* DB clear/export/import-ish流程相关联
* 临时 MemorySDK 生命周期

能力很强，但文件已经非常大了 

### 判断

这不是现在必须删代码，但已经进入“可维护性风险区”。

### 建议

下一轮可以拆成：

* `recordEditor/chats.ts`
* `recordEditor/maintenanceView.ts`
* `recordEditor/diagnosticsView.ts`
* `recordEditor/rawView.ts`

---

## 我认为可以删，但先确认调用方的

### 可以优先检查删除的候选

1. settings relation tab 里的旧逻辑表 UI 链
2. `logicTableEntitySelectId / logicTableRefreshBtnId / logicTableContainerId` 相关 DOM 和样式
3. `recordEditor.ts` 兼容转发壳
4. `worldInfo.getLogicTable()` 的旧实现逻辑
5. `index.ts` 里专门给 settings 逻辑表服务的 helper

### 1. `recordEditor.ts` 如果外部 import 还很多

这种壳文件删了容易造成一串改 import，收益不一定最高。

### 2. `worldInfo.getLogicTable()` 的类型声明

接口层可以暂时保留 deprecated，先删内部旧实现，再观察外部调用。

## 我这次没看到的明显语法级 bug

这次我没再看到上次那个 `SourceRef` 字段解析错位的问题了。
你现在 settings 页的来源详情已经按新 `SourceRef` 结构在读：

* `label`
* `kind`
* `recordId`
* `path`
* `note`
* `ts` 

而类型定义里也确实就是这套字段 

所以这个点算是修掉了。

* 删 settings 页 relation tab 里的逻辑表维护链

* 只保留 deprecated facade 或直接转新逻辑

## 一句话结论

**你现在最大的冗余不是某个小函数，而是“settings 页还保留了一套旧逻辑表维护台”。**
这块最该删。删完以后，代码结构会明显干净很多，很多潜在状态不同步问题也会一起消失。相关证据主要在 settings 的 `index.ts` 仍然刷新和渲染逻辑表 、新编辑器已经完整承接维护职责 ，以及 SDK 已经把旧 `worldInfo.getLogicTable()` 标成 deprecated
