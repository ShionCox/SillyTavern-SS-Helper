## 二、现在离你要的功能还差什么

## A. “角色的记忆”现在还不是角色级主模型

现在虽然有 `PersonaMemoryProfile`，但它更像：

* **当前聊天的角色性格记忆画像**
* 或主角色/当前视角角色画像

而不是“每个角色各自一份独立遗忘配置”。

目前类型里并没有我想看到的这种结构：

* `actorKey -> persona profile`
* `recordKey -> ownerActorKey`
* `recordKey -> forgotten / visible / hiddenByOwner`

现在的 `MemoryLifecycleState` 只有：

* `recordKey`
* `recordKind`
* `stage`
* `strength`
* `salience`
* `rehearsalCount`
* `lastRecalledAt`
* `distortionRisk`
* `emotionTag`
* `relationScope`
* `updatedAt`

**没有“这条记忆属于谁”**，也没有“是否已遗忘”的显式布尔/状态位，只是通过 `stage` 间接表达模糊/扭曲 

这会直接卡住你的目标：

> “这条记忆始于哪个角色”“每条记忆是否已经遗忘”“不同角色按性格各自遗忘”

现在底层还不够。

---

## B. 记忆类型太粗

你现在的记录类型只有：

* `fact`
* `summary`
* `state`
* `relationship`

这对系统内部够用，但对你要做“按类型设定不同遗忘公式”来说太粗了。因为你想区分的其实更细：

* 重大剧情事件
* 日常闲聊
* 世界规则
* 城市设定
* 角色印象
* 好感/信任变化
* 物品知识
* 任务/目标
* 谣言/不可靠传闻

现有 `MemoryRecordKind` 只是技术层分类，不够做产品层遗忘策略 

---

## C. 遗忘不是“概率事件”，现在更像“连续衰减阶段”

你想要的是：

* 每种记忆类型有遗忘公式
* 根据角色性格生成遗忘概率
* 会因为重大事件触发“忘记/加深/覆盖”
* 可以真的标记“已遗忘”

而你现在这版更偏：

* 基于 age / strength / salience / forgettingSpeed 算出阶段
* 召回时对 `blur/distorted` 降权

这更像“连续衰减模型”，不是“离散概率遗忘事件”。
所以如果你要做“有概率遗忘”，最好不要直接推翻现有 stage，而是**在它上面再加一层 forget status / forget score / forget roll**。

---

## D. 记录编辑器还看不到“生命周期明细”

现在记录编辑器已经很强了，但它的重心还是：

* 逻辑维护
* 诊断维护
* 原始库表

并没有把每条记忆的这些东西清楚展开出来：

* ownerActorKey
* memory subtype
* lifecycle stage
* forgotten flag
* forget probability
* derived formula / reasons
* affected by event X

所以从“功能存在”到“可管理、可编辑”之间，编辑器还差一层专门的记忆视图。

---

## E. world_state 还没有你要的三级分类

你现在 `world_state` 依然是“path + value”风格，提议写入时也是按 patch path 写进去的 
这适合机器更新，但不适合你说的这种管理方式：

* 规则是什么类型
* 是全局设定 / 城市设定 / 物品设定
* 有关键词
* 便于向量检索
* 编辑器里按层查看

所以这里确实应该升级结构。

---

## 三、我建议你怎么设计“角色记忆 + 遗忘”

## 第一层：不要改掉现有 `MemoryLifecycleState`

它已经在工作，而且召回链已经依赖它。
应该把它保留为**底层连续衰减层**。

### 保留它负责

* `clear / blur / distorted`
* recency 衰减
* distortion 风险
* tone 映射
* recall penalty

---

## 第二层：新增“角色记忆归属 + 遗忘决策层”

建议新增一套显式结构，比如：

```ts
interface OwnedMemoryState {
  recordKey: string;
  ownerActorKey: string | null;
  memoryType: MemoryType;
  memorySubtype: string;
  sourceScope: 'self' | 'target' | 'group' | 'world';
  importance: number;
  forgetProbability: number;
  forgotten: boolean;
  forgottenAt?: number;
  forgottenReasonCodes: string[];
  lastForgetRollAt?: number;
  reinforcedByEventIds: string[];
  invalidatedByEventIds: string[];
  updatedAt: number;
}
```

### 这层解决什么

* 这条记忆属于哪个角色
* 这条记忆是否已经遗忘
* 为什么容易忘
* 是否被重大事件强化/覆盖
* 是否属于世界知识而不是角色私有记忆

---

## 第三层：把“记忆类型”从 4 大类扩成业务子类型

建议预设一套稳定子类型，不要让 AI 自由发明。

例如：

### 角色向

* `identity`
* `trait`
* `preference`
* `bond`
* `emotion_imprint`
* `goal`
* `promise`
* `secret`
* `rumor`

### 事件向

* `major_plot_event`
* `minor_event`
* `combat_event`
* `travel_event`
* `conversation_event`

### 世界向

* `global_rule`
* `city_rule`
* `location_fact`
* `item_rule`
* `faction_rule`
* `world_history`

### 状态向

* `current_scene`
* `current_conflict`
* `temporary_status`

然后让每个 `recordKey` 附：

* `recordKind` 保留技术大类
* `memoryType / memorySubtype` 作为业务层分类

---

## 第四层：遗忘公式不要“一条公式打天下”

建议按类型有不同基线。

例如：

### 1. 身份/核心设定

* 几乎不遗忘
* 主要受“强重写事件”影响
* 公式偏向低 forgetting

### 2. 关系与情绪烙印

* 受 `relationshipSensitivity / emotionalBias` 影响大
* 会因重大冲突或重大和解强化
* 会因长时间无互动弱化

### 3. 普通事件

* 最适合走你说的“概率遗忘”
* 时间衰减快
* 重大事件可压掉旧小事件

### 4. 世界规则

* 几乎不该按角色遗忘
* 应该属于 world_state / semantic
* 角色是否“记得”可以另建“认知覆盖层”，不要直接删规则本身

这个边界很关键：
**世界规则本身不应该因为角色忘了就从 world_state 消失。**
应该是：

* `world_state` 保存客观设定
* `owned_memory_state` 保存角色是否仍记得它

---

## 第五层：重大事件要作为“强化/覆盖触发器”

你说得对，“重大剧情事件会遗忘什么/记住什么”很重要。

建议不要直接“随机忘记”，而是设计成：

### 事件触发器

重大事件落库后，扫描受影响的记忆：

* 同主题旧事件
* 同关系对象记忆
* 同地点设定
* 目标/誓言/冲突类记忆

然后做三种动作：

* `reinforce`
* `blur`
* `overwrite`

例如：

* “角色被背叛” → 与该对象的旧信任记忆强化为负向关系变化
* “城市规则被推翻” → city_rule 旧规则标记为 superseded
* “得到重大创伤” → 部分 minor_event 忘得更快

---

## 四、我建议你怎么升级 world_state

你这个判断是对的：
**现在 world_state 不该只是 path/value 树，它应该再多一层结构分类。**

建议至少做三级：

## Level 1：域

* `global`
* `region`
* `city`
* `location`
* `faction`
* `item`
* `character`
* `scene`

## Level 2：类型

* `rule`
* `constraint`
* `history`
* `status`
* `capability`
* `ownership`
* `culture`
* `danger`
* `relationship_hook`

## Level 3：具体键

比如：

* `city/aurora/rule/night_curfew`
* `item/relic_x/constraint/cannot_be_destroyed`
* `global/world/rule/magic_requires_contract`

但别只存 path。
建议 value 也变成结构化对象：

```ts
interface WorldStateNodeValue {
  title: string;
  summary: string;
  scopeType: 'global' | 'city' | 'location' | 'item' | 'faction' | 'character';
  stateType: 'rule' | 'constraint' | 'history' | 'status' | 'culture' | 'danger';
  subjectId?: string;
  regionId?: string;
  cityId?: string;
  locationId?: string;
  itemId?: string;
  keywords: string[];
  tags: string[];
  confidence?: number;
  sourceRefs?: string[];
  updatedAt: number;
}
```

### 这会带来什么

1. 编辑器能按：

* 全局规则
* 城市规则
* 物品规则
* 派系规则

分组看

2. 向量检索/关键词检索更好做
   因为你已经明确提出关键词需求，而当前召回里也确实会按 query 拆关键词做打分 

3. 后续可以做“角色认知覆盖层”
   比如：

* 世界里有一条 rule
* 某角色是否知道它、记不记得它，是另一层

---

## 五、编辑器应该怎么改

## 1）记录编辑器新增“角色记忆”视图

我建议你在现有：

* 逻辑维护
* 诊断维护
* 原始库表

之外，再加一个：

* **角色记忆**

这个页面按角色查看：

### 左侧

角色列表

* 主角色
* 次角色
* 群聊成员

### 右侧

该角色记忆表

字段至少要有：

* 记忆标题
* ownerActorKey
* recordKind
* memorySubtype
* sourceScope
* importance
* lifecycle.stage
* forgetProbability
* forgotten
* lastRecalledAt
* rehearsalCount
* distortionRisk
* reasonCodes

### 可编辑动作

* 标记已遗忘 / 恢复
* 调整类型
* 调整 owner
* 调整重要度
* 强化 / 覆盖 / 锁定
* 重新计算遗忘概率

---

## 2）世界状态页加分类浏览

不要只让用户看原始 `world_state` 行。

加三层筛选：

* 作用域：全局 / 城市 / 地点 / 物品 / 派系 / 角色
* 类型：规则 / 历史 / 状态 / 约束
* 关键词搜索

再点开详情：

* path
* value
* keywords
* tags
* 来源
* 相关角色是否知晓

---

## 3）诊断页加“遗忘分布”

这会很好用。

显示：

* clear / blur / distorted / forgotten 数量
* 每角色遗忘率
* 每记忆类型遗忘率
* 最近被重大事件强化/覆盖的记忆
