# 一、页面与信息架构

## 顶层改造目标

把现在的：

* 记录编辑器

  * 逻辑表

改成：

* 记录编辑器

  * 设定总览
  * 角色与地点
  * 逻辑表维护
  * 诊断与维护

也就是把“查看”和“维护”拆开。

---

## 1）设定总览

这是默认页，解决“打开编辑器看不到关键信息”的问题。

### 页面结构

#### A. 世界概览卡

显示：

* 当前世界/模板
* 当前城市或主要地点
* 世界规则
* 硬约束
* 活跃 lorebooks
* 群组成员

### 交互

* 每项都能点开“来源”
* 每项都显示更新时间
* 空值时展示“尚未稳定抽取”而不是空白

---

#### B. 角色概览卡

显示：

* 主要角色
* 别名
* 身份摘要
* 关系锚点
* 当前所在地点
* 最近活跃

### 交互

* 支持按角色展开
* 支持“跳到逻辑表维护”
* 支持“标记为主角色 / 次角色 / 待确认”

---

#### C. 当前场景卡

显示：

* 当前场景
* 当前冲突
* 待处理事件
* 当前参与者

### 交互

* 一键刷新场景聚合
* 查看来源消息 / 来源摘要 / 来源状态

---

#### D. 聊天上下文卡

显示：

* 当前有效消息数
* 最近编辑次数
* 最近删除次数
* 分支根数量
* 是否建议 rebuild

### 交互

* 一键 rebuild chat view
* 查看最近 mutation 类型

---

#### E. 健康度卡

显示：

* 孤儿事实数量
* 待处理设定草稿
* 群聊状态分裂风险
* 设定一致性风险

### 交互

* 一键进入诊断与维护页
* 展开查看建议动作

### 两列布局

左边角色，右边地点。

* 别名

* 关系

* 相关地点

#### 地点列

这是原“逻辑表”重构后的定位：高级维护。

### 页面结构

上部是表选择器：

* 角色表
* 地点表
* 派系表
* 物品表
* 事件表
* 其他模板表

中部是表状态区：

* 表健康度
* 映射行数
* tombstone 数
* redirect 数
* alias 数
* derived row 数

下部是行表格。

### 每行除了字段值，还要显示：

* 来源类型
* alias
* redirect 目标
* tombstone 状态
* warnings
* 最近更新时间

### 行操作

* 合并
* 恢复
* 删除
* 设 alias
* 提升 derived row 为正式 row
* 修复字段

### 空表态要改成：

不是“空白表格”，而是：

> 当前表暂无稳定结构化行
> 可能存在于：语义种子 / 世界状态 / 群聊状态 / 已被合并隐藏的逻辑行

并给按钮：

* 查看候选来源
* 从 seed 生成候选
* 从 world_state 回填候选
* 查看已隐藏行

## 4）诊断与维护页

这个页专门解决“为什么数据对不上”。

### 模块

#### A. 数据分层诊断

显示：

* facts 数量
* world_state 条目数
* summary 数量
* active template
* semantic seed 是否存在
* logical chat view 是否存在
* group memory 是否存在

#### B. 问题列表

例如：

* orphan facts 偏多
* 角色重复命名
* 地点未归类
* alias 冲突
* redirect 循环
* 场景状态落后
* summary 与 facts 不一致

#### C. 修复动作

* rebuild logical chat view
* regenerate semantic seed
* recompute scene snapshot
* normalize aliases
* compact tombstones
* promote candidates

# 二、状态流与数据流

这里是最关键的，因为这决定你 Phase 1 和 Phase 2 怎么不互相打架。

---

## 总原则

### 1. 真相层

只把这些当主真相源：

* facts
* world_state
* summaries
* events

### 2. 运行时层

这些是聊天解释层：

* semanticSeed
* groupMemory
* logicalChatView
* chatProfile
* adaptivePolicy / maintenanceInsights

### 3. 展示层

新增聚合模型：

* CanonSnapshot
* EntityDirectorySnapshot
* LogicTableViewModel
* DiagnosticsSnapshot

也就是：

**UI 不直接拿某一张“逻辑表”当聊天真相。**

---

## Phase 1 数据流

### 打开编辑器

1. 读取 chatKey
2. 读取 `semanticSeed`
3. 读取 `groupMemory`
4. 读取 `logicalChatView`
5. 读取 `/semantic/world/*`
6. 读取角色/地点相关 facts
7. 读取维护洞察
8. 聚合为 `CanonSnapshot`

### 展示逻辑

* 总览页优先展示 `CanonSnapshot`
* 逻辑表页保持旧 rows API，但包一层“空态解释”

---

## Phase 2 数据流

### 打开逻辑表维护页

1. 读取 active template table defs
2. 读取 table 对应 facts
3. 读取 alias / redirect / tombstone
4. 读取 seed/world_state 中可映射的候选
5. 聚合为 `LogicTableViewModel`

### 行级状态判断

每个 row 有 4 类：

* `materialized`：真实 fact row
* `redirected`：已并入别的 row
* `tombstoned`：已逻辑删除
* `derived`：从 seed/world_state 推导出的候选 row

### 写入策略

只有 `materialized` 和用户确认后的 `derived -> promoted` 才写回事实层。
`derived` 只做只读候选，不自动写盘。

这个边界很重要，不然很容易把“展示聚合结果”污染成“事实数据”。

# 四、两阶段实施边界

这个边界要守住，不然很容易 Phase 1 做过头。

---

## Phase 1 边界

### 要做

* 新默认页：设定总览
* 角色与地点页基础版
* 健康度卡
* 逻辑表空态解释
* rebuild / refresh seed 按钮
* 只读聚合接口

### 不做

* 新 row 类型入库
* derived row promotion
* 表健康评分算法复杂化
* 深度自动修复
* 批量结构迁移

### 成功标准

* 用户打开编辑器能看到城市、角色、场景、规则等
* 不再出现“主页面几乎全空”
* 即使逻辑表为空，也能解释为什么空

---

## Phase 2 边界

### 要做

* 新逻辑表 ViewModel
* derived rows 候选
* 来源解释
* 表健康诊断
* 行级维护动作
* 候选转正

### 不做

* 全量重写底层 facts schema
* 把所有 world_state 自动强转为 facts
* 自动批量写盘

### 成功标准

* 逻辑表即使 facts 稀疏也不会完全空白
* 用户能看懂每行为什么存在、为什么隐藏、为什么不能编辑
* 逻辑表真正变成维护台，而不是失败的总览页

# 七、我对你这次重构的最终建议

你这次不要把目标定义成：

**“把逻辑表修到能显示更多内容”**

而应该定义成：

**“把记录编辑器重构成以世界观总览为主、逻辑表维护为辅的双层编辑器”**

这会更符合现在 MemoryOS 的真实结构，也更符合你的目标。

因为你真正要的是：

* 完整查看聊天世界观
* 看到城市、角色、场景、规则
* 看懂重要设定
* 必要时再维护逻辑行

而不是把所有责任都压在一张“逻辑表”上。
